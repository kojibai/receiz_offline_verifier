# Receiz v107.0.0 Release Process

## Goal

The v107.0.0 release process starts from the v106.0.0 baseline at `477713ae2`, closes inherited v106 implementation gaps, then implements canonical operation schemas, Receiz.com bindings, SDK methods, offline queueing, MCP tools, AI skills, emulator scenarios, generated integrations, frontend parity, migration, registry, and evidence.

## What Worked

Each mutation rail is tested at its authority boundary: expected heads, idempotency, identity/tenant control, enclosing proof, atomic effects, canonical receipt, and projection-after-verification. Visual evidence remains a blocking truth source for identity, ownership, media, hierarchy, and deterministic first paint.

Release qualification runs focused v107 gates, legacy v106 constitutional and migration gates, lint, type checks, production build, performance budgets, conformance suites, security/chaos contracts, governance controls, package tarball installation, and attestation verification. A passing unit test cannot override contradictory visible evidence.

The package gate installs `@receiz/sdk`, `@receiz/mcp-server`, and `@receiz/ai-skills` tarballs in a clean external project and imports their public entry points. The final release freeze accepts no workspace-only dependency or source-only success.

## What Did Not Work

Initial evidence captures exposed nested-scroll and compositor instability, so those screenshots were rejected. The final evidence used a settled opaque shell, wrapping operation rail, exact Playwright viewports, fresh DOM snapshots, and pixel inspection. The package gate also exposed missing package-level v107 documentation and a source/import assertion that still expected the historical TypeScript extension.

## Practical Rule For Future Work

The release process does not publish npm packages or deploy production. Those are distinct authorized mutations after repository qualification. History remains first admission only, then append forever.
