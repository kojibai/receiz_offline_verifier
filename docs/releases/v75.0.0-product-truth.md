# Receiz v75.0.0 Product Truth

Release date: May 2, 2026
Status: Locked for `v75.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v75.0.0`, the local proof, backup sign-in, market continuity, and Explore release after the `v74.0.0` Receiz Key / identity-artwork baseline.

`v75.0.0` extends product coverage for local Receiz ID proof, durable account binding registers, Identity Record / Identity Seal / Receiz Key restore parity, old-browser backup sign-in, restored-account profile / wallet / upgrade authority, deterministic profile identity downloads, Plans / Upgrade payment parity, market snapshot continuity, stable market navigation, and Explore mobile packaging. It does not weaken the `v70.0.0` public alpha law or the `v74.0.0` Receiz Key / identity-artwork baseline.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, shipped HTML version markers, and package-metadata alignment only. The verifier proof-format and producer-payload contract remain unchanged relative to `v60.0.0`.

## Core Product Truth

`v75.0.0` carries forward the `v70.0.0` public alpha law and the `v74.0.0` Receiz Key / identity-artwork baseline:

> Identity is local. Proof is in the file. Verified truth is append-only. `/<username>` is canonical profile truth. Settlement and Reserve preserve funded causality. The network propagates and appends; it does not get to downgrade stronger proof already held by the device.

## Locked Release Surface

### 1. Receiz ID Login Is Local-First

- The device-held Receiz ID key admits local identity truth.
- Local identity truth projects the signed-in shell before network reconfirmation.
- Local identity truth seeds profile and wallet route truth.
- Server cookie and account propagation start in the background.
- Server Receiz ID continuation may publish, bind, mint cookies, and return stronger additions.
- Server continuation must not replace a stronger local account binding with a weaker or stale session payload for a different account.

### 2. Account Bindings Are Durable Register Truth

- Local account bindings are durable register truth across Receiz device, Receiz Key, PBI, email, and server-switch sources.
- Switching bindings projects the selected account.
- Switching bindings clears account-scoped continuity when the actor changes.
- Restored-account truth outranks stale old-cookie echoes.
- Account, managed profile, wallet, and upgrade surfaces recognize locally restored account authority before stale remote session payloads.

### 3. File Restore Has PBI Parity

- Identity Record, Identity Seal, and Receiz Key restore project the verified file session directly.
- Verified file restore admits the restored account locally.
- Verified file restore seeds profile, wallet, and upgrade truth.
- The restored account remains active when import, bind, wallet refresh, or remote reconfirmation is delayed.
- First-time offline Receiz Key restore requires verified portable state.
- Later restores may reuse admitted local key truth.
- Optional local PBI publication is a background append, not restore permission.

### 4. Backup Sign-In Is Explicit And Sanitized

- Browsers that cannot support the Receiz ID crypto path route to `/signin/receiz-id-fallback`.
- Return paths are sanitized before backup sign-in continues.
- Primary sign-in surfaces stay Receiz ID pass-through.
- The backup route owns PBI/passkey, magic-link, and Identity Record / Seal / Receiz Key file restore.

### 5. Profile Identity Downloads Preserve Canonical Truth

- Profile identity artwork preserves deterministic profile truth.
- Signed original-download paths preserve owner proof, purchase context, and PBI download authorization.
- Identity download hardening must not redefine the base profile shell.
- Identity Record and Identity Seal artifacts append proof rather than replacing profile truth.

### 6. Plans And Upgrade Preserve Payment Parity

- Plans and Upgrade keep parity for immediate payment and upgrade actions.
- The two surfaces must not diverge on the payment path.
- The two surfaces must not diverge on the old-browser fallback affordance.

### 7. Market Continuity Preserves Stronger Local Truth

- Market snapshots preserve stronger signed-in wallet, owned-item, prediction-history, and share-trade truth across lagging or anonymous payloads.
- Market snapshots refuse to preserve private state across viewer changes.
- Sold share positions remain closed when local sell history is newer than incoming market payloads.
- Weaker snapshots must not resurrect closed ownership.
- Market navigation uses stable internal navigation for record cards, verified routes, live players, and asset links.
- Market pointer handling prevents mobile press behavior from double-firing or falling through.
- Market mobile inputs and certificate panels keep no-zoom viewport behavior, 16px editable controls, and wallet-density typography.

### 8. Explore Is Local-First And Dock-Safe

- Explore opens as a card-first, dock-safe mobile surface.
- Swipe rendering is bounded.
- Preview fallbacks are deterministic.
- In-card read, play, save, share, and proof actions are first-class.
- Route entry is local-first.
- Server discovery remains a background recovery source.
- Already-held view state and deterministic previews are reused instead of replaced by richer rediscovery.

### 9. Evidence And Version Surfaces Move Together

- Product doctrine explicitly names Identity Record, Identity Seal, and Receiz Key restore parity with PBI recovery.
- The HPL continuity review and Receiz identity continuity one-pager are release evidence for zero-regression local-proof upgrades.
- Package metadata, service-worker defaults, public version badges, verifier surfaces, README release pointer, release notes, changelog, docs indexes, and release docs must agree on `v75.0.0`.
- Historical release documents preserve their own version truth.

## Important Non-Changes

- The verifier proof-format and producer-payload contract remain unchanged.
- Trusted-signature success still requires verified `signatureV4`.
- Effective anchor context remains required for verified outcomes.
- Groth16 fields remain required and only real `g16:` proof payloads are accepted.
- The `v64.0.0` value-loop invariants remain in force.
- The `v70.0.0` public alpha invariant register remains the baseline law.
- The `v74.0.0` Receiz Key / identity-artwork baseline remains in force.

## Non-Regression Rule

A change is out of bounds for `v75.0.0` if it:

- makes Receiz ID login wait for network reconfirmation before local truth can project
- lets server Receiz ID continuation replace a stronger local account binding with a weaker or stale session payload for a different account
- treats local account bindings as disposable UI cache instead of durable register truth
- preserves account-scoped private continuity after the actor changes
- requires PBI publication as permission for a verified Identity Record, Identity Seal, or Receiz Key restore
- blanks a restored account while import, bind, wallet refresh, or remote reconfirmation is delayed
- sends unsupported browsers into the primary Receiz ID crypto path instead of `/signin/receiz-id-fallback`
- accepts unsafe return paths from backup sign-in
- lets stale cookie echoes outrank locally restored account authority
- redefines base profile truth through identity artwork or signed-original download hardening
- lets Plans and Upgrade diverge on payment or old-browser fallback behavior
- replaces stronger signed-in market truth with lagging or anonymous market payloads
- resurrects closed share ownership from weaker snapshots
- lets market mobile pointer handling double-fire or fall through
- replaces already-held Explore view state with rediscovery payloads that are merely richer, not stronger
- allows version drift between package metadata, README, public badges, verifier surfaces, and release docs
