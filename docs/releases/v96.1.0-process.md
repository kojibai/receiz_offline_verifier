# Receiz v96.1.0

Release date: 2026-06-30

## Goal

Release the Twin and World `@receiz/sdk` expansion without changing Receiz authority. Developers should be able to build real Receiz-powered apps through typed SDK clients and documented Connect scopes while the product remains governed by proof objects, identity proofs, durable local truth, verified appends, and Kai/Klok state.

## What Worked

- `v96.0.0` already locked the SDK as an executable developer surface with CLI conformance and durable proof memory.
- The existing Connect model already provided delegated API access, so Twin read/write could be exposed as scoped capability instead of inventing a parallel login or authority model.
- The existing World/Twin server routes could be composed through SDK convenience rather than duplicated.
- OpenAPI, status catalog, Connect docs, identity conformance, and SDK docs could all name the same route surface, keeping developer discovery coherent.
- Version locking the app and SDK prevents the package from underclaiming or drifting from the shipped product.

## What Did Not Work

- After the Twin and World SDK work, leaving the package at `96.0.0` would have made npm identity lag behind implemented developer capability.
- A patch-level SDK expansion without release docs would have hidden an important public developer surface.
- The release lock had to cover public version labels, service-worker/runtime coherence, SDK package version, SDK runtime constant, and release documents together.

## Practical Rule For Future Work

When a developer capability is added beneath an implemented Receiz primitive, release it as a coherent package:

1. Typed SDK client.
2. Route/OpenAPI/status visibility.
3. Scope and token lifecycle documentation.
4. Conformance or contract coverage.
5. Version-locked app and SDK release identity.

The SDK remains convenience. It must make proof-native development easier without creating a new source of truth. The durable proof memory law remains first admission only, then append forever.
