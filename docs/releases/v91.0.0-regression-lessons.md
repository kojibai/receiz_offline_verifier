# Receiz v91.0.0 Regression Lessons

Release date: June 16, 2026
Status: Required release-lock artifact for `v91.0.0`

This document is part of the release lock. It records the regression classes found or carried into the `v91.0.0` release arc and the rules that must prevent the same mistakes.

## What Regressed

### Scale Reasoning Almost Removed The Primitive

The Sports value chart became lighter, but the first optimization pass removed visible LiveChart function: range controls, tooltip, full chart surface, and prior visual behavior.

That was a product regression. The chart primitive was not the problem. The unnecessary timer/rebuild work underneath it was the problem.

Correct rule:

```txt
keep the exact visible primitive
remove unnecessary weaker work beneath it
```

### Deterministic Chart Data Was Treated Like Runtime State

The value series can be reconstructed from deterministic Kai pulse math and known card/value inputs.

Treating that data as something to poll, fetch, accumulate, or store adds weight and weakens the model.

Correct rule:

```txt
if deterministic state can compute the series, compute it locally
```

### Expired Or Final Sports State Stayed Active

Old rewards, reveals, entries, tournaments, and challenges can keep pressure on first paint and gameplay when they stay active after truth is final.

The concrete v91 cases were:

- opened reward receipts needed to close stale finalized recovery
- FIFA tournament reads needed closeout and pending old-date sweep
- soccer/UFC entry routes needed claim/finalize PATCH
- the client claim path needed the active sport route instead of baseball
- Pack Derby challenge settlement needed deadline-or-attempts behavior
- Pack Derby run recording needed a post-deadline guard

Correct rule:

```txt
final truth closes stale active state
```

### Settlement Conformance Was Too Weak

Settlement conformance was not a distinct mechanical gate. That was weaker than the primitive.

Correct rule:

```txt
settlement conformance is a required public suite
```

### Public Sports Proof Projection Needed Sport Awareness

Shared public card, QR, vault proof, and value surfaces must carry the sport primitive they project.

Hardcoding baseball for market depth or claim/finalize behavior made soccer and UFC proof surfaces weaker than their implemented primitive.

Correct rule:

```txt
shared Sports surfaces are sport-aware proof surfaces
```

## Why It Regressed

### Legacy Performance Assumptions Entered The Reasoning

Legacy apps often make rich UI lighter by removing UI. Receiz must not inherit that assumption.

Receiz has deterministic primitives. The scale path is usually:

```txt
use the stronger known truth
bound or remove weaker work
keep the settled surface
```

### Timer Work Was Mistaken For Live Motion

A surface can look live because deterministic pulse math changes its projection. It does not need chart-local timers for every visual point when the state machine already supplies the pulse.

### Stale Rows Were Treated As Harmless Data

Old active Sports rows are not harmless. They affect first paint, recovery, selected entry state, reveal state, and the user’s belief that rewards are locked forever.

### Shared Routes Lagged Behind Shared Primitives

The codebase had shared Sports entry primitives, but soccer/UFC did not expose the same finalization surface. That mismatch created stale state and wrong-route pressure.

## Locked Rules

### Scale Rules

- Rich deterministic primitives stay rich.
- Performance fixes remove weaker work beneath the primitive.
- Do not remove chart controls, proof media, ownership state, settlement state, or first-paint truth as an optimization.
- Deterministic math is not fetched state.
- If a state machine can compute it, do not poll or accumulate it.

### Sports Closeout Rules

- Old active game/reward/reveal/tournament/challenge rows must close when final truth exists.
- Opened reward receipts must keep opened rewards out of stale recovery.
- FIFA tournament routes must run closeout and pending old-date sweeps.
- Soccer/UFC entry routes must expose sport-specific finalization.
- Client claim routes must use the active sport route.

### Pack Derby Rules

- Challenge settlement follows server truth: deadline or attempts.
- Post-deadline receipts are rejected.
- Challenge HUD affordance must match the server settlement contract.
- Pack Derby remains a Sports reward game with proof-backed scoring, not client-only animation.

### Conformance Rules

- Settlement conformance is distinct from economy conformance.
- Release freeze and governance must run settlement conformance explicitly.
- Public conformance suite surface contracts must list settlement as its own suite.

### Public Proof Rules

- Public Sports card, QR, vault, and event routes are public proof surfaces.
- Market depth and value projection must be sport-aware.
- Metadata and social images project proof truth only.

## Release-Lock Requirement

`pnpm release:lock 91.0.0` must require this document for `v91.0.0` and later release cuts. A release that omits regression lessons is incomplete because the scale, settlement, Pack Derby, public proof, and sport-specific route regression boundaries have not been made explicit.

## No Downgrade Statement

The lessons above preserve Receiz proof-native artifact system truth, deterministic value chart truth, full LiveChart value surface behavior, Sports public proof surfaces, Pack Derby Sports reward game truth, Pack Derby challenge settlement truth, sport-specific Sports entry settlement, FIFA closeout composition, opened reward truth, settlement conformance, and service-worker release truth. They do not downgrade proof objects to cards, public proof surfaces to share pages, settlement primitives to balance displays, ownership or reward truth to badges, deterministic first paint to loading state, verified history to activity feed, route truth to SEO payload, or service-worker release truth to incidental cache behavior.
