## Preparation Steps

1. Preserve `v85.0.0` as the prior current-release archive.
   - Keep `docs/releases/v85.0.0.md`.
   - Keep `docs/releases/v85.0.0-product-truth.md`.
   - Keep `docs/releases/v85.0.0-checklist.md`.
   - Keep `docs/releases/v85.0.0-process.md`.
   - Preserve `v85.0.0` as historical release-law truth.

2. Publish canonical `v86.0.0` release artifacts.
   - Add `docs/releases/v86.0.0.md`.
   - Add `docs/releases/v86.0.0-product-truth.md`.
   - Add `docs/releases/v86.0.0-checklist.md`.
   - Add `docs/releases/v86.0.0-process.md`.

3. Align repository release law and product-truth references.
   - Update `AGENTS.md` release law to `v86.0.0`.
   - Update product-truth reference status markers carried forward for the current release.
   - Preserve the primitive decision procedure, stronger/weaker truth hierarchy, and forbidden downgrade boundary.

4. Align repository release surfaces.
   - Update `package.json` to `86.0.0`.
   - Update root `README` current-release references.
   - Update `RELEASE_NOTES.md` and `CHANGELOG.md`.
   - Update `docs/README.md`, `docs/governance/README.md`, and `docs/FORMAT.md`.
   - Update shipped version markers in `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.
   - Ensure static service-worker registrations request `sw.js?v=86.0.0`.
   - Ensure `site/sw.js` carries service-worker runtime version `86.0.0`.
   - Ensure local release-surface checks can execute through `pnpm lint`, `pnpm build`, and `pnpm release:lock`.

5. Validate primitive naming gates.
   - Receiz is named as a proof-native artifact system.
   - Profile-visible proof objects are named as proof objects and public profile truth.
   - Market and wallet are named as ownership and settlement projection surfaces.
   - Explore is named as a deterministic proof first-paint surface.
   - Sports Arena is named as a live proof economy.
   - Main Event is named as a payment-confirmed settlement primitive.
   - World and public twin truth retain direct primitive names.
   - Public proof, identity, route, and service-worker truth retain direct primitive names.

6. Validate deterministic first-paint and speed gates.
   - Known truth paints before hydration.
   - Profile, Market, and Explore cold loads reuse canonical/deterministic truth before owner/session/server enrichment.
   - Sports first paint and live appends use all relevant source dates before hidden hydration.
   - World profile and public twin surfaces keep stronger preview/portrait truth before weaker presentation.
   - SEO, analytics, session checks, lazy imports, dynamic JSON, global sync, server reconciliation, tracking, marketing surfaces, and framework convention do not gate first paint.
   - Hidden hydration appends verified additions only.

7. Validate post-v85 stabilization gates.
   - Visible profile proof objects project into Market and hidden proof objects leave Market.
   - Signed-in Market private graph hydration cannot erase stronger public or local ownership truth.
   - Explore preserves deterministic proof truth and profile image data needed by username pills.
   - Market chart, certificate/share, wallet, position, and history paths reuse deterministic market math.
   - Sports user-live hydration covers every relevant active, zero-append, and recovery source date.
   - Main Event checkout cancellation does not enter the event, and successful payment enters through the existing paid-settlement continuation.
   - Sports single-card QR proof pages exist without modifying existing vault links.

8. Validate version lockstep.
   - Confirm all governed version markers show `v86.0.0`.
   - Confirm all current-release docs and indices point to `v86.0.0` artifacts.
   - Confirm historical `v80.0.0` through `v85.0.0` artifacts remain archived release truth.
   - Run `pnpm release:lock`.
   - Run whitespace diff check.

9. Validate broader platform gates before production promotion.
   - Run release-freeze and page-first-paint checks.
   - Confirm focused Sports, Market, Profile, World, and release contracts.
   - Confirm target production DB migrations named by the release are applied.

10. Generate signed release attestation.
    - Run `pnpm release:attest` for the final commit in the environment that holds the release attestation signing key.
    - Retain the signed attestation artifact under the governance release-attestation policy.
    - Do not tag or ship without the signed attestation.

## Focused Verification Matrix

Broader platform release validation should include the gates named in `docs/releases/v86.0.0-checklist.md`, including:

- direct primitive naming and no forbidden downgrade
- deterministic first-paint truth before hidden hydration
- Profile, Market, Explore, Sports, World, public witness, and release-law contracts
- Main Event payment-confirmed entry and checkout-cancel contracts
- service-worker version coherence
- release freeze, page first paint, TypeScript, release lock, whitespace diff, and release attestation gates

## Tag Criteria

Do not tag `v86.0.0` until:

- all required `v80.0.0`, `v81.0.0`, `v82.0.0`, `v83.0.0`, `v84.0.0`, `v85.0.0`, and `v86.0.0` release artifacts exist
- version surfaces are aligned to `v86.0.0`
- release gates in `docs/releases/v86.0.0-checklist.md` pass from the exact release candidate
- governed surfaces all show `v86.0.0`
- `pnpm lint` has zero warnings/errors
- `pnpm build` completes
- `pnpm release:lock` confirms the release surfaces
- whitespace diff check passes
- `pnpm release:attest` emits a signed attestation for the final commit

## Release Rule

Do not ship `v86.0.0` unless the release gates in this checklist pass from the exact release candidate, `pnpm release:lock` confirms the release surfaces, whitespace diff passes, first-paint and release-freeze checks pass before production promotion, focused Sports, Market, Profile, World, and release contracts pass, and production migrations named by the release are applied in the target environment.
