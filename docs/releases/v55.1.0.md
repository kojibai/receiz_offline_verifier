# Receiz v55.1.0

Release date: April 16, 2026

## Overview

`v55.1.0` is the continuity-and-execution release on top of the `v55.0.0` profile/world/market baseline. The work since the last bumped version surfaces has concentrated on one theme: once Receiz already knows the truth for a surface, the product should stop behaving like it needs to rediscover it.

This release removes profile-shell drift, closes second-settle and hydration regressions, makes profile/search/player navigation reuse deterministic route truth, turns market/chart trading into a single passkey-first continuation flow, moves market execution and settlement onto exact USD intent for non-entitled wallets, and hardens live-player reliability and card fit.

There is no dedicated `v55.0.1` release note or tag in the repository. This `v55.1.0` note therefore rolls up the full repository delta after `v55.0.0`, including the intervening `55.0.1` version-surface prep state, into one coherent release record.

The product-truth freeze for this release is defined in `docs/releases/v55.1.0-product-truth.md`.

## What Defines v55.1.0

- Profile truth is now treated as a reusable live graph: `/profile`, public profiles, search handoff, the global composer host, and bottom-bar/profile navigation all reuse the same already-loaded owner or route truth instead of re-resolving it through extra shells.
- Public profile navigation now lands with the same real page truth whether it comes from direct load, signed-in navigation, or floating people search, including header media and joined metadata.
- Market entry and chart popovers now behave like one continuation system: `Buy` starts the whole ceremony, passkey resolves in place, funding covers only the exact shortfall, and the pending order resumes automatically on the same surface.
- Sell is now budget-first in both the chart and the full desk, and the sell lane only exists when the viewer actually holds shares in that asset.
- Non-entitled share and prediction execution is now USD-authoritative at the database layer, so wallet availability, settlement, and post-trade balances derive from exact USD flows instead of loosely inferred phi balances.
- The live player is more deterministic: the back card fits inside frame, the public twin preview chooser prefers a valid public twin over stale hidden preview state, and live-player entry keeps continuity through prepared route handoff.
- Route feedback now happens on the buttons themselves, without artificial waits, across account/dashboard entry, bottom-bar market entry, profile artifact/live-player actions, and chart-to-market handoff.

## Since The Last Bumped Version Surfaces

### 1. Profile Became A Single Continuous Truth Surface

The profile stack now behaves much more like one mounted system and much less like several adjacent loaders.

The current repository state adds:

- server-resolved owner `/profile` entry instead of the extra fast-entry shell path
- reactive session bootstrap hydration so passkey-created sessions update chrome immediately
- search warming of the full public profile route truth instead of only a tiny avatar/name seed
- public-profile hydration that keeps client-only seeds out of the initial render tree
- local profile publish continuity so posting updates the active profile graph instead of immediately refetching the same surface
- composer host reuse of owner workspace truth instead of running its own disconnected owner-profile load
- deterministic warming for the full visible profile artwork window rather than a tiny above-the-fold subset

The net effect is that profile no longer needs a reload to look complete, and profile-adjacent actions no longer knock the user back into a weaker copy of truth.

### 2. Navigation Paths Now Reuse Known Truth Instead Of Rediscovering It

The route work since `v55.0.0` focused on removing second-click and second-settle behavior:

- profile artifact and live-player buttons now warm deterministic route snapshots before navigation
- floating people search preloads the richer public-route profile payload before push
- bottom-bar market/profile/world routes moved onto the same route-feedback + stable-navigation discipline
- account-to-dashboard now prewarms and animates on real route pending state instead of leaving the user with a dead tap
- market page entry prefers the richest warmed snapshot, including fast-entry handoff without a route loading shell
- market/chart `Full` transitions reuse the already loaded market snapshot so the desk can render immediately and reconcile in the background

This release treats known route truth as something to carry forward, not something to throw away and re-fetch after the click.

### 3. Market Buy Flow Became One Continuation Instead Of Separate Auth And Funding Detours

The largest product change after `v55.0.0` is the market execution model.

The current state now provides:

- passkey-first buy continuation directly from the chart popover and full market desk
- exact-shortfall funding instead of a separate `Add funds` lane
- a dedicated funding modal that reuses the same Stripe checkout pattern as the lock flow rather than stuffing a second funding card into the chart sheet
- automatic replay of the pending order after funding settles
- magic-link fallback only when passkey is unsupported or fails, with continuation routed back into the same pending order context
- session-bootstrap updates that keep the newly created account signed in even if the user closes funding before payment completes
- retry-based post-passkey session pickup so successful passkey completion does not incorrectly fall back to email during cookie propagation

This is the difference between "go sign in, then maybe fund, then maybe buy again" and one uninterrupted buy action.

### 4. Market Execution And Wallet Accounting Became More Exact

This release also tightened what the trade actually means on the server:

- share buy and prediction wager execution now preserve a locked USD spend intent and derive the live share outcome from that budget
- sell now accepts a USD receive target on both chart and market, with share quantity derived from the live bid and clamped to held position
- non-entitled wallet execution now computes available wallet value, next wallet value, and settlement from exact USD balances in SQL for both share trades and prediction trades
- API routes, hydration, compose logic, and wallet history all pass entitlement metadata so the correct settlement path is used everywhere
- share-certificate trading, import, and market-position hydration were updated to stay aligned with the USD-authoritative server path

This makes the market stack more defensible: what the user commits is the dollar intent, and the system derives the rest truthfully from the live book and the actual wallet state.

### 5. Live Player And Chart Presentation Were Tightened

The presentation layer also closed some visible regressions:

- the live-player back card now fits within the frame without clipped text or required internal scroll
- public twin preview precedence now prefers a valid public-ready twin over stale hidden preview state
- chart pill height now matches the real available lanes, so logged-in users without a position do not get a blank extended sell area
- sell only appears when the viewer actually has shares in that asset
- route-pending feedback now stays on the button surface itself, mobile-first, instead of relying on cursor-only affordances
- the live chart contract and pixel-text chart rendering were tightened alongside the market-view presentation updates

## Public Summary

Receiz `v55.1.0` turns continuity into a hard product rule: profiles, market routes, player routes, and composer flows now reuse already-loaded truth instead of rediscovering it; buy becomes one passkey-first continuation that funds only the shortfall and resumes automatically; sell becomes a dollar-intent action tied to real held position; and live-player/twin presentation is tighter and more deterministic.
