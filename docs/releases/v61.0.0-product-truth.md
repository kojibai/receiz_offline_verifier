# Receiz v61.0.0 Product Truth

Release date: April 19, 2026
Status: Locked for `v61.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v61.0.0`. It carries forward the `v60.0.0` stable baseline and the `v60.1.0` public-market continuity floor while recording governed historical continuity for pre-v4 receiz, deterministic market exit continuity, harder settled-shell runtime boundaries, published operating standards around hidden-only warming and expensive truth, and release-surface lockstep as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, shipped HTML version markers, and package-metadata alignment only. The verifier trust contract remains unchanged relative to `v60.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- File-backed proof remains the authority for artifact truth.
- Authentic pre-v4 records may be governed and presented as historical, but they must not be silently recast as current verified proof.
- Stronger already-known market witness truth and deterministic route truth remain the authority over weaker fallback clocks, late anonymous payloads, or weaker navigation paths.
- Once a visible home or profile shell has settled, explicit user intent remains the authority for promoting a richer runtime.
- Historical packaging may add governed transport and explanation artifacts, but it must preserve the original artifact bytes and identity.
- Public product routes, release metadata, governance artifacts, and offline verifier entrypoints remain operating or governance surfaces around those authorities, not replacements for them.

## Locked Release Surface

### 1. The v60.0.0 Baseline And v60.1.0 Floor Must Remain In Force

- Release-governed version surfaces must remain lockstep and current-release docs must stay aligned.
- Live front-door proof/twin presentation, canonical original upload/download authority, explicit twin portrait publishing, compact wallet/certificate framing, and unified `v60.0.0` release truth remain in force.
- Holder-aware public market witness truth, tighter compact/public market presentation, hidden-only preload behavior for settled twin shells, lighter public-profile route assembly, and deterministic market sigil hydration from `v60.1.0` remain in force.
- The offline verifier trust contract must remain file-authoritative, deterministic, and fail-closed.

### 2. Pre-v4 Receiz Must Remain A Governed Historical Cohort

- Historical pre-v4 status must derive from the explicit frozen public cohort, not from broad legacy-shape inference.
- The governed historical cohort membership and governance payload must remain immutable, fingerprinted, and hash-locked.
- Historical trust presentation must keep `Historical` distinct from current `Verified` and generic warning states.
- Non-members must not inherit historical status just because they resemble legacy records.
- Historical download and package flows must preserve untouched raw artifact bytes while attaching separately sealed governance evidence.

### 3. Settled Twin Shells Must Stay Hard Runtime Boundaries

- A visible settled home or profile shell must stay mounted until explicit open intent chooses the richer twin runtime.
- Compact touch devices must keep full twin background hydration on-demand instead of auto-starting the richer runtime graph.
- Hidden preparation may prefetch code or data, but it must stay hidden and must not visually promote the settled shell.
- The home dock audit shell must remain the visible authority until explicit activation opens the deferred module.
- Runtime warming may respond to hover, focus, touch, or pointer intent, but it must not replace the visible shell before explicit activation.

### 4. Public Market Truth And Exit Continuity Must Stay Deterministic

- Public market holder count and latest witness timing must continue to derive from real holder and trade activity when that truth is already known.
- Witnessed duration must not remain stranded at zero when the product already knows there was real holder activity.
- `Verify asset`, `Open source`, `/v/...`, `/r/...`, verified-player, and original-viewer exits must stay on the shared stable client-navigation path for deterministic Receiz routes.
- Buy continuation must remain coherent through the same surface model, and sell must stay hidden when there is no position.
- Compact market polish must not weaken witness truth, deterministic exits, or public-proof legibility.

### 5. Historical Continuity And Fast-Page Discipline Must Remain Published Standards

- The release line must continue to publish the operating standards that define hidden-only warming, stronger-known-truth preference, and governed historical continuity.
- Deterministic surfaces must not regress into fake loading theater or visible second-settle behavior.
- Historical continuity must remain additive and governed rather than inferred by loose legacy exception logic.
- Saved evidence and rendered performance collateral must continue to support the published fast-page and expensive-truth model.

### 6. Release-Governed Version Surfaces Must Move Together

- Release-governed version surfaces must move together to `v61.0.0` across package metadata, service-worker defaults, public verifier artifacts, public release badges, and current release documentation.
- Public-facing version truth must not drift between shipped HTML entrypoints, docs indexes, release artifacts, and repository metadata.
- This offline verifier repository must reflect that lockstep rule in `package.json`, `README.md`, `RELEASE_NOTES.md`, `CHANGELOG.md`, `docs/README.md`, `docs/governance/README.md`, `docs/FORMAT.md`, `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.

### 7. Release Governance And Verifier Truth

- Release-scoped note, product-truth, and checklist documents are required for `v61.0.0`.
- Root package metadata must resolve to `v61.0.0`.
- Shipped verifier, studio, and settlement entrypoints must visibly display `v61.0.0`.
- No new database migrations are added beyond the `v60.0.0` baseline.
- The offline verifier trust contract remains unchanged relative to `v60.0.0`.

## Allowed Evolution

Implementation details may change after `v61.0.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v61.0.0` if it:

- grants historical status by vague legacy shape instead of the governed cohort boundary
- rewrites original historical artifact bytes or blurs historical packaging into current verified proof identity
- lets settled home or profile shells visually remount, auto-upgrade, or promote richer runtime before explicit user intent
- lets public market exits fall off the stable client-navigation path or regress witness truth back to weaker creator-side fallback clocks
- drops the published operating standards or their supporting evidence out of the governed release record
- allows package metadata, public badges, shipped verifier artifacts, or current-release docs to advertise different release numbers
- changes release truth without updating the versioned release docs and aligned repository version surfaces
