## Frozen Primitives

| Primitive | v94.0.0 truth | Must not regress |
| --- | --- | --- |
| Proof object | A Receiz object carries deterministic identity, verifiable payload/proof material, provenance, inspection behavior, and durable state. | No API response, UI cache, index, or SDK projection may replace sealed or admitted proof truth. |
| Durable proof memory | First verified admission creates known local truth; later verified additions append after the known head. | Partial newer payloads must never erase, shrink, or rediscover known truth before render. |
| Kai state coordinate | Kai pulse/upulse orders Receiz proof objects and settlement ledger additions. Chronos is display/fallback context. | Backend ordering must not use Chronos as the primary state machine where Kai exists. |
| Official SDK | `@receiz/sdk` is the official developer convenience package for Receiz proof objects, identity artifacts, wallet ledgers, Sports cards, public proofs, Connect, webhooks, projections, and proof memory. | SDK helpers must not become authority over sealed artifacts, proof bundles, verified appends, ownership appends, or settlement ledger rows. |
| Sports proof economy | Sports cards, Sports Economy, tournament projections, event proofs, card appends, ownership appends, and showcase projections are proof surfaces over ledger truth. | Read hardening must not drop history, hide appends, change ownership truth, or replace proof payloads with summary-only state. |
| Wallet settlement ledger | Wallet history is settlement proof projection ordered by Kai heads and append windows. | The ledger must not ask whether old truth is still true; it asks for verified additions after the known head. |
| Calendar/account event projection | Calendar points and account brain activity nodes are projections of admitted proof objects. | The same proof object must not be duplicated into competing route-local truth objects. |

## Frozen Invariants

- Receiz is not a generic app with proof features.
- Known verified truth paints first.
- Verified stored truth is not cache and is not stale mutable data.
- Once admitted, truth is kept and extended; it is not rediscovered before use.
- Background sync discovers additions beneath settled truth.
- Weak partial payloads never outrank stronger admitted proof memory.
- Kai is the Receiz ordering coordinate when present.
- Chronos is user-facing display context and compatibility fallback.
- Database indexes and RPCs are projection mechanics, not proof authority.
- The SDK is official developer infrastructure, not a new truth engine.
- Developers should be able to build Receiz-native apps without re-learning these invariants through production failure.

## Release-Lock Requirement

`pnpm release:lock 94.0.0` must require this document for `v94.0.0` and later release cuts. A release that omits the product-truth freeze is incomplete because proof object authority, durable proof memory, Kai ordering, official SDK convenience boundaries, Sports proof economy, wallet settlement ledger projection, and calendar/account event projection would not be explicit release truth.

## Residual Risks

- Publishing `@receiz/sdk@94.0.0` is an external npm registry action and must occur after broader platform SDK gates pass.
- Live Vercel and Supabase behavior must be observed after deployment for runtime pressure evidence.
- Service-worker uptake depends on installed clients receiving the new versioned worker.
- SDK users still must verify untrusted artifact bytes; the SDK makes the correct path easy but does not turn transport into authority.
