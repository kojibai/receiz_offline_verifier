Ship the signed merchant public-store rail and public proof graph without downgrading any existing Receiz primitive.

## What Worked

- Identity Seal / Receiz Key proof authority already carried the merchant-owned publish boundary.
- The public proof rail could admit signed public-store envelopes without creating a new authority.
- The SDK could expose a one-shot helper while preserving lower-level `signPublish` and `publishSigned` composition.
- The MCP docs could name the delegated-token boundary without changing MCP runtime behavior.
- The Sports saved-card surface could keep exact `1:1` base probability while correcting the label from odds to base issue.

## What Did Not Work

- A generic token-first mental model would have made merchant storefront sync depend on `RECEIZ_ACCESS_TOKEN`.
- Labeling base `1:1` probability as `ODDS` was technically numeric but ontologically wrong for base issue cards.
- Release docs with non-standard headings failed the release-lock contract.

## Practical Rule For Future Work

When the merchant proof object already carries the authority, expose a helper that uses it directly. Do not make developers reach for a delegated token, server signer, database row, or MCP tool to prove merchant-owned public state.

## Process Law

1. Identify the implemented primitive before adding a new layer.
2. Reuse Identity Seal / Receiz Key proof authority for merchant-owned public-store sync.
3. Keep delegated Connect/OIDC tokens scoped to agent/server work.
4. Preserve deterministic first paint and durable local truth: first admission only, then append forever.
5. Keep service-worker continuity below proof object truth.
6. Validate release identity across app, SDK, MCP, offline verifier, and service worker.
