# Receiz v113.0.0 Regression Lessons

## What Regressed

Pre-release qualification found a historical v112 test that denied the newly implemented v113 surface and release records that did not satisfy the current release-lock document contract.

## Why It Regressed

The historical assertion scanned current source rather than the frozen v112 export, and the new records predated the generic release-lock template.

## Locked Rules

- Registry effective Kai cannot substitute for a key-specific signed revocation observation.
- A caller-shaped revoked-key set cannot manufacture historical revocation evidence.
- A Connect token cannot silently become operation authority.
- A serialized server artifact cannot become runtime verified merely because it arrived over an authenticated channel.
- Plan, stage, commit, attempt resolution, and effect resolution cannot collapse into one mutation call.
- `idempotent` and `already-accepted` describe different facts and must remain distinct.
- A lost commit response is indeterminate, not failed.
- Accepted does not mean effects delivered.
- Private database rows coordinate one domain; they cannot manufacture artifact, identity, ownership, or history truth.
- Documentation that does not execute against packed packages is not release evidence.

## Contract Gates

The frozen-v112/current-v113 matrix tests, packed README examples, v113 release lock, reconciliation suite, and umbrella release freeze prevent these regressions from shipping.

## Release-Lock Requirement

`pnpm release:lock 113.0.0` must require this regression record. Key-specific revocation evidence, Connect/capability separation, returned-artifact reverification, phase separation, indeterminate attempt recovery, effect status, and coordination-row non-authority must remain independently reviewable boundaries.
