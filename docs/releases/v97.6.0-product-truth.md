## Frozen Primitives

| Primitive | v97.6.0 truth | Must not regress |
| --- | --- | --- |
| Proof object | Receiz proof objects carry deterministic identity, verifiable payload, provenance, inspection behavior, and Kai/Klok state. | Do not let SDK, MCP, DB, service worker, or projection rows treat proof truth as generic app data. |
| Kai Klok | Kai Klok is the proof object state machine. `kaiPulseEternal` / `kai_upulse` is the pulse unit. | Do not use Chronos as the primary state coordinate for admitted proof objects. |
| Merchant Identity Seal / Receiz Key publish | Merchant-owned public-store state can be signed by the merchant proof object and appended to the public proof rail. | Do not require a delegated agent/server token where the merchant proof object is the authority. |
| Public-store projection rail | Public-store records are public recovery projections for storefront first paint and cold-start recovery. | Do not let public-store projections outrank sealed artifacts, identity proofs, ownership appends, settlement rows, verified local truth, or verified appends. |
| MCP agent rail | `@receiz/mcp-server` exposes Receiz SDK/API rails to agents. | MCP must never replace, rename, or outrank Receiz proof objects, identity primitives, ownership surfaces, settlement primitives, public proof surfaces, or verification primitives. |
| Service worker runtime continuity | Service worker state preserves runtime continuity, offline access, and projection availability. | Do not let service-worker state become source of truth above proof objects or verified local truth. |
| Public proof graph | `/proof-graph` is a public proof surface for inspecting the Receiz network. | Do not turn the graph into decorative marketing or static simulation. It must carry real release/API/product proof data and live Kai pulse state. |
| Sports vault card proof display | Saved Sports cards must distinguish base issue probability from scarce rarity odds. | Do not label base `1:1` issue as `ODDS`; reserve odds language for rarity variants. |

## Frozen Invariants

- Use known verified truth immediately.
- Durable proof memory remains first admission only, then append forever.
- Ask only for verified additions after the known proof/Kai head.
- Merchant proof authority can sign merchant public-store publish without `RECEIZ_ACCESS_TOKEN`.
- Delegated agent tokens are for scoped agent/server operations only.
- Public-store publish must return append coordinate evidence: Kai pulse, anchor, append proof, and known head.
- MCP tools must expose the authority boundary instead of becoming authority.
- Service worker release identity must match app release identity.
- Public proof graph navigation must remain in-page and mobile-first.
- Base cards can show `1:1`, but the label must be base issue, not rarity odds.

## Residual Risks

- npm publication is an external registry action and must be run after local gates pass.
- Production database and Vercel health remain runtime evidence after deploy, but they do not become proof authority.
- Live proof graph visual verification should be repeated after deploy because animation, viewport fit, and browser compositor behavior are visible evidence.
- Full `pnpm test:release-freeze` can be long-running; any skipped umbrella gate must be named before release.

## Release-Lock Requirement

`pnpm release:lock 97.6.0` must require this document for `v97.6.0` and later release cuts. A release that omits the product-truth freeze is incomplete because merchant Identity Seal / Receiz Key publish authority, public-store projection limits, proof graph public witness behavior, service-worker continuity boundaries, MCP/SDK convenience limits, Sports base-card copy truth, and app/SDK/MCP version lock cannot remain explicit release truth.
