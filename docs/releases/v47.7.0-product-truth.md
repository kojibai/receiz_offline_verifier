# Receiz v47.7.0 Product Truth

Release date: April 9, 2026
Status: Locked for `v47.7.0` release prep

## Purpose

This document defines the patch-level product-truth freeze for Receiz `v47.7.0`. It preserves the current `v47.x` product truth while recording durable contacts persistence, historical relationship backfill, live person-first write-through actions, and portable import/export as explicit non-regression behavior at release time.

For this offline verifier repository, the patch is represented as current-release documentation and version-surface alignment only. The verifier trust contract remains unchanged relative to `v47.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- The file remains the authority for artifact truth.
- Deterministic manifests remain the authority for historical route truth.
- Durable contacts CRM state, timeline rollups, and import-run observability must run on top of the existing trust, auth, and operator primitives, not weaken or replace them.
- Signed-in account state and server-truth persistence remain the authority for contact identities, notes, stages, invite state, timeline history, and import lineage.
- Public twins, operator surfaces, offline verifier entrypoints, and release metadata remain transport and governance around those boundaries, not replacements for them.

## Locked Patch Surface

### 1. `/contacts` Must Remain Durable Operator Infrastructure

- Contacts must persist in dedicated contact, identity, event, and import-run tables instead of local-only client state.
- Person records must remain shaped around full-name, company, role, notes, next-step, stage, and invite state.
- Bulk hydration and imports must remain observable and reproducible through tracked import runs.

### 2. Historical Relationship Memory Must Backfill Into Contacts

- Signed-in users must not start from zero when historical participants or collectors already exist.
- Historical conversation participants must materialize into contacts with message-history context.
- Historical original collectors must materialize as person-linked timeline events, and owner/twin interaction counts must roll up from stored events rather than client-side guesses.

### 3. Contacts Actions Must Stay Live And Person-First

- Inline edits must patch back to the contact record through server truth.
- Invite launches and manual interaction logs must append real timeline events.
- Refresh, sync, and operator handoff must preserve notes, stages, invite status, last-meeting state, and person timelines instead of collapsing back to local cache.

### 4. Contacts Must Remain Portable

- Import and export must continue to support CSV, JSON, and vCard as first-class flows.
- Browser-native phone-contact picking must be used where available.
- Unsupported clients must retain a clean file-based fallback, and all supported import sources must dedupe into the same server-backed contact graph.

### 5. Release Governance And Version Truth

- Release-scoped note and product-truth documents are required for the patch.
- Release-governed version surfaces must move together across package metadata, service worker defaults, public badges, offline verifier surfaces, and release documents.
- This repository records the public release documents and shipped HTML version markers for that lockstep requirement.

## Allowed Evolution

Implementation details may change after `v47.7.0` only if they preserve the authority boundaries and locked patch behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v47.7.0` if it:

- pushes contacts persistence back into local-only state or drops import-run observability
- stops historical participants or original collectors from backfilling into signed-in CRM state
- loses write-through edits, live timeline events, or durable per-person state across refresh, sync, or operator handoff
- traps contacts in one UI by regressing supported import/export paths, dedupe behavior, or device fallback behavior
- changes release truth without updating the versioned release docs and aligned version surfaces
