# Receiz v47.3.0 Product Truth

Release date: April 9, 2026
Status: Locked for `v47.3.0` release prep

## Purpose

This document defines the patch-level product-truth freeze for Receiz `v47.3.0`. It preserves the current `v47.x` product truth while recording the custom-domain continuity, embedded-browser sign-in, and trusted-host media behaviors that are now treated as explicit non-regression behavior at release time.

For this offline verifier repository, the patch is represented as current-release documentation and version-surface alignment only. The verifier trust contract remains unchanged relative to `v47.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- The file remains the authority for artifact truth.
- Deterministic manifests remain the authority for historical route truth.
- Canonical Receiz identity, passkey, WebAuthn, and PBI authority remain rooted in the Receiz auth core even when a custom domain participates in the surface flow.
- Public readers, verified-player surfaces, storage proxies, custom-domain bridges, world embodiment, and release metadata remain transport and governance around those boundaries, not replacements for them.

## Locked Patch Surface

### 1. Verified Custom Domains Reuse Receiz Authority

Verified custom domains are now part of the governed authenticated surface.

- A signed-in Receiz user may arrive on a verified custom domain and reuse existing Receiz session authority.
- The custom domain may mint a local domain session only from that canonical Receiz authority.
- The local domain session must not weaken or replace the canonical Receiz identity, passkey, or signing model.
- Logout and sign-in flows must remain symmetric across the business host and `receiz.com`.

### 2. Custom-Domain Signing Keeps Canonical Authority

Business-host signing may feel native, but authority remains canonical.

- Custom-domain signing flows must keep Receiz WebAuthn / PBI as the canonical authority.
- The bridge flow must return the user to the business host after canonical auth completes.
- The business host must not behave like an independent passkey authority when canonical Receiz authority is required.

### 3. Embedded Browsers Must Not Dead-End Passkeys

Likely in-app browsers are now an explicit governed sign-in condition.

- Sign-in surfaces must detect likely embedded browsers before prompting for passkeys.
- Email fallback must open immediately instead of waiting for a WebAuthn failure path to teach the user what went wrong.
- The UI must tell the truth: continue with email here, or open the page in Safari, Chrome, or the default browser to use passkeys.

### 4. Custom-Domain First Paint Must Trust Branded Host Media

Custom-domain profile rendering keeps the same trusted-host treatment as the main Receiz domain.

- The profile route on a custom domain must trust its own branded host for proxied hero media fetches.
- First avatar and header render must not depend on reload to recover from a trusted-host mismatch.
- Trusted-host media handling on custom domains must match the governed first-load behavior of the main Receiz domain.

### 5. Release Governance and Version Truth

The release contract remains locked across visible version surfaces.

- Release-scoped note and product-truth documents are required for the patch.
- Release-governed version surfaces must move together across package metadata, service worker defaults, public badges, and release documents.
- This repository records the public release documents and shipped HTML version markers for that lockstep requirement.

## Allowed Evolution

Implementation details may change after `v47.3.0` only if they preserve the authority boundaries and locked patch behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v47.3.0` if it:

- forces already-signed-in users on verified custom domains to reauthenticate because existing Receiz authority cannot be reused
- weakens or forks canonical Receiz WebAuthn, PBI, passkey, or signing authority during custom-domain auth or signing flows
- presents passkey sign-in inside likely embedded browsers instead of expanding email fallback immediately with truthful guidance
- breaks first-load custom-domain hero media because branded hosts are not treated as trusted fetch origins
- changes release truth without updating the versioned release docs and aligned version surfaces
