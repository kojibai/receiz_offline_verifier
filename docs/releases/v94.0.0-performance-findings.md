# Receiz v94.0.0 Performance Findings

Release date: 2026-06-27

## Executive Finding

`v94.0.0` locks the production performance rule learned across the v93 arc: use known verified truth immediately, then ask only for verified additions after the known head.

The performance improvement is not generic caching, optimistic UI, or database authority. It is proof-native projection: admitted proof truth paints first, while RPCs, indexes, SDK helpers, and background work discover verified additions beneath that truth.

## Findings

### 1. Durable Proof Memory Removes Rediscovery Lag

Sports card history proved that admitted event history should not be re-fetched before render. The same rule applies to wallet ledgers, calendar points, account activity nodes, public proof objects, and SDK registers.

### 2. Bounded Projection Turns Database Work Into Mechanics

Owner, sport, collectible, tournament, vault, and Kai-head bounds reduce pressure by asking for the next proof additions instead of broad rediscovery. Those paths are projection mechanics. They do not replace proof payloads, verified appends, ownership appends, or settlement ledger rows.

### 3. Kai Ordering Prevents State Collisions

Kai pulse/upulse is the Receiz state coordinate when present. Preserving app-supplied child upulses prevents multiple proof children born in one Chronos instant from collapsing into the same stored coordinate.

### 4. SDK Helpers Carry The Fast Path To Developers

The official `@receiz/sdk@94.0.0` launch packages validation, deterministic projection, admission, persistence, append windows, and test fixtures so developers do not rebuild newest-response-wins behavior around Receiz objects.

### 5. First Paint Must Project Known Truth

No release surface should block first paint on session checks, analytics, global sync, server reconciliation, lazy imports, or full-history rediscovery when deterministic file proof, verified local register truth, route-bound payloads, or admitted proof memory already exist.

## Locked Performance Invariants

- Known verified truth paints first.
- Proof memory is not cache.
- Background work discovers verified additions only after the known head.
- Bounded RPCs and indexes are projection mechanics beneath proof authority.
- Kai is the state coordinate where present; Chronos is display/fallback context.
- SDK helpers must preserve admitted fields and append verified additions.
- Weak partial payloads may fill missing fields but must not erase stronger admitted proof truth.
- No release may claim SDK publication until build, test, package, and registry evidence exist in the platform release environment.

## Release-Lock Requirement

`pnpm release:lock 94.0.0` must require this document for `v94.0.0` and later release cuts. A release that omits performance findings is incomplete because durable proof memory, bounded projection, Kai ordering, SDK fast-path helpers, first-paint truth, and verified-addition windows would not be explicit release truth.
