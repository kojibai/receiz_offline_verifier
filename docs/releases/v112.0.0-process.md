# Receiz v112.0.0 Release Process

The `@receiz/sdk` v112.0.0 release retains **first admission only, then append forever**.

## Goal

Make the v112 artifact authority hierarchy executable across SDK, MCP, AI skills, packaged registry identity, durable browser admission coordination, and release evidence without introducing remote global authority.

## What Worked

Implementation proceeded in dependency order: exact-byte input and verification custody; verified actor/history evidence; portable transition planning; deterministic effects derivation; capability signing and verification; sealing and neutral staging; independent named-domain commit; asynchronous admission-store boundaries; browser IndexedDB durability; v111 containment; typed MCP coordination; canonical registry generation; AI/package coordination; distribution; and final release freeze.

Independent reviews guarded verification/admission separation, transition-plan-command separation, namespace preservation/reducers, neutral staging, independent commit verification, named-domain atomicity, and receipt non-authority. v111 compatibility remained exact-byte readmission only. No current operation received authority from a historical object.

The registry process is deterministic and unsigned: generate canonical JSON, verify the pinned predecessor digest, calculate the single canonical digest, embed the exact value/laws in generated SDK source, rebuild declarations/runtime, then test detached recursive freezing and digest equality. No native `.receiz` or signed wrapper is created or claimed.

## What Did Not Work

Treating a package path identifier as a repository filesystem read produced an overbroad clean-room assertion; the corrected contract rejects actual filesystem APIs and independently proves the installed consumer loads the frozen registry. A separate packed-distribution gate also found stale pre-v112 MCP helper names in human AI instructions even though manifest parity was correct; those instructions were corrected to the exact five v112 tools and nine application operations.

Package qualification builds and typechecks `@receiz/sdk` and `@receiz/mcp-server`, installs packed tarballs into clean temporary consumers, loads the generated registry without repository-relative access, composes the browser store and admission engine, checks identity capability signing surfaces, and verifies the deferred global rails are absent.

The final gate runs focused v112 tests, v111 compatibility, package release contracts, doctrine contracts, full release freeze, and visual non-regression evidence. Task 14 records the full release freeze and confirming desktop/mobile evidence passing on 2026-07-19. The signed governance attestation runs after push through the existing GitHub Environment workflow so it can bind the final commit SHA. The visible change is version identification only; page hierarchy, first paint, media, routing, and service-worker behavior remain unchanged. No new product HTTP route or migration exists.

The process therefore records every required boundary directly: exact-byte artifact truth; verification/admission separation; verified actor/history; transition-plan-command separation; namespace preservation/reducers; registry chain; neutral staging; independent commit verification; named-domain atomicity; receipt non-authority; v111 compatibility; no new product HTTP route; no migration; checks; visual evidence; and exact uncertainty boundaries. The only remaining external evidence boundary is the post-push signed governance attestation.

## Practical Rule For Future Work

Run machine-manifest parity, human-instruction distribution, clean-room package installation, first-paint evidence, and umbrella release freeze as separate gates. None substitutes for another, and no gate may weaken exact artifact truth or runtime custody to make packaging easier.

## Release-Lock Requirement

`pnpm release:lock 112.0.0` must require the complete v112 record set. Omitting a record would leave the registry chain, compatibility boundary, staging/commit separation, or external attestation boundary outside explicit release truth.
