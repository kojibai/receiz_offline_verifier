# Receiz v105.0.0 Invariant Register

`v105.0.0` carries forward the proof-native artifact system and locks a browser-safe SDK architecture beneath authenticated native Record-before-Seal truth. Runtime, compiler, testing, error, capability, generation, MCP, AI-skill, and packaging rails remain mechanics.

## Locked Invariants

1. Receiz remains a proof-native artifact system. Product truth is carried by files, identities, vault objects, ownership states, settlement states, public witness surfaces, verification routes, public proof surfaces, and append-only history.
2. Proof objects remain stronger truth than SDK helpers, MCP tools, OIDC tokens, webhook endpoints, server rows, service-worker state, public-store projections, app-state rows, charts, analytics, SEO, session state, or generic display state.
3. Kai Klok remains the proof object state machine. `kaiPulseEternal` / `kai_upulse` remains the pulse unit for complete proof objects.
4. Durable proof memory remains first admission only, then append forever. Hidden work may only append verified additions after the known head.
5. Deterministic first paint must project known proof, identity, ownership, settlement, profile, market, ledger, transaction, Sports, public graph, and archive truth before weaker work.
6. `@receiz/sdk` remains typed developer convenience beneath proof objects, verified appends, identity primitives, ownership appends, settlement rows, public proof surfaces, and offline verification.
7. `@receiz/mcp-server` remains an agent rail over SDK/API mechanics and must name its authority boundary.
8. Root and React SDK graphs cannot reach compiler, CLI, testing, filesystem, or Node built-ins through transitive exports.
9. Compiler code is available only through the explicit Node-only compiler subpath; browser-safe testing resolves separately.
10. Emulator and conformance output is sandbox evidence and cannot claim production artifact verification.
11. Stable errors redact secrets while retaining documented codes; capability describe/probe remains backward compatible and lower authority.
12. Developer-owned extension siblings are never overwritten; protected generation must fail or append safely.
13. Semantic MCP writes require explicit confirmation and cannot infer permission from inspection or planning.
14. SDK, MCP, and AI skills align exactly at 105.0.0 with no `workspace:` protocol in source, packed, or externally installed manifests.
15. Authenticated real-owner Record-before-Seal creation, continuity-bound verification, and historical proof compatibility remain unchanged; no database migration is required.
16. Static verifier, offline verifier, record-seal studio, settlement entrypoint, Sports card verifier, package metadata, release docs, and service-worker runtime version must move together.
