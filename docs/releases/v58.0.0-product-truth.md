# Receiz v58.0.0 Product Truth

Release date: April 17, 2026
Status: Locked for `v58.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v58.0.0`. It carries forward the `v57.0.0` baseline while recording authoritative signed-in market first paint, coherent wallet and chart-buy hydration, lean deterministic market transport, correct live tracked-position semantics, durable market continuity, and release-surface lockstep as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, shipped HTML version markers, and package-metadata alignment only. The verifier trust contract remains unchanged relative to `v57.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- File-backed proof remains the authority for artifact truth.
- Deterministic market seed truth plus the live signed-in user overlay remain the authority for market first paint when that truth is already knowable.
- Wallet, position, prediction-history, and share-trade state remain durable user-visible truth, not disposable background fetch hints.
- Live refresh may hydrate on top of deterministic baseline state, but it must not replace or discard already-known stronger market or wallet truth as the source of truth.
- Public product routes, conformance surfaces, release metadata, and offline verifier entrypoints remain operating or governance surfaces around those authorities, not replacements for them.

## Locked Release Surface

### 1. The v57.0.0 Baseline Must Remain The Floor

- Release-governed version surfaces must remain lockstep and current-release docs must stay aligned.
- Canonical managed-owner routing, world/market fast-entry continuity, certificate-native custody/history, and market-conformance guarantees from `v57.0.0` remain in force.
- Market truth already known at navigation time must continue to win over weaker anonymous or late-arriving refresh state.
- The offline verifier trust contract must remain file-authoritative, deterministic, and fail-closed.

### 2. Signed-In Market Entry Must Be Authoritative When Knowable

- Signed-in `/market` entry must render authoritative wallet, position, and history truth on first paint.
- Signed-in `/market/assets/[receizId]` entry must render authoritative wallet, position, and history truth on first paint.
- Route hydration must preserve stronger signed-in wallet/history truth instead of replacing it with a weaker later snapshot.
- Signed-in market truth must be treated as durable route truth rather than a disposable bootstrap hint.

### 3. Wallet And Chart-Buy Hydration Must Share One Coherent Path

- Market wallet surfaces and chart-buy surfaces must hydrate through one shared wallet-summary path.
- Wallet hydration must patch the live signed-in market snapshot directly instead of waiting on an oversized desk refresh.
- Chart-buy preload must seed wallet truth before the sheet opens when that truth is already available.
- Retry and continuation flows must preserve wallet coherence through funding, passkey re-entry, and snapshot refresh.

### 4. Market Transport Must Stay Deterministic And Lean

- Market transport must carry deterministic public inputs plus the live user overlay instead of shipping the fully expanded desk snapshot.
- Server and client must materialize market snapshots locally from the same transport contract.
- `/market`, `/api/market/snapshot`, route prewarm, market refresh, and chart-buy refresh paths must rebuild from the same deterministic transport shape.
- Transport slimming must not drop authoritative wallet, position, pricing, or history truth.

### 5. Tracked Positions Must Match Live Contract Truth

- Tracked positions must equal live share positions plus still-open contracts.
- Expired or non-open contracts must not count as live tracked positions.
- Wallet rails for open contracts must filter through real open-contract semantics rather than any attached prediction position.
- Prediction-history and share-history rails must remain visible without inflating the live tracked-position count.

### 6. Market Continuity Must Preserve Durable User Truth

- Market snapshot continuity must preserve newer wallet, prediction-history, and share-trade truth when a weaker refresh arrives later.
- Prediction history and share-trade history must be treated as durable user-visible truth, not temporary background fetch state.
- Route refresh must not regress the visible signed-in market surface to a lighter shell once stronger truth has been established.

### 7. Release-Governed Version Surfaces Must Move Together

- Release-governed version surfaces must move together to `v58.0.0` across package metadata, service-worker defaults, public verifier artifacts, public release badges, and current release documentation.
- Public-facing version truth must not drift between shipped HTML entrypoints, docs indexes, release artifacts, and repository metadata.
- This offline verifier repository must reflect that lockstep rule in `package.json`, `README.md`, `RELEASE_NOTES.md`, `CHANGELOG.md`, `docs/README.md`, `docs/governance/README.md`, `docs/FORMAT.md`, `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.

### 8. Release Governance And Verifier Truth

- Release-scoped note, product-truth, and checklist documents are required for `v58.0.0`.
- Root package metadata must resolve to `v58.0.0`.
- Shipped verifier, studio, and settlement entrypoints must visibly display `v58.0.0`.
- The offline verifier trust contract remains unchanged relative to `v57.0.0`.

## Allowed Evolution

Implementation details may change after `v58.0.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v58.0.0` if it:

- reintroduces a light signed-in market shell that hides already-known positions, history, or wallet truth until a later settle
- lets wallet and chart-buy surfaces diverge so visible buying power depends on which market surface hydrated first
- expands market transport back into an oversized desk snapshot when the client can deterministically derive the same public state
- counts expired or otherwise non-open contracts as live tracked positions
- discards stronger wallet, prediction-history, or share-trade truth during later route refresh
- allows package metadata, public badges, shipped verifier artifacts, or current-release docs to advertise different release numbers
- changes release truth without updating the versioned release docs and aligned repository version surfaces
