## Preparation Steps

1. Preserve `v81.0.0` as the prior current-release archive.
   - Keep `docs/releases/v81.0.0.md`.
   - Keep `docs/releases/v81.0.0-product-truth.md`.
   - Keep `docs/releases/v81.0.0-checklist.md`.
   - Keep `docs/releases/v81.0.0-process.md`.
   - Preserve `v81.0.0` as historical Signal Card asset truth.

2. Publish canonical `v82.0.0` release artifacts.
   - Add `docs/releases/v82.0.0.md`.
   - Add `docs/releases/v82.0.0-product-truth.md`.
   - Add `docs/releases/v82.0.0-checklist.md`.
   - Add `docs/releases/v82.0.0-process.md`.

3. Align repository release surfaces.
   - Update `package.json` to `82.0.0`.
   - Update root `README` current-release references.
   - Update `RELEASE_NOTES.md` and `CHANGELOG.md`.
   - Update `docs/README.md`, `docs/governance/README.md`, and `docs/FORMAT.md`.
   - Update product-truth reference status markers carried forward for the current release.
   - Update shipped version markers in `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.
   - Ensure local release-surface checks can execute through `pnpm lint`, `pnpm build`, and `pnpm release:lock`.

4. Validate Sports Arena destination gates.
   - `/game` points to Sports.
   - `/game/sports` and `/game/sports/baseball` render sport navigation, MLB rows, Pick 5, field, cockpit, leaders, proofs, cards, profile access, and mobile chrome.
   - Global navigation does not cover or destabilize the game surface.

5. Validate MLB card inventory and rarity gates.
   - MLB registry ingestion and player images are present.
   - Base-set cards avoid fake rarity labels.
   - Owned rarity variants render only when owned.
   - Issue numbers, legacy issue backfill, rookie registry, pack origin, downloaded metadata, rarity switching, highest-rarity/lowest-issue ordering, and premium exports remain deterministic.
   - Player strength and card rarity stay separate scarcity gates.

6. Validate pack and odds gates.
   - Pack purchase, reveal, sync, reward packs, free packs, starter packs, premium packs, and chase packs share the same conceptual model.
   - Published and observed odds match deterministic issuance math.
   - Simulation output covers player scarcity, spread, duplicate pressure, top-player concentration, rarity availability, and registry basis hashes.
   - Pack health catches misleading concentration or rarity availability before launch.

7. Validate Pick 5 and live cockpit gates.
   - Per-game play uses owned matchup-team cards and may accept fewer than five.
   - Main Event uses the user's best five across the slate.
   - Selected cards rise to the top, unselected cards remain identifiable, scores are scoped to selected game/date/context, and button language matches the mode.
   - Final games cannot be entered as live games.
   - Live cockpit shows selected scoring cards, scoped deltas, total, rank, payout position, payout distance, scoring story, line-chart movement, and value cues.

8. Validate deterministic live game state and pregame gates.
   - Selected game field data matches the selected game.
   - Entry lock state, active entries, first-live-game fallback, and finalized proof fallback cannot rewrite live scoreboards.
   - Pregame shows matchup, venue, first-pitch readiness, top team star cards, ratings, projected scores, stats, and card access without pretending a live play occurred.

9. Validate tournament, finality, reward, and card-memory gates.
   - Daily tournament entry, wallet funding, Stripe delta recovery, minimum participant logic, underfilled refund support, top-percent payout, result pages, settlement payout, last-winner sharing, and play-again reset pass.
   - Per-game and Main Event lock domains remain separate.
   - Final games remain selectable when rewards need claiming and clear after claims.
   - Entry finality can rank, issue rewards, open reward packs, update proof history, and append long-term card-career memory.

10. Validate Sports market and trading gates.
    - Sports card sales, wallet-first purchase, Stripe delta funding, provenance routes, public stages, value panels, bids, asks, floors, spread, volume, value movement, listings, sale history, and market conformance pass.
    - Trade proposals, action routes, baskets, counterproposal direction, accept, decline, cancel, stale-state, and status handling pass.
    - Non-open decline/cancel returns `409 sports_trade_not_open`; unexpected lookup failures remain `500 sports_trade_lookup_failed`.

11. Validate conformance, scale, and PWA gates.
    - Sports conformance routes, badge, history, rollups, rarity scoring, odds, source checks, hot fanout, checkout recovery, and signed history pass.
    - Hot public snapshots, Vercel Cron warming, runtime cache helpers, warmed stream reads, account overlay throttling, route backpressure, worker jobs, observability, Vercel build-cost guard, and `vercel-ignore-build` support pass.
    - Sports first paint projects account, entered, cockpit, card catalog, and logged-in state early.
    - PBI signing, optimistic post anchors, signing control prewarm, and newly posted profile content continuity pass.

12. Validate release-surface lockstep.
    - Confirm all governed version markers show `v82.0.0`.
    - Confirm all current-release docs and indices point to `v82.0.0` artifacts.
    - Confirm `v80.0.0` and `v81.0.0` artifacts remain historical and untouched.
    - Run `pnpm release:lock`.

13. Generate signed release attestation.
    - Run `pnpm release:attest` for the final commit in the environment that holds the release attestation signing key.
    - Retain the signed attestation artifact under the governance release-attestation policy.
    - Do not tag or ship without the signed attestation.

Broader platform release validation should include the gates named in `docs/releases/v82.0.0-checklist.md`, including:

- Sports card career append and provenance coverage

- trade proposal, accept, decline, cancel, stale-state, and lookup-failure coverage
- Sports conformance badge, history, rollup, odds, source, hot fanout, checkout recovery, and signed-history coverage
- hot snapshot, runtime cache, stream fanout, route backpressure, worker hardening, and observability coverage
- PWA first paint, signing continuity, optimistic anchor hydration, profile route continuity, and release-surface lock checks

## Tag Criteria

Do not tag `v82.0.0` until:

- all required `v80.0.0`, `v81.0.0`, and `v82.0.0` release artifacts exist
- version surfaces are aligned to `v82.0.0`
- release gates in `docs/releases/v82.0.0-checklist.md` pass from the exact release candidate
- governed surfaces all show `v82.0.0`
- `pnpm lint` has zero warnings/errors
- `pnpm build` completes
- `pnpm release:lock` confirms the release surfaces
- `pnpm release:attest` emits a signed attestation for the final commit

## Release Rule

Do not ship `v82.0.0` unless the release gates in this checklist pass from the exact release candidate, `pnpm release:lock` confirms the release surfaces, and `pnpm release:attest` emits a signed attestation for the final commit.
