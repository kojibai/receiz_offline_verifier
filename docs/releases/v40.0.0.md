# Receiz v40.0.0

Release date: March 27, 2026

## Overview

`v40.0.0` is the release where Receiz stops looking like a single proof product with adjacent tools and starts behaving like a full trust runtime: hardened signatures, stronger offline issuance and verification, operational wallet and payment controls, richer public artifact delivery, broader admin and business infrastructure, and an entirely new world stack for owned runtime, rendering, identity, and presence.

The core rule does not change: the file remains the authority. Record, seal, share, and verify are still deterministic. Public `/v` routes, online viewers, profile delivery, and world surfaces provide context, portability, and operational reach around the same underlying artifact truth.

## What Defines v40.0.0

- `signatureV4` and release-governed trust are now central platform primitives, not edge capabilities.
- Offline verification and offline studio surfaces are now substantially more self-contained and operational.
- Wallet, payments, settlement, and admin controls have matured into real operational infrastructure.
- Verified artifact, original delivery, and profile-publication surfaces are now deeper, broader, and more productized.
- Business distribution now includes custom domains and stronger account/admin rails.
- Receiz now includes a first-class world runtime stack across public world surfaces, owned model packs, render services, and conformance history.

## Major Accomplishments Since v30.0.0

### 1. Security, Trust, and Release Integrity Tightened Materially

The entire `30.x` line pushed the trust boundary inward:

- stronger artifact verifier hardening
- broader API security contracts
- expanded `signatureV4` server and enrollment testing
- stronger governance controls and artifact-signature verification
- release lock and attestation scripts tied directly to versioned release notes and checklists
- additional chaos and failure-path coverage around critical runtime behavior

This matters because Receiz now treats trust not just as a product claim, but as a continuously enforced property of issuance, verification, deployment, and release itself.

### 2. Offline Proof and Document-Seal Delivery Became Real Product Surfaces

Since `v30.0.0`, offline proof tooling moved from useful support tooling into a durable product layer:

- offline verifier surfaces were carried forward as release-versioned public artifacts
- offline record and seal flows matured further
- document-seal proof generation and verification routes expanded
- embedded proof assets and offline bundle verification became more portable
- trusted signer enrollment and offline signing behavior became more operational for standalone/offline studio paths

The result is a platform that can keep the proof model intact even when the network is unreliable, unavailable, or intentionally removed from the flow.

### 3. Wallet, Payments, and Settlement Became Operational Control Planes

The settlement stack expanded far beyond a simple payment-proof story:

- Stripe user payment flows and session routes landed
- wallet Stripe deposit events and credit rollups were added
- admin balance adjustment and wallet-post adjustment flows landed
- public ledger and wallet surfaces expanded
- Connect payment/session routes were deepened
- pay-via-Receiz flows matured

This is the difference between a proof of payment product and an operational money-and-ledger surface that can actually support business workflows.

### 4. Admin, Business, and Account Infrastructure Expanded

`30.x` added a much stronger operational shell around the core trust system:

- admin dashboard expansion
- admin user directory
- admin Stripe user lookup
- stronger account surfaces
- business custom-domain support
- custom-domain API routes and documentation

Receiz is now in a stronger position to serve teams, operators, and businesses that need administrative visibility and controlled branded distribution, not just end-user proof flows.

### 5. Verified Artifact and Profile Delivery Reached a New Level

Public artifact delivery grew substantially:

- verified artifact route clients and snapshots expanded
- original artifact viewers improved
- verified original player experiences deepened
- profile published manifests and route hydration expanded
- profile original delivery, playback, and checkout/context routes improved
- richer profile/public delivery plumbing landed

Receiz now delivers proof-bearing artifacts with better public presentation, stronger delivery contracts, and better downstream consumption surfaces.

### 6. Real-Time Communication and Media Surfaces Continued to Mature

The platform added or deepened:

- video and voice chat infrastructure
- call quality gates and topology policy enforcement
- better media delivery and verified-player experiences
- stronger chat attachment and conferencing routes
- broader call/session operational coverage

This extends Receiz beyond static proof issuance into live collaborative surfaces that can still remain rooted in deterministic artifact truth.

### 7. The World Stack Landed as a First-Class Platform Layer

The biggest step-change after `v30.0.0` is the arrival of the world runtime:

- world core schema and migrations
- world agent memory and chat-channel infrastructure
- district access grants and avatar versioning
- public world routes and world module surfaces
- owned model-pack infrastructure
- resident and owned twin runtime scripts
- Runpod bootstrap and render-backend tooling
- CUDA and native renderer services
- world realism, blueprint, and production docs
- world conformance history

This is not a small adjacent feature. It turns Receiz into a broader identity-and-runtime platform where public embodiment, owned computation, deterministic packaging, and trust surfaces all connect.

### 8. Performance, PWA, and Deterministic Delivery Were Tightened

Across the `30.x` line, Receiz kept improving product sharpness:

- page-first-paint testing
- performance budget enforcement
- deterministic page-to-page speed improvements
- PWA and service-worker behavior refinements
- better offline shell coverage
- route and hydration cleanup across high-traffic surfaces

This matters because proof systems only feel trustworthy when the product itself feels stable, fast, and consistent.

The major themes of the release are:

- hardened trust and signing
- offline-capable proof tooling
