# Receiz v95.0.0 Invariant Register

Release date: 2026-06-27

This register makes explicit the v95.0 invariants proven by the Official Public Table Release.

## 1. Receiz Is A Proof-Native Artifact System

Receiz is not a generic app with proof features. Product truth is carried by files, proof objects, identities, ownership states, settlement states, public witness surfaces, and verification routes.

Violation: release language reduces Receiz to an app shell, SDK wrapper, API product, cache strategy, or generic display experience.

## 2. Complete Proof Objects Carry Kai/Klok

A complete Receiz proof object carries deterministic identity, sealed/verifiable payload, provenance, inspection behavior, and Kai/Klok state.

Violation: a canonical proof object contract is exposed without Kai/Klok, except where explicitly marked as legacy compatibility.

## 3. Kai Klok Is The State Machine

Kai Klok is the proof object state machine. `kaiPulseEternal` / `kai_upulse` is the pulse unit.

Violation: Kai is described as a label, metadata, vibe, timestamp helper, optional tie breaker, or SDK-only field.

## 4. Chronos Is Not Backend Authority

Chronos is human display, bridge, import context, and debugging context.

Violation: Chronos is used as the primary ordering authority for Kai-bearing proof objects.

## 5. Offline Proof Requires A Living Coordinate

Offline proof works because the object carries its own proof coordinate without consensus.

Violation: complete offline proof depends on server, DB, SDK, API, or route rediscovery before use.

## 6. Durable Proof Memory Uses Known Heads

First admission persists verified truth. Later sync asks only for verified additions after the known proof/Kai head.

Violation: known proof truth is rediscovered before projection or replaced by a partial later payload.

## 7. SDK Mirrors Canonical Proof Law

The SDK is developer convenience over the canonical primitive contract, not a softer model.

Violation: SDK proof types, fixtures, schemas, examples, or docs loosen requirements that internal proof bundle law requires.

## 8. Public Reports Are Doctrine Surfaces

The public architecture report must name the same primitives as the release docs.

Violation: report language reintroduces optional Kai/Klok wording, generic proof-feature framing, or SDK-only ontology.

## 9. Projection Mechanics Stay Beneath Proof Authority

Server, DB, SDK, API, and UI routes project, sync, index, and publish proof truth. They do not replace it.

Violation: a route payload, SDK response, DB row, API response, report summary, or UI state outranks sealed/admitted proof truth.

## 10. Offline Verification Remains File-Authoritative

This offline verifier can document platform SDK behavior, but verification authority remains in selected file bytes and embedded proof material.

Violation: release metadata, SDK docs, service-worker cache state, external registry state, or generated report state is presented as stronger than file proof.

## Release-Lock Requirement

`pnpm release:lock 95.0.0` must require this document for `v95.0.0` and later release cuts. A release that omits the invariant register is incomplete because proof-native system identity, complete proof object Kai/Klok state, Kai Klok state-machine truth, Chronos limits, offline proof completeness, durable proof memory heads, SDK canonical parity, public report doctrine, projection boundaries, and offline file authority would not be explicit release truth.
