# Receiz v64.0.0 Product Truth

Release date: April 21, 2026
Status: Locked for `v64.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v64.0.0`. It carries forward the governed proof, market, certificate, wallet, and settlement model while recording the complete value loop as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, shipped HTML version markers, and package-metadata alignment only. The verifier trust contract remains unchanged relative to `v60.0.0`.

The canonical invariant reference for this release is `docs/value-loop-invariants.md`.

## Core Product Truth

`v64.0.0` is the release where Receiz becomes one complete value system instead of separate advanced primitives.

The product now has a closed causal loop:

> Create value, hold Settlement, send proof-native value, use Reserve for notes and wire transfers, issue shares into certificates, let buyers fund certificates, and credit sellers only when value is actually funded.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- File-backed proof remains the authority for artifact truth.
- Settlement remains proof-native liquid value.
- Reserve remains funded external-conversion value.
- Wallet account-lane truth remains the authority for Settlement and Reserve display.
- Account quote math remains the authority for USD and phi presentation.
- Certificate issue locks value, and funded certificate sale credits the seller only when buyer funding exists.
- Notes issue from Reserve only.
- Wire transfer draws from Reserve.
- Provider names remain implementation detail, not user-facing liquidity language.

## Locked Release Surface

### 1. Settlement Must Stay Primary And Proof-Native

- Wallet must lead with Settlement balance.
- Settlement balance must be USD-first, with phi equivalent shown beneath where phi context is exposed.
- Settlement must remain the proof-native liquid value lane.
- Price movement remains Settlement until funded.
- Future value surfaces must keep Settlement primary and proof-native.

### 2. Reserve Must Stay The Funded External-Conversion Lane

- Reserve funds notes, sends, positions, and wire transfers.
- Reserve is shown beneath Settlement as the funded external-conversion lane.
- Reserve proceeds issue notes.
- Reserve builds Settlement shares.
- Wire transfer must draw from Reserve.
- Future wire improvements must preserve funded causality.

### 3. Notes Must Stay Reserve-Only

- Notes must issue from Reserve only.
- Notes must not issue from unfunded Settlement.
- Note issuance must preserve the Reserve provenance that made the note fundable.

### 4. Sends And Buys Must Deploy Reserve First

- Sends deploy available funded Reserve first.
- Buys deploy available funded Reserve first.
- Settlement covers the remainder only after Reserve is deployed.
- Reserve-first behavior must preserve lane metadata through execution and receipt surfaces.

### 5. Market Sells Must Split Funded And Unfunded Value

- Market sells split into funded Reserve and unfunded Settlement.
- Funded Reserve may credit the seller as funded value.
- Unfunded Settlement remains Settlement until buyer funding exists.
- Product surfaces must not merge funded and unfunded sale proceeds into one misleading balance.

### 6. Certificates Bridge Settlement To Buyer-Funded Reserve

- Settlement shares and certificates are the bridge from Settlement to buyer-funded Reserve.
- Certificate issue locks value.
- Buyer-funded certificate sale credits the seller.
- Unfunded certificate sale state must not be presented as funded Reserve.
- Peer certificate liquidity remains buyer-funded or Receiz-funded.

### 7. Account-Lane Charts Must Use Account Truth

- Settlement and Reserve popover charts must use account-lane truth and account quote math.
- Chart surfaces must not reconstruct misleading spikes from partial receipt history.
- Market wallet panels and portfolio charts must use real phi conversion math through the wallet account quote rather than symbol-swapping USD into phi.

### 8. Release-Governed Version Surfaces Must Move Together

- Release-governed version surfaces must move together to `v64.0.0` across package metadata, shipped verifier artifacts, public release badges, current release documentation, and canonical release docs.
- Public-facing version truth must not drift between shipped HTML entrypoints, docs indexes, release artifacts, and repository metadata.
- This offline verifier repository must reflect that lockstep rule in `package.json`, `README.md`, `RELEASE_NOTES.md`, `CHANGELOG.md`, `docs/README.md`, `docs/governance/README.md`, `docs/FORMAT.md`, `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.

## Important Non-Changes

- No new value state was added.
- Notes issue from Reserve only.
- Wire transfer draws from Reserve.
- Certificate issue locks value; sale credits the seller only when funded.
- Peer certificate liquidity stays buyer-funded or Receiz-funded.
- Provider names remain implementation detail, not user-facing liquidity language.
- The offline verifier proof-format and producer-payload contract remain unchanged by this documentation cut.

## Allowed Evolution

Implementation details may change after `v64.0.0` only if they preserve the authority boundaries and locked value-loop behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v64.0.0` if it:

- moves value without preserving Reserve/Settlement metadata
- treats provider names as user-facing liquidity language
- lets notes issue from anything other than Reserve
- lets wire transfer draw from anything other than Reserve
- credits a certificate seller before buyer funding exists
- fails to split market sells into funded Reserve and unfunded Settlement
- lets sends or buys consume Settlement before available funded Reserve
- reconstructs Settlement or Reserve charts from partial receipt history when account-lane truth is available
- symbol-swaps USD into phi instead of using wallet account quote math
- changes release truth without updating the versioned release docs and aligned repository version surfaces
