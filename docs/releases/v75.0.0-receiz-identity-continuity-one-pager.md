# Receiz v75.0.0 Identity Continuity One-Pager

Release date: May 2, 2026
Status: Evidence artifact for `v75.0.0`

## Thesis

Receiz identity continuity is local-first: the device may use proof it already holds, and the network may append or strengthen that truth but may not downgrade it.

## Identity Rails

- Receiz ID is the everyday identity rail.
- PBI/passkey, magic-link, Identity Record, Identity Seal, and Receiz Key are backup, recovery, and interoperability rails.
- `/signin` remains Receiz ID pass-through.
- `/signin/receiz-id-fallback` owns old-browser backup sign-in with sanitized return paths.

## Account Binding

- Local account bindings are durable register truth.
- Binding sources include Receiz device, Receiz Key, PBI, email, and server-switch sources.
- Switching bindings projects the selected account.
- Switching bindings clears account-scoped continuity when the actor changes.
- A stale old-cookie echo does not outrank locally restored account authority.

## File Restore

- Identity Record, Identity Seal, and Receiz Key restore project verified file sessions directly.
- Verified file restore admits the restored account locally.
- Verified file restore seeds profile, wallet, and upgrade truth.
- Restored authority remains active while import, bind, wallet refresh, or remote reconfirmation is delayed.
- First-time offline Receiz Key restore requires verified portable state.
- Later restores may reuse admitted local key truth.
- Optional local PBI publication is a background append, not restore permission.

## Surface Expectations

- Account, managed profile, wallet, and upgrade surfaces must recognize locally restored account authority before weaker server echoes.
- Profile identity artwork and signed original downloads preserve deterministic profile truth, owner proof, purchase context, and PBI authorization.
- Identity artifacts append proof without replacing the base profile shell.
- Plans and Upgrade keep payment and old-browser fallback behavior in parity.

## Release Gate

Do not ship `v75.0.0` if any identity surface treats verified local proof as provisional, creates a competing identity for proof that verifies the existing root, or requires remote reconfirmation before locally restored account authority can remain visible.
