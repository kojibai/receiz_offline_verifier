# Receiz v98.0.0 Regression Lessons

## What Regressed

- Minor release documents split the post-`v97.0.0` work into smaller locks, but the final `98.0.0` release needed one complete review over the full commit range.
- Developer-runtime work increased the number of lower-authority rails that could be mistaken for truth: SDK helpers, MCP tools, OIDC agent tokens, webhook endpoints, public-store projections, and service worker/runtime labels.
- Public proof graph and market-cap work became visually and computationally rich enough that chart construction, ledger pagination, and viewport fit could threaten first paint or visible continuity if not bounded.
- Ledger and transaction surfaces risked language drift if wallet rows, transfer snapshots, or API responses were described as balances instead of settlement, transfer, proof-bundle, and Kai/proof projections.
- Sports arena and vault performance work risked moving known proof truth behind idle/background machinery if deferred surfaces, rollup workers, and active-job dedupe were not contract-guarded.
- Pitch Command archive catch-up created large generated proof archive changes that could be misread as static asset churn instead of proof archive material.

## Why It Regressed

- Release scope grew across SDK, MCP, Connect, webhooks, graph, ledger, sports, market/profile, and archive work faster than a single release-lock narrative was updated.
- Generic developer-platform convention would naturally frame SDK/MCP/OIDC/webhooks as authority rails, but Receiz law makes them mechanics beneath proof objects and verified appends.
- Generic chart/performance convention would optimize by reducing visible function or waiting for fetched state, but Receiz scale law requires preserving the visible primitive and reducing weaker work beneath it.
- Generic ledger convention would call value rows balances; Receiz settlement law requires naming settlement, transfer, reserve, certificate, note, proof-bundle, and Kai/proof state directly.
- Generic archive convention would treat generated gzip/projection files as static assets; Receiz proof law treats Pitch Command day proofs as proof archive material.

## Locked Rules

- `v98.0.0` is locked against `v97.0.0..HEAD`, covering 65 commits before release-lock edits.
- `@receiz/sdk` remains a convenience rail beneath proof objects, and durable proof memory remains first admission only, then append forever.
- SDK and MCP are convenience/agent rails beneath Receiz primitives; they are not truth authority.
- Delegated OIDC agent tokens authorize scoped agent/server mechanics; they do not replace Receiz ID, PBI, Identity Seal, Receiz Key, proof objects, ownership appends, settlement rows, or verified local truth.
- Developer webhooks sign delivery events beneath delegated scopes; endpoint state and delivery logs do not outrank proof objects or verified appends.
- Merchant-owned public-store state must remain publishable with Identity Seal / Receiz Key proof authority and no delegated bearer token.
- Public proof graph must remain a public proof surface rooted in real proof/economy/network data and live Kai pulse state.
- Ledger, wallet, transaction, and market-cap surfaces must preserve Kai/proof coordinates and settlement language.
- Sports vault and arena performance work must preserve known proof, media, ownership, closeout, and settlement truth before background work.
- Profile and market public projections are write-side obligations after deterministic read paths are protected from rediscovery.
- Pitch Command day-proof archives must remain strict proof archive material.
- Service worker/runtime release identity must match app release identity but remains below proof and verified local truth.

## Release-Range Lesson

The full release should not be described as an SDK release, a graph release, a sports performance release, or an archive release alone. It is a proof-native artifact-system release that added many lower-authority mechanics beneath existing primitives. The lock must preserve the primitives first and describe the mechanics second.

## Visual Evidence Lesson

The reviewed working set includes Playwright and Lighthouse evidence under `output/`, but fresh release-candidate browser evidence still matters. If `/proof-graph`, `/transactions/[slug]/[code]/[pulse]`, `/developers/connect`, `/developers/webhooks`, Sports Arena, `/market`, or live player evidence contradicts this lock, the visual evidence wins and the release is blocked.

## Release-Lock Requirement

`pnpm release:lock 98.0.0` must require this document for `v98.0.0` and later release cuts. A release that omits regression lessons is incomplete because post-`v97.0.0` scope consolidation, SDK/MCP/OIDC/webhook authority boundaries, proof graph performance, ledger/settlement language, Sports Arena first-paint pressure, Pitch Command archive classification, and visual-evidence law would not be explicit release truth.
