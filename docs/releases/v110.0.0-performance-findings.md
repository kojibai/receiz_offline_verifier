# Receiz v110.0.0 Performance Findings

`v110.0.0` composes v109 local verification into a unified admission and recovery rail. Exact enclosing artifact bytes are verified once before classification; `admitAndRecover` remains read-only, zero-network, and zero-write; mutation is isolated behind an explicit capability-bound atomic commit.

## Findings

- `artifacts.admit` exposes one typed boundary for native identity/proof objects, supported legacy artifacts, portable vault/proof objects, foreign ownership, bearer recovery, and invalid card-only payloads.
- Classification yields exactly one of `canonical-identity`, `bearer-recovery`, `verified-legacy-read`, `foreign-owner`, or `invalid`, with explicit action flags rather than inferred UI or session state.
- `artifacts.planRecovery` derives prior/next head, recovered state, parent-linked history, and bounded explanation deterministically without writes.
- `artifacts.admitAndRecover` composes admission and planning once, avoiding duplicate verification, network coordination, store initialization, capability reads, idempotency work, or hidden mutation.
- Standard proof history preserves parent links, Kai ordering, ownership transitions, terminal events, receipts, and unknown namespaces while rejecting missing parents, cycles, stale branches, mixed identities, receipt failures, divergence, and terminal reversal.
- `artifacts.commitRecovery` performs one caller-local atomic transaction for verification evidence, projected state, history, head, receipt, idempotency record, outbox intent, and audit reference—or changes nothing.
- Stable operation identity, expected-head binding, idempotency, and replay protection prevent duplicated or reordered SDK/MCP/AI recovery execution.
- Multi-application conformance covers reverse transport order, reconnect, unknown namespaces, transfer, restore, and deterministic convergence without promoting transport order into truth.
- Receiz.com behavior, APIs, database schema, migrations, queries, production data, and native Record -> Seal issuance remain unchanged; visible release identity advances without adding runtime application work.

## Performance Boundary

- Known verified artifact bytes and proof history project before session, server, registry, network, explanation, plan, or UI work.
- Bearer, legacy, foreign-owner, and invalid verdicts remain bounded; classification cannot escalate signing, publishing, settlement, identity, or ownership authority.
- Read-only admission/planning accepts no store, capability, idempotency key, or mutation callback. Commit accepts only SDK plan custody, verified capability, expected head, stable idempotency, and caller-local atomic storage.
- This repository validates static release surfaces and constitutional records; it does not recreate upstream package/runtime benchmarks or authorize signing, publication, deployment, production database work, production verification, or Git push.

## Release-Lock Requirement

`pnpm release:lock 110.0.0` must require this document for `v110.0.0` and later release cuts. Without it, single-pass read-only recovery, deterministic history, atomic commit, idempotency, replay protection, convergence, bearer bounds, and unchanged application/database cost would lack an explicit scale boundary.
