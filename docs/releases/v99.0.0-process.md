# Receiz v99.0.0 Release Process

This process locks the exact post-`v98.0.0` working set as `v99.0.0`. `@receiz/sdk` remains a developer convenience rail, and durable proof memory remains **first admission only, then append forever**.

## Goal

Promote the complete local-proof, identity-continuity, deterministic-player, Explore, and whole-Market public-register work without letting version mechanics, cache invalidation, session state, server publication, or DB projection outrank the implemented proof primitives.

## What Worked

- Starting from the annotated `v98.0.0` tag produced an exact 39-commit release range rather than a narrative based on the last version bump.
- Deterministic live-player first paint reused shipped and held Market/profile proof truth before network rediscovery.
- Whole-Market membership introduced a complete durable register and explicit tombstones, separating immutable proof history from current public projection.
- Receiz Key export expanded the existing portable proof primitive instead of inventing a new recovery authority.
- Local PBI authorship reused existing PBI action/receipt verification and added durable append/custody rails beneath that primitive.
- Service-worker runtime epochs remained separate from app release identity and durable verified truth.
- Direct contract failures exposed obsolete historical literals instead of hiding the mismatch behind a release label.

## What Did Not Work

- Interim `98.0.1`, `98.0.2`, and `98.0.3` app/runtime bumps did not move SDK, MCP, public labels, offline verifier labels, or active release documents together, so they were not clean full release locks.
- The release lock failed because the root version named `98.0.3` while the required `v98.0.3` record did not exist.
- The service-worker continuity contract still required the `sports-pack-account-session-v1` epoch after the implementation intentionally moved to `pbi-proof-persistence-v3`.
- A public-player revocation contract froze `APP_VERSION` to historical `98.0.0`, conflating “this repair must not independently bump a release” with “the service worker can never join a later package-aligned release.”
- A brief PBI signature-history change was reverted, then rebuilt around permanent append preservation and profile snapshot persistence; the release record must document that correction chain rather than treating the final commit alone as the history.

## Practical Rule For Future Work

1. Treat a version bump as valid only when app, SDK, MCP, runtime, service worker, public labels, verifier labels, and active release documents move together.
2. Keep runtime cache epochs independent from release identity and lower than durable verified truth.
3. Contracts should assert the invariant—package alignment, append retention, exact-route continuity—not freeze a historical release literal inside unrelated behavior tests.
4. Record correction chains, including reverts, in the release history.
5. Run the full release freeze from the exact candidate after the final evidence/checklist edit.
6. Publish `@receiz/sdk`, MCP, app deployment, and tag only after the signed attestation verifies.

## Candidate Procedure

```bash
node scripts/check_release_lock.mjs 99.0.0
pnpm test:local-truth-release-contracts
pnpm test:page-first-paint
pnpm test:deterministic-surfaces
pnpm test:release-freeze
pnpm release:attest
pnpm release:verify-attestation <path>
```

Fresh browser evidence must cover Market, Explore, Account/Receiz Key identity continuity, wallet, canonical profile, and deterministic live player. Any contradiction blocks promotion.
