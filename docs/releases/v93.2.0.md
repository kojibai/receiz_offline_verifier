## Title

Receiz `v93.2.0` - SDK Proof Developer Experience Release

## Summary

Receiz `v93.2.0` is the SDK Proof Developer Experience release.

This release keeps `@receiz/sdk` aligned with the app version after `@receiz/sdk@93.1.0` was published. The SDK now exposes the primitives developers need to build Receiz-native apps without rebuilding Receiz law themselves:

- runtime schema exports for asset manifests, Sports card manifests, and webhook events
- deterministic render-ready projections for proof objects and Sports card proof objects
- an append-only proof memory register with first-admission-then-append-forever behavior
- SDK docs that teach verify, project, admit, persist, and append as the default integration path

The SDK remains a convenience layer. It does not become proof authority. Sealed artifacts, proof bundles, verified appends, ownership appends, and settlement ledger rows remain the stronger sources of truth.

The release-lock edit itself moves the governed local public version surfaces from `93.1.0` to `93.2.0`: package metadata, static service-worker version, shipped offline verifier labels, release notes, changelog, agent law, docs indices, and release documents. Broader app runtime storage, public labels, `packages/receiz-sdk/package.json`, SDK exported version constants, SDK build output, npm publication, and package-registry state remain platform evidence boundaries for this offline verifier checkout.

## Product Lock

`v93.2.0` locks the SDK developer primitive:

```txt
verify proof object -> project deterministic rows -> admit into durable proof memory -> append verified additions
```

Developer apps should be able to render Receiz proof objects on first paint from admitted proof memory, then ask Receiz only for verified additions after their known head.

## Why This Matters

Receiz is easier for developers when the SDK carries the correct shape by default. A developer should not need to learn the hard way that proof memory is not stale cache, that duplicate transport must not drop known truth, or that a partial server response cannot outrank a full admitted proof object.

The SDK now gives them those rails directly.
