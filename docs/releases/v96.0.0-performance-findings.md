# Receiz v96.0.0 Performance Findings

Release date: 2026-06-29

`v96.0.0` keeps the `v95.0.0` public table performance law intact and extends it to developer tooling: executable SDK evidence must start from admitted proof truth, project known heads locally, and avoid weaker network or database rediscovery in local conformance.

## Findings

- The `receiz` CLI improves developer first contact by running inspection, conformance, and starter generation from local proof material instead of asking developers to assemble proof memory from docs alone.
- `receiz conformance` is performance evidence because it proves packaged fixtures can be validated and admitted with zero network calls and zero DB calls.
- `receiz inspect <file>` is performance evidence because it projects a proof payload and known Kai/proof head without turning CLI output into authority.
- `receiz init <dir>` is integration performance evidence because it starts from local durable proof memory instead of a rediscovery-first client pattern.
- `ReceizSdkObservation` is passive instrumentation. It can report admission and persistence timing, but it cannot gate first paint or alter proof admission.
- Copy-paste integrations preserve Receiz speed only when they render known verified truth first and ask for verified additions after the known head.

## Preserved Rules

- First paint must project known proof truth before hydration, analytics, SEO, session checks, lazy imports, global sync, server reconciliation, or SDK convenience work.
- Hidden work may append verified additions only.
- CLI and SDK evidence must not become a new conformance authority.
- Local SDK conformance must remain free of network, Supabase, service-role, and database dependencies.
- Package evidence must remain release-blocking when developer-facing primitives change.
