# Receiz v54.0.0 Product Truth

Release date: April 15, 2026
Status: Locked for `v54.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v54.0.0`. It carries forward the existing file-authoritative, deterministic-route, ledger-backed, and governed-market model from `v53.0.0` while recording the broader public product map, the dedicated Business front door, the admin command deck, Stripe-to-ledger wallet transparency, shared session bootstrap, deterministic market-preview first paint, world-twin runtime continuity, and Kai sigil SVG serialization stability as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, and visible version-surface alignment only. The verifier trust contract remains unchanged relative to `v53.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- The file remains the authority for artifact truth.
- Deterministic route state remains the authority for first paint and historical route truth.
- Ledger-backed settlement remains the authority for money state.
- Live session, market, and twin deltas may hydrate on top of deterministic baseline state, but they must not replace that baseline as the source of truth.
- Public product routes, admin surfaces, release metadata, and offline verifier entrypoints remain operating or governance surfaces around those authorities, not replacements for them.

## Locked Release Surface

### 1. The Public Product Map Must Stay Broad And Plain-English

- `/business` must remain a dedicated business-front-door narrative rather than a buried upgrade concept.
- `/everything-receiz-lets-you-do`, `/what-can-you-do-with-a-receiz`, `/what-is-a-receiz`, `/start-here`, `/public-twin`, and `/receiz-world` must continue to describe the product in layered, plain-English terms.
- Updated `/developers`, `/developers/receiz-connect`, `/faq`, `/powered-by`, claim-ownership surfaces, and generated system-brief collateral must continue to reflect the broader public product map.
- Public explanation must stay layered instead of collapsing the system back into one or two overloaded gateway pages.

### 2. Admin Must Stay An Operational Command Deck

- `/admin` must remain a command-deck shell rather than regressing into a single long scroll report.
- Operators must continue to have one-view access to payments, users, wallet operations, growth, integrity, connect, performance, and backfill or maintenance controls.
- Action-center style controls for backfill, compaction, rollup drain, and snapshot refresh must remain available where those operations exist.
- Per-user Stripe USD visibility and clearer payment-source visibility must remain part of the admin operating surface.

### 3. Money Operations Must Stay Explicit, Ledger-Backed, And Replayable

- Admin wallet adjustments must continue to record explicit ledger entries rather than silently mutating balances.
- Stripe-funded wallet deposits, payment-embed flows, and profile purchase flows must continue to land in wallet or ledger-backed paths that can be replayed and audited.
- Admin visibility into wallet-credit rollups, Stripe-funded USD deposits, and Stripe-derived ledger backfill state must remain available.
- Money state must remain explainable after the fact, not merely inspectable in the moment.

### 4. Signed-In Routes Must Reuse Shared Session Bootstrap

- Signed-in routes such as account, wallet, calendar, contacts, upgrade, plans, make, seal, affiliate, profile entry, embed scheduler, payments builder, analytics, and adjacent onboarding surfaces must continue to prefer one shared session/bootstrap graph when deterministic baseline state is already available.
- Route entry must continue to render from deterministic signed-in baseline state before reconciling live updates where possible.
- Cold identity refetch behavior such as mount-time `/api/session` probing must not re-expand across these surfaces as the default route-entry model.

### 5. Market Preview Must Stay Deterministic Before Live Hydration

- Profile, artifact viewer, live player, and related market surfaces must continue to share deterministic `marketPreview` state.
- Profile originals and route snapshots must continue to carry deterministic market-preview value needed for first paint.
- Viewer and live-player hydration guards must continue to reject stale snapshots that are missing required deterministic market-preview state.
- Price pills must continue to render deterministic value immediately and then hydrate live afterward instead of waiting for a click to wake the market graph.

### 6. World Twin Continuity Must Stay A Runtime Contract

- World twin continuity must continue across profile, artifact viewer, and live player transitions through session reuse and cached runtime reuse.
- Deferred twin loading must continue to reuse already loaded rig or module runtime across route remounts where applicable.
- Artifact routes must remain on the same deferred twin continuity path as profile and live-player surfaces rather than bypassing continuity logic.
- The dock must not leak the square render-image fallback before the rigged twin is ready.

### 7. Interaction Discipline And Hydration Stability Must Stay Deterministic

- Haptics policy/runtime support may expand, but interaction behavior must remain disciplined and deterministic.
- Touch-safe market and profile controls must remain improved rather than regressing into accidental or unstable interaction paths.
- The removed bottom-bar swipe experiment path must not reappear as a half-active behavior.
- Kai sigil SVG builders must continue to emit serialization-stable path data so the server and client produce identical deterministic sigils without hydration mismatch caused by floating-point drift.

### 8. Release Governance And Version Truth

- Release-scoped note and product-truth documents are required for `v54.0.0`.
- Release-governed version surfaces must move together across current-release docs, verifier-contract docs, and shipped offline HTML entrypoints.
- This repository records the public release documents and shipped HTML version markers for that lockstep requirement.

## Allowed Evolution

Implementation details may change after `v54.0.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v54.0.0` if it:

- collapses the broader public product explanation back into a narrow or jargon-heavy front door
- regresses admin from a command deck into a long mixed audit report without one-view operating access
- reintroduces silent wallet-balance mutation or hides Stripe-funded wallet state outside explicit ledger-backed behavior
- re-expands cold mount-time identity probing across signed-in routes that already have deterministic bootstrap state
- removes deterministic market-preview first paint or makes price pills wait on live graph wake-up before rendering value
- breaks world-twin continuity across profile, artifact viewer, and live player or leaks the square fallback before the rigged twin is ready
- reintroduces hydration drift in Kai sigil SVG output or otherwise weakens deterministic UI serialization
- changes release truth without updating the versioned release docs and aligned version surfaces
