# Receiz v47.8.0 Product Truth

Release date: April 10, 2026
Status: Locked for `v47.8.0` release prep

## Purpose

This document defines the patch-level product-truth freeze for Receiz `v47.8.0`. It preserves the current `v47.x` product truth while recording guest-contact materialization, public-twin continuity, deterministic action-ledger projection into Contacts CRM, mobile-visible identity context, and guest-thread follow-up continuity as explicit non-regression behavior at release time.

For this offline verifier repository, the patch is represented as current-release documentation and version-surface alignment only. The verifier trust contract remains unchanged relative to `v47.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- The file remains the authority for artifact truth.
- Deterministic manifests remain the authority for historical route truth.
- World action-ledger events, public-twin continuity keys, and calendar projections must materialize into durable contact records without becoming separate competing sources of person truth.
- Signed-in account state and server-truth persistence remain the authority for contact identities, continuity keys, notes, stages, invite state, timeline history, and direct-handoff enrichment.
- Mobile contact snapshots, twin follow-up surfaces, offline verifier entrypoints, and release metadata remain transport and governance around those boundaries, not replacements for them.

## Locked Patch Surface

### 1. Guest Visitors Must Remain First-Class Contacts

- Guest visitors must materialize into durable contact records instead of remaining receipt-only or calendar-only artifacts.
- Saved `visitorKey` and `threadKey` continuity must remain attached to the same relationship record.
- Later authenticated identity enrichment must upgrade the existing record instead of fragmenting history into duplicates.

### 2. Twin, Calendar, And Contact Continuity Must Remain Deterministic

- World action-ledger entries must materialize deduped contact events with deterministic event keys.
- Calendar projections must continue to attach meeting events to the same contact record.
- Contact history must remain reconcilable from stored continuity identifiers rather than operator guesswork.

### 3. Mobile Contact Handling Must Stay Legible

- Mobile contact snapshots must show whether the record is guest, authenticated, or direct.
- Visitor and thread continuity keys must remain searchable and inspectable when needed.
- Follow-up actions must remain available from the same contact surface.

### 4. Follow-Up Continuity Must Survive Guest-Only Lanes

- Message and call follow-up must reopen the saved public twin thread when a direct chat account, email, or phone lane is unavailable.
- Direct messaging, email, phone, and twin-thread handoff must operate as alternate lanes on the same relationship record rather than separate contact identities.

### 5. Release Governance And Version Truth

- Release-scoped note and product-truth documents are required for the patch.
- Release-governed version surfaces must move together across package metadata, service worker defaults, public badges, offline verifier surfaces, and release documents.
- This repository records the public release documents and shipped HTML version markers for that lockstep requirement.

## Allowed Evolution

Implementation details may change after `v47.8.0` only if they preserve the authority boundaries and locked patch behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v47.8.0` if it:

- stops guest visitors from materializing into durable contacts or strips saved visitor/thread continuity keys
- forks authenticated upgrades into duplicate records instead of enriching the same relationship record
- breaks deterministic event projection or calendar continuity across twin, world, and contacts surfaces
- hides continuity identity context or removes searchability/inspectability of visitor/thread keys on mobile contact surfaces
- loses guest-only follow-up continuity when no direct lane is available
- changes release truth without updating the versioned release docs and aligned version surfaces
