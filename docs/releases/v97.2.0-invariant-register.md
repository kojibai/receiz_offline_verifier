# Receiz v97.2.0 Invariant Register

Release date: 2026-06-30

`v97.2.0` carries forward the proof-native artifact system and public app-state projection invariants from `v97.0.0` and adds the enterprise app runtime SDK invariants needed for production developer apps.

## Locked Invariants

1. Receiz remains a proof-native artifact system. Product truth is carried by files, identities, vault objects, ownership states, settlement states, public witness surfaces, verification routes, Twin surfaces, World projections, public app-state projections, and verified appends.
2. Complete proof objects carry deterministic identity, sealed/verifiable payload, provenance, inspection behavior, and Kai/Klok state.
3. Kai Klok is the proof object state machine. `kaiPulseEternal` / `kai_upulse` is the pulse unit.
4. Durable proof memory is not cache. First admission creates known truth, and later work appends verified additions.
5. Public app-state projections are durable public projection/index rows for public app recovery. They must not become proof authority.
6. Enterprise app runtime helpers are SDK convenience beneath proof objects, identity primitives, ownership appends, settlement rows, durable proof memory, public projections, Kai/Klok state, and verified appends.
7. Customer session helpers and customer portal helpers are tenant-scoped access mechanics, not identity proof roots.
8. Merchant onboarding helpers compose identity, ownership, settlement, and public projection primitives without replacing them.
9. Commerce helpers for refunds, subscriptions, shipping, tax, discounts, gift cards, access passes, inventory, fulfillment, and payouts must preserve ownership and settlement authority.
10. Every new enterprise write helper accepts idempotency material.
11. Offline proof queues replay verified additions after known heads and must not duplicate side effects.
12. Audit logs, event replay, jobs, release pinning, search, notifications, media transforms, risk checks, compliance exports, and portability expose or recover proof-native state without replacing it.
13. `@receiz/sdk/react` hooks and hosted component factories remain UI/developer convenience beneath Receiz primitives.
14. Connect is delegated API access. It must not become the identity proof root or replace Receiz ID, Identity Record, Identity Seal, Receiz Key, PBI, or artifact proof.
15. SDK clients, React components, docs, CLI, OpenAPI, status catalog, server, database, SEO, analytics, session, and framework routing conventions remain weaker than admitted proof truth.
16. `@receiz/sdk` release version, app release version, SDK runtime constants, release documents, public labels, and service-worker/runtime version must not diverge before promotion.
17. The operating law remains first admission only, then append forever.

## Forbidden Regressions

- proof object to card
- artifact system to UI experience
- durable proof memory to cache
- public app-state projection to proof authority
- enterprise SDK helper to issuer, verifier, database authority, or settlement authority
- customer portal to identity proof root
- settlement primitive to wallet balance display
- React hosted component to proof authority
- offline proof queue to rediscovery-first loading
- idempotent replay to duplicate append behavior
- audit log to verified history replacement
- search surface to proof authority
- release pinning to service-worker cache folklore
- release evidence to optional package hygiene

## Release-Lock Requirement

`pnpm release:lock 97.2.0` must require this document for `v97.2.0` and later release cuts. A release that omits the invariant register is incomplete because proof-native system identity, complete proof object Kai/Klok state, durable proof memory, public app-state projection rail, enterprise app runtime SDK helpers, idempotent replay, offline proof queues, React subpath limits, Connect delegated access, version lock, and broader package evidence would not be explicit release truth.
