# Receiz v101.0.0 Invariant Register

`v101.0.0` carries forward the proof-native artifact system and locks Portable Asset Continuity: only a complete sealed proof object carrying payload, ownership/custody, provenance/appends, and settlement may cross the new SDK asset-admission boundary.

## Locked Invariants

1. Receiz remains a proof-native artifact system. Product truth is carried by files, identities, vault objects, ownership states, settlement states, public witness surfaces, verification routes, public proof surfaces, and append-only history.
2. Proof objects remain stronger truth than SDK helpers, MCP tools, OIDC tokens, webhook endpoints, server rows, service-worker state, public-store projections, app-state rows, charts, analytics, SEO, session state, or generic display state.
3. Kai Klok remains the proof object state machine. `kaiPulseEternal` / `kai_upulse` remains the pulse unit for complete proof objects.
4. Durable proof memory remains first admission only, then append forever. Hidden work may only append verified additions after the known head.
5. Deterministic first paint must project known proof, identity, ownership, settlement, profile, market, ledger, transaction, Sports, public graph, and archive truth before weaker work.
6. `@receiz/sdk` remains typed developer convenience beneath proof objects, verified appends, identity primitives, ownership appends, settlement rows, public proof surfaces, and offline verification.
7. `@receiz/mcp-server` remains an agent rail over SDK/API mechanics and must name its authority boundary.
8. A self-hashable or shape-valid manifest is not a complete proof object and cannot enter the proof/custody register as an asset.
9. SDK creation returns only after seal, local file reopen, existing-verifier pass, continuity validation, and authenticated admission succeed.
10. Server admission repeats existing artifact verification and binds continuity ownership to the authenticated Receiz identity.
11. Canonical payload bytes/hash, ownership/custody, provenance with append history, and settlement state travel together inside the artifact.
12. Database admission accepts only continuity-complete v2 requests; execution of the manifest-only v1 admission path remains revoked.
13. Historical valid artifacts and Receiz.com single-verdict verification behavior remain unchanged.
14. Offline verification remains primary; online verification mirrors it for automation and cannot become stronger authority.
15. Runtime cache epochs and payload schema versions remain separate from app/package release identity.
16. Static verifier, offline verifier, record-seal studio, settlement entrypoint, Sports card verifier, package metadata, release docs, and service-worker runtime version must move together.

## Release-Lock Requirement

`pnpm release:lock 101.0.0` must require this document for `v101.0.0` and later release cuts. A release that omits the invariant register is incomplete because sealed file authority, complete payload/ownership/provenance/settlement continuity, revoked manifest admission, historical verifier compatibility, offline primacy, and coordinated version lock would not be explicit release truth.
