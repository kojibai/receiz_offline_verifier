# Receiz v60.0.0 Product Truth

Release date: April 18, 2026
Status: Locked for `v60.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v60.0.0`. It carries forward the `v58.0.0` baseline while recording a live-product homepage, stronger canonical original upload/download authority, first-class twin portrait publishing, compact wallet and certificate presentation, production schema hardening, and release-surface lockstep as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, shipped HTML version markers, and package-metadata alignment only. The verifier trust contract remains unchanged relative to `v58.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- File-backed proof remains the authority for artifact truth.
- Canonical sealed originals remain the authority for downloadable original recovery when a legacy streamed path needs repair.
- The live twin render and the profile-ready portrait are separate truth objects with separate publishing intent.
- Deterministic market seed truth plus the live signed-in user overlay remain the authority for market first paint when that truth is already knowable.
- Wallet, custody, and certificate state remain durable user-visible truth, not decorative presentation hints.
- Public product routes, release metadata, and offline verifier entrypoints remain operating or governance surfaces around those authorities, not replacements for them.

## Locked Release Surface

### 1. The v58.0.0 Baseline Must Remain The Floor

- Release-governed version surfaces must remain lockstep and current-release docs must stay aligned.
- Authoritative signed-in market first paint, coherent wallet/chart-buy hydration, lean deterministic market transport, and correct live tracked-position semantics from `v58.0.0` remain in force.
- Market truth already known at navigation time must continue to win over weaker anonymous or late-arriving refresh state.
- The offline verifier trust contract must remain file-authoritative, deterministic, and fail-closed.

### 2. The Front Door Must Show Live Product

- The homepage hero proof module must load from real verified-player route truth rather than a static-only marketing shell.
- The homepage must present a live twin dock as part of the real front-door product experience.
- Direct market action must be exposed inside the live proof experience instead of being deferred behind detached framing.
- Proof, identity, embodiment, and market must remain visible together at the public boundary.

### 3. Original Uploads And Original Recovery Must Stay Canonical

- Owner profile uploads must continue to flow through the canonical `/seal` document path.
- Hash-verified streamed originals must remain trusted when the original bytes already match the expected canonical file truth.
- Legacy streamed originals must be able to reseal and backfill on demand when the canonical artifact needs to be restored.
- Expected code and pulse continuity must be checked before any fallback artifact is accepted.
- Any repaired or recovered original must be re-verified before it is returned.

### 4. Twin Portrait Publishing Must Remain Explicit

- World avatar generation must continue to render a distinct white-backed profile portrait alongside the live twin asset.
- The profile portrait must be able to mint its own Receiz original with its own verify path and verify URL.
- World forge output must keep the portrait distinct from the live twin render.
- Applying the profile portrait to the public profile picture must remain an explicit user-controlled action rather than an automatic side effect.

### 5. Market, Wallet, And Certificate Surfaces Must Stay Coherent On Real Devices

- The market buy flow must continue through passkey and Stripe funding fallback when wallet bootstrap truth is temporarily missing.
- Compact wallet presentation must keep the Receiz ID pill centered, legible, and coherent on real device widths.
- Certificate mobile presentation must remain tight and readable.
- Certificate framing must continue to use `CERTIFICATE CODE` language rather than weaker ambiguous footer labeling.

### 6. Production Hardening Must Include Policy And Index Boundaries

- `supabase/migrations/20260418001500_market_conformance_history_rls.sql` must remain part of the governed release delta.
- `supabase/migrations/20260418003000_auth_rls_initplan_policy_fix.sql` must remain part of the governed release delta.
- `supabase/migrations/20260418004500_high_value_fk_indexes.sql` must remain part of the governed release delta.
- Conformance history access policy, auth init-plan behavior, and higher-value relational joins must stay aligned with the stronger governed runtime boundary.

### 7. Release-Governed Version Surfaces Must Move Together

- Release-governed version surfaces must move together to `v60.0.0` across package metadata, service-worker defaults, public verifier artifacts, public release badges, and current release documentation.
- Public-facing version truth must not drift between shipped HTML entrypoints, docs indexes, release artifacts, and repository metadata.
- This offline verifier repository must reflect that lockstep rule in `package.json`, `README.md`, `RELEASE_NOTES.md`, `CHANGELOG.md`, `docs/README.md`, `docs/governance/README.md`, `docs/FORMAT.md`, `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.

### 8. Release Governance And Verifier Truth

- Release-scoped note, product-truth, and checklist documents are required for `v60.0.0`.
- Root package metadata must resolve to `v60.0.0`.
- Shipped verifier, studio, and settlement entrypoints must visibly display `v60.0.0`.
- The offline verifier trust contract remains unchanged relative to `v58.0.0`.

## Allowed Evolution

Implementation details may change after `v60.0.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v60.0.0` if it:

- regresses the homepage back to static framing that hides already-known live proof, twin, or direct market action
- lets owner uploads or original-download recovery bypass canonical sealing, continuity checks, or re-verification
- silently overwrites the public profile picture as part of twin generation instead of preserving explicit operator intent
- breaks compact-device wallet, certificate, or buy-continuation coherence in real money or custody flows
- drops the governed schema hardening that protects conformance history, auth init-plan policy, or high-value relational joins
- allows package metadata, public badges, shipped verifier artifacts, or current-release docs to advertise different release numbers
- changes release truth without updating the versioned release docs and aligned repository version surfaces
