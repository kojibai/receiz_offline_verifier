# Receiz v96.0.0

Release date: 2026-06-29

## Goal

Make `@receiz/sdk` unmistakably A+: not by adding a new Receiz authority, but by making the existing proof object, durable proof memory, local conformance, projection, and Connect convenience rails executable, copy-pasteable, and release-blocking.

## What Worked

- `v95.0.0` already made the canonical proof language precise: Kai Klok is the state machine and complete proof objects carry Kai.
- The SDK already had validators, projections, typed clients, identity helpers, webhook signatures, fixtures, and durable proof memory.
- Adding a CLI around those existing primitives produced a better developer experience without creating a new authority layer.
- Running local conformance through packaged fixtures proved that developer tooling can admit and project truth without network or DB permission.
- Passive observations gave developers timing/evidence hooks without touching proof admission or visible product behavior.
- Copy-paste examples made the correct order obvious: verify, project, admit, persist, ask for additions after knownHead.

## What Did Not Work

- Before this release, SDK package evidence was still partly manual: tests and build existed, but pack dry-run and built CLI conformance were not release-blocking.
- The SDK had strong primitives, but a lazy developer still had to compose several docs before building a complete local-first integration.
- Observability was implicit in tests and command output; there was no typed passive observation event for proof memory admission.

## Practical Rule For Future Work

When Receiz exposes a primitive to developers, ship all four layers together:

1. Typed API.
2. Copy-paste integration.
3. Local executable proof.
4. Release-blocking contract.

The operating model remains first admission only, then append forever. The SDK may help developers move faster, but it must never ask weaker systems whether already-admitted proof truth is true.
