## Release Framing

Title: Receiz `v87.0.0` - Offline Proof Links, QR Proof Transport, And Post-v86 Public Proof Lock

`v87.0.0` locks the post-v86 proof arc: offline sealed proof objects and offline moments produce compact self-contained `/v` proof links, `/v` verifies embedded Signature V4 proof bundles as `Proof`, and coordinate-only historical evidence remains lower authority.

## Preparation Steps

1. Preserve `v86.0.0` as the prior current-release archive.
   - Keep `docs/releases/v86.0.0.md`.
   - Keep `docs/releases/v86.0.0-product-truth.md`.
   - Keep `docs/releases/v86.0.0-checklist.md`.
   - Keep `docs/releases/v86.0.0-process.md`.
   - Preserve `v86.0.0` as historical release-law truth.

2. Publish canonical `v87.0.0` release artifacts.
   - Add `docs/releases/v87.0.0.md`.
   - Add `docs/releases/v87.0.0-product-truth.md`.
   - Add `docs/releases/v87.0.0-checklist.md`.
   - Add `docs/releases/v87.0.0-process.md`.

3. Align repository release law and product-truth references.
   - Update `AGENTS.md` release law to `v87.0.0`.
   - Update product-truth reference status markers carried forward for the current release.
   - Preserve the primitive decision procedure, stronger/weaker truth hierarchy, and forbidden downgrade boundary.

4. Align repository release surfaces.
   - Update `package.json` to `87.0.0`.
   - Update root `README` current-release references.
   - Update `RELEASE_NOTES.md` and `CHANGELOG.md`.
   - Update `docs/README.md`, `docs/governance/README.md`, and `docs/FORMAT.md`.
   - Update shipped version markers in `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.
   - Ensure static service-worker registrations request `sw.js?v=87.0.0`.
   - Ensure `site/sw.js` carries service-worker runtime version `87.0.0`.
   - Ensure local release-surface checks can execute through `pnpm lint`, `pnpm build`, and `pnpm release:lock`.

5. Validate primitive naming gates.
   - Offline proof is named as proof, not fallback mode.
   - Compact proof token is named as transport, not authority.
   - Sealed proof bundle is named as the authority source.
   - `/v` is named as a public proof surface.
   - Coordinate-only evidence is named as historical compatibility evidence.
   - Sports single-card QR/value surfaces are named as public proof surfaces.
   - World Estate is named as a World identity/public witness surface.
   - Public proof, identity, route, and service-worker truth retain direct primitive names.

6. Validate compact offline proof-link gates.
   - Offline sealing produces proof-bearing `/v` URLs.
   - Offline moments preserve proof-bearing `/v` URLs.
   - `/record` and `/make` keep proof-bearing URLs instead of reconstructing coordinate-only links.
   - `/v` decodes the compact proof token, reconstructs the signed `receiz.proof_bundle`, verifies Signature V4, checks route binding, and classifies valid pages as `Proof`.
   - Legacy `rpb1:` compressed JSON proof tokens remain decodable.
   - Coordinate-only evidence remains historical compatibility evidence.
   - Malformed or mismatched proof material is rejected or not elevated.

7. Validate post-v86 surface gates.
   - Live offline moment card QR points to the compact self-contained proof URL.
   - Sports single-card QR cockpit preserves vault card/export/link authority.
   - Sports value popovers remain readable and mobile-closeable after portal movement.
   - World Estate carries its own route surface while `/world` remains unchanged.
   - Signed-in identity inspector X closes the inspector instead of reopening identity state.
   - Market public profile asset loading preserves public proof objects under RLS constraints.
   - First-paint proof object import contracts remain strengthened.
   - Managed profile route contracts remain covered.
   - RLS/security cleanup and browser storage contracts are applied.

8. Validate version lockstep.
   - Confirm all governed version markers show `v87.0.0`.
   - Confirm all current-release docs and indices point to `v87.0.0` artifacts.
   - Confirm historical `v80.0.0` through `v86.0.0` artifacts remain archived release truth.
   - Run `pnpm release:lock`.
   - Run whitespace diff check.

9. Validate broader platform gates before production promotion.
   - Run release-freeze and page-first-paint checks.
   - Confirm compact offline proof-link, `/v`, Sports, Market, Profile, World Estate, security, and release contracts.
   - Confirm target production DB migrations named by the release are applied.

10. Generate signed release attestation.
    - Run `pnpm release:attest` for the final commit in the environment that holds the release attestation signing key.
    - Retain the signed attestation artifact under the governance release-attestation policy.
    - Do not tag or ship without the signed attestation.

## Focused Verification Matrix

Broader platform release validation should include the gates named in `docs/releases/v87.0.0-checklist.md`, including:

- direct primitive naming and no forbidden downgrade
- sealed proof bundle authority above compact URL transport
- Signature V4 and route-binding verification on `/v`
- coordinate-only historical fallback boundaries
- offline moment QR proof-bearing URL contracts
- Sports single-card QR/value public proof contracts
- Market/Profile first-paint and RLS contracts
- World Estate identity/public witness contracts
- service-worker version coherence
- release freeze, page first paint, TypeScript, release lock, whitespace diff, and release attestation gates

## Tag Criteria

Do not tag `v87.0.0` until:

- all required `v86.0.0` and `v87.0.0` release artifacts exist
- version surfaces are aligned to `v87.0.0`
- release gates in `docs/releases/v87.0.0-checklist.md` pass from the exact release candidate
- governed surfaces all show `v87.0.0`
- `pnpm lint` has zero warnings/errors
- `pnpm build` completes
- `pnpm release:lock` confirms the release surfaces
- whitespace diff check passes
- `pnpm release:attest` emits a signed attestation for the final commit

## Release Rule

Do not ship `v87.0.0` unless the release gates in this checklist pass from the exact release candidate, `pnpm release:lock` confirms the release surfaces, whitespace diff passes, compact offline proof-link and `/v` proof-token checks pass before production promotion, focused Sports, Market, Profile, World Estate, security, and release contracts pass, and production migrations named by the release are applied in the target environment.
