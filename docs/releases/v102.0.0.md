## Title

Receiz `v102.0.0` — Indivisible Verification And Continuity

## Summary

`v102.0.0` closes every known path where valid bytes, a signature, a manifest, a public proof lookup, or an integrity seal could be reported as a verified Receiz proof object without ownership continuity.

The shared verifier now returns top-level success only when the enclosing artifact proves integrity and carries a valid existing Receiz continuity carrier. That carrier binds a real profile-backed Receiz ID, deterministic artifact identity, the profile continuity namespace, and the prior-head/genesis relationship before sealing. Receiz.com, the downloadable offline verifiers, the SDK, Connect/API verification, admission, World packages, and MCP language use the same meaning of verified.

Native Receiz profile composition, Record, Seal, ownership provenance, and card continuity remain the authority. The SDK composes those existing semantics through `receiz.portable_asset.v1`; it does not invent an owner, namespace, issuer, or registry. Generic local sealing remains an explicit integrity seal and cannot return a verified proof-object verdict.

Across Receiz and `@receiz/sdk`, durable proof memory remains **first admission only, then append forever**.

## Breaking Boundary

- `@receiz/sdk` removes generic `sealArtifact` proof-object entry points; `assets.createProofObject(...)` uses the authenticated continuity preflight and secure seal route.
- `verification.verifyArtifact(...)` succeeds only for integrity plus ownership continuity.
- `@receiz/mcp-server` removes `receiz_verify_asset` and `receiz_verify_offline_file`; public-proof lookup is resolution, and local manifest checking is `receiz_inspect_offline_file` with `verified: false`.
- Generic `/api/document-seal` rejects portable proof-object payloads and remains integrity-only.
- Historical artifacts with integrity but no file-carried continuity return an integrity diagnostic and fail the proof-object verdict.
