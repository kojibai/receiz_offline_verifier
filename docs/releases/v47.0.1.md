# Receiz v47.0.1

Release date: April 5, 2026

## Overview

`v47.0.1` is a stability patch focused on the profile composer and profile-page loading discipline. It keeps the `v47.0.0` product truth intact while fixing the real browser path for profile video uploads and removing non-critical artwork head preloads that were generating unused-preload warnings.

For this offline verifier repository specifically, `v47.0.1` is a release-surface alignment patch. The verifier trust contract and runtime behavior remain unchanged relative to `v47.0.0`; this repository carries the current release note, product-truth freeze, and visible version-surface bump so the public release artifacts stay locked together.

The product-truth freeze for this release is defined in `docs/releases/v47.0.1-product-truth.md`.

## What Defines v47.0.1

- Profile composer video uploads keep the resumable storage handoff available even while the UI is reporting upload progress.
- Storage-backed sealed video handoffs resolve to a real download target instead of being falsely treated as failed uploads.
- Profile artwork warming is constrained to the visible showcase window, and only explicit high-priority images emit head preload links.
- Release-governed version surfaces stay locked together across package metadata, service worker defaults, public badges, and release documents.

## Targeted Patch Outcomes

### 1. Profile Composer Video Uploads Are Real Browser-Safe Again

The main patch fixes the exact path a signed-in owner uses on `/profile?compose=1`:

- original video upload keeps the intended Supabase-backed handoff path
- storage-confirmed seals are accepted as valid even when the response does not hand back a public URL immediately
- the composer no longer reports a false failure after the object is already present in authoritative storage

### 2. Profile Preload Noise Is Reduced Without Weakening the Page

The browser warnings about preloaded profile artwork were real. They signaled that some non-critical images were being head-preloaded and then not consumed quickly enough after load. `v47.0.1` limits warming to visible showcase items and reserves head preload links for truly high-priority images, which improves bandwidth discipline instead of degrading the page.

### 3. Release Truth Stayed Locked During the Fix

This patch also rolls the visible release surfaces and service worker version to `v47.0.1`, so stale cached clients do not present an older release badge while running a newer app build.

In this repository, that alignment shows up as updated release documentation plus the shipped version markers in the published verifier, studio, and settlement entrypoints.
