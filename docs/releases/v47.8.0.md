# Receiz v47.8.0

Release date: April 10, 2026

## Overview

`v47.8.0` is the contacts continuity hardening release. It keeps the durable CRM foundation from `v47.7.0`, then closes the public-twin and guest-contact gap so the same relationship record can carry twin exchanges, guest identifiers, direct messaging handoff, calendar continuity, and mobile-visible identity context without splitting the person across separate lanes.

For this offline verifier repository specifically, `v47.8.0` is a release-surface alignment patch. The guest-contact continuity, action-ledger projection, direct handoff, and mobile CRM behavior live in the broader Receiz application; this repository carries the canonical release note, product-truth freeze, and visible version-surface bump so the public release artifacts stay locked together.

The product-truth freeze for this release is defined in `docs/releases/v47.8.0-product-truth.md`.

## What Defines v47.8.0

- Public-twin and guest interactions now project into the same Contacts CRM from the action ledger instead of living only in world receipts or calendar projections.
- Guest visitors are materialized into durable contact records by their saved visitor and thread continuity keys, and authenticated visitors upgrade the same record rather than fragmenting into duplicates.
- The contacts mobile UI now exposes continuity identity context directly so an operator can see whether the record is guest, authenticated, or direct, and search against visitor/thread continuity keys when needed.
- Message and call follow-up now preserve continuity for guest-only twin visitors by reopening the saved public twin thread when a direct chat account, email, or phone lane is not available.
- Release-governed version surfaces move together to `v47.8.0` across package metadata, service worker defaults, public badges, offline verifier surfaces, and release docs.

## Targeted Release Outcomes

### 1. Guest Visitors Become First-Class Contacts

Contacts must no longer treat public-twin guests as second-class history:

- guest visitors are saved as contact records
- the same visitor and thread continuity keys stay attached to that record
- later authenticated identity can enrich the same person instead of forking history

### 2. Twin, Calendar, And Contact Continuity Stay Deterministic

The operator should not have to mentally reconcile separate systems:

- world action-ledger entries materialize contact events
- calendar projections continue to attach meeting events to the same contact
- the contact history remains deduped by deterministic event keys

### 3. Mobile Contact Handling Stays Legible

The mobile CRM view should explain who this person is:

- guest and authenticated continuity status is visible in the snapshot
- visitor/thread keys are searchable and inspectable when needed
- follow-up actions stay available from the same contact surface
