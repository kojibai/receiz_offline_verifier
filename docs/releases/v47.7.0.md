# Receiz v47.7.0

Release date: April 9, 2026

## Overview

`v47.7.0` is the durable contacts production release. It keeps the `v47.6.x` world-command and concierge operating model intact while turning `/contacts` into a migration-backed CRM with historical backfill, robust import/export, live per-person timelines, and server-truth persistence that survives refresh, sync, and operator handoff.

For this offline verifier repository specifically, `v47.7.0` is a release-surface alignment patch. The durable contacts schema, historical backfill, import/export portability, and write-through contacts API live in the broader Receiz application; this repository carries the canonical release note, product-truth freeze, and visible version-surface bump so the public release artifacts stay locked together.

The product-truth freeze for this release is defined in `docs/releases/v47.7.0-product-truth.md`.

## What Defines v47.7.0

- `/contacts` now persists into dedicated contact, identity, event, and import-run tables instead of behaving like a local-only shell.
- Historical message participants and historical original collectors are automatically materialized into the contacts CRM for signed-in users.
- CSV, JSON, vCard, phone-contact, and local-cache imports all dedupe into the same server-backed contact graph.
- Contact notes, stages, invite status, last-meeting state, and person-first timelines survive refresh because edits and logged interactions now write through the contacts API.
- Release-governed version surfaces stay locked together across package metadata, service worker defaults, public badges, offline verifier surfaces, and release documents.

## Targeted Release Outcomes

### 1. Contacts Become Durable Operator Infrastructure

The contacts app is no longer just a polished UI shell:

- contacts are persisted in durable tables with identity dedupe and event rollups
- person records are shaped around full-name, company, role, notes, next-step, stage, and invite state
- import runs are tracked so bulk hydration is observable and reproducible

### 2. Historical Relationship Memory Is Backfilled

Signed-in users should not start from zero:

- conversation participants are backfilled into contacts with message-history context
- original-collection history is backfilled into contacts as person-linked timeline events
- owner and twin interaction counts are rolled up from timeline events rather than guessed on the client

### 3. Contacts Actions Stay Live And Person-First

The one-viewpoint contacts surface now writes through to server truth:

- inline edits are patched back to the contact record
- invite launches and manual interaction logs append real timeline events
- refresh and periodic reload preserve the same contact state instead of collapsing back to local cache

### 4. Import And Export Stay Portable

Contacts remain portable rather than becoming trapped in one UI:

- CSV, JSON, and vCard import/export remain first-class flows
- browser-native phone contact picking is used where available
- unsupported devices still fall back to file-based import cleanly

## Deployment Notes

The release adds a durable contacts schema and expects the production database to be aligned:

- apply `supabase/migrations/20260409163000_contacts_crm.sql` before routing production traffic to the new contacts API
