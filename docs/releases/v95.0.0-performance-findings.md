# Receiz v95.0.0 Performance Findings

Release date: 2026-06-27

## Executive Finding

`v95.0.0` makes the public performance model precise: a complete Receiz proof object carries Kai/Klok state, and verified systems resume from the known proof/Kai head instead of rediscovering old truth.

This is not a cache optimization. Kai Klok is the proof object state machine. `kaiPulseEternal` / `kai_upulse` is the pulse unit that lets Receiz order proof truth without treating Chronos, server arrival time, SDK response order, or database scan order as authority.

## Findings

### 1. Public SDK Contracts Must Match Canonical Proof Bundles

SDK developers move faster when the public type matches the internal proof object contract. Optional or vague Kai/Klok fields invite integrators to build fallback ordering systems that later conflict with proof truth.

### 2. Kai Heads Bound Addition Discovery

Durable proof memory is fastest when known admitted truth paints first and sync asks for verified additions after the known proof/Kai head. Full-history rediscovery before render is both slower and weaker.

### 3. Chronos Stays Human-Facing

Chronos is useful for human display, import bridging, and debugging. It must not become the backend state machine for Kai-bearing proof objects.

### 4. Offline Proof Completeness Depends On The Living Coordinate

Offline proof works because the proof object carries its own verification coordinate. A complete offline-verifiable Receiz proof object cannot depend on server, DB, SDK, or API rediscovery before use.

### 5. Public Reports Are Release Surfaces

The public architecture report is a release surface. If it presents Kai as decorative, optional, or merely SDK-specific, it creates implementation risk even when code is stricter.

## Locked Performance Invariants

- Complete proof objects carry Kai/Klok.
- Kai Klok is the proof object state machine.
- `kaiPulseEternal` / `kai_upulse` is the pulse unit.
- Known verified truth paints first.
- Verified additions are requested after the known proof/Kai head.
- Chronos is display, bridge, import context, and debugging context.
- SDK, API, server, DB, report, and UI surfaces must not weaken canonical proof bundle truth.

## Release-Lock Requirement

`pnpm release:lock 95.0.0` must require this document for `v95.0.0` and later release cuts. A release that omits performance findings is incomplete because Kai/Klok state-machine precision, proof/Kai head append windows, Chronos limits, offline proof completeness, SDK contract parity, and public-report wording would not be explicit release truth.
