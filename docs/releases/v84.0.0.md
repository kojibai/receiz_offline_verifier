## Title

Receiz `v84.0.0` - Proof-Native Artifact System Stable Release

## Summary

`v84.0.0` is the release lock where Receiz names and protects what the implementation already carries: a proof-native artifact system.

Receiz is not a generic app with proof features. Receiz is a system where files, identities, vault objects, ownership states, settlement states, public witness surfaces, and verification routes carry product truth. The release locks that truth into repo law, public version surfaces, service-worker versioning, and release documentation.

The release candidate is current `main` at `d9490705d7` plus the `v84.0.0` release-lock worktree. The committed arc from tag `v83.0.0` to `d9490705d7` contains 17 commits across 48 files, with 3,122 insertions and 416 deletions before the final `v84.0.0` lock edits.

The outcome is direct:

```txt
stronger truth stays above weaker truth
deterministic first paint remains protected
proof objects are named as proof objects
vault objects remain artifact-system objects
ownership and settlement surfaces keep their primitive names
external systems provide mechanics only
service worker, package version, public badges, and release law move together
```

## What v83.0.0 Did

`v83.0.0` was the stable live-proof release after the `v82.0.0` Sports launch.

`v82.0.0` made Sports a real proof-backed live game economy: collect cards, pick a lineup, watch live sports, move on the board, seal proof, earn reward and value, and play again.

`v83.0.0` made that economy stable enough to hold:

- Sports first paint became deterministic.
- Main Event and single-game play stopped fighting over selected-game and lock state.
- Known score truth projected before hidden hydration.
- Legitimate zero scores stayed visible as score truth.
- Card reveal, reward reveal, saved card issuance, card memory, and market history stayed connected.
- Live-player sharing opened from iMessage again.
- SEO and structured proof enrichment moved behind first paint without moving canonical truth.
- Explore avoided white-screen races.
- Market conformance protected the fast `/market` entry path.
- Callback policy became centralized.
- OTC and Cash Image flows became offline-verifiable proof-object upgrade paths.
- Account, calendar, profile, and navigation surfaces stopped treating richer background state as permission to replace settled visible truth.

The `v83.0.0` law was:

```txt
paint known truth first
hydrate hidden data next
append verified additions only
never replace a settled shell to prove something already known
```

## v84.0.0 Lock Outcomes

- Primitive acceptance criteria are written into `AGENTS.md` and `docs/literal-product-law.md`.

- Release output must state changed surface, preserved primitive, risk boundary, checks run, and no downgrade.
- Public frontend surfaces display `v84.0.0`.
- `package.json`, service-worker registration, and `site/sw.js` carry `84.0.0`.
- Offline verifier pages carry the same release version.

## Product Statement

Receiz is a proof-native artifact system.

The vault is an artifact system.

Vault objects are proof objects when they satisfy deterministic identity, proof material, verification behavior, durable state, and a source of truth stronger than generic display state.

Ownership surfaces are ownership surfaces.

Settlement primitives are settlement primitives.

Identity primitives are identity primitives.

Public proof surfaces are public proof surfaces.

External systems may inform implementation mechanics. They may never define Receiz truth.

Generic convention is the weaker prior. Hedge weak priors before strong primitives.
