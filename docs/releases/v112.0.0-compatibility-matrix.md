# Receiz v112.0.0 Compatibility Matrix

**Status:** Frozen pre-implementation compatibility contract approved on 2026-07-18.

## Governing Rule

Historical sealed artifact bytes remain proof. Historical runtime authority objects do not cross a current release boundary.

V112 re-verifies and readmits exact artifact bytes under the v112 verifier, registry, digest domains, admission profiles, and runtime custody. It never rewrites historical provenance to manufacture current authority.

## Artifact Byte Compatibility

V112 must accept a v111 artifact without changing one byte when:

- its carrier remains supported;
- its Signature V4 and proof suite verify;
- its verifier context is explicitly allowed by v112 historical compatibility rules;
- its payload and continuity bindings recompute correctly;
- no compatibility fallback is used to create current authority.

Readmission produces new v112 runtime objects and v112 authority digests. It does not rewrite the v111 artifact, provenance, claim, timestamp, history, or registry evidence.

## Runtime Incompatibility

The following mechanisms must reject v111 runtime objects even when their visible shapes match v112 types:

- module-private custody sets or equivalent unforgeable runtime state;
- v112 registry digest checks;
- v112 authority-domain digest checks;
- v112 plan schema and plan digest;
- v112 capability schema and exact plan constraint;
- v112 verified-history evidence roots;
- v112 candidate transition digest;
- v112 commit-domain identity and binding;
- v112 staged-candidate digest and durable reference;
- v112 MCP attempt identity and lifecycle.

Object spread, JSON, `structuredClone`, prototype copying, copied symbols, cross-package imports, database reconstruction, RPC property bags, and MCP responses cannot upgrade a historical runtime object.

## Versioned Outcomes

V112 retains one unversioned current SDK entry point at package version `112.0.0`. It does not introduce `receiz.v111` or `receiz.v112` client namespaces.

Historical version identifiers appear only in compatibility, migration, provenance, fixture, registry, and release evidence. Current authority uses the v112 registry and package contract.

## Breaking Developer Changes

V112 intentionally breaks developer APIs where v111 types permit authority ambiguity:

- `DocumentVerifyResponse` becomes a strict verification result with `invalid`, `unsupported`, and `verified-artifact` states;
- admission inputs become profile-discriminated;
- generic `can*` booleans become primitive-specific assessments;
- `admittedActorId` becomes non-authoritative `actorConstraint` while planners require separate verified identity-admission evidence and plans bind its actor reference;
- authoritative recovery accepts only verified history;
- append requires an SDK-issued plan and a verified capability bound to that plan;
- public proof records become projection-only types;
- digest domains become typed and non-interchangeable;
- process-boundary readmission becomes explicit;
- application contracts declare operation authority matrices.
- append stores expose a canonical commit domain and durable candidate resolution;
- target namespaces evolve only through registered law/reducer versions while every non-target namespace remains byte-identical.

These changes do not alter existing Receiz.com end-user HTTP product behavior or production user workflows.

## Compatibility Acceptance Tests

The release is blocked unless tests prove:

1. A representative v111 native artifact readmits under v112 with byte identity preserved.
2. A representative v111 verified historical carrier receives the exact supported historical disposition.
3. A structurally detectable carrier or proof profile not implemented by v112 receives `unsupported` without being called valid.
4. A tampered supported carrier receives `invalid`.
5. Every v111 runtime authority object is rejected by every corresponding v112 authoritative operation.
6. V111 exact bytes sent through worker, file, RPC, and storage transports can be reverified.
7. V111 projections remain locators and cannot satisfy v112 artifact parameters.
8. V111 registry evidence verifies historically but cannot authorize a v112 plan or commit.
9. V111 capability and plan digests fail v112 domain and registry checks.
10. No compatibility path rewrites v111 provenance or silently grants current authority.
11. V111 receipts, store identifiers, candidate records, and accepted-head references cannot supply a v112 commit domain or accepted head.
12. A v112 result from commit domain A cannot satisfy domain B, even when actor, resource, key, and visible receipt shape match.
13. An uncommitted candidate remains a sealed proof object but cannot satisfy an accepted-head parameter.
14. Exact candidate bytes must be durably staged and resolvable by digest before v112 head acceptance.
15. Same-command replay returns the originally staged bytes; resubmitted bytes cannot replace them.
16. Idempotency and external effects do not alter the artifact transition digest; different retry keys for the same proposed successor produce the same transition digest.
17. A domain-A plan, capability, or command presented to domain B returns `COMMIT_DOMAIN_MISMATCH` before idempotency lookup.
18. Separately authorized operations in domains A and B may use the same raw caller key without collision.
19. Within one domain, reuse of the same scoped key with a different command returns `IDEMPOTENCY_CONFLICT`.
20. Commit independently resolves and reverifies staged bytes, and committed history is derived from the exact accepted artifact.

## No Downgrade

V112 compatibility preserves sealed artifact truth and historical proof. It rejects historical runtime authority rather than promoting server, database, session, package, projection, or reconstructed object state above exact artifact bytes.

## Release-Lock Requirement

`pnpm release:lock 112.0.0` must preserve this compatibility matrix with the release archive. Exact v111 bytes may be reverified and readmitted, but historical runtime authority, receipts, store references, projections, and process-boundary property bags must remain rejected by v112 operations.
