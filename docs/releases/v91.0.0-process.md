# Receiz v91.0.0

Release date: June 16, 2026
Status: Release prep

Release-lock references:

- `docs/releases/v91.0.0-regression-lessons.md`
- `docs/releases/v91.0.0-commit-history.md`

## Goal

Ship `v91.0.0` as the release that documents and locks the post-`v90.0.0` Sports scale arc:

```txt
document every commit since v90.0.0
make deterministic scale reasoning explicit
preserve the full visible value chart primitive
close stale Sports settlement and reward state
lock Pack Derby challenge settlement and deadline truth
promote settlement conformance into its own release gate
move package, service worker, runtime, public labels, offline verifier labels, and release docs together
```

The release goal is not to add a new story. It is to name what the repository now proves:

```txt
Receiz already has advanced deterministic primitives.
Scale comes from using them directly.
```

## What Worked

- The v90 product-truth docs gave an exact primitive model, so v91 could extend rather than re-argue Receiz identity.
- The post-v90 commit range is short enough to document mechanically and conceptually.
- The Pack Derby work kept game state tied to Sports reward/proof truth.
- Sports route contracts caught the exact challenge settlement and post-deadline receipt bugs.
- The value-loop guard and LiveChart contracts gave a way to restore full chart behavior without accepting a visual regression.
- The shared entry route made soccer/UFC claim/finalize behavior composable without inventing a new settlement authority.
- The release-lock script already enforced version-surface coherence, so v91 release docs can be checked mechanically.

## What Did Not Work

- The first chart optimization reduced the visible primitive instead of removing only the unnecessary work beneath it.
- Some stale Sports settlement paths treated old active rows as tolerable UI state instead of first-paint and continuity pressure.
- FIFA tournament reads returned `closeout: null` even though the settlement primitive had a closeout path ready to compose.
- Soccer/UFC entry routes lacked the same claim/finalize `PATCH` surface, and the client claim path still called baseball.
- Settlement conformance was not originally enforced as its own suite.
- Visual evidence could not be captured from the local dev server during the v91 documentation pass because the existing sports route timed out before DOMContentLoaded.

## Practical Rule For Future Work

For scale:

```txt
do not shrink the primitive
remove weaker work beneath it
```

For deterministic value:

```txt
if the state machine can compute it, do not fetch, store, poll, or accumulate it
```

For stale game/reward/tournament state:

```txt
final truth must close old active rows
opened rewards must not reappear as recovery pressure
sport-specific entry truth must use sport-specific routes
```

For conformance:

```txt
every primitive with release law needs its own mechanical gate
```

For release docs:

```txt
document the exact git range
document the exact lessons
document the exact product truth
name the exact uncertainty boundary
move version surfaces together
```

## Release-Lock Requirement

`pnpm release:lock 91.0.0` must require `docs/releases/v91.0.0-regression-lessons.md` and `docs/releases/v91.0.0-commit-history.md`. A `v91.0.0` release that omits either document is incomplete because the regression lessons and exact commit-history boundary would not be explicit release truth.
