Release date: 2026-06-29

`v96.0.0` preserves Receiz as a proof-native artifact system and locks `@receiz/sdk` as an executable developer integration surface beneath the same primitive law.

## Frozen Primitives

| Primitive | v96.0.0 truth | Must not regress |
| --- | --- | --- |
| Proof object | Complete Receiz proof objects carry deterministic identity, sealed/verifiable payload, provenance, inspection behavior, and Kai/Klok state. | Do not let the SDK, CLI, docs, or examples treat proof truth as generic app state. |
| Kai Klok | Kai Klok is the proof object state machine. `kaiPulseEternal` / `kai_upulse` is the pulse unit. | Do not use Chronos as the primary state machine for admitted proof objects. |
| Durable proof memory | The SDK exposes the same law Receiz uses internally: first admission only, then append forever. | Do not call admitted proof memory cache, stale data, hydration state, or rediscovery state. |
| SDK CLI | The `receiz` CLI inspects, validates, projects, and starts integrations from existing SDK primitives. | Do not let the CLI become a new authority, issuer, verifier, database confirmation, or server confirmation layer. |
| Local conformance | SDK local conformance validates packaged fixtures without network or DB authority. | Do not add network, Supabase, service-role, or database dependency to local SDK conformance. |
| Passive observation | `ReceizSdkObservation` reports operation evidence for proof memory admission and persistence. | Do not let observations change proof admission, leak proof payloads, or gate first paint. |
| Copy-paste integrations | Developer examples must start from proof objects, durable proof memory, and known heads. | Do not teach developers to rediscover known truth before projection. |
| Release evidence | SDK tests, typecheck, build, pack dry-run, CLI conformance, and package contents are release-blocking. | Do not release a version where the app, SDK package, and SDK runtime constant diverge. |

## Frozen Invariants

- Use known verified truth immediately.
- Ask only for verified additions after the known proof/Kai head.
- SDK convenience stays beneath proof objects, verified appends, ownership appends, settlement rows, and identity artifacts.
- Local SDK conformance must report zero network calls and zero DB calls.
- Developer examples must not import Supabase or service-role authority.
- Passive observation must not alter proof admission or user experience.
- A partial later payload may fill missing fields, but it must not downgrade stronger admitted truth.
- `@receiz/sdk` release version must match the app release version.

## Release Gates

Required repo-local checks for this release:

```bash
pnpm release:lock 96.0.0
node scripts/check_release_lock.mjs 96.0.0
git diff --check
```

Broader platform confidence requires:

```bash
pnpm test:sdk-release-contracts
npm test --prefix packages/receiz-sdk
npm run build --prefix packages/receiz-sdk
pnpm test:release-freeze
```

## Release-Lock Requirement

`pnpm release:lock 96.0.0` must require this document for `v96.0.0` and later release cuts. A release that omits the product-truth freeze is incomplete because proof object authority, Kai/Klok state-machine doctrine, durable proof memory law, SDK CLI boundaries, local conformance limits, passive observation, copy-paste integration order, and release evidence cannot remain explicit release truth.

## Residual Risks

- npm publication is an external registry action and must be run after local and broader platform gates pass.
- Production DB and Vercel health remain runtime evidence after deploy, but they do not become proof authority.
- Full `pnpm test:release-freeze` can be long-running; any skipped umbrella gate must be named before release.
