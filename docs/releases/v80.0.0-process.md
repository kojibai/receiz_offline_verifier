## Release Framing

Title: Receiz `v80.0.0` - Major Stable Release

`v80.0.0` locks Receiz as a stable proof-native operating platform. Local identity proof, file restore, profile truth, market/wallet state, card ownership, vault export, creator production, public proof, conformance, and runtime continuity all follow the same law: known verified truth moves first, network paths append verified additions, and visible surfaces do not visibly rediscover what the system already knows.

## Preparation Steps

1. Preserve `v78.0.0` as the prior current-release archive.
   - Keep `docs/releases/v78.0.0.md`.
   - Keep `docs/releases/v78.0.0-product-truth.md`.
   - Keep `docs/releases/v78.0.0-checklist.md`.
   - Keep `docs/releases/v78.0.0-process.md`.
   - Preserve `v78.0.0` as historical release truth.

2. Publish canonical `v80.0.0` release artifacts.
   - Add `docs/releases/v80.0.0.md`.
   - Add `docs/releases/v80.0.0-product-truth.md`.
   - Add `docs/releases/v80.0.0-checklist.md`.
   - Add `docs/releases/v80.0.0-process.md`.

3. Align repository release surfaces.
   - Update `package.json` to `80.0.0`.
   - Update root `README` current-release references.
   - Update `RELEASE_NOTES.md` and `CHANGELOG.md`.
   - Update `docs/README.md`, `docs/governance/README.md`, and `docs/FORMAT.md`.
   - Update product-truth reference status markers carried forward for the current release.
   - Update shipped version markers in `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.
   - Add or update local release-surface checks so `pnpm lint` and `pnpm build` can execute for this static repository.

4. Validate verified-truth-first gates.
   - Accepted identity proof projects the correct account immediately.
   - Background import, bind, mirror, sync, refresh, and enrichment cannot make accepted proof look provisional.
   - Weaker stale echoes cannot take away account, profile, wallet, market, card, vault, or creator truth.

5. Validate value, card, vault, and creator gates.
   - Settlement, Reserve, market, wallet, cards, auctions, packs, transfers, and battle stakes preserve funded causality.
   - Signal Cards remain owned Receiz assets across game, world, vault, discovery, transfer, auction, battle, market, export, share, and conformance surfaces.
   - Vault export, download, transfer, and restore remain explicit user-intent flows.
   - Creator OS and World agent loops produce durable artifacts and charge or hold wallet value explicitly.

6. Validate public proof, conformance, and runtime gates.
   - Public Proof, Signal Circuit conformance, Status, and Exchange publish public-safe inspection without exposing private/admin/worker paths.
   - Runtime, service-worker, PWA, stale header, stale chunk, and stale session recovery do not create visible second-load behavior.
   - Game-front, card pages, popovers, battle queues, and card-stage controls remain mobile-fit and official.

7. Validate release-surface lockstep.
   - Confirm all governed version markers show `v80.0.0`.
   - Confirm all current-release docs and indices point to `v80.0.0` artifacts.
   - Confirm `v78.0.0` artifacts remain historical and untouched.

## Focused Verification Matrix

Broader platform release validation should include the gates named in `docs/releases/v80.0.0-checklist.md`, including:

- local-proof identity, account binding, file restore, and stale echo precedence
- canonical profile and deterministic identity truth
- wallet, Settlement, Reserve, market, cards, auctions, packs, transfers, and battle stakes
- World agent, revenue, Sales Brain, campaign, and public-safe trace flows
- Public Proof registry, rights, identity, source, and status inventory flows
- Creator OS durable production, wallet-backed AI work, managed launch, and worker authorization
- Signal Card World, Vault Card export/restore, transfer packages, Arena appends, card flip gestures, and mobile/PWA download intent
- public `/signal-circuit`, `/exchange`, conformance, Status, and public/private route exclusions
- runtime, service-worker, PWA header, stale chunk/session, and release-worker marker coherence
- `pnpm lint`
- `pnpm build`
- release-surface lock checks where available

## Tag Criteria

Do not tag `v80.0.0` until:

- all required `v78.0.0` and `v80.0.0` release artifacts exist
- version surfaces are aligned to `v80.0.0`
- release gates in `docs/releases/v80.0.0-checklist.md` pass from the exact release candidate
- governed surfaces all show `v80.0.0`
- `pnpm lint` has zero warnings/errors
- `pnpm build` completes

## Release Rule

Do not ship `v80.0.0` unless the release gates in this checklist pass from the exact release candidate, `pnpm lint` has zero warnings/errors, `pnpm build` completes, and the governed surfaces all show `v80.0.0`.
