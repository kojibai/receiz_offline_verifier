# Receiz v30.0.0

Release date: March 16, 2026

## Overview

`v30.0.0` is the public release standard for the modern Receiz platform.

This release formalizes Receiz as a file-first trust system with public conformance, canonical settlement surfaces, operational payment-proof tooling, embedded builder rails, stronger identity and auth controls, and significantly deeper public proof delivery across profile, wallet, and verification routes.

The core rule remains unchanged: the file is the authority. Record, seal, share, and verify remain deterministic, and public `/v` routes continue to exist as convenience context for the same file, not as a replacement for the file itself.

For this offline verifier repository specifically, `v30.0.0` is primarily a release and documentation expansion. The verifier trust contract itself remains aligned with `v29.0.0`.

## What Defines v30.0.0

- Public conformance is now a product surface, not just an internal check.
- Settlement is now expressed as a complete standard surface, not a partial supporting page.
- Builder rails now include login, payments, calendar, Connect, and public badge/embed surfaces.
- Wallet, WireProof, AssetSeal, and profile delivery have matured into operational products.
- Governance, release attestation, and evidence verification are now explicit release controls.

## Major Accomplishments Since v27

### 1. Conformance Became Public and Verifiable

Since `v27`, Receiz introduced a complete public conformance layer:

- Conformance Center
- Settlement Conformance
- Verification Conformance
- Identity Conformance
- Issuance Conformance
- Interoperability Conformance

Each suite now includes live dashboards, badge endpoints, signed history feeds, rollups, and raw JSON artifacts. This turns standards from static claims into continuously inspectable runtime surfaces.

This is one of the most important product shifts in the platform. Receiz no longer asks users or integrators to trust documentation alone; it publishes live, inspectable conformance evidence.

### 2. The Settlement Standard Became a Full Canonical Surface

The economy and settlement stack expanded materially after `v27`:

- Settlement conformance routes and badges
- Value Source refinements
- Settlement ledger surfaces
- Transfers surfaces
- Sealed money surfaces
- Offline settlement notes
- Stronger compliance and conformance linking from legal and standards routes

This moves settlement from a conceptual standard into a complete, navigable, operational standard with public proof, runtime checks, and implementation evidence.

### 3. Profile, Artifact, and Proof Delivery Reached a New Level

Public profile and proof routes were expanded across:

- richer profile route hydration
- viewer/bootstrap APIs
- verified artifact route snapshots
- verified player experiences
- live player routes
- artifact contract and proof manifest support
- improved original artifact delivery and checkout flows
- profile analytics surfaces

This means Receiz is no longer just creating proofs. It is now delivering proof-bearing artifacts through stronger public routes, stronger player experiences, and stronger downstream verification context.

### 4. Builder and Embed Infrastructure Expanded Significantly

Since `v27`, builder-facing surfaces broadened into a multi-surface platform:

- Receiz Connect payment, transfer, record, and seal rails
- login embed builder
- payments embed builder
- calendar embed builder and scheduler
- developer communications surfaces
- Receiz Connect documentation improvements
- public badge/embed support for conformance distribution

This matters because Receiz is now easier to integrate into external products without requiring a full custom implementation from scratch.

### 5. Wallet, Payment-Proof, and Operational Claim Surfaces Matured

Wallet and payment-proof functionality expanded across:

- wallet note mint, claim, download, verify, and status routes
- public wallet ledger surfaces
- wallet wire request APIs
- payout profile and payout account surfaces
- WireProof dashboard expansion
- WireProof org archive support
- payment embed flows
- payment proof route improvements

This is the operationalization of the Receiz payment-proof model. The platform now supports a much broader set of post-issuance actions, proof views, claim states, and operational handoff flows.

### 6. Identity, Sign-In, and OIDC Security Grew Up

Identity and auth work since `v27` includes:

- Receiz Key auth surfaces
- identity-key enrollment, challenge, import, export, and finish routes
- passkey flow hardening
- OIDC client and token route expansion
- OIDC guardrails and security operations helpers
- consent surface improvements
- admin security controls for OIDC clients

This gives the platform much stronger footing for external identity integrations, higher-assurance auth, and institutional security posture.

### 7. Live Communication and Media Capabilities Expanded

Receiz added deeper real-time communication infrastructure:

- call panels and dock host improvements
- chat call lifecycle routes
- ICE and SFU session support
- transcript, notes, moderation, export, and participant APIs
- topology policy controls
- audio and video search APIs
- upgraded verified-player experiences

This is a substantial increase in platform depth. Receiz is no longer limited to static proof issuance; it now supports richer live and collaborative flows around verifiable artifacts.

### 8. Calendar, Scheduling, and Public Time-Bound Delivery Surfaces Landed

A full calendar surface was added with:

- calendar studio
- public calendar pages
- calendar embed pages
- event APIs
- embed scheduler tooling

This extends Receiz from file verification into scheduled and public-facing time coordination surfaces, while keeping the same proof-oriented architecture.

### 9. Governance and Release Integrity Became First-Class

Receiz now ships with explicit governance release controls and artifact integrity controls, including:

- governance checks
- governance evidence checks
- release attestation generation
- release attestation verification
- governance artifact signature verification
- documented release approval and signing requirements

This is not a cosmetic process. It is a direct step toward making the release itself part of the trust model.

### 10. Product Surface and UX Quality Improved Across the Platform

Major page and shell improvements since `v27` include:

- stronger global auth chrome and footer structure
- bottom-bar route gating and swipe navigation
- home and trust presentation refinements
- improved `what is a receiz`, `how do I receiz`, `what can you do with a receiz`, FAQ, and Start Here flows
- dashboard loading and route polish
- service worker registration and PWA behavior updates
- performance observation hooks

The result is a more coherent public product, not just a larger one.

## By The Numbers

Measured from the `v27.0.0` platform baseline on March 8, 2026 to the `v30.0.0` release snapshot on March 16, 2026:

- `411` files changed
- `94,843` insertions
- `6,268` deletions
- `5` major public conformance suites added
- `1` public Conformance Center added
- `20` conformance API surfaces added across suite routes, badge routes, history routes, and rollup routes

## Public Release Framing

`v30.0.0` should be understood as the release where Receiz crossed from a powerful proof product into a standards-governed platform.

The major themes of the release are:

- proof in the file
- public conformance
- canonical settlement
- operational payment proof
- embedded builder distribution
- stronger identity and release integrity

## Suggested Public Summary

Receiz `v30.0.0` turns the platform into a fully public, standards-governed proof system. Since `v27`, we have added live conformance across settlement, verification, identity, issuance, and interoperability; expanded the settlement standard into a full operational surface; strengthened wallet, WireProof, and profile delivery; launched broader builder rails across Connect, payments, login, and calendar embeds; and formalized governance and release attestation as part of the product trust model itself.
