# Receiz v47.0.0 Product Truth

Release date: April 5, 2026
Status: Locked for `v47.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v47.0.0`. It records which post-`v46.0.0` behaviors are now treated as explicit non-regression product truth at release time.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- The file remains the authority for artifact truth.
- Deterministic manifests remain the authority for historical route truth.
- Public readers, verified-player surfaces, storage proxies, world embodiment, and release metadata remain transport and governance around those boundaries, not replacements for them.

## Locked Product Surface

### 1. Original and Document Delivery

Original-file and document behavior is now treated as a first-class product surface.

- Documents must classify deliberately and route into truthful reader modes.
- Inline document reading must stay openable instead of collapsing back into attachment-only fallback behavior.
- Download precedence between original, derivative, and related artifact forms must remain explicit.
- Verified-original delivery and profile-original delivery must stay coordinated.
- Large-media storage-object delivery behavior must remain explicit and governed.

### 2. Verified Player and Route Continuity

The verified-player route is now a release-governed destination.

- Route hydration and continuity between profile, artifact, and live-player paths must remain deterministic.
- Live-player share metadata and artwork behavior must stay deliberate.
- Queue handling and playback handoffs must remain coherent.
- Profile-to-player and player-to-artifact navigation contracts are part of the locked product surface.
- Paid-download and ownership-aware viewer behavior must remain explicit rather than incidental.

### 3. Storage, Streaming, and Client Freshness

Delivery correctness is part of product truth, not an implementation detail.

- Signed storage-object routes must preserve `HEAD` and byte-range semantics.
- Profile original delivery contracts must remain explicitly guarded.
- Service-worker freshness must stay tied to release-version truth.
- Stale asset replay across releases must remain a release-governed risk, not an ignored browser quirk.
- Upload and storage paths must stay aligned with the known-good original bucket contract.

### 4. Public Twin and World Authority

Public Twin and world presentation remain constrained by explicit authority and freshness rules.

- Render-locked preview freshness must remain authoritative.
- Photoreal shell fit and face-feature alignment may use fallback paths, but those authority rules must stay explicit.
- Public world/profile payloads must remain sanitized.
- Stage and dock rendering must stay separated.
- Public live-video presentation must not fake or degrade delivery that is not backed by a real transport path.

### 5. Release Governance and Version Truth

The release contract itself is now more explicit and more machine-checkable.

- A release-scoped product-truth document is required.
- A canonical product-surface inventory contract is required.
- A release-freeze gate grouping major local blockers is required.
- Release attestation must remain tied to versioned docs and signing requirements.
- Version badges, package version, release docs, and service-worker defaults must move together.

## Allowed Evolution

Implementation details may change after `v47.0.0` only if they preserve the authority boundaries and locked behaviors above. If a future change materially alters one of these product truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v47.0.0` if it:

- weakens file or deterministic-manifest authority boundaries
- turns deliberate original/document delivery back into attachment passthrough
- breaks verified-player route continuity, ownership-aware behavior, or public share framing
- relaxes storage, byte-range, `HEAD`, or release-freshness correctness expectations
- permits Public Twin/world presentation to lose authority, freshness, sanitization, or stage-separation guarantees
- changes release truth without updating the versioned freeze and release-governance layer
