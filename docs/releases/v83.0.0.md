## Title

Receiz `v83.0.0` - Stable Live Proof Release

## Summary

`v83.0.0` is the release where the Sports Arena, profile live-player route, Explore, Market, OTC proof objects, and conformance system are hardened into one stable public build.

The release base before the final `v83.0.0` lock edits is `223b2ddbbd`, 123 commits after tag `v82.0.0`. That pre-lock arc from `v82.0.0` to `82.9.4` spans 188 files, 28,716 insertions, and 3,485 deletions.

`v82.0.0` made Sports a real proof-backed live game economy.

`v83.0.0` makes that economy feel stable, instant, and honest in daily use:

```txt
first paint shows the truthful shell
known game state is used before hydration
live score and lock state stay scoped to the selected game
cards keep issue, proof, career, and market history
conformance catches drift before release
```

The outcome is a release that can be held for a while: Sports is faster and less ambiguous, live-player sharing works in iMessage again, Market keeps its fast entry path, Explore avoids white-screen races, and release conformance now protects the exact laws that made the product feel settled.

## The Full Arc From v82.0.0

### 82.0.0: Sports Became The Daily Game

`v82.0.0` shipped the Sports Arena as a first-class product: MLB games, owned cards, pack issuance, rarity odds, Pick 5, Main Event, per-game play, live scoring, settlement, rewards, market value, trades, vault history, and Sports conformance.

That release established the core loop:

```txt
collect cards
pick a lineup
watch live sports
move on the board
seal proof
earn reward and value
play again
```

`v83.0.0` does not replace that loop. It makes the loop reliable enough to be the stable baseline.

### Sports First Paint Became Deterministic

The largest product law of this release is simple:

```txt
Sports must feel already there.
```

The Arena no longer treats first paint as a place for avoidable checking, shell swapping, or vague "locked" state. Main Event and single-game paths now preserve the fast Sports shell, project known entry and score truth immediately, and hydrate hidden data behind the visible stage.

Specific outcomes:

- Main Event and single-game selection share the same scoreboard selection path.
- First selected single game receives the same immediate score path as scoreboard-clicked games.
- Active entry lock state no longer rewrites unrelated field or scoreboard state.
- Known zero scores remain legitimate zero scores instead of being hidden behind loading doubt.
- The visible Sports shell stays instant while next-game hydration warms behind it.

### Card Reveal Became Stable Again

The card stage received the interaction fixes needed for a real collectible surface.

The reveal spread now keeps click, popover, drag, drop, selection, deselection, and reward-card behavior aligned across normal packs, multi-card spreads, middle cards, and reward reveals.

The important outcome is not animation. The important outcome is control:

- opening a card viewer no longer damages later drag/drop behavior
- reward cards can be selected, opened, dragged, and dropped
- 1-card through 10-card reveals preserve the same interaction contract
- deselection stays deselected until the user selects again
- card reveal remains a game surface, not a one-shot modal that breaks the stage

### Saved Cards Keep Issuance Truth

Saved card proof continuity was tightened so the issue number survives the viewer-to-saved-card path.

The release makes issuance an appendable proof event, not loose display text. That matters because Sports cards are assets. Issue number, pack origin, card identity, proof object, current owner, and card memory must move together or the market and vault become untrustworthy.

Outcome:

```txt
issued card
proof event
saved card
viewer
vault
market
```

now carries the same issuance truth.

### Player Availability Became Product Truth

Pick 5 now moves toward the truth users actually need before locking a lineup: whether a player is a starter, bench/role player, backup, or injured/out.

The release work identifies lineup and availability as first-class Sports information for both Main Event and single-game play. This prevents users from treating every owned player as equally likely to appear in a live game.

This is the right product framing:

- starters are expected active contributors
- bench and role players are playable but carry risk
- backup players should be identified before selection surprise
- injured or unavailable starters must be called out clearly
- Main Event and single-game cards must use the same availability language

### Live Player Sharing Was Restored

The live-player route is back to the behavior that matters: it can be opened from iMessage and play without turning profile or SEO cleanup into a first-paint delay.

This release keeps the canonical public profile route as the source of live-player truth while avoiding hot-path work that would make users wait on a white or broken shell.

Outcome:

- iMessage live-player previews and opens work again
- newly posted profile originals can navigate into live player without leave-and-return
- profile live-player truth first-paints from canonical public truth
- structured proof and manifest work remains behind the shell instead of blocking it

### SEO Hot Path Was Cleaned Without Moving Truth

The SEO cleanup removed expensive work from the hot path while preserving canonical profile and market truth.

The important release law is:

```txt
SEO enrichment may describe truth, but it must not make first paint wait for truth it already has.
```

Outcome:

- profile and market public routes avoid unnecessary structured-data work on the first visible path
- JSON-LD does not become the permission gate for the shell
- public profile remains canonical at `/<username>`
- `?manage=1` remains owner permissions layered over canonical public truth

### Explore Became Instant Enough To Trust

Explore had a server race when a build was already in flight. `v83.0.0` hardens that path so Explore can paint an immediate deterministic shell and keep richer discovery behind it.

Outcome:

- no white-screen dependency on a build already running elsewhere
- no Three.js, remote media, or heavy enrichment on the first shell
- discovery can hydrate after first paint without replacing the visible surface

### Market Stayed Fast And Conformance Caught Up

The Market conformance failure was not permission to slow `/market`. The product was already fast; the old test was carrying an outdated assumption after the SEO hot-path cleanup.

`v83.0.0` aligns the conformance suite with the actual product law:

- `/market` keeps the fast entry path
- conformance checks transport truth instead of requiring removed hot-path JSON-LD
- Market proof, provenance, value, and asset surfaces stay governed
- the test protects speed and truth together instead of choosing one

### Interoperability Was Centralized

Callback URL policy moved into a centralized allowlist path and Vercel trace inclusion now covers that source of truth.

Outcome:

- payment and embed callbacks share one policy
- interoperability conformance checks the policy users actually rely on
- callback behavior becomes auditable instead of scattered route logic

### OTC Proof Objects Became A Real Upgrade Path

The OTC and Cash Image work added the next proof object class: crypto and wallet state can be upgraded into offline-verifiable Receiz proof objects before optional chain reconciliation.

This release includes:

- image-native Receiz ID wallet creation
- private-key import and connected-wallet add paths
- wallet bundle sealing into `receiz.wallet.bundle-proof.v1`
- Cash Image proof objects
- offline pulse transfer before chain reconciliation
- wallet proof export and import handling
- OTC UI that keeps wallet actions in the wallet panel instead of fighting the command bar

Outcome:

```txt
chain asset or wallet state
offline Receiz proof
portable Cash Image
optional later reconciliation
```

### Account, Calendar, Profile, And Navigation Were Stabilized

Several release patches fixed places where larger account state was being treated as all-or-nothing.

Outcome:

- Account Brain and Calendar handle multi-page loads without failing the whole surface
- calendar keyset pagination is hardened
- bottom navigation points to canonical profile truth
- owner/manage hydration warms in the background
- account route feedback and one-viewport contracts remain covered
- profile showcase and profile analytics paths no longer destabilize the visible screen

### Conformance Became A Release Boundary

The release closes by treating conformance as product law enforcement, not a decorative dashboard.

The gates now cover:

- Market conformance
- Sports conformance
- Interoperability conformance
- profile first-paint continuity
- service-worker version coherence
- release freeze
- page first paint
- TypeScript
- release lock
- release attestation

The key outcome is cultural as much as technical:

```txt
if a law is true, the test must protect it
if a test is stale, it must be corrected
if code violates the law, code must move
```

## Release Boundary

`v83.0.0` is the stable release boundary after the `v82.0.0` Sports launch.

Future work may modularize large Sports files, extend player availability intelligence, add richer market depth, and broaden proof object classes. That work must stay behind this boundary:

- no slower first paint
- no visible shell swap after a page feels loaded
- no rediscovery before known deterministic truth
- no weakened Sports lock or score truth
- no proof object that drops issue, owner, provenance, or append history
- no conformance update that ratifies a visible regression
