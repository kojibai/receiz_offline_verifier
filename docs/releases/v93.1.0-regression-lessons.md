# Receiz v93.1.0 Regression Lessons

Release date: June 26, 2026

## What Regressed

- Some routes still treated admitted history as temporary hydration state instead of durable proof memory.
- Calendar and Account projected the same event data but did not fully share the same durable proof register.
- Wallet history could fall back to Chronos-window thinking even though Kai head state is the settlement ordering primitive.
- Some Sports read hardening reduced pressure but still left fallback loops that could rediscover old proof rows.
- Release lock drifted when `93.0.1` version surfaces existed without matching release docs.
- Sports TypeScript fixtures became stale after production proof objects became stricter.

## Why It Regressed

- The older app pattern treated route hydration as harmless even when it was re-proving known truth.
- Multiple screens displayed the same proof objects but the implementation sometimes stored route-shaped memory rather than object-shaped memory.
- Backend ordering inherited Chronos habits even though Receiz proof objects carry Kai state.
- Some fixes optimized query cost before asking whether the query should run at all after proof admission.
- Release version metadata moved faster than the release-lock document set.
- Tests were allowed to keep shortcut fixture objects after the source model required complete proof identity.

## Locked Rules

- Verified stored truth is not cache.
- Full-history hydration is first admission and recovery only.
- After admission, network reads discover additions after the known proof head.
- The same proof object must not be forked into separate route authorities.
- Kai pulse/upulse is ordering state for proof and settlement objects that carry it.
- Chronos is display/fallback context, not the first backend ordering authority when Kai exists.
- Bounded RPCs and indexes are mechanics beneath proof truth; they cannot become proof authority.
- Tests must move toward stronger proof object construction, not weaker production fields.
- Release docs, package version, service worker version, runtime storage version, SDK version, and public labels must move together.

## Contract Gates

- `app/calendar/__tests__/calendarActivityProjectionContract.test.ts` locks durable calendar/account event proof memory.
- `app/lib/account/__tests__/accountOneViewportContract.test.ts` locks account brain projection over admitted event truth.
- `app/lib/wallet/__tests__/walletRouteHydrationContract.test.ts` locks wallet route activity register and Kai/Chronos ordering.
- `app/lib/runtime/__tests__/localTruthStoreContract.test.ts` locks durable local truth rank and overwrite behavior.
- `app/game/__tests__/sportsArenaCardValue.behavior.test.ts` locks strict Sports proof fixture construction and deterministic value behavior.
- `app/lib/game/__tests__/sportsArenaPlayCommand.test.ts` locks recorded and live play/pitch witness source snapshots.
- `scripts/check_release_lock.mjs` locks release-version surface coherence.

## Release-Lock Requirement

`pnpm release:lock 93.1.0` must require this document for `v93.1.0` and later release cuts. A release that omits regression lessons is incomplete because the durable proof memory, shared event projection, Kai-ledger ordering, bounded Sports projection, stale fixture, and version-drift failure modes would not be explicit release truth.
