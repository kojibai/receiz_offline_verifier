## Preparation Steps

1. Preserve `v82.0.0` as the prior current-release archive.
   - Keep `docs/releases/v82.0.0.md`.
   - Keep `docs/releases/v82.0.0-product-truth.md`.
   - Keep `docs/releases/v82.0.0-checklist.md`.
   - Keep `docs/releases/v82.0.0-process.md`.
   - Preserve `v82.0.0` as historical Sports Arena launch truth.

2. Publish canonical `v83.0.0` release artifacts.
   - Add `docs/releases/v83.0.0.md`.
   - Add `docs/releases/v83.0.0-product-truth.md`.
   - Add `docs/releases/v83.0.0-checklist.md`.
   - Add `docs/releases/v83.0.0-process.md`.

3. Align repository release surfaces.
   - Update `package.json` to `83.0.0`.
   - Update root `README` current-release references.
   - Update `RELEASE_NOTES.md` and `CHANGELOG.md`.
   - Update `docs/README.md`, `docs/governance/README.md`, and `docs/FORMAT.md`.
   - Update product-truth reference status markers carried forward for the current release.
   - Update shipped version markers in `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.
   - Ensure local release-surface checks can execute through `pnpm lint`, `pnpm build`, and `pnpm release:lock`.

4. Validate Sports first-paint gates.
   - Sports first paint shows the truthful shell.
   - Known game state is used before hydration.
   - Main Event and single-game selection share the same scoreboard selection path.
   - First selected single game receives the same immediate score path as scoreboard-clicked games.
   - Active entry lock state cannot rewrite unrelated field or scoreboard state.
   - Known zero scores remain legitimate zero scores.

5. Validate card reveal and issuance gates.
   - Opening a card viewer does not damage later drag/drop behavior.
   - Reward cards can be selected, opened, dragged, and dropped.
   - 1-card through 10-card reveals preserve the same interaction contract.
   - Deselection stays deselected until the user selects again.
   - Saved cards keep issue number, pack origin, card identity, proof object, current owner, vault truth, market truth, and appendable memory.

6. Validate player availability gates.
   - Starters, bench/role players, backup players, and injured/out players are called out clearly.
   - Main Event and single-game cards use the same availability language.
   - Player availability is available before lineup lock decisions.

7. Validate live-player and SEO gates.
   - iMessage live-player previews and opens work.
   - Newly posted profile originals can navigate into live player without leave-and-return.
   - Profile live-player first-paints from canonical public truth.
   - Structured proof, manifest, JSON-LD, SEO enrichment, and owner/manage hydration remain behind the first visible shell.
   - Public profile remains canonical at `/<username>`, and `?manage=1` remains owner permissions layered over canonical public truth.

8. Validate Explore and Market gates.
   - Explore paints an immediate deterministic shell even when build work is already in flight.
   - Explore first shell avoids Three.js, remote media, and heavy enrichment.
   - `/market` keeps the fast entry path.
   - Market conformance checks transport truth without requiring removed hot-path JSON-LD.
   - Market proof, provenance, value, and asset surfaces stay governed.

9. Validate interoperability and OTC proof-object gates.
   - Payment and embed callbacks share the centralized allowlist policy.
   - Interoperability conformance checks the callback policy users rely on.
   - Vercel trace inclusion covers the allowlist source of truth.
   - Wallet and crypto state can upgrade into offline-verifiable Receiz proof before optional chain reconciliation.
   - Image-native Receiz ID wallet creation, private-key import, connected-wallet add, wallet bundle sealing, Cash Image proof objects, offline pulse transfer, proof export/import, and wallet-panel UI placement pass.

10. Validate account, calendar, profile, and navigation gates.
    - Account Brain and Calendar handle multi-page loads without failing the whole surface.
    - Calendar keyset pagination is hardened.
    - Bottom navigation points to canonical profile truth.
    - Owner/manage hydration warms in the background.
    - Account route feedback, profile showcase, profile analytics, and one-viewport contracts remain covered.

11. Validate conformance boundary gates.
    - Market conformance passes.
    - Sports conformance passes.
    - Interoperability conformance passes.
    - Profile first-paint continuity passes.
    - Service-worker version coherence passes.
    - Release freeze, page first paint, TypeScript, release lock, and release attestation gates pass.

12. Validate release-surface lockstep.
    - Confirm all governed version markers show `v83.0.0`.
    - Confirm all current-release docs and indices point to `v83.0.0` artifacts.
    - Confirm `v80.0.0`, `v81.0.0`, and `v82.0.0` artifacts remain historical and untouched except for index references.
    - Run `pnpm release:lock`.

13. Generate signed release attestation.
    - Run `pnpm release:attest` for the final commit in the environment that holds the release attestation signing key.
    - Retain the signed attestation artifact under the governance release-attestation policy.
    - Do not tag or ship without the signed attestation.

## Focused Verification Matrix

Broader platform release validation should include the gates named in `docs/releases/v83.0.0-checklist.md`, including:

- Sports first paint, selected-game score, lock state, zero score, and next-game hydration
- Main Event and single-game scoreboard selection
- card reveal, reward reveal, saved-card issue/proof/owner/vault/market continuity
- Pick 5 player availability language and lineup truth
- iMessage live-player, canonical public profile first paint, owner/manage hydration, SEO hot-path cleanup
- Explore immediate shell and no-white-screen race behavior
- Market fast entry, transport proof conformance, proof/provenance/value/asset surfaces
- centralized callback allowlist and Vercel trace inclusion
- OTC wallet bundle proof, Cash Image proof, offline pulse transfer, export/import, and wallet-panel UI
- Account Brain, Calendar keyset pagination, profile showcase, profile analytics, navigation, and one-viewport contracts
- Market, Sports, interoperability, profile first-paint, service-worker, release freeze, page first paint, TypeScript, release lock, and release attestation gates

## Tag Criteria

Do not tag `v83.0.0` until:

- all required `v80.0.0`, `v81.0.0`, `v82.0.0`, and `v83.0.0` release artifacts exist
- version surfaces are aligned to `v83.0.0`
- release gates in `docs/releases/v83.0.0-checklist.md` pass from the exact release candidate
- governed surfaces all show `v83.0.0`
- `pnpm lint` has zero warnings/errors
- `pnpm build` completes
- `pnpm release:lock` confirms the release surfaces
- `pnpm release:attest` emits a signed attestation for the final commit

## Release Rule

Do not ship `v83.0.0` unless the release gates in this checklist pass from the exact release candidate, `pnpm release:lock` confirms the release surfaces, and `pnpm release:attest` emits a signed attestation for the final commit.
