# Receiz v105.0.0 Performance Findings

`v105.0.0` preserves Receiz scale law by separating browser-safe SDK runtime graphs from Node-only compiler code and a browser-safe testing emulator. Consumers load only the rail they request while proof creation and verification semantics remain unchanged.

## Findings

- Root and React emitted graphs contain no transitive compiler, CLI, testing, filesystem, or `node:*` imports.
- Compiler functionality resolves from the explicit Node-only `@receiz/sdk/compiler` subpath rather than burdening browser package resolution.
- The testing emulator provides bounded sandbox evidence with zero production network or database calls and never returns a production verified verdict.
- Stable redacted errors and capability describe/probe allow consumers to fail or adapt without leaking secrets or probing unsupported rails repeatedly.
- Protected `receiz.extensions.ts` generation preserves developer-owned siblings, and semantic MCP writes require explicit confirmation boundaries.
- Packed tarball installation into an empty external project validates exact 105.0.0 exports and dependencies outside monorepo resolution.

## Performance Boundary

- Known verified truth must paint before hydration, analytics, SEO, session checks, lazy imports, global sync, chart hydration, ledger pagination, or server reconciliation.
- Offline artifact verification remains primary and separate from emulator or conformance evidence.
- Export-map resolution, compiler execution, capability probes, error transport, protected generation, MCP confirmation, package installation, and service-worker continuity remain weaker mechanics beneath proof primitives.
- Visual evidence can block release even when static contracts pass.

## Release-Lock Requirement

`pnpm release:lock 105.0.0` must require this document for `v105.0.0` and later release cuts. A release that omits performance findings is incomplete because browser/Node graph separation, emulator isolation, stable error handling, capability discovery, protected generation, semantic MCP confirmation, external packed installation, and verifier/service-worker identity would lack explicit proof-authority boundaries.
