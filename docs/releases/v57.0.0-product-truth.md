# Receiz v57.0.0 Product Truth

Release date: April 17, 2026
Status: Locked for `v57.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v57.0.0`. It carries forward the `v56.0.0` version-surface baseline while recording canonical managed-owner routing, world/market fast-entry route continuity, share-certificate live custody/history, market balance and inventory coherence, and market-conformance boundary locking as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, shipped HTML version markers, and package-metadata alignment only. The verifier trust contract remains unchanged relative to `v56.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- File-backed proof remains the authority for artifact truth.
- Deterministic managed-owner, world, market, and player route snapshots remain the authority for first paint and navigation continuity truth when that route truth is already knowable.
- Wallet, ledger, and note-event custody state remain the authority for money, share, and certificate truth.
- Live world, market, certificate, player, and visibility deltas may hydrate on top of deterministic baseline state, but they must not replace or discard already-known route or custody truth as the source of truth.
- Public product routes, conformance surfaces, release metadata, and offline verifier entrypoints remain operating or governance surfaces around those authorities, not replacements for them.

## Locked Release Surface

### 1. The v56.0.0 Baseline Must Remain The Floor

- Release-governed version surfaces must remain lockstep and current-release docs must stay aligned.
- Logged-in world hydration must continue to reject malformed or stale cached route snapshots before bootstrap truth is adopted.
- Concierge polling must continue to respect the real Automation entitlement.
- Fresh-account analytics must continue to degrade cleanly when the database schema is briefly behind the app event model.
- The offline verifier trust contract must remain file-authoritative, deterministic, and fail-closed.

### 2. Managed Owner Routing Must Be Canonical When Knowable

- Runtime profile entry must resolve through the canonical managed owner route `/${username}?manage=1` when owner-route truth is already available.
- Auth flows, global auth chrome, chart continuation, and market continuation paths must preserve already-known owner truth instead of falling back through a generic `/profile` destination.
- Manage-route clients must reuse deterministic owner snapshots instead of discarding them and forcing a second settle.

### 3. World And Market Must Reuse Prepared Route Truth

- World and market entry surfaces must prime route snapshots before navigation and reuse warmed truth on first paint.
- Authenticated world truth must take precedence over stale anonymous world truth when both are available.
- Cold market entry must seed a light entry snapshot before background bootstrap instead of blocking navigation on full desk hydration.
- Already-hydrated chart or pill truth must promote into market route truth rather than be discarded during market navigation.

### 4. Share Certificates Must Behave As Live Custody

- Share certificates must remain chart-first, compact, passkey-first, and USD-first on the public certificate surface.
- Certificate trade gating and lane visibility must derive from real certificate balance rather than weak warmed state.
- Public certificate links, OG surfaces, and live route metadata must resolve to canonical production URLs and live certificate summary truth.
- Certificate history and basis semantics must remain certificate-native rather than thin projections of market truth.

### 5. Certificate Custody Accounting Must Be Note-Native

- Note-event ledger guarantees are required for certificate mint, restore, buy, and sell mutations.
- Import-delta recovery must remain authoritative from note metadata and real mutation results instead of optimistic assumptions.
- Certificate sells must execute exact imported quantity instead of re-deriving a second quantity from USD after import.
- Certificate chart history, basis overlays, and tooltip semantics must remain tied to certificate-native custody truth.

### 6. Market Balances, Inventory, And Conformance Must Stay Coherent

- Export and import flows must hydrate market and certificate balances immediately without requiring a reload to discover the new state.
- Public market inventory must revalidate against profile visibility, archive/unarchive, and public receiz publication changes deterministically.
- Chart and market auth continuation must clear sign-in CTAs and preserve wallet entry usability immediately after successful passkey login.
- The market conformance boundary must include certificate note-event ledger coverage, authoritative import-delta recovery, exact sell quantity, certificate-native history/basis semantics, fast-entry route continuity, suite-level drift checks, and required trace coverage.

### 7. Release-Governed Version Surfaces Must Move Together

- Release-governed version surfaces must move together to `v57.0.0` across package metadata, service-worker defaults, public verifier artifacts, public release badges, and current release documentation.
- Public-facing version truth must not drift between shipped HTML entrypoints, docs indexes, release artifacts, and repository metadata.
- This offline verifier repository must reflect that lockstep rule in `package.json`, `README.md`, `RELEASE_NOTES.md`, `CHANGELOG.md`, `docs/README.md`, `docs/governance/README.md`, `docs/FORMAT.md`, `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.

### 8. Release Governance And Verifier Truth

- Release-scoped note, product-truth, and checklist documents are required for `v57.0.0`.
- Root package metadata must resolve to `v57.0.0`.
- Shipped verifier, studio, and settlement entrypoints must visibly display `v57.0.0`.
- The offline verifier trust contract remains unchanged relative to `v56.0.0`.

## Allowed Evolution

Implementation details may change after `v57.0.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v57.0.0` if it:

- reintroduces weak `/profile` fallback when canonical managed-owner route truth is already available
- discards warmed world or market route truth and returns to second-settle or blank-shell entry behavior
- treats certificates like thin export artifacts instead of live custody surfaces with note-native history and basis truth
- allows market balances or public inventory to remain stale after certificate or profile-visibility mutations that the product already knows about
- narrows market conformance below the real runtime boundary for certificate custody, import recovery, exact sell execution, or fast-entry route continuity
- allows package metadata, public badges, shipped verifier artifacts, or current-release docs to advertise different release numbers
- changes release truth without updating the versioned release docs and aligned repository version surfaces
