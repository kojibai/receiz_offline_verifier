# Receiz v50.0.0

Release date: April 11, 2026

## Overview

`v50.0.0` is the world revenue OS release and the major recap cut for everything shipped after `v40.0.0`. It takes the platform from a strong trust-and-world foundation into a product that can actually operate: durable contacts CRM, live world command surfaces, concierge-run lead sourcing, campaign persistence, scheduled autopilot, provider-backed enrichment, truthful rail readiness, deliverability telemetry, reply handling, and closed-loop revenue attribution back into booking and settlement.

The core Receiz rule does not change. The file remains the authority. Deterministic manifests remain the authority for route truth. Ledger-backed payment state remains the authority for settlement. The new twin, campaign, outreach, and operator surfaces are execution and continuity layers around those trust boundaries, not replacements for them.

The product-truth freeze for this release is defined in `docs/releases/v50.0.0-product-truth.md`.

## What Defines v50.0.0

- Receiz now has a real revenue operating system inside `/world`, not just a conversational twin surface.
- Contacts, campaigns, outreach, replies, bookings, and revenue attribution are now tied together instead of living as separate product islands.
- The twin can source leads, enrich contact paths, import them into Contacts, stage and send outreach, track provider events, surface reply work, and keep campaign continuity alive across runs.
- Rail truth is now explicit: blocked rails, missing provider setup, missing endpoints, and delivery failures surface as blocked operator state instead of optimistic "success."
- A standalone enrichment service can now run on its own deployment and feed provider-backed public contact enrichment into the main app without changing the main app deployment boundary.
- Release-governed version surfaces move together to `v50.0.0` across package metadata, service worker defaults, public badges, offline verifier surfaces, and release docs.

## Major Accomplishments Since v40.0.0

### 1. Receiz Moved From Trust Runtime To Trust Platform

The `40.x` baseline established the core modern Receiz substrate:

- hardened signature and release-governance primitives
- richer offline verification and seal tooling
- stronger wallet, payment, settlement, and admin controls
- deeper public artifact, profile, and verified-original surfaces
- the first serious world runtime stack across owned compute, rendering, embodiment, identity, and presence

That work mattered because it made Receiz more than a single proof feature. It established the platform boundary that the later world and revenue layers could safely build on.

### 2. Public Twin And World Became Product-Grade Surfaces

The `45.x` through `47.0.x` span took the early world runtime and made it product-real:

- stronger Public Twin bootstrap, continuity, stage separation, and preview locking
- more deliberate photoreal shell fit, face-feature authority, and freshness behavior
- better rendered portrait overlays, voice playback, and streaming behavior
- tighter route hydration and world/profile/public payload discipline
- stronger trust, governance, legal, developer, and public framing around the runtime

The result is that Public Twin and world stopped feeling like a backend demonstration layer and started behaving like governed product surfaces.

### 3. Originals, Documents, Verified Playback, And Public Delivery Matured

The `46.x` and `47.0.x` releases materially expanded the artifact-facing side of the product:

- richer original and document reading instead of attachment passthrough
- stronger verified-original player continuity and sharing
- harder-to-break signed storage, byte-range, and `HEAD` delivery behavior
- clearer download precedence and route truth across originals, profiles, and player surfaces

That matters because Receiz still has to be truthful at the file and delivery layer while the world/runtime layers grow around it.

### 4. Contacts Became Durable Operator Infrastructure

The `47.6.x` through `48.0.0` span turned Contacts into a real system of record:

- `/contacts` became a standalone operating surface instead of a convenience panel
- dedicated CRM tables, import runs, event history, and person-first persistence landed
- historical message and original-collection data started backfilling into the same contact graph
- mobile contacts became compact enough to operate from one viewport without dropping any primary actions
- linked-account import merges became durable instead of silently dropping identity upgrades

This is the first big reason `v50.0.0` is possible. The twin cannot operate as a revenue system if its people layer is not durable.

### 5. World Became A Real Command Center

The world shell itself changed from atlas-only presentation into a living operator console:

- district and atlas surfaces became cleaner, more legible, and more authoritative on mobile
- concierge gained tracked live runs, task history, restart/clear controls, and more truthful blocked/failure handling
- the system now exposes revenue launchers, next-money-moves, deal-room surfaces, and live campaign cards instead of hiding execution behind generic prompts
- swipe-navigation conflicts, cramped headers, sticky task cards, and false-mobile stack patterns were tightened so the live operator shell has more usable space and fewer UI traps

The twin now feels less like a modal assistant and more like a workspace with continuity.

### 6. The Twin Can Now Run Revenue Campaigns Instead Of One-Off Demos

The largest post-`48.0.0` change is the campaign substrate:

- world campaigns, campaign contacts, and campaign runs now persist in dedicated schema
- campaigns now score lead quality, deliverability, response momentum, follow-up pressure, and revenue grade
- the campaign model can compute next-best moves for the operator and the twin
- blocked states, scheduled states, hot leads, due follow-ups, and reply queues are surfaced as explicit operating state

This is the release where the twin stops being "chat plus helper actions" and becomes "campaign memory plus operator execution."

### 7. Autopilot, Deliverability, And Provider Telemetry Landed

The product now has a real execution loop, not just an initial send:

- campaign autopilot cadence, quiet hours, pass caps, and next-run scheduling landed
- due-campaign heartbeat routes can execute scheduled passes server-side
- provider webhook ingestion now records Resend and Twilio delivery/status events into campaign state
- campaign events keep enrichment, provider, reply, and commercial activity in one timeline
- rail readiness distinguishes saved policy from actually configured provider infrastructure

This closes a major credibility gap. The system can now say whether it is truly ready to send, what happened after send, and why a campaign is blocked.

### 8. Enrichment, Reply Handling, And Money Attribution Closed The Loop

The current `50.0.0` state also closes the most important operator loop:

- lead research now widens briefs automatically instead of failing on the first thin search
- concierge dispatch can retry enrichment inline, switch rails when the wrong channel is selected, and keep blocked campaigns alive instead of dying silently
- a standalone enrichment service now exists for provider-backed public contact and buyer-signal enrichment on its own subdomain
- reply handling and campaign inbox views now distinguish live reply work from sequence work
- booking signals, Stripe settlement signals, and Receiz wallet transfer context now reconcile back into campaign contacts as booked / won revenue

This is the practical difference between "AI lead generation theater" and "a system that can keep working the list until there is a usable path or a real commercial outcome."

## Public Release Framing

`v50.0.0` should be understood as the release where Receiz turns the world stack into a governed revenue operating system while preserving the original trust model.

The major themes are:

- trust boundaries stayed deterministic while the operator layer expanded
- world and Public Twin became real product surfaces
- Contacts became the durable memory layer for people and work
- campaigns, outreach, replies, bookings, and revenue attribution now move through one system
- rail truth, enrichment, and provider telemetry make the operator surface more honest

## Suggested Public Summary

Receiz `v50.0.0` turns the world stack into a real revenue operating system. The twin can now source and enrich leads, import them into durable Contacts, run tracked outreach campaigns, keep reply and follow-up continuity alive, react to provider telemetry, and reconcile bookings and revenue back into campaign state, all while keeping Receiz's core trust rule unchanged: the file remains the authority.
