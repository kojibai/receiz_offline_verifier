## Title

Receiz `v93.1.0` - Durable Proof Memory Release

## Summary

`v93.1.0` is the first locked minor release after `v93.0.0`. It freezes the breakthrough learned from Sports card history and applies it to the broader Receiz operating model: once proof truth is admitted, it is durable local truth, not stale cache. Routes project admitted proof objects. Servers, DBs, and global sync discover verified additions beneath that truth.

The documented implementation baseline is `v93.0.0..465fde3f5`. That range spans `29` commits, `145` files changed, `75,532` insertions, and `836` deletions. The mechanical commit ledger is preserved in `docs/releases/v93.1.0-commit-history.md`.

The release-lock edit itself moves the governed local public version surfaces from `93.0.x` to `93.1.0`: package metadata, static service-worker version, shipped offline verifier labels, release notes, changelog, agent law, docs indices, and release documents. Broader app runtime storage, public labels, and SDK package/version constants remain named product-truth surfaces for the platform release and are external evidence boundaries for this offline verifier checkout.

## What Changed Since v93.0.0

### Durable Proof Memory

Sports card history proved the correct model: first admission only, then append forever. Cards that needed missing event memory could hydrate once, then reopen from durable proof memory. Known history no longer needed to be rediscovered by broad DB reads before the user could see it.

`v93.1.0` carries that model forward. Calendar and Account now project the same admitted event proof objects through different surfaces. Calendar renders them as calendar points. Account renders them as activity proof nodes. They are not separate truth objects. They are one admitted proof register with multiple projections.

### Wallet Ledger Kai Ordering

Wallet settlement history now carries Kai pulse/upulse ordering through ledger heads, cursor windows, and activity registers. Chronos remains display and fallback context. Kai state is the ordering primitive beneath settlement projection.

The key rule is now explicit: the app asks what verified additions exist after the known ledger head. It does not ask whether old ledger truth is still true.

### Sports Hot-Path Hardening

Sports vault, event proof, visible append, and showcase paths were hardened around bounded reads, RPC projection, timeout containment, and durable append memory. The fix was not to remove proof data. The fix was to stop re-querying old proof and to bound the paths that discover additions.

### Developer Surface And SDK

The developer suite moved from concept to implemented surface. The OpenAPI console, developer documentation, SDK package, fixtures, conformance tests, and quickstart structure now give external developers a proof-native integration path. The SDK remains convenience beneath Receiz primitives; it is not authority.

### Release Hygiene

The release closes stale TypeScript test debt in Sports proof fixtures. The production sports proof object model stayed strict. The tests were upgraded to construct complete media, base card, pack, proof, and witness objects instead of weakening the production contracts.
