# Receiz v50.0.0 Product Truth

Release date: April 11, 2026
Status: Locked for `v50.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v50.0.0`. It carries forward the existing file-authoritative, route-deterministic, and settlement-truth platform model while recording durable contacts CRM, live world command surfaces, persistent campaigns, scheduled autopilot, truthful rail readiness, provider-backed enrichment, reply handling, and revenue attribution as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, and visible version-surface alignment only. The verifier trust contract remains unchanged relative to `v47.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- The file remains the authority for artifact truth.
- Deterministic manifests remain the authority for historical route truth.
- Ledger-backed payment state remains the authority for settlement truth.
- Contacts, campaigns, outreach, replies, bookings, and world operator surfaces must execute around those trust boundaries rather than replace them.
- Provider telemetry, webhook events, enrichment outputs, and operator recommendations may advance execution state, but they must not fabricate settlement, delivery, or booked / won truth without durable backing records.
- Offline verifier entrypoints, release metadata, and public release documents remain transport and governance around those boundaries, not replacements for them.

## Locked Release Surface

### 1. `/world` Must Remain A Real Operator Command Center

- `/world` must continue to expose live operator state for contacts, campaigns, outreach, replies, bookings, and next-money-moves instead of collapsing back into generic conversational prompts.
- Concierge must preserve tracked runs, task history, restart and clear controls, and truthful blocked or failed execution state.
- Mobile world surfaces must remain legible and actionable, with atlas, district, and campaign surfaces behaving like workspace infrastructure instead of demo-only presentation.

### 2. Contacts Must Remain Durable Operator Infrastructure

- `/contacts` must remain a standalone operating surface instead of a convenience side panel.
- Dedicated CRM persistence, import runs, event history, and person-first records must continue to back operator workflows.
- Historical message and original-collection continuity must keep materializing into the same durable contact graph.
- Identity upgrades and linked-account merges must continue to enrich the existing person record instead of fragmenting continuity.

### 3. Campaigns Must Remain Persistent System State

- World campaigns, campaign contacts, and campaign runs must remain durable schema-backed records rather than transient helper state.
- Campaign state must continue to track lead quality, deliverability, response momentum, follow-up pressure, and revenue grade.
- Next-best moves, blocked state, scheduled state, hot leads, due follow-ups, and reply queues must remain explicit operating state for the twin and the operator.

### 4. Rail Readiness And Provider Telemetry Must Stay Truthful

- Saved policy must remain distinct from actually configured provider infrastructure.
- Missing provider setup, blocked rails, missing endpoints, and delivery failures must surface as blocked operator state rather than optimistic success.
- Provider webhook ingestion must continue to write delivery and status events into campaign state and timeline history.

### 5. Autopilot, Enrichment, Replies, And Attribution Must Remain Closed-Loop

- Campaign autopilot cadence, quiet hours, pass caps, and next-run scheduling must remain first-class release behavior.
- Due-campaign heartbeat execution must continue to run scheduled passes server-side when configured.
- Standalone enrichment deployment must remain able to feed provider-backed public contact enrichment into the main operator system without requiring the main app deployment boundary to change.
- Reply handling must continue to distinguish live reply work from sequence work.
- Booking signals, Stripe settlement signals, and Receiz wallet transfer context must continue reconciling into campaign contacts as booked / won revenue.

### 6. Release Governance And Version Truth

- Release-scoped note and product-truth documents are required for `v50.0.0`.
- Release-governed version surfaces must move together across package metadata, service worker defaults, public badges, offline verifier surfaces, and release documents.
- This repository records the public release documents and shipped HTML version markers for that lockstep requirement.

## Allowed Evolution

Implementation details may change after `v50.0.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v50.0.0` if it:

- regresses `/world` back into a generic assistant shell that hides campaign, revenue, or blocked-state operator truth
- weakens Contacts from a durable CRM system of record into transient twin memory or drops merge and identity-upgrade continuity
- makes campaign runs, campaign contacts, or revenue state non-durable or no longer explicit to the operator
- reports rail readiness, provider delivery, or autonomous execution success without reflecting missing configuration, blocked rails, or delivery failures
- breaks scheduled autopilot, standalone enrichment continuity, reply-work separation, or revenue attribution back into campaign state
- changes release truth without updating the versioned release docs and aligned version surfaces
