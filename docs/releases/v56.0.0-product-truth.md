# Receiz v56.0.0 Product Truth

Release date: April 17, 2026
Status: Locked for `v56.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v56.0.0`. It carries forward the continuity architecture locked in `v55.1.0` while recording canonical version-surface alignment, guarded logged-in world hydration, entitlement-aware concierge polling, and fresh-account analytics schema-drift tolerance as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, shipped HTML version markers, and package-metadata alignment only. The verifier trust contract remains unchanged relative to `v55.1.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- File-backed proof remains the authority for artifact truth.
- Deterministic route state and already-known route snapshots remain the authority for first paint and navigation continuity truth.
- Wallet, ledger, and exact USD execution state remain the authority for money truth.
- Live session, market, player, media, analytics, and world deltas may hydrate on top of deterministic baseline state, but they must not replace or discard that baseline as the source of truth.
- Public product routes, admin surfaces, release metadata, analytics, and offline verifier entrypoints remain operating or governance surfaces around those authorities, not replacements for them.

## Locked Release Surface

### 1. The v55.1.0 Continuity Baseline Must Remain The Floor

- `/profile`, public profiles, search handoff, the global composer host, and profile navigation must continue to reuse already-known owner or route truth instead of re-resolving through extra shells.
- Known route truth must continue across navigation, with route feedback staying on-button and tied to real pending state rather than dead taps or artificial waits.
- Market `Buy` must remain one passkey-first continuation with exact-shortfall funding and automatic order replay.
- Sell and settlement must remain USD-intent authoritative, position-aware, and deterministic.
- Live player fit and public twin precedence must remain deterministic.

### 2. Release-Governed Version Surfaces Must Move Together

- Release-governed version surfaces must move together to `v56.0.0` across package metadata, service-worker defaults, public verifier artifacts, public release badges, and current release documentation.
- Public-facing version truth must not drift between shipped HTML entrypoints, docs indexes, release artifacts, and repository metadata.
- This offline verifier repository must reflect that lockstep rule in `package.json`, `README.md`, `RELEASE_NOTES.md`, `CHANGELOG.md`, `docs/README.md`, `docs/governance/README.md`, `docs/FORMAT.md`, `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.

### 3. Logged-In World Hydration Must Reject Malformed Cached Truth

- The world route must no longer trust malformed or stale cached snapshots before bootstrap shape is validated.
- Logged-in world hydration must protect itself before adopting cached route truth.
- The product must not regress into client adoption of incomplete cached world state that can crash or poison initial render truth.

### 4. Concierge Polling Must Respect Automation Entitlement

- World concierge task polling must check the real Automation entitlement before polling or rendering live task controls.
- Signed-in users without Automation access must not repeatedly hit `/api/world/concierge/tasks` with `403 world_automation_required` churn.
- The product must not render a broken concierge lane that implies live task access when entitlement is absent.

### 5. Fresh-Account Analytics Must Degrade Cleanly Under Schema Drift

- Fresh-account owner analytics events such as `profile_activation_checkpoint` must not break login or first-use flows when the database schema is temporarily behind the app event model.
- The server analytics path must degrade cleanly to `tracked: false` instead of throwing a `500` when the older analytics event-name constraint is still active.
- The release includes the migration that brings analytics constraints and rollups back into sync with the current event model.

### 6. Release Governance And Verifier Truth

- Release-scoped note, product-truth, and checklist documents are required for `v56.0.0`.
- Root package metadata must resolve to `v56.0.0`.
- Shipped verifier, studio, and settlement entrypoints must visibly display `v56.0.0`.
- The offline verifier trust contract remains file-authoritative, deterministic, and fail-closed relative to `v55.1.0`.

## Allowed Evolution

Implementation details may change after `v56.0.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v56.0.0` if it:

- breaks the `v55.1.0` continuity baseline by reintroducing profile-shell drift, second-settle route behavior, split buy continuation, non-USD settlement truth, or nondeterministic player presentation
- allows package metadata, public badges, shipped verifier artifacts, or current-release docs to advertise different release numbers
- allows the logged-in world route to adopt malformed or stale cached snapshots before validating route truth
- allows non-Automation users to keep polling concierge tasks or render live concierge controls without entitlement
- lets fresh-account analytics throw `500` responses instead of degrading cleanly when database constraints are stale
- changes release truth without updating the versioned release docs and aligned repository version surfaces
