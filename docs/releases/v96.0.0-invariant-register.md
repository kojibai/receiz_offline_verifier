# Receiz v96.0.0 Invariant Register

Release date: 2026-06-29

`v96.0.0` carries forward the proof-native artifact system invariants from `v95.0.0` and adds the SDK A+ release-evidence invariants needed for executable developer surfaces.

## Locked Invariants

1. Receiz remains a proof-native artifact system. Product truth is carried by files, identities, vault objects, ownership states, settlement states, public witness surfaces, and verification routes.
2. Complete proof objects carry deterministic identity, sealed/verifiable payload, provenance, inspection behavior, and Kai/Klok state.
3. Kai Klok is the proof object state machine. `kaiPulseEternal` / `kai_upulse` is the pulse unit.
4. Durable proof memory is not cache. First admission creates known truth, and later work appends verified additions.
5. The SDK and CLI are developer convenience surfaces beneath sealed artifacts, proof bundles, verified appends, ownership appends, settlement rows, and identity artifacts.
6. The `receiz` CLI may inspect, validate, project, and initialize local proof memory from SDK primitives. It must not become an issuer, verifier, server confirmation layer, database authority, or conformance authority.
7. SDK local conformance must run from packaged fixtures without network calls, Supabase calls, service-role credentials, or database calls.
8. Passive `ReceizSdkObservation` events are evidence only. They must not alter proof admission, leak proof payloads, or gate first paint.
9. Copy-paste integrations must teach verify, project, admit, persist, and ask for verified additions after the known head.
10. SDK tests, typecheck, build, pack dry-run, CLI conformance, published package contents, and no-authority-leak checks are release-blocking in the broader platform workspace.
11. `@receiz/sdk` release version, app release version, SDK runtime constants, and release documents must not diverge before promotion.
12. Server, database, SDK, CLI, SEO, analytics, session, and framework routing conventions remain weaker than admitted proof truth.

## Forbidden Regressions

- proof object to card
- artifact system to UI experience
- durable proof memory to cache
- SDK CLI to verifier authority
- local conformance to network or database confirmation
- passive observation to admission control
- known-head sync to rediscovery-first loading
- release evidence to optional package hygiene

## Release-Lock Requirement

`pnpm release:lock 96.0.0` must require this document for `v96.0.0` and later release cuts. A release that omits the invariant register is incomplete because proof-native system identity, complete proof object Kai/Klok state, durable proof memory, SDK CLI boundaries, zero network/DB local conformance, passive observation, copy-paste integration order, version lock, and broader package evidence would not be explicit release truth.
