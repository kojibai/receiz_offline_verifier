# Receiz v107.0.0 Performance Findings

`v107.0.0` preserves Receiz scale law while exposing one proof-native operation language across identity, profile, media, portable continuity, generic bearer ownership, receipts, proof heads, offline commands, SDK, MCP, AI skills, emulator, generator, migration, frontend, service worker, and release evidence.

## Findings

- Known verified carried identity, profile, ownership, app-state, receipt, and proof-head truth paints immediately; network discovery and server reconciliation may append verified additions only after the known head.
- Raw profile media and existing media proof references share one staged atomic profile/media/head append, avoiding split mutations and partial public projection.
- Username availability remains advisory; commit-time uniqueness and same-key identity append avoid reservation state becoming authority.
- Bearer ownership verifies the enclosing artifact before interpreting embedded material and commits one append before deactivating the prior projection.
- Offline queues preserve FIFO order per proof object, allow independent lanes to resume separately, stop conflicted lanes deterministically, and never represent queued work as globally committed.
- Canonical receipt verification gates public projection, so SDK, MCP, frontend, cache, and emulator results cannot produce a faster but weaker committed state.
- The deterministic emulator carries `productionVerified: false`; generator preview/apply remains digest-bound, root-contained, and protective of developer-owned extensions.
- Exact public package versions, registry digest parity, clean external installation, runtime labels, service-worker identity, visual evidence, and release freeze move together without changing file-authoritative verification semantics.

## Performance Boundary

- Deterministic first paint must precede hydration, analytics, SEO, session checks, lazy imports, global sync, media discovery, queue resume, or server reconciliation.
- Registry evaluation, SDK/MCP operations, AI-skill contracts, emulator fixtures, generated integrations, migration reports, caches, and release evidence remain weaker mechanics beneath proof identity, ownership, settlement, verified receipts, and append-only history.
- Visual evidence may block promotion when static or unit contracts pass but the visible proof, identity, ownership, media, hierarchy, or first-paint primitive regresses.
- Publication, deployment, production database/Redis state, signed production attestation, and authenticated production-browser evidence remain separate external boundaries.
