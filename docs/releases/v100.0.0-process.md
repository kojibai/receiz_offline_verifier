## Goal

Lock the exact post-`v99.0.0` Living Proof working set as `v100.0.0`, move every established release identity surface together, and verify that living Twin and SDK runtime mechanics remain beneath the proof objects they compose.

`@receiz/sdk` remains a convenience rail, and durable proof memory remains **first admission only, then append forever**.

## What Worked

- The annotated `v99.0.0` tag provided an exact baseline rather than relying on the interim `99.0.1` package label.
- Character, atlas, motion, and Twin mind work reused the existing proof-bound Twin identity.
- Existing-object backfill runs after paint and binds appends to the current portrait rather than creating replacement authority.
- SDK proof/custody/escrow/event rails compose admitted proof objects and existing authentication/custody boundaries.
- Profile, Market, and Sports repairs preserve stronger known truth while removing foreground rediscovery.
- A named Living Proof release carries milestone magnitude while strict SemVer preserves compatibility meaning after 100.

## What Did Not Work

- The interim root/service-worker `99.0.1` bump did not move SDK, MCP, public labels, verifiers, or release records together and therefore was not a full release lock.
- Treating a subpayload, atlas, SDK route, or escrow response as its own authority would have bypassed the enclosing proof object.
- Running backfill before settled first paint would have turned additive continuity into a visible reload.
- A passing static contract without fresh browser evidence cannot prove visible Twin, Profile, Market, or Sports continuity.

## Practical Rule For Future Work

1. Use patch versions for compatible corrections and minor versions for compatible capability.
2. Reserve the next major version for an intentional breaking artifact or public contract boundary.
3. Keep named release themes separate from compatibility numbering.
4. Treat atlases and SDK payloads as deterministic projections beneath enclosing Receiz proof objects.
5. Keep backfill, hydration, session, and API work behind settled first paint.
6. Move all established version surfaces together and verify from the exact final candidate.

## Candidate Procedure

Run the gates recorded in `docs/releases/v100.0.0-checklist.md`. Any failed contract or contradictory visual evidence blocks the lock. Tagging, publishing, pushing, deployment, and production smoke verification remain separate external actions until actually completed.
