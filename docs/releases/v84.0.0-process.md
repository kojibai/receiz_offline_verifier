## Preparation Steps

1. Preserve `v83.0.0` as the prior current-release archive.
   - Keep `docs/releases/v83.0.0.md`.
   - Keep `docs/releases/v83.0.0-product-truth.md`.
   - Keep `docs/releases/v83.0.0-checklist.md`.
   - Keep `docs/releases/v83.0.0-process.md`.
   - Preserve `v83.0.0` as historical stable live-proof truth.

2. Publish canonical `v84.0.0` release artifacts.
   - Add `docs/releases/v84.0.0.md`.
   - Add `docs/releases/v84.0.0-product-truth.md`.
   - Add `docs/releases/v84.0.0-checklist.md`.
   - Add `docs/releases/v84.0.0-process.md`.

3. Align repository release law.
   - Add or update `AGENTS.md` with primitive acceptance criteria.
   - Update `docs/literal-product-law.md` with the stronger/weaker truth hierarchy, primitive decision procedure, and forbidden downgrade boundary.
   - Update product-truth reference status markers carried forward for the current release.

4. Align repository release surfaces.
   - Update `package.json` to `84.0.0`.
   - Update root `README` current-release references.
   - Update `RELEASE_NOTES.md` and `CHANGELOG.md`.
   - Update `docs/README.md`, `docs/governance/README.md`, and `docs/FORMAT.md`.
   - Update shipped version markers in `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.
   - Ensure `site/sw.js` carries service-worker runtime version `84.0.0`.
   - Ensure local release-surface checks can execute through `pnpm lint`, `pnpm build`, and `pnpm release:lock`.

5. Validate primitive naming gates.
   - Proof objects are not renamed as cards.
   - Vault objects are not downgraded to UI experiences.
   - Public proof surfaces are not downgraded to share pages.
   - Settlement primitives are not downgraded to balance displays.
   - Ownership, identity, route, and service-worker truth retain direct primitive names.

6. Validate deterministic first-paint gates.
   - Known truth paints before hydration.
   - SEO, analytics, session checks, lazy imports, dynamic JSON, global sync, server reconciliation, tracking, marketing surfaces, and framework convention do not gate first paint.
   - Hidden hydration appends verified additions only.

7. Validate version lockstep.
   - Confirm all governed version markers show `v84.0.0`.
   - Confirm all current-release docs and indices point to `v84.0.0` artifacts.
   - Confirm historical `v82.0.0` and `v83.0.0` artifacts remain archived release truth.
   - Run `pnpm release:lock`.
   - Run whitespace diff check.

8. Validate broader platform gates before production promotion.
   - Run release-freeze and page-first-paint checks.
   - Confirm Sports, Signal Run vault, profile/public asset, live-player, service-worker, and refunded tournament entry cleanup contracts.
   - Confirm target production DB migrations named by the release are applied.

9. Generate signed release attestation.
   - Run `pnpm release:attest` for the final commit in the environment that holds the release attestation signing key.
   - Retain the signed attestation artifact under the governance release-attestation policy.
   - Do not tag or ship without the signed attestation.

## Focused Verification Matrix

Broader platform release validation should include the gates named in `docs/releases/v84.0.0-checklist.md`, including:

- primitive decision procedure and direct primitive naming
- deterministic first-paint truth before hidden hydration
- Sports refunded tournament entry cleanup and lock-state display contracts
- live-player first-paint, share, OG, and deterministic payload contracts
- Signal Run vault write-pressure contracts
- profile/public asset and original-engagement read-path contracts
- service-worker version coherence
- release freeze, page first paint, TypeScript, release lock, whitespace diff, and release attestation gates

## Tag Criteria

Do not tag `v84.0.0` until:

- all required `v82.0.0`, `v83.0.0`, and `v84.0.0` release artifacts exist
- version surfaces are aligned to `v84.0.0`
- release gates in `docs/releases/v84.0.0-checklist.md` pass from the exact release candidate
- governed surfaces all show `v84.0.0`
- `pnpm lint` has zero warnings/errors
- `pnpm build` completes
- `pnpm release:lock` confirms the release surfaces
- whitespace diff check passes
- `pnpm release:attest` emits a signed attestation for the final commit

## Release Rule

Do not ship `v84.0.0` unless the release gates in this checklist pass from the exact release candidate, `pnpm release:lock` confirms the release surfaces, first-paint and release-freeze checks pass before production promotion, and production migrations named by the release are applied in the target environment.
