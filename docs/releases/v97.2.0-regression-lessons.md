# Receiz v97.2.0 Regression Lessons

Release date: 2026-06-30

## What Regressed

No runtime product behavior is intended to regress in this release. The concrete risks closed by this release were:

- App builders still had to wire tenant-scoped customer sessions, customer portals, merchant onboarding, media flows, events, jobs, permissions, audit, search, compliance, release pinning, and offline proof queues by hand.
- Enterprise writes could be replayed after cold starts, browser interruptions, or offline recovery without a consistent idempotency contract.
- React apps had to recreate proof-aware identity, checkout, app-state, media, domain, wallet, and Twin writer controls.
- Developer docs did not yet show a full Commerce Cloud grade app runtime path across app-state, checkout, media, events, jobs, RBAC, audit, search, compliance, offline replay, and release pinning.

## Why It Regressed

The `v97.0.0` app-state rail made Receiz practical for public app recovery, but enterprise apps need more than cold-start state. They need retryable writes, tenant-aware sessions, commerce workflows, media workflows, audit surfaces, notification/search recovery, compliance exports, and replayable offline queues while preserving Receiz truth hierarchy.

The regression risk was not missing primitives. The risk was making every developer reassemble primitive law from scattered routes and examples, then accidentally treating SDK helper state, hosted React controls, search rows, job rows, or queue rows as stronger than proof truth.

## Locked Rules

- `@receiz/sdk` must be version-locked to the app release.
- `RECEIZ_SDK_VERSION` must match `packages/receiz-sdk/package.json`.
- Public app release labels must match `package.json`.
- Service-worker and runtime coherence defaults must match the release.
- Every new enterprise write helper must accept idempotency material.
- Offline proof queue replay may append verified additions only.
- React hooks and hosted component factories must name proof primitives directly and remain below proof authority.
- Commerce helpers must not outrank ownership appends or settlement rows.
- Customer session helpers and customer portals must not become identity proof roots.
- Audit, risk, compliance, search, notification, job, media, and release pinning rails must expose or recover proof-native state without replacing it.

## Contract Gates

- `scripts/check_release_lock.mjs`
- `scripts/test_sdk_release_contracts.ts`
- `packages/receiz-sdk/test/sdk.test.ts`
- `packages/receiz-sdk` typecheck
- `packages/receiz-sdk` build
- `scripts/test_release_freeze.ts`

These gates fail if the app and SDK version identity diverge, SDK release evidence drifts, React subpath evidence disappears, idempotency headers disappear, offline queue replay breaks, or public developer route documentation stops matching the implemented surface.

## Release-Lock Requirement

`pnpm release:lock 97.2.0` must require this document for `v97.2.0` and later release cuts. A release that omits regression lessons is incomplete because enterprise SDK helper drift, idempotency replay behavior, React subpath boundaries, offline proof queue replay, customer/merchant/commerce helper authority limits, and app/SDK version lock lessons would not be explicit release truth.
