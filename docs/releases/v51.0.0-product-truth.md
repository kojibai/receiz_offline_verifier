# Receiz v51.0.0 Product Truth

Release date: April 12, 2026
Status: Locked for `v51.0.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v51.0.0`. It carries forward the existing file-authoritative, route-deterministic, and settlement-truth platform model while recording the public product-ladder realignment, stable public-to-billing plan mapping, consistent entitlement framing, and continued Contacts/world operator hardening as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, and visible version-surface alignment only. The verifier trust contract remains unchanged relative to `v50.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- The file remains the authority for artifact truth.
- Deterministic manifests remain the authority for historical route truth.
- Ledger-backed settlement remains the authority for payment truth.
- Public plan language, upgrade copy, and entitlement framing may clarify the product model, but they must not weaken or misstate the authority model beneath it.
- Contacts continuity, campaign visibility, activity history, pacing controls, and operator-surface hardening must execute around those trust boundaries rather than replace them.
- Offline verifier entrypoints, release metadata, and public release documents remain transport and governance around those boundaries, not replacements for them.

## Locked Release Surface

### 1. The Public Ladder Must Stay Ordered And Honest

- Public plan order must remain Free -> Authorship -> Business -> Automation -> Enterprise.
- The product must continue to describe those layers as try it, prove it, own it, run it, and scale it, or equivalent language with the same meaning.
- WireProof and adjacent proof utilities may remain product components, but they must not distort the public ladder back into a legacy Pro-first story.

### 2. Public Names Must Map Safely To Stable Billing Keys

- Public plan names must remain `Authorship`, `Business`, and `Automation`.
- Internal billing and subscription keys may remain `premium`, `business`, and `pro`.
- Public naming changes must not force a destructive billing-schema reset or break subscription continuity.

### 3. Upgrade And Entitlement Surfaces Must Speak With One Voice

- `/upgrade`, billing, account, FAQ, explainer, authorship, AssetSeal, and WireProof-adjacent prompts must describe the ladder in the same proof -> ownership -> automation order.
- Signed-out and signed-in users must both be able to understand the pricing model before committing to purchase or upgrade.
- The pricing surface must read like a coherent product model rather than a disconnected feature inventory.

### 4. Free Must Stay Usable While Paid Tiers Open Deeper Layers

- Free must remain a usable try-the-surface product rather than a crippled teaser.
- Paid tiers must deepen operating layers rather than merely repackage the free baseline.
- Automation and Enterprise must continue to read as active operating layers instead of renamed proof add-ons.

### 5. Contacts And World Truth Hardening Must Remain In Scope

- `/contacts` must continue paging through real owner CRM state rather than stopping at arbitrary backend chunks.
- World campaign contacts already linked into the owner CRM must remain visible from the Contacts frontend.
- Placeholder rows with no real identity must remain archived out of active CRM views.
- Guest/profile continuity, user-facing activity history, and calmer same-day repeat-touch pacing must remain part of the release truth.

### 6. Release Governance And Version Truth

- Release-scoped note and product-truth documents are required for `v51.0.0`.
- Release-governed version surfaces must move together across current-release docs and offline verifier HTML entrypoints.
- This repository records the public release documents and shipped HTML version markers for that lockstep requirement.

## Allowed Evolution

Implementation details may change after `v51.0.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v51.0.0` if it:

- reverts the public product ladder away from Free -> Authorship -> Business -> Automation -> Enterprise or returns to a legacy Premium/Pro-first presentation
- exposes public plan names that do not map cleanly and safely onto the stable internal billing keys
- makes `/upgrade`, billing, account, FAQ, or entitlement prompts disagree about proof, ownership, or automation
- turns Free into an intentionally crippled teaser instead of a usable product layer
- brings placeholder CRM rows, fragmented guest/profile continuity, opaque history copy, or overly aggressive repeat-touch behavior back into the live operator surface
- changes release truth without updating the versioned release docs and aligned version surfaces
