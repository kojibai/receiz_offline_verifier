# Receiz v47.6.0

Release date: April 9, 2026

## Overview

`v47.6.0` is the world command-center and contacts-operations release. It keeps the current `v47.x` product truth intact while turning the district workspace into a live atlas, letting the concierge quarterback tracked twin runs, introducing a standalone contacts CRM, and hardening outreach plus embedded checkout completion so operator state stays truthful under failure.

For this offline verifier repository specifically, `v47.6.0` is a release-surface alignment patch. The richer world atlas, concierge twin-run tracking, standalone contacts surface, outreach integrity hardening, and Stripe checkout completion verification live in the broader Receiz application; this repository carries the canonical release note, product-truth freeze, and visible version-surface bump so the public release artifacts stay locked together.

The product-truth freeze for this release is defined in `docs/releases/v47.6.0-product-truth.md`.

## What Defines v47.6.0

- `/world` districts are now map-first, with a richer atlas surface instead of a stacked card list.
- The existing concierge shell can now launch live twin runs from detailed operator prompts and show task progress inside the transcript.
- `/contacts` is now a standalone operating surface with import, export, notes, last-meeting memory, invite drafting, and person-first interaction history.
- World outreach now blocks private/internal webhook destinations and only persists email-thread artifacts after successful delivery.
- Embedded checkout completion now verifies Stripe session state before telling the parent surface that payment succeeded.
- Release-governed version surfaces stay locked together across package metadata, service worker defaults, public badges, offline verifier surfaces, and release documents.

## Targeted Release Outcomes

### 1. World Is Now A Real Command Map

The district workspace is no longer a flat stack:

- the world surface now opens as a richer atlas with landmass, route-pressure, and district focus rails
- district selection stays tied to one live world map instead of splitting attention between disconnected cards and detail panes
- the more cinematic atlas shell remains additive to the existing world runtime rather than replacing the underlying world model

### 2. Concierge Can Quarterback Twin Work

The concierge stays visually familiar but acts more like an operator console:

- detailed prompts can start a structured twin run instead of only returning a static answer
- live task cards show intent, stages, logs, next unlocks, and recent progress inside the same transcript shell
- active task state feeds back into concierge context so follow-up prompts stay grounded in the run already in motion

### 3. Contacts Become A First-Class Operating Surface

Receiz now has a dedicated contacts app:

- account now links into `/contacts` as a standalone CRM-style page
- contacts support CSV, JSON, and vCard import/export plus browser-native contact picking where the device/browser exposes it
- notes, stage filters, last-meeting fields, next steps, invite history, and person-first timelines all live in one surface instead of being scattered

### 4. Outbound And Checkout State Fail More Truthfully

Two operator-facing integrity holes are now closed:

- app-webhook outreach rejects localhost, private-network, link-local, and other non-public destinations before any server-side send
- email outreach checks delivery readiness first and only creates chat persistence after the email send succeeds
- embedded checkout completion-only flows now verify the Stripe session and only emit success when the session is actually paid

## Deployment Notes

The release keeps the current world-outreach production requirements:

- Receiz email: `RESEND_API_KEY`, `RECEIZ_FROM_EMAIL`
- SMS relay: `RECEIZ_WORLD_TWILIO_ACCOUNT_SID` or `TWILIO_ACCOUNT_SID`; `RECEIZ_WORLD_TWILIO_AUTH_TOKEN` or `TWILIO_AUTH_TOKEN`; and either `RECEIZ_WORLD_TWILIO_FROM_PHONE` / `TWILIO_FROM_PHONE` or `RECEIZ_WORLD_TWILIO_MESSAGING_SERVICE_SID` / `TWILIO_MESSAGING_SERVICE_SID`
- Favorite-app webhooks should be public, operator-controlled destinations
- Browser-native phone-contact import depends on client support for the Contacts Picker API; unsupported clients still use file import/export
