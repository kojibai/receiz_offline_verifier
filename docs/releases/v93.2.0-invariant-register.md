# Receiz v93.2.0 Invariant Register

Release date: 2026-06-26

This register makes explicit the v93.2 invariants proven by the SDK Proof Developer Experience release.

## 1. SDK Is Convenience, Not Authority

`@receiz/sdk` helps developers use Receiz primitives faster. It does not outrank sealed artifacts, proof bundles, verified appends, ownership appends, settlement ledger rows, or verified local registers.

Violation: an SDK response is treated as proof authority when stronger proof material exists.

## 2. Runtime Schemas Are Validation Mechanics

Runtime schema exports validate asset manifests, Sports card manifests, and webhook event shapes.

Violation: schema success is presented as equivalent to artifact byte verification, ownership proof, settlement proof, or verified append truth.

## 3. Deterministic Proof Object Projection Preserves Identity

Proof object projections carry deterministic identity, media, owner, proof, append, settlement, and verification rows from validated Receiz objects.

Violation: SDK projection splits a proof object into unrelated UI state or drops proof identity from the render-ready row.

## 4. Deterministic Sports Card Projection Preserves Proof Context

Sports card projections carry collectible identity, claim hash, ownership, value basis, append summary, and event proof summary together.

Violation: card identity is separated from claim, ownership, value, or proof append context.

## 5. SDK Proof Memory Is Append-Only Admission

The SDK proof memory register admits first truth once, preserves it, and appends verified additions.

Violation: duplicate or newer partial transport replaces a fuller admitted proof object.

## 6. First Paint Uses Admitted Proof Memory

Developer apps should render known proof truth from admitted proof memory before asking Receiz for verified additions.

Violation: a developer integration blocks first paint on rediscovering already-admitted proof truth.

## 7. Verified Additions Follow Known Heads

After admission, SDK-backed apps ask for additions after the known proof head.

Violation: SDK docs or helpers encourage full-history rehydration as the normal reopen path.

## 8. Webhook Helpers Do Not Define Truth

Webhook helpers parse and validate transport events. The event payload remains subordinate to proof bundles, verified appends, ownership appends, and settlement rows.

Violation: webhook receipt alone is treated as final proof truth without verifying the underlying primitive.

## 9. SDK Version Coherence Is Release Truth

Publishable SDK changes must move package version, exported constants, docs, app release labels, and release docs together.

Violation: SDK package version advances without aligned app/release docs, or app release docs claim SDK primitives not present in the package.

## 10. Offline Verification Remains File-Authoritative

The offline verifier release surface can document SDK behavior, but verification authority remains in selected file bytes and embedded proof material.

Violation: release metadata, SDK docs, service-worker cache state, or external registry state is presented as stronger than file proof.

## Release-Lock Requirement

`pnpm release:lock 93.2.0` must require this document for `v93.2.0` and later release cuts. A release that omits the invariant register is incomplete because SDK convenience boundaries, runtime schema mechanics, deterministic proof projections, append-only SDK proof memory, verified-addition heads, webhook helper limits, version coherence, and offline file authority would not be explicit release truth.
