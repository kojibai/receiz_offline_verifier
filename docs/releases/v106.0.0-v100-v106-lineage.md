# Receiz v100.0.0–v106.0.0 Release Lineage

This record is the inherited product-truth chain for v106. It does not reinterpret prior releases. Each release remains governed by its own product-truth, process, regression, checklist, and commit-history documents. Across the entire line, durable proof memory is **first admission only, then append forever**.

## v100.0.0 — The Living Proof Release

v100 joined sealed Twin identity, deterministic living proof media, existing-object continuity, Profile and Market continuity, Sports proof recovery, and authenticated `@receiz/sdk` application-state, custody, asset-admission, escrow, and event-proof rails. Proof objects became capable of living, moving, acting, and composing without giving atlas, UI, routes, databases, sessions, SDK, MCP, or transport authority over their sealed identity. Known proof painted first; Twin atlas backfill remained a post-paint verified append.

Inherited into v106:

- sealed proof and proof-bound Twin identity outrank projections;
- custody, ownership, settlement, and verification stay explicit;
- Kai/Klok remains proof-object state and Chronos remains bridge/display time;
- deterministic first paint cannot be traded for rediscovery.

## v101.0.0 — Portable Asset Continuity

v101 closed manifest-only admission. New SDK asset admission required a complete sealed proof object whose existing Receiz verifier proved canonical payload, ownership, provenance/appends, and settlement continuity. The client sealed, reopened, and verified the file before authenticated admission; the server repeated verification before append. A self-hashable manifest was never sufficient authority.

Inherited into v106:

- `assets.createProofObject(...)` is the safe creation rail;
- complete enclosing artifact proof outranks embedded manifest hashes;
- historical artifacts retain their existing verification behavior;
- route or database success cannot invent ownership.

## v102.0.0 — Indivisible Verification And Continuity

v102 made a verified verdict indivisible from artifact integrity and ownership continuity across Receiz.com, offline verifiers, SDK, Connect/API, admission, World packages, and MCP language. It removed generic SDK seal-as-proof entry points, made public proof lookup resolution rather than verification, and prevented integrity-only seals from becoming proof-object verdicts.

Inherited into v106:

- integrity alone is not verified ownership continuity;
- MCP lookup or inspection is not artifact verification;
- AI, metadata, manifests, APIs, and databases remain beneath the enclosing proof object;
- SDK, MCP, and AI skill versions move as a coordinated set.

## v103.0.0 — Native Record And Seal Continuity Correction

v103 corrected v102’s parallel-carrier regression. Native Profile Composer, Record, and Seal proof objects again verified through their existing sealed proof bundle and canonical coordinates. SDK creation resolved the authenticated account’s real Receiz ID, created native Record before Seal, and removed developer-supplied owner or parallel continuity authority.

Inherited into v106:

- native Record before Seal is mandatory;
- the real account Receiz ID owns continuity;
- no new wrapper, carrier, issuer, or registry may replace an implemented primitive;
- correction contracts preserve working historical proof instead of ratifying a regression.

## v104.0.0 — Application Contract Compiler

v104 introduced the typed application contract compiler. The SDK compiled deterministic scopes, environments, framework wiring, findings, safe change previews, conformance, and upgrade guidance. CLI and MCP remained thin interfaces; inspection was not verification; writes required the exact reviewed preview digest and stayed repository-root confined. Packed packages used exact registry versions with no `workspace:` leakage.

Inherited into v106:

- the SDK owns compilation and product-law machinery;
- MCP plans and previews are lower-authority interfaces;
- automatic change requires exact digest confirmation;
- witnessed history is never rewritten; missing truth appends and projections rebuild from stronger truth.

## v105.0.0 — Browser-Safe SDK And Conformance

v105 separated the browser-safe runtime, React entry point, Node-only compiler, and browser-safe deterministic testing emulator. It added stable errors, capability discovery, protected code generation, expanded conformance, semantic MCP operations, and externally installable coordinated packages. Emulator output stayed sandbox evidence and never became a production verdict.

Inherited into v106:

- browser graphs contain no Node-only authority or dependencies;
- compiler and testing boundaries remain explicit;
- conformance is deterministic, local, and performs zero network/database calls;
- errors, capabilities, package exports, and packed installation are public contracts.

## v106.0.0 — Constitutional Application Platform

v106 makes those inherited boundaries executable as one enforcement chain:

1. `@receiz/ai-skills` loads the exact registry digest, writes an implementation contract, and refuses completion without required evidence.
2. `@receiz/mcp-server` injects laws, ruleset, state machine, authority, allowed commands, forbidden mutations, and verification requirements; it exposes law-aware semantic operations and digest-confirmed plan/permit/execute.
3. `@receiz/sdk` compiles laws, verifies capabilities, admits commands, commits atomically, emits signed receipts, preserves causal history, simulates and replays deterministically, and migrates forward without rewriting proof history.
4. The private independent verifier replays the resulting history outside the coding agent’s control.
5. The release lock rejects package, ruleset, registry, skill, migration, conformance, performance, governance, or visible-continuity drift.

v106 also corrects the owner-local profile regression: finding a canonical global row no longer short-circuits local proof admission. The newest published proof object is merged with accepted PBI authorship, appended to owned authority, written to deterministic profile and Live Player snapshots, and retained against older snapshot rollback.

That owner-local Showcase/PBI/Live Player correction is v106's only new main-application capability. The rest of v106 is the coordinated SDK, MCP server, independent verifier, AI-skill, migration, and release-enforcement system; it does not add features to or redesign other application surfaces. The Sports safe-area CSS correction restores an inherited continuity contract rather than defining new v106 product behavior.

## The Complete Inherited Contract

- Sealed artifact truth remains first.
- Verified local truth is durable truth, not cache.
- Native Record precedes Seal and binds the real Receiz ID.
- Ownership, custody, settlement, provenance, verification, and public witness remain named primitives.
- Integrity-only payloads and self-hashes cannot become authority.
- Commands are the only mutation rail; direct state writes and authority bypass fail closed.
- Causal history appends; replay is deterministic; migration is forward-only.
- MCP and AI guide and automate safe mechanics but never become proof authority.
- Independent evidence, not an agent’s assertion, qualifies the release.
- Known verified truth paints immediately; background work only appends.
- Showcase, PBI signing, and Live Player project the same newest admitted proof object.

This is the v106 release standard: every inherited primitive remains direct, every authority boundary is executable, every refusal is explainable, every package is digest-aligned, and no contract may be changed to bless weaker behavior.
