# Receiz v60.1.0 Product Truth

Release date: April 18, 2026
Status: Locked for `v60.1.0` release prep

## Purpose

This document defines the product-truth freeze for Receiz `v60.1.0`. It carries forward the `v60.0.0` baseline while recording holder-aware public market witness truth, tighter compact/public market presentation, hidden-only preload behavior for home and profile twin shells, lighter public-profile route assembly with preserved SSR truth, deterministic market sigil hydration, and release-surface lockstep as explicit non-regression behavior at release time.

For this offline verifier repository, the release is represented as current-release documentation, canonical release artifacts, shipped HTML version markers, and package-metadata alignment only. The verifier trust contract remains unchanged relative to `v60.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- File-backed proof remains the authority for artifact truth.
- Market witness truth for public asset markets must prefer real holder positions and buy activity over creator-side timestamps or decorative fallback clocks when that holder-side truth is already knowable.
- Once a visible home or profile shell has settled, that settled shell remains the authority until explicit open intent chooses a richer surface.
- SSR-rendered public profile truth remains the authority for initial route truth even if heavier client modules warm later in the background.
- Stable surface-scoped sigil identity is part of hydration truth, not incidental render noise.
- Public product routes, release metadata, and offline verifier entrypoints remain operating or governance surfaces around those authorities, not replacements for them.

## Locked Release Surface

### 1. The v60.0.0 Baseline Must Remain The Floor

- Release-governed version surfaces must remain lockstep and current-release docs must stay aligned.
- Live front-door proof/twin presentation, canonical original upload/download authority, explicit twin portrait publishing, and compact wallet/certificate framing from `v60.0.0` remain in force.
- Already-known stronger route truth must continue to win over weaker anonymous or late-arriving fallback state.
- The offline verifier trust contract must remain file-authoritative, deterministic, and fail-closed.

### 2. Public Market Witness Truth Must Be Holder-Aware

- Public profile asset holder count must derive from live public asset positions with non-zero quantity.
- Latest witness timing must derive from real public buy activity, owner position open time, and matching share-trade history when those facts are available.
- Witness detail surfaces must prefer holder- or trade-side truth over creator seal time or post time when stronger holder evidence exists.
- Witnessed duration must not remain stranded at zero when the system already knows there was real purchase or holder activity.

### 3. Compact And Public Market Surfaces Must Stay Tight And Legible

- The market desk, chart dock, and public market metadata surfaces must stay dense and legible on compact devices.
- Signed-out and public-profile market headers, captions, and summary tiles must avoid wasted space without dropping market truth.
- Timestamp presentation, proof metadata tiles, artifact labeling, and UTC chronos lines must remain readable at narrow widths.
- Compact polish must not weaken market continuity, holder truth, or proof detail.

### 4. Twin Continuity Must Not Promote Visibly Without Intent

- The visible home and profile dock shell must stay on the settled shell while richer code and data warm in the background.
- Remembered twin session state must not auto-promote the full module on load.
- Home-route auth prewarming must stay hidden and must not disturb a settled front door.
- The richer twin module may only take over after explicit open intent.

### 5. Public Profile Assembly And Sigil Hydration Must Stay Deterministic

- `UserProfileRouteClient` may lazy-load the heavy client and trusted-stats merge path only if initial SSR route truth remains intact.
- `MarketValuePill` sigils must use stable surface-scoped IDs rather than hydration-order-dependent IDs.
- Stable sigil scope must remain threaded through profile, player, hero, and artifact/original-viewer surfaces.
- Copy around extra showcase hydration must remain honest about what has and has not fully hydrated after settlement.

### 6. Release-Governed Version Surfaces Must Move Together

- Release-governed version surfaces must move together to `v60.1.0` across package metadata, service-worker defaults, public verifier artifacts, public release badges, and current release documentation.
- Public-facing version truth must not drift between shipped HTML entrypoints, docs indexes, release artifacts, and repository metadata.
- This offline verifier repository must reflect that lockstep rule in `package.json`, `README.md`, `RELEASE_NOTES.md`, `CHANGELOG.md`, `docs/README.md`, `docs/governance/README.md`, `docs/FORMAT.md`, `site/index.html`, `apps/offline-verifier.html`, `apps/offline-record-seal.html`, and `apps/offline-settlement.html`.

### 7. Release Governance And Verifier Truth

- Release-scoped note, product-truth, and checklist documents are required for `v60.1.0`.
- Root package metadata must resolve to `v60.1.0`.
- Shipped verifier, studio, and settlement entrypoints must visibly display `v60.1.0`.
- No new database migrations are added beyond the `v60.0.0` baseline.
- The offline verifier trust contract remains unchanged relative to `v60.0.0`.

## Allowed Evolution

Implementation details may change after `v60.1.0` only if they preserve the authority boundaries and locked release behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v60.1.0` if it:

- lets public asset markets narrate witness truth from creator-side timestamps or decorative fallbacks when stronger holder or trade truth already exists
- regresses compact/public market surfaces so proof metadata, timestamps, or summary truth become harder to read on real device widths
- visually remounts or upgrades a settled home or profile shell before explicit user open intent
- weakens SSR route truth by making heavier public-profile client assembly the source of initial truth
- reintroduces hydration-order-dependent market sigil IDs and causes SVG namespace drift between server and client
- allows package metadata, public badges, shipped verifier artifacts, or current-release docs to advertise different release numbers
- changes release truth without updating the versioned release docs and aligned repository version surfaces
