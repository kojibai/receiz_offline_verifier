# Receiz v70.0.0 Growth From v60.0.0

Release date: April 24, 2026
Status: Locked for `v70.0.0` release prep

`v70.0.0` packages the work from `v60.0.0` through the final `v69.9.x` stabilization line into one governed public release. It does not redefine the product at the last second. It records the product system that now exists.

## Summary

Since `v60.0.0`, Receiz moved from a strong live proof product into a complete public operating surface:

- live proof front door
- holder-aware market truth
- historical continuity governance
- Settlement/Reserve value loop
- one-click Receiz ID login
- local-first/server-always recovery
- profile-law restoration
- conformance isolation
- custom-domain and in-app browser visual normalization
- truthful logout final state
- follow-login continuation
- route-preserving stale-runtime recovery
- proof-sealed world trail bundles
- `v70.0.0` invariant register

## Release-Line Growth

### v60 Baseline

`v60.0.0` made the live proof front door coherent: stronger canonical original upload/download authority, explicit twin portrait publishing, compact wallet and certificate framing, schema hardening, and release-surface lockstep.

### Continuity And Market Proof

The release line then hardened already-rendered truth. Public profile asset markets became holder-aware, market witness timing moved to real holder activity, profile-route assembly became lighter, and deterministic market sigils stopped drifting between hydration passes.

### Historical Continuity Governance

Pre-v4 and historical proof moved under governed continuity. Historical records could be explained and carried forward without being silently recast as current verified proof or rewritten for convenience.

### Complete Value Loop

The value loop became a single causal system: Settlement as proof-native liquid value, funded Reserve as external-conversion value, Reserve-only notes, Reserve-funded wire transfer, Reserve-first sends and buys, funded/unfunded market sale splits, and buyer-funded certificates.

### Local Proof Continuity

Receiz then made local proof continuity explicit: identity and proof held by the node are authoritative, deterministic state stays visible, verified snapshots become offline truth carriers, and sync propagates stronger known truth instead of making local truth wait for the network.

### Receiz ID And Profile Law

The final stabilization line restored everyday identity and profile law:

- Receiz ID became the one-click everyday login rail.
- `/signin` became a pass-through route.
- Cold devices could create a local Receiz ID and continue directly to the managed profile.
- PBI, keyfile, email, and magic-link rails stayed available as recovery and interoperability.
- `/<username>` became canonical profile truth.
- `/<username>?manage=1` became owner powers layered onto the same profile truth.
- Owner-only data could append controls, but not rewrite the public shell.

### Runtime And Offline Resilience

Runtime recovery moved from generic fallback toward route-preserving recovery. Stale runtime, stale service-worker, and stale session states recover through local proof or server resync. Hard runtime recovery returns to the route the user was on instead of dumping the user to home.

### World, Twin, And Proof Bundles

World and live-player identity became stricter. Live twins cannot fall back to profile pictures as identity substitutes, twin images stay locked to current per-user/per-twin images, and mind trail ZIP downloads package files with Receiz proof material so the downloaded bundle verifies as a proof bundle.
