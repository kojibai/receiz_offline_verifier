# Receiz v47.2.0 Product Truth

Release date: April 9, 2026
Status: Locked for `v47.2.0` release prep

## Purpose

This document defines the patch-level product-truth freeze for Receiz `v47.2.0`. It preserves the current `v47.x` product truth while recording the custom-domain routing and setup-surface fixes that are now treated as explicit non-regression behavior at release time.

For this offline verifier repository, the patch is represented as current-release documentation and version-surface alignment only. The verifier trust contract remains unchanged relative to `v47.0.0`.

## Core Authority Boundaries

The release does not relax the core Receiz authority model:

- The file remains the authority for artifact truth.
- Deterministic manifests remain the authority for historical route truth.
- Public readers, verified-player surfaces, storage proxies, world embodiment, custom-domain routing, and release metadata remain transport and governance around those boundaries, not replacements for them.

## Locked Patch Surface

### 1. Effective Forwarded Host Determines Custom-Domain Root Routing

Root routing for a custom domain remains part of the governed product surface.

- Root resolution must use the effective forwarded host, not only the internal request hostname.
- Proxy-forwarded custom hostname traffic must rewrite `https://custom.host/` to the bound `/_business-domain/...` profile route.
- Connected custom-domain requests must not fall through to the Receiz 404 surface when the domain binding is already authoritative.

### 2. Bound Custom Domains Resolve At `/`

Connected Business domains must remain usable from the root path.

- A connected Business custom domain must land visitors on the bound public profile at `/`.
- The root-path experience must not require the `/<username>` fallback when the custom-domain binding is authoritative.
- Existing attached DNS-live domains must continue to work without new registrar changes for this patch.

### 3. DNS Setup Execution Is Copy-Safe

The `/account` setup flow must remain easier to execute directly.

- DNS setup guidance must expose direct copy controls for each record name and record value.
- Copyable DNS fields must reduce registrar copy/paste mistakes without changing provider behavior.
- Visual refinements may evolve only if the direct-execution copy affordances remain intact.

### 4. Release Governance and Version Truth

The release contract remains locked across visible version surfaces.

- Release-scoped note and product-truth documents are required for the patch.
- Release-governed version surfaces must move together across package metadata, service worker defaults, public badges, and release documents.
- This repository records the public release documents and shipped HTML version markers for that lockstep requirement.

## Allowed Evolution

Implementation details may change after `v47.2.0` only if they preserve the authority boundaries and locked patch behaviors above. If a future change materially alters one of these truths, it should ship under a new explicit release note and updated product-truth freeze.

## Non-Regression Rule

A change is out of bounds for `v47.2.0` if it:

- resolves custom-domain root traffic from the wrong host context and falls through to the 404 surface
- requires the `/<username>` fallback for already-bound, DNS-live Business custom domains
- removes direct copy access to DNS record names or values in `/account`
- changes release truth without updating the versioned release docs and aligned version surfaces
