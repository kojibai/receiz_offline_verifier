1. Receiz remains a proof-native artifact system. Product truth is carried by files, identities, vault objects, ownership states, settlement states, public witness surfaces, verification routes, canonical receipts, proof heads, and append-only history.
2. 3. Kai Klok remains the proof-object state machine. `kaiPulseEternal` / `kai_upulse` remains the pulse unit for complete proof objects.
4. Durable proof memory remains first admission only, then append forever. Hidden work may only append verified additions after the known head.
5. Deterministic first paint projects known proof, identity, ownership, settlement, profile, market, ledger, transaction, Sports, public graph, portable state, and archive truth before weaker work.
6. The Receiz key remains the identity. A username is a mutable projection, and a rename preserves the same key while releasing the prior username only after canonical commitment.
7. Username availability is advisory; commit-time global uniqueness decides the winner and cannot be replaced by reservation state.
8. Profile text, username, display name, raw or prepublished avatar proof, and proof head commit atomically; public projection changes only after verified canonical receipt and matching head.
9. Receiz Keys, Identity Records, Identity Seals, and verified Vault enclosures carry portable identity and namespace-isolated app state that projects locally before network discovery.
10. Generic bearer ownership verifies the enclosing qualifying artifact before embedded material, verifies claimant key control and the expected ownership head, and seals one committed ownership append before ending the prior owner projection.
11. Offline commands are signed proposals. `status: "queued"` always carries `globallyCommitted: false`; only a verified canonical receipt can establish a committed append.
12. Offline queues preserve FIFO causality per proof object, permit independent lanes, stop conflicted lanes, and never reorder witnessed history.
13. MCP planning, AI-skill guidance, generator preview, emulator simulation, migration verification, cache retention, and route success never become admission, ownership, settlement, or verification authority.
14. Emulator results remain `emulator_simulation` with `productionVerified: false`; generated files remain root-contained, digest-confirmed, and protective of developer-owned extensions.
15. Forward-only v106-to-v107 migration preserves sealed artifacts, receipts, heads, verified local projections, and witnessed history byte-for-byte; pending operations remain signed uncommitted proposals.
16. SDK, MCP, AI skills, frontend, runtime, service worker, verifier, registry, capability ranges, manifests, generator, migration, and release records align exactly at `107.0.0` with registry digest `4d0caa6172a69c3bf5817c1c35db5630e555b5d6d824091d45a90fb426b86ef6`.
