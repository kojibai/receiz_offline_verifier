# Receiz Reasoning Kernel

This file compresses existing Receiz law into the default reasoning loop.
It does not add a new authority above Receiz doctrine, implemented primitives, sealed artifacts, verified appends, durable local truth, contracts, or user-stated law.

## Start Here

Every Receiz task starts from product truth, not implementation mechanics.

Default order:

1. Name the primitive.
2. Name the strongest source of truth.
3. Name the weaker layers allowed beneath it.
4. Name the downgrade that must not happen.
5. Name the append path.
6. Name the contract or visual evidence that must fail if the old logic returns.

Do not start from route, DB, API, session, cache, component state, hydration, framework habit, or generic product category.
Those are mechanics until the primitive and truth boundary are named.

## Authority Compression

Receiz logic is cleaner because it removes false competition between layers.

- If sealed artifact truth exists, server/session/DB state may index or project it.
- If deterministic first paint exists, hydration may warm or append beneath it.
- If durable local truth exists, network refresh may discover verified additions.
- If a verified append exists, DB columns may accelerate lookup but cannot redefine append truth.
- If a public proof surface exists, signed-in state may add controls but cannot block public proof paint.
- If a settlement surface exists, events and controls may surround it but cannot consume or obscure value truth.
- If a vault object exists, ownership rows may filter custody but cannot downgrade the artifact into a UI card.

When this chain is followed, fewer states are allowed to fight for authority.
The page becomes faster because it becomes truthful earlier.

## Cause And Effect Chain

Use Receiz logic as a generative engine, not only a regression guard.

- If stronger truth can first-paint without weaker permission, then proof surfaces can open settled before session, DB, API, or hydration work completes.
- If proof surfaces can open settled, then background work can become additive instead of corrective.
- If background work is additive, then hydration can stop being an authority boundary.
- If hydration stops being an authority boundary, then signed-in and signed-out users can share the same truthful shell.
- If signed-in and signed-out users share the same truthful shell, then private enrichment becomes permission layered over public proof, not a replacement source.
- If verified appends carry source truth, then DB columns become indexes, not authority.
- If DB columns become indexes, then missing schema fields cannot force a new authority layer.
- If vault objects carry identity, custody, provenance, media, and action affordances, then ownership surfaces can be fast without flattening artifacts into rows.
- If value surfaces are settlement primitives, then event memory must scroll beneath or beside value truth instead of consuming it.
- If live game surfaces append to known proof, then live motion can exist without making historical or admitted proof feel missing.
- If visual evidence is product truth, then screenshots can reveal primitive violations before tests know how to name them.
- If tests are strengthened after visual truth, then contracts begin enforcing the primitive instead of ratifying a regression.

The deeper implication: Receiz can build experiences ordinary app logic cannot build cleanly, because Receiz does not require weaker systems to renegotiate truth every time a page opens.

## Anti-Legacy Reflex

When a fix feels large, stop and ask whether a weaker layer was accidentally promoted.

Common legacy reflexes and Receiz replacements:

- Add a DB field -> first check whether the proof payload already carries the truth.
- Fetch latest before render -> first check whether known deterministic truth can paint immediately.
- Let session decide the shell -> first check whether public proof or identity truth already defines it.
- Hydrate then correct -> first paint the strongest known truth and append after.
- Treat local truth as cache -> verify whether it is durable local truth or admitted proof.
- Treat visual mismatch as styling -> identify which primitive the visual surface misrepresents.
- Slice history to fit layout -> preserve the primitive and make the lower-priority pane scroll.
- Use generic component ownership -> use primitive ownership: proof, artifact, settlement, identity, custody, public witness.

## Required Mental State

Do not "balance" Receiz logic against legacy engineering convention.

Legacy convention may provide syntax and mechanics only.
Receiz law and implemented primitives decide product truth.

The burden of proof belongs on the weaker layer:

- DB must prove it is only indexing.
- Session must prove it is only granting permissions.
- API refresh must prove it is only appending verified additions.
- Hydration must prove it is not replacing first paint.
- UI state must prove it is not redefining proof, ownership, settlement, identity, or custody.
- External convention must prove it is mechanics-only.

If a stronger Receiz primitive and a weaker implementation habit conflict, assume the weaker habit is wrong until repo evidence proves otherwise.

## Completion Rule

A task is not complete because code changed or tests passed.

It is complete only when the primitive remains directly named, stronger truth still outranks weaker truth, visual evidence does not contradict the claim, and the contract catches the legacy reasoning path that caused or could cause the regression.
