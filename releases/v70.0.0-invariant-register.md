# Receiz v70.0.0 Invariant Register

Status: v70 release-prep register. This document does not bump the app version, cut the release, or declare v70 shipped.

Authority order:

1. Explicit user law.
2. Doctrine documents in `docs/`.
3. Historical baseline release truth.
4. Release-blocking contract tests.
5. Current code.

If code, tests, and this register disagree, compare them to the higher authority first. Do not weaken a contract to preserve a regression.

## Purpose

This register pulls the solid primitives that were previously scattered across doctrine, release notes, route contracts, and implementation details into one release-blocking map.

Every primitive below is release-blocking for v70. A future change may append stronger law, but it must not silently remove or weaken any row.

## Previously Implicit Invariants Now Locked

- Explicit logout preserves Receiz ID but suppresses automatic session projection until the user takes a new explicit action.
- Logout recovery prepares PBI/Receiz binding when possible, but keyfile download is presented as a final logged-out choice.
- Profile follow can be login, account creation or resume, and follow in one press without route navigation or reload.
- Custom-domain and local conformance checks default to repo-local origins unless an override is explicit.
- GTM is not a reserved future permission. If it is not intentionally enabled, it does not execute.
- Stripe cannot become generalized identity, analytics, settlement, or permission infrastructure.
- Market predictions are not a separate UX law from buy/sell; higher/lower must obey the same continuation and value rules.
- Wallet no-funds guidance cannot trap the user in an unhideable add-funds panel.
- Kai-Klok/KKS v1 deterministic state-machine logic is canonical and frozen; styling or placement may wrap it, but the logic itself must not be edited.
