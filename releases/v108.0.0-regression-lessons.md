# Receiz v108.0.0 Regression Lessons

## What Regressed

Commit `9611cb8f57c587eae1b6ee8079ee6d7bd2a02637` replaced the long-standing first-party profile calls with a developer-operation client. Before the save, that client called `/api/receiz/identity/profile`; the projection selected a latest `identity_keys` row and mapped missing or failed projection reads to `identity_key_mismatch`.

That changed the identity authority boundary. An already-admitted account could be denied by a weaker database projection, and a developer receipt/head protocol became a prerequisite for an established Receiz.com action. The exact unaffected behavior is preserved at parent commit `a1081796e90a65124d952ce5907512999dd16cb0`.

The swallowed production error does not prove that a SQL migration was missing or unapplied. A missing table or projection could amplify the regressed developer path, but no migration may be used to make that path a prerequisite. The restored first-party operation is required to save with zero `identity_keys` rows and introduces no new profile-authority migration.

## Why It Regressed

V107 developer operations were treated as if a coordinated SDK/MCP/AI operation language had authority to redefine Receiz.com. It did not. V107's lawful scope was additive developer rails and compatibility mechanics beneath implemented Receiz primitives.

The same inversion appeared in related assumptions:

- recency ordering was used to infer the active identity key;
- every projection failure was softened into one false mismatch label;
- a registry row was allowed to imply verified portable state;
- proof heads and receipts were treated as preconditions instead of additive developer evidence;
- a developer command queue leaked toward first-party runtime and service-worker behavior;
- version-branded developer presentation obscured the unversioned implemented law.

## Locked Rules

1. Proof is first. Receiz.com is a reference implementation beneath proof, not proof authority.
2. An accepted identity proof and admitted same-account binding cannot be re-decided by an `identity_keys` row.
3. First-party profile UI uses `PATCH /api/profile/me`; SDK/MCP/AI adapt to that operation and never become its prerequisite.
4. A rename changes the mutable username projection, not the account actor. The account UID remains identical.
5. Availability is advisory. Global uniqueness is coordinated at commit, and the old username is released only after success.
6. A missing key row is `identity_projection_unavailable`; service failure is `profile_service_unavailable`; mismatch requires explicit contradictory evidence.
7. Retired v107 key-selection behavior is historical evidence only. The current profile operation accepts authenticated profile intent and never selects an identity key.
8. A registry row is a projection, not a portable proof. Verified portable state requires the existing artifact or admitted-block verifier.
9. Retired v107 heads and receipts are historical witnesses only. They do not qualify, authorize, or describe a current v108 profile or ownership operation.
10. Historical schema names and witnessed artifacts remain preserved without becoming current capability inventory or authority.
11. A developer queue belongs in developer tooling. It cannot remount or redefine an established first-party service worker, first paint, or PWA runtime.
12. `/developers` states current implemented law without turning that law into a version-branded product page.
13. Real Kai-Klok seal time is computed from the sealing instant. A release label is not an effective Kai pulse.
14. Passing tests cannot overrule a failed mutation recording, changed UID, developer-endpoint network request, or contradictory screenshot.

## Contract Gates

V108 locks the exact profile client hashes, scans first-party source for forbidden developer dependencies, tests same-UID projection, exercises zero-key-row behavior, verifies the neutral SDK/MCP adapter composition, validates every AI manifest and skill boundary, recomputes the registry pulse and digest, and requires desktop/mobile visual evidence.

Historical v107 gates preserve witnessed history only. They are not current defaults and cannot require retired key, head, receipt, claimant, idempotency, or RPC behavior. The full release freeze combines current correction gates with historical evidence checks so a test cannot be changed to bless a visible or architectural downgrade.

## Durable rule

Receiz proof memory remains first admission only, then append forever. A weaker projection may append or display beneath accepted proof; it cannot make admitted identity untrue.

`@receiz/sdk` adapts to this law; it cannot redefine it.

A release that omits these lessons is incomplete because a database key projection, developer endpoint, SDK/MCP/AI rail, fabricated head/receipt prerequisite, developer queue, versioned presentation, or release label could again be allowed to redefine admitted identity, first-party behavior, proof authority, or real Kai-Klok time.
