# Receiz v112.0.0 Invariant Register

`v112.0.0` locks the executable artifact authority hierarchy. Runtime custody protects accepted inputs but never outranks exact verified sealed bytes and independently verified history.

## Locked Invariants

1. Exact verified sealed bytes establish artifact truth above admission, plans, capabilities, staging references, receipts, storage, sessions, servers, databases, SDK/MCP/AI projections, and UI state.
2. Admission establishes profile membership only; it never grants signing, transition, publishing, ownership, settlement, or commit authority.
3. Verified identity admission supplies actor evidence, and verified history supplies predecessor truth. Structural objects and copied runtime shapes cannot substitute for either.
4. Portable transition identity binds successor meaning while excluding idempotency and external effects.
5. The operation plan binds actor, stable resource, current registry, registry-derived effects, idempotency identity, and one named commit domain.
6. A verified capability authorizes exactly that plan within actor, tenant, operation, resource, registry, effects, Kai validity, algorithm, key, and revocation bounds.
7. Ed25519 and P-256 Identity Seals sign capability payloads without elevating key wrappers or signature transport into identity truth.
8. Neutral staging preserves exact sealed candidate bytes durably and advances no accepted head.
9. Commit independently resolves and verifies staged bytes before accepting history or advancing a head.
10. One named commit domain atomically changes head, event, receipt, derived effects, idempotency, and outbox state—or changes nothing.
11. A receipt reports accepted artifact, transition, actor evidence, history, effects, and head; it is never artifact, actor, history, plan, capability, or commit authority.
12. Unknown namespaces survive byte projection and transition. Known namespaces change only through registry-authorized reducers.
13. Committed history derives from the independently accepted artifact, never from a plan, caller report, receipt, cache, locator, or database projection.
14. Durable browser admission state is not cache, but its IndexedDB ledger stores neither proof objects nor sealed artifact bytes.
15. Known exact artifact truth paints first; hydration and hidden work may append only verified additions.
16. The current deterministic unsigned registry contains 57 laws, is recursively frozen in the SDK, and is available through a detached loader.
17. Registry digest `1356f8122d0b5fcbe891d7e6ed1e75faca827f15d63d1ed5d950664e11c146ee` binds predecessor `cf02d0bce6ad1541cfe84e27bfb1036777b29616bf8a1e5aeafb899a945e359a` and canonical effective Kai `13192000`.
18. The registry payload is executable law metadata beneath sealed proof truth; it does not become proof-object authority. No native `.receiz` or signed wrapper is required or claimed.
19. v111 exact sealed bytes remain verifiable and eligible for current profile admission after v112 verification/admission.
20. v111 runtime admissions, verified-history objects, plans, capabilities, confirmations, receipts, store references, and projections have no v112 mutation authority.
21. MCP exposes only the five typed v112 artifact transition tools and remains orchestration beneath SDK admission and sealed artifact truth.
22. Remote global admission and offline-to-global reconciliation are outside v112.0 and remain absent.
23. No product HTTP route, application behavior, database schema, migration, query path, production datum, or native issuance behavior changes in this release.
24. Current package metadata, documentation, verifier labels, service worker, and release records align at `112.0.0`.
