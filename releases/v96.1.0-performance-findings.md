# Receiz v96.1.0 Performance Findings

Release date: 2026-06-30

`v96.1.0` keeps the `v96.0.0` SDK evidence law intact and extends it to Twin and World developer integrations. Typed SDK clients can reduce integration work only if they preserve proof-native order: render known verified truth, keep Connect delegated, and append verified additions beneath durable proof memory.

## Findings

- Twin SDK clients improve developer speed by wrapping existing delegated Twin routes instead of creating a second identity or authority model.
- World SDK clients improve public integration speed by exposing existing World/Twin projection routes without making public reads a proof issuer.
- Connect scopes `receiz:twin.read` and `receiz:twin.write` make delegated capability explicit and bounded for third-party apps.
- OpenAPI and status catalog entries are discovery and contract evidence. They do not outrank route contracts, proof objects, identity proofs, or verified appends.
- Identity conformance visibility is useful only when it records the new Connect/Twin surface as a public developer contract beneath Receiz proof authority.
- Version-locking the app and SDK prevents developer integrations from guessing whether Twin and World clients belong to the shipped product.

## Preserved Rules

- First paint must project known proof truth before hydration, analytics, SEO, session checks, lazy imports, global sync, server reconciliation, or SDK convenience work.
- Hidden work may append verified additions only.
- Connect grants delegated route access only; it must not become an identity proof root.
- Twin and World SDK helpers must preserve existing server route boundaries.
- OpenAPI/status docs must not become truth authority above implemented primitives.
- SDK package evidence must remain release-blocking when developer-facing primitives change.
