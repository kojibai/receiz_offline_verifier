# Receiz v73.0.0

## Goal

Ship a major release lock for the ownership, vault, account, profile, receipt, and world twin work accumulated after the `v72.0.0` stable continuity baseline, while preserving the public alpha product law from `v70.0.0`.

## What Worked

- Measuring the release from commit `61ae0ef2b` made the v72-to-v73 diff explicit even though this checkout has no `v72.0.0` tag.
- Business account ownership fit the existing account server model when ownership, holdings, and certificate routes were treated as account truth instead of UI decoration.
- Receipt ownership provenance worked as an append around existing artifact truth, which preserved route identity and download verification.
- Vault portability improvements aligned with verified-history law because decoded records stay portable across verifier, download, and account surfaces.
- Account brain proof-bundle detail stayed safe when it was treated as selected-node data only: no first-paint panel, no automatic first-node takeover, and no Open action unless a real route target exists.
- World twin fit improvements stayed manageable when face features became explicit authority with contracts instead of broad viewport offsets.

## What Did Not Work

- A label-only release bump would hide a large post-v72 product diff, so the release docs must state the baseline and changed product areas directly.
- Account and profile UI growth can easily become visible churn unless it remains one-viewport, actor-scoped, and first-paint truthful.
- Ownership and provenance flows can regress if transfer state is allowed to replace the original receipt or artifact identity.
- Vault verifier improvements can regress if transport state is treated as stronger than verified portable record truth.

## Practical Rule For Future Work

When Receiz adds richer ownership, vault, account, or twin capabilities, classify each new payload before it reaches the visible shell: canonical truth first, verified additions second, augmentation third, and experience state last. Richer UI may append controls and context, but it must not redefine settled truth or hide release version drift.
