# Receiz v55.0.0 Release Lock Checklist

Release date target: April 15, 2026

## Purpose

This checklist locks `v55.0.0` to one exact release state without allowing regression in guided owner setup, sealed profile media, world avatar freshness, stable market/profile/player navigation, canonical performance route analytics, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v55.0.0-product-truth.md`

Run from the exact release commit only:


## Production Environment + Governance Preflight

The production database must already include the active market/world rails from earlier releases.

For the profile guided-setup path present in this release-prep state, confirm the target environment also includes:

- `supabase/migrations/20260415120000_profile_guided_setup_completion.sql`

Before release, confirm:

- `RECEIZ_SIGNING_KEY` is present so profile avatar and cover uploads can mint sealed originals
- the profile media path can create or update the public `receiz-profile-media` bucket
- historical `perf_web_vitals.route_group` values are backfilled with `pnpm perf:backfill-route-groups` if admin analytics need canonical route history for older rows
- `public/governance/governance-keyring.v1.json` still contains the active release-attestation key ID if release attestation is generated for this cut
- public/offline verifier surfaces, service worker version, and public version badges all show `v55.0.0`

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- `/profile` can open from a primed fast-entry path without losing the signed-in workspace shell
- the owner setup checklist stays complete after refresh once username, twin, cover, proof, and link steps are all finished
- the inline username editor can change the username cleanly and move the route to the new canonical profile path
- choosing a twin source photo from `/profile` opens World directly in the avatar forge with that file staged
- avatar and cover uploads update the live profile and still mint a sealed original with a working verify path
- proof manifests show a usable media SHA-256 even when the original row needed the derived-hash fallback
- world/profile/public-twin surfaces all prefer the freshest saved avatar render after avatar generation
- compact touch devices do not background-load the full world module before explicit intent
- the bottom bar exposes Market, and profile navigation from `/market` stays on app navigation instead of a browser-level reload
- Viewer opens from market immediately from the local seeded handoff while the authoritative player fetch settles
- admin performance analytics aggregate canonical route groups instead of raw user-specific URLs
- public version badges, service worker version, offline verifier surfaces, and release docs all show `v55.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v55.0.0`.
2. Run the `Release Governance Attestation` workflow if the release process for this cut still requires a signed attestation.
3. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
4. Confirm the attestation references the published key ID and the expected release version.
5. Store the attestation path and approval reference with the release record.

## Release Rule

If any guided-setup regression, profile-media sealing regression, avatar-freshness regression, world-hydration budgeting regression, market-route continuity regression, performance-route canonicalization regression, version-surface mismatch, or release-lock failure occurs, `v55.0.0` is not released from that commit.
