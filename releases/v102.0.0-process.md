## Goal

Make it impossible for developers, agents, Receiz.com, or an offline verifier to create or receive a verified proof-object verdict unless real Receiz ownership continuity existed before sealing and verifies from the artifact.

`@receiz/sdk` composes this boundary, while durable proof memory remains **first admission only, then append forever**.

## What Worked

- Reusing the existing portable asset and ownership-provenance carriers preserved the implemented Record, Seal, profile composition, and card continuity model.
- Moving the SDK proof-object operation behind authenticated profile-backed ownership closed arbitrary owner and UID fallback paths before bytes were produced.
- Hard-gating the shared artifact verifier propagated one meaning of verified through the API, Connect, SDK, web verifier, and downloadable offline verifiers.
- Keeping integrity diagnostics separate preserved useful local byte checks without letting them become proof-object authority.
- The full release freeze exposed a World package that still labeled integrity-sealed members verified; binding the package root to the real profile/action-ledger continuity closed it.
- Auditing MCP terminology exposed shape validation and public-proof resolution masquerading as verification; v102 removes those tool names.
- Packaging the authoritative `@receiz/ai-skills@102.0.0` bundle as an exact-version SDK and MCP dependency keeps installed agent doctrine aligned with the enforced rails.

## What Did Not Work

- v101 allowed the general verifier to succeed before complete continuity was considered at admission.
- Generic SDK sealing let an application create signed bytes without authenticated Receiz ownership continuity.
- Offline verifier branches accepted trusted integrity when continuity was absent.
- Agent-token and signer enrollment paths could derive or omit ownership without proving a real profile-backed Receiz ID.
- MCP returned `ok: true` from manifest shape validation and called public-proof resolution verification.
- The first v102 freeze run stopped on the World trail-package contract because that package still expected integrity-only members to verify independently.

## Practical Rule For Future Work

1. Resolve a real profile-backed Receiz owner first.
2. Establish deterministic identity, profile namespace, and prior head or valid genesis before sealing.
3. Seal the enclosing artifact around that existing continuity.
4. Verify enclosing integrity and continuity together through the shared verifier.
5. Persist or mirror only the already-verified projection beneath the artifact.
6. Name lookup, parsing, projection, and integrity inspection directly; reserve verified for the indivisible proof-object verdict.
7. Move app, SDK, MCP, AI skills, runtime, service worker, offline verifier labels, and release records together.

Promotion follows `docs/releases/v102.0.0-checklist.md`; no external action is complete until its command and read-back succeed.
