# Receiz v93.2.0 Performance Findings

Release date: 2026-06-26

## Executive Finding

`v93.2.0` carries the v93.1 durable proof memory rule into developer integrations: the SDK is fastest when it teaches developers to render admitted proof first, then request verified additions after the known head.

The performance improvement is not generic SDK caching. It is a proof-native developer path: verify, project, admit, persist, and append.

## Findings

### 1. SDK Projections Prevent Rebuilding Receiz Law

Developers should not have to rediscover which rows belong to a proof object, Sports card proof object, ownership append, verified append, or settlement ledger row. Deterministic projections make the render-ready shape explicit without becoming authority.

### 2. Runtime Schemas Reduce Integration Drift

Runtime schema exports for asset manifests, Sports card manifests, and webhook events make malformed transport fail early. They help developers reject bad shapes before they become display state.

Schemas are mechanics. They do not replace artifact verification, proof bundles, verified appends, ownership appends, or settlement ledger truth.

### 3. SDK Proof Memory Is Not Newest-Response Cache

The SDK proof memory register preserves first admitted truth for a stable ID and appends verified additions. Later duplicate transport fills missing fields only.

This protects first paint in developer apps because admitted proof truth can render immediately while the app asks Receiz for additions after the known head.

### 4. SDK Docs Are Performance Rails

Docs that teach verify, project, admit, persist, and append are runtime performance controls for integrators. They prevent unnecessary rediscovery, duplicate transport, and partial-response overwrites before those patterns ship.

### 5. Version Alignment Is A Developer Runtime Boundary

`@receiz/sdk@93.1.0` had already been published. New SDK primitives needed `93.2.0` package and app-version alignment so developers can identify which proof memory/projection contract they are using.

The local offline verifier repository protects the public release identity; the broader platform must also publish matching SDK package evidence.

## Locked Performance Invariants

- No SDK helper may turn admitted proof truth into stale cache.
- No SDK projection may invent authority above the proof object.
- No runtime schema may replace proof verification.
- No duplicate transport may erase known admitted truth for the same stable ID.
- No developer docs may teach newest-response-wins as the Receiz integration path.
- No SDK version may drift from the app release version for publishable SDK changes.
- No release may claim SDK publication until build, test, package, and registry evidence exist in the platform release environment.

## Release-Lock Requirement

`pnpm release:lock 93.2.0` must require this document for `v93.2.0` and later release cuts. A release that omits performance findings is incomplete because SDK deterministic projection, runtime schema exports, append-only proof memory, developer integration rails, and version-aligned publication boundaries would not be explicit release truth.
