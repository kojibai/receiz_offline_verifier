# Receiz v108.0.0 migration

V108 is a forward-only coordinated package and authority correction. It preserves sealed artifact bytes, verified local truth, carried continuity, and witnessed history while removing retired v107 logic from current application and developer surfaces.

## Upgrade coordinated packages

```sh
pnpm add @receiz/sdk@108.0.0
pnpm add @receiz/mcp-server@108.0.0
pnpm add @receiz/ai-skills@108.0.0
pnpm add @receiz/constitutional-verifier@108.0.0
```

The maintained application and package defaults are v108. Historical v107 source and witnessed release artifacts are not current capability inventory.

## Migrate profile updates

Old developer authority inputs must move to authenticated profile intent:

```ts
const updated = await receiz.profile.update({
  username: "new_name",
  displayName: "New Name",
});
```

The server derives the actor from the admitted session or scoped OIDC bearer, executes the same profile command as Receiz.com, coordinates uniqueness at commit, preserves the UID, and returns the refreshed projection.

First-party Receiz.com continues to use `PATCH /api/profile/me`. Do not add a developer preflight to that path.

## Migrate proof-object creation and opening

Call `assets.createProofObject()` with payload bytes and keep the returned complete sealed artifact. Creation parameters are standard transport mechanics beneath the artifact and do not supply proof, ownership, or continuity authority. Use `artifacts.download()` for exact artifact bytes and `artifacts.verifyAndOpen()` before reading a payload projection.

```ts
const sealed = await receiz.assets.createProofObject(createInput, createOptions);
/*
createInput = {
  assetType: "document",
  payload: { bytes, mimeType },
};
createOptions identifies the output filename and standard transport mechanics.
*/

const opened = await receiz.artifacts.verifyAndOpen(sealed.artifact);
```

Do not persist or pass the extracted payload as if it were the proof object.

## Migrate bearer ownership

Remove every caller-supplied ownership authority. Verify and open the complete artifact, then pass the runtime-issued artifact custody object:

```ts
const opened = await receiz.artifacts.verifyAndOpen(file);
const claimed = await receiz.ownership.claimBearerAsset({
  artifact: opened.sealedArtifact,
});
```

The server derives prior ownership from the verified carried document, binds the authenticated Receiz owner, appends continuity, and returns a newly verified complete Record -> Seal artifact.

## Migrate MCP callers

Keep the unversioned profile and bearer plan/execute tool names, but update inputs:

- profile plan: `{ profile }`;
- profile execute: exact local plan confirmation;
- bearer plan: complete artifact bytes plus filename and MIME type;
- bearer execute: exact local plan confirmation.

Current default MCP dispatch accepts only profile intent or complete artifact bytes with carrier metadata. Planning is confirmation mechanics, not proof authority.
