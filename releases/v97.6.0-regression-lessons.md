# Receiz v97.6.0 Regression Lessons

## What Regressed

- Merchant app developers could still infer that public-store sync required `RECEIZ_ACCESS_TOKEN` because the typed convenience method did not make signed merchant publish the obvious one-call path.
- Base Sports cards displayed `1:1` under the `ODDS` stat label in saved-card exports.
- The new release documents initially missed the exact release-lock headings.

## Why It Regressed

- The public-store rail already existed, but `@receiz/sdk` did not yet expose the no-env-token merchant proof publish recipe as a first-class helper.
- The base issue probability and rarity odds mechanics shared a display tile, so base probability inherited rarity language.
- The first draft of 97.6.0 release docs described the correct facts but did not match the repo's release-lock document contract.

## Locked Rules

- Merchant-owned public-store state must be publishable with Identity Seal / Receiz Key proof authority and no delegated bearer token.
- Durable proof memory remains first admission only, then append forever.
- Delegated tokens remain agent/server mechanics beneath proof authority.
- Base cards may show `1:1`, but the label is `BASE` / `Base issue`, not `ODDS`.
- Release docs must satisfy the release-lock contract before the umbrella release gate can proceed.

## Contract Gates

- `packages/receiz-sdk/test/sdk.test.ts` locks signed merchant public-store publish and token suppression.
- `app/lib/public-proof/__tests__/publicProofContract.test.ts` locks signed envelope verification and public-proof route/store behavior.
- `app/game/__tests__/sportsArenaRouteContract.test.ts` locks the base-card `BASE 1:1` saved-card language.
- `scripts/check_release_lock.mjs` locks release document shape and release identity.

## Delegated Tokens Are Not Merchant Proof Authority

The SDK already had public-store and app-state primitives, but developers could still reach for `RECEIZ_ACCESS_TOKEN` when publishing merchant-owned storefront state. `v97.6.0` closes that confusion by exposing `publishWithIdentityProof`, which signs with the merchant Identity Seal / Receiz Key and posts a signed envelope to the public proof rail.

Lesson: when a merchant proof object carries the authority, do not substitute a server token.

## Public Graph Must Be Proof Surface, Not Decoration

The proof graph can be visually rich, but it is still a public proof surface. The graph uses actual proof/release/API/product network data and a live `☤KAI` pulse. It is not a static marketing animation.

Lesson: visual impact is allowed when it names and preserves proof truth.

## Base Issue Is Not Rarity Odds

Base cards legitimately carry `1:1` issue probability, but labeling that as `ODDS` made the display sound like scarce rarity odds. `v97.6.0` uses `BASE 1:1` / `Base issue 1:1` for base cards and keeps `ODDS` for rarity cards.

Lesson: truthful copy can preserve exact data while correcting the primitive label.

## Service Worker Is Continuity, Not Authority

Service worker version alignment is required for release continuity, but the service worker remains a runtime projection layer. It can preserve and serve known truth; it cannot outrank proof objects, sealed artifacts, verified local truth, or verified appends.

Lesson: offline speed and continuity must reinforce proof law, not become a new authority layer.

## MCP Must Teach The Boundary

Agents need the MCP server to build advanced apps quickly. They also need the MCP docs and tool responses to name the authority boundary directly. MCP wraps SDK/API rails for agents; it does not create product truth.

Lesson: agent ergonomics must make lawful use easier than accidental authority drift.

## Release-Lock Requirement

`pnpm release:lock 97.6.0` must require this document for `v97.6.0` and later release cuts. A release that omits regression lessons is incomplete because merchant proof authority, delegated-token boundaries, proof graph public witness behavior, base issue probability language, service-worker authority limits, and MCP authority boundaries would not be explicit release truth.
