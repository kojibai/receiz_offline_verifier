## Frozen Invariants

- Law first, user experience second, and never violate either.
- `v64.6.0` stability is the baseline; local-first work must be additive to it.
- If stronger verified truth is already local, nothing weaker remote may delay, downgrade, replace, or reinterpret it.
- If network exists, server propagation starts in the same action path.
- Sync is propagation, discovery, and verified addition. It is not permission.
- A settled visible shell is immutable unless the user explicitly interacts.
- Background preload, hydration, lazy imports, and remembered session state may warm hidden code, data, and state only.
- Durable session bootstrap is projection, not proof; durable signed-in persistence requires proof backing.
- A stale cookie, stale service-worker runtime, stale app chunk, or stale server projection must recover through local proof first, server session resync second, and stable home third.
- Wallet, Market, World, Explore, Account, Profile, Receipt, Viewer, and Player surfaces must project known local truth before discovering additions.
- Explore single-view cards must reserve identity space; media is not allowed to consume the creator `@` handle.
- A missing, anonymous, or weaker session payload must not erase locally proven identity/account state.
- Brand-new Receiz ID and local PBI enrollment write/project local truth first. Global registration is a verified append.
- Signed-in Market entry must carry active position ownership on first paint when local action truth exists.
- Public twin local register projection is visible truth. Online stream output is allowed when no local projection exists or when a stronger verified append contract exists.
- Public twin visual identity is part of the deterministic projection when profile/twin image truth is known.
- Live player first paint is the selected original. Full queue, world profile, artwork enrichment, and chat/twin augmentation are follow-on work.
- Valid live-player URLs must render the player route shell first; absence is a weaker conclusion until local snapshots and server resolution have had a deterministic path to prove it.
- Live-player queue hydration may extend or complete an artist/video flow, but it may not shrink an admitted multi-track flow into a single-track projection.
- Responsive layout is part of user truth: a control the user cannot reach is a broken action surface.
- Same-rank model output must not overwrite verified profile, receipt, memory, wallet, market, or twin-mind truth without domain-specific merge and proof rules.
- Deterministic trust surfaces must never use spinners, skeletons, "Loading...", "Preparing...", or equivalent fake progress for already-held truth.
- User actions that create, sign, post, trade, hide, unhide, or switch account state must land locally first and publish globally after.

## v64 To v68 Authority Map

### `v64.0.0`

The value loop became complete:

- Settlement is proof-native.
- Reserve is funded.
- Notes stay Reserve-only.
- Certificates are buyer-funded bridges.
- Market sells preserve funded/unfunded causality.

### `v64.6.0`

The app was stable in production behavior:

- stale-cookie PWAs did not crash
- server session behavior was reliable
- user-facing routes worked well

This release keeps that behavior as the baseline.

### `v65` through `v66`

The offline/local primitives were introduced:

- offline verified register language
- local truth store
- PBI local verifier
- offline identity sessions
- service-worker explicit snapshot persistence
- wallet/market/explore/world route hydration improvements
- public twin local continuity
- twin mind portable verification

Correct direction:

- honoring local proof
- treating network as verified additions
- preserving local positions and wallet graph truth

Risk introduced:

- durable session projection was too broad before proof boundaries and recovery were strict enough

### `v67` branch work

The official Receiz ID rail was added:

- one-click Receiz ID login
- server continuation route
- multi-account bindings
- Account/Profile switching
- PBI profile signing prompt
- local profile/wallet route seeds
- server publication after local projection
- stale runtime recovery hardening

Correct direction:

- local identity is real before server publication
- server still mints cookies and syncs
- legacy PBI/email remains recovery/interop

Risk corrected in `v68`:

- unbacked bootstrap cannot skip normal server session path
- durable signed-in bootstrap requires proof backing
- `v64.6.0` server function is explicitly preserved

### `v68.0.0`

The release freezes the final shape:

```txt
proof held locally -> project/action now -> server propagation starts -> verified additions admitted -> visible truth remains coherent
```

## Residual Risks

- Universal local signed outboxes are started as a product direction, but every private server action still needs its final local-append contract before the system is complete everywhere.
- Cross-device Receiz ID/account binding hydration should continue to improve so restored keyfiles can rebuild binding lists cleanly.
- Browser local model readiness depends on device WebGPU support, memory, and model availability. Deterministic local fallback remains required.
- Peer-to-peer presence still needs transport. Transport loss is not identity loss, but live communication still requires a path between devices.
- Public-key rotation for portable twin-mind artifacts must preserve historical verification.
- Cold opens with no local block still need server resolution; this is not a primitive violation because the node does not yet hold that truth.
