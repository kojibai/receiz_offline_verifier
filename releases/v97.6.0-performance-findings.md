# Receiz v97.6.0 Performance Findings

`v97.6.0` keeps the `v97.5.0` public app-state, enterprise SDK, and MCP agent rail law intact and extends the developer platform with signed merchant public-store publishing plus a public proof graph surface. The performance boundary remains primitive-first: first paint may use known verified truth and route-bound release snapshots, while hidden work may only append verified additions after the known head.

## Findings

- Signed merchant public-store publish removes the token-first round trip for merchant-owned storefront sync. The merchant Identity Seal / Receiz Key signs the public-store envelope directly, and the public proof rail verifies feed digest, challenge digest, tenant host, merchant Receiz ID, public key identity, and signature before append.
- Public-store recovery remains a projection for cold-start first paint. It can restore storefront state quickly, but it does not outrank sealed artifacts, identity proofs, ownership appends, settlement rows, verified local truth, or verified appends.
- `/proof-graph` is a public proof surface with live Kai pulse state and in-page graph navigation. Rich rendering is acceptable only while it carries release/API/product proof data and avoids becoming static decoration.
- Service-worker release identity must match app release identity so runtime continuity, offline access, and projection availability do not drift from the shipped verifier surface.
- Sports saved-card export avoids a copy-induced primitive downgrade by labeling base cards as `BASE 1:1` / `Base issue 1:1`; scarce rarity variants keep `ODDS`.

## Performance Boundary

- Known verified truth must paint before hydration, analytics, SEO, session checks, lazy imports, global sync, or server reconciliation.
- Signed merchant public-store publish may improve developer speed, but it is a proof-authority lane, not a delegated-token shortcut.
- MCP and SDK helpers may reduce integration work, but they remain beneath proof objects, verified appends, public proof surfaces, ownership states, settlement states, and durable proof memory.
- Service-worker continuity may preserve local availability, but it cannot become authority above deterministic file proof or verified local register state.
