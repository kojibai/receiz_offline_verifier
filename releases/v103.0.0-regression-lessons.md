# Receiz v103.0.0 Regression Lessons

## What Regressed

Valid Profile Composer, Record, and Seal proof objects failed verification and Profile showcase downloads under v102.

## Why It Regressed

The v102 failure came from promoting the pre-existing parallel SDK portable carrier into a universal verification requirement above implemented Profile Composer, Record, and Seal truth. The code paths that created native proof objects were unchanged; the shared verifier rejected them afterward.

The production rollback to v101 restored verification and downloads without changing any artifact. That proves the defect was exclusively the v102 verifier boundary, not native artifact continuity.

## Locked Rules

For `@receiz/sdk`, `@receiz/mcp-server`, and `@receiz/ai-skills`, “mirror Receiz” means direct composition of the existing primitive. A new wrapper may never become an authority requirement for older native artifacts.

Passing release gates cannot overrule a live 409, broken download, or offline rejection of a valid native proof object. Durable proof memory remains **first admission only, then append forever**.

## Contract Gates

- Shared verification must accept the existing native proof bundle without a portable carrier.
- SDK creation must persist the authenticated Record before sealing at the same coordinates.
- Offline verification must preserve historical native proof objects.
- Profile release and verification-conformance suites must remain green.

## Release-Lock Requirement

`pnpm release:lock 103.0.0` must require these regression lessons in the standalone offline-verifier checkout. Omitting them would sever the correction from production rollback evidence proving that v102's verifier boundary—not the saved native artifacts—caused the failure.
