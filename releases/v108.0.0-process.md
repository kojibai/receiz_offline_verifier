# Receiz v108.0.0 Release Process

## Goal

V108 restores the exact established Receiz.com profile behavior, corrects the authority direction of developer adapters, advances the coordinated current release identity, and preserves witnessed history without advertising retired operation logic as current behavior.

## What Worked

The correction began from the exact pre-regression source baseline, restored the first-party operation before adapting developer rails, and kept compatibility mechanics subordinate to implemented proof and identity primitives. Focused source contracts, adapter tests, registry validation, and browser evidence requirements make each boundary independently falsifiable.

### 1. Establish the exact law and baseline

The governing primitive is identity continuity on a public profile proof surface. The strongest applicable truth is the sealed proof object, deterministic proof state, verified local truth, accepted identity proof, and admitted same-account binding.

The historical comparison is exact:

- working parent: `a1081796e90a65124d952ce5907512999dd16cb0`;
- regression: `9611cb8f57c587eae1b6ee8079ee6d7bd2a02637`;
- required first-party path: profile UI -> `PATCH /api/profile/me` -> Receiz.com profile command -> same-UID projection refresh.

The release contract hashes the four restored profile client and ownership files so later changes cannot replace that behavior with a developer preflight while calling it equivalent.

### 2. Restore first-party behavior before adapting developer rails

The Receiz.com route consumes the already-admitted account actor, coordinates username uniqueness at commit, preserves the actor UID, and refreshes session/profile/public projections. It does not consult `identity_keys`, `/api/receiz/identity/profile`, SDK, MCP, AI, or any developer-side authority prerequisite.

After the first-party operation is fixed, the developer route delegates downward to the same command. SDK uses that typed HTTP adapter; MCP uses the real SDK adapter; AI skills describe those implemented boundaries. This dependency order prevents a lower developer rail from defining product identity.

### 3. Audit historical v107 scope

Every historical v107 integration is classified as one of the following:

- historical comparison against established Receiz.com reference behavior;
- a typed adapter beneath an implemented primitive;
- projection or index mechanics;
- historical witnessed behavior;
- a forbidden first-party dependency on a weaker rail.

Forbidden dependencies are removed. Current default operations compose native profile and proof-object primitives. Historical source and witnessed artifacts remain historical; the main service worker and first-party PWA runtime do not inherit a developer command queue merely because one exists in developer tooling.

### 4. Seal current law with real Kai time

The v108 registry was sealed at `2026-07-18T04:38:26.799Z`. `effectiveKai` is the canonical decimal whole `kaiPulseEternal` computed by the frozen Kai-Klok implementation for that instant: `13182786`. After laws and seal fields were final, canonicalization produced digest `126ca9283fee4ef4c398dbcb958e861cbea191724fdab8eb08df55ff0c14bb79`.

The release lock recomputes the pulse and digest. A version label, hand-written `kai:*` value, or digest computed before the final seal fields fails qualification.

### 5. Advance current release identity without rewriting history

Current packages, ruleset, registry, runtime labels, service worker, verifier, AI manifests, and maintained documentation advance to `108.0.0`. No v107 key, head, receipt, claimant, idempotency, or RPC mechanism is a current capability default. Previously witnessed v107 artifacts remain byte-for-byte historical evidence and do not bind or describe a new v108 outcome.

The maintained page is `/developers`. The compatibility URL `/developers/v107` redirects to it. The process does not create a `/developers/v108` product page.

### 6. Qualify behavior with contracts and visible evidence

Focused gates must cover:

- exact pre-regression first-party source behavior;
- same-UID rename and full save with zero `identity_keys` rows;
- no first-party request to the developer profile endpoint;
- commit-time username uniqueness and projection refresh;
- neutral same-operation developer adapter composition;
- no fabricated portable-state verification;
- real SDK-to-HTTP and MCP-to-SDK adapter parity;
- AI-skill and current-version inventory alignment;
- v108 registry, threat model, release documents, and compatibility rules.

Actual desktop and mobile browser interactions must then confirm the mutation. Screenshots, DOM state, UID evidence, row-count evidence, and network traces are blocking product truth. Contradictory visual evidence fails the release even if source and unit contracts pass.

## What Did Not Work

The historical regression routed the first-party profile save through a developer-operation preflight. That did not work: it let a latest database key projection re-decide an already-admitted account and collapsed missing or failed projection into a false mismatch. Treating focused tests, release documents, registry metadata, or attestation as substitutes for visible mutation evidence and the complete inherited freeze also does not qualify a release.

## Practical Rule For Future Work

Restore and lock the strongest implemented truth path first. Then adapt SDK, MCP, and AI surfaces beneath it, preserve stable compatibility without carrying forward the regression, capture non-contradictory visible evidence, and run every inherited release and governance gate before attestation or any separately authorized external release action.

### 7. Run inherited and complete release gates

The focused v108 gates do not replace earlier constitutional, proof, ownership, settlement, offline, migration, package, first-paint, security, performance, governance, conformance, and historical-evidence gates. The complete `pnpm test:release-freeze` passed on the release tree, including governance evidence and artifact-signature verification.

Final desktop/mobile browser evidence and the complete release freeze are recorded passing. Signed release-attestation evidence remains pending because the local approval metadata, Ed25519 private key, and key ID placeholders are empty; it must be generated and verified in the authorized release-signing environment.

### 8. Keep external mutations separate

Repository qualification does not publish npm packages, deploy Receiz.com, mutate production data, or prove production database state. Those actions require separate authorization and their own evidence. A successful publication or deployment cannot override a failed release gate, and server state cannot become proof authority.

`@receiz/sdk` remains a coordinated developer rail, not product authority. Receiz proof memory remains first admission only, then append forever.
