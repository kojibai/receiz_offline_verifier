# Receiz v60.1.0 Release Lock Checklist

Release date target: April 18, 2026

## Purpose

This checklist locks `v60.1.0` to one exact release state without allowing regression in public market witness truth, compact market readability, twin shell continuity, sigil hydration stability, profile route continuity, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v60.1.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
npx tsx --test app/lib/market/__tests__/marketTradeContinuationContract.test.ts app/lib/market/__tests__/marketWitnessContract.test.ts
npx tsx --test app/lib/profile/__tests__/profileRouteContinuityContract.test.ts
npx tsx --test app/lib/world/__tests__/publicTwinBootstrapReliabilityContract.test.ts app/lib/world/__tests__/worldAvatarFreshnessContract.test.ts app/lib/world/__tests__/worldTwinSessionReuseContract.test.ts app/lib/world/__tests__/worldTwinStageSeparationContract.test.ts
pnpm build
pnpm release:lock
pnpm test:release-freeze
```

If the candidate deploy also requires broader governance or attestation checks beyond the list above, those must still pass before tag creation.

## Production Environment + Governance Preflight

The production environment must already include the active `v60.0.0` baseline. `v60.1.0` adds no new database migrations on top of that cut.

Before release, confirm:

- public profile asset markets show holder count and latest witness timing from live public holder activity
- market buy continuation still survives passkey and funding fallback, and sell remains hidden when the viewer has no position
- the market desk still renders cleanly on compact devices, including public chart dock stats, timestamps, and proof metadata tiles
- home and profile twin shells do not visually remount or upgrade after the page has already settled
- explicit twin open still hands off cleanly into the full module
- profile, player, artifact, and home market pills render without hydration ID drift or missing sigils
- public and offline verifier surfaces, service worker version, `package.json`, and release docs all show `v60.1.0`

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- load `/market` at compact width and confirm witness timing, timestamps, summary tiles, and proof metadata fit cleanly
- open a public profile market asset and confirm holder count and latest witness detail are not stuck on creator seal time when market activity exists
- load `/` and a profile route, let both pages fully settle, and confirm the visible twin shell does not perform a second visual reload or upgrade
- explicitly open the twin from home and profile and confirm the full twin module still opens on demand
- open the home proof trigger and confirm the handoff into the richer proof surface stays clean
- open profile, original-viewer, and verified-player surfaces that render `MarketValuePill` and confirm there is no hydration mismatch or missing sigil art
- public version badges, service worker defaults, offline verifier surfaces, package metadata, and release docs all show `v60.1.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v60.1.0`.
2. Run the `Release Governance Attestation` workflow if the release process for this cut still requires a signed attestation.
3. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
4. Confirm the attestation references the published key ID and the expected release version.
5. Store the attestation path and approval reference with the release record.

## Release Rule

If any public-witness regression, compact-market layout regression, trade-continuation regression, twin-shell continuity regression, sigil-hydration regression, profile-route continuity regression, version-surface mismatch, release-lock failure, or release-freeze failure occurs, `v60.1.0` is not released from that commit.
