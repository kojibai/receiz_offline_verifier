## Purpose

This checklist locks `v68.0.0` as the production release that preserves the `v64.6.0` UX/server baseline while fully honoring the local-first proof model added through the official Receiz ID branch.

Do not release `v68.0.0` unless the product does both:

1. Known local proof projects immediately.
2. Server propagation/session/global sync still starts and remains functional.

## Manual Launch Tests

### `v64.6.0` baseline preservation

- Open the app with a normal server session cookie.
- Confirm logged-in chrome appears without application failure.
- Confirm server session/cookie calls still happen in the background.
- Confirm APIs are not skipped as a consequence of local-first projection.
- Confirm the app never asks the server for permission before projecting already-held proof.

### Stale cookie and stale PWA recovery

- Open a PWA/browser instance with old service-worker state.
- Sign in, close, resume, and verify no React #185 application crash.
- Confirm recovery order:
  - valid local proof keeps the user continuous
  - valid server session can re-seed when local runtime state is suspect
  - missing unsafe state lands stable on home
- Confirm app-shell JS/CSS/worker assets update from network while online.

### Receiz ID one-click login

- Click login on a device without a Receiz ID.
- Confirm Receiz ID is created locally.
- Confirm user lands on the managed profile route without waiting for server publication.
- Confirm `/api/auth/receiz-id/continue` still runs and mints a server cookie when online.
- Reload and confirm the next login is one press.

### Existing PBI/email users

- Sign in with legacy PBI.
- Confirm account recovery works.
- Confirm the current Receiz ID binds to the recovered account.
- Sign in with a Receiz email link.
- Confirm the session is created/restored and Receiz ID binding can happen naturally.
- Confirm future login uses Receiz ID without repeating recovery.

### Multi-account binding

- Bind more than one account to one Receiz ID.
- Switch between bound accounts from Account/Profile.
- Confirm the visible account changes locally first.
- Confirm server switch route verifies the local signature.
- Confirm private routes use the active account UID.

### Profile PBI signing

- Create a new Receiz ID profile.
- Confirm the compact PBI signing action appears on the profile surface above the fold.
- If a passkey exists, confirm the ceremony selects/recovers through it first.
- If no passkey exists, confirm passkey creation/signing works.
- Confirm the prompt disappears after proof.
- Confirm the profile shows PBI-backed verification.

### Profile post and visibility actions

- Post from profile while logged in and online.
- Confirm local proof/action truth appears before upload propagation completes.
- Confirm server upload/publication still starts immediately.
- Confirm upload errors do not make the user appear logged out.
- Hide and unhide posts without glyph-economy entitlement.
- Confirm visibility syncs globally when online.

### Wallet

- Open Wallet after local wallet graph exists.
- Confirm Account Value paints immediately.
- Confirm no `Loading...`/sync theater appears for known signed-in zero state.
- Confirm a missing balance displays zero, not a fallback label.
- Confirm received funds update the active Wallet balance through live summary sync without route refresh.
- Confirm Market Wallet add funds / open wallet / import certificate actions sit above the wallet chart and add funds opens directly underneath the action row.
- Confirm wallet API still syncs ledger/global state.

### Market

- Buy and sell.
- Confirm local position updates immediately below the chart and in tracked positions.
- Confirm lagging `/api/market/snapshot` cannot drop known local positions.
- Confirm received funds update the active Market wallet balance without route refresh.
- Confirm a second buy after draining balance opens checkout for the actual shortfall instead of stopping on balance-short copy.
- Confirm logged-out market wallet state shows login affordance and Receiz ID badge correctly.
- Confirm wallet tabs remain compact and one-row where intended.
- Confirm market snapshot still syncs global additions.

### Explore

- Open Explore with local content already visible.
- Confirm `/api/explore/view` discovers additions without replacing useful visible content with weaker transport.
- Confirm no missing export/function errors.
- Confirm local snapshot remains stable through background hydration.
- Confirm single-view image/audio cards keep the creator avatar and `@` handle in a reserved lane with no overlap from the media card.

### World and twin

- Open World signed in.
- Confirm Launchpad/profile state appears from local truth and does not reset after hydration.
- Open a public twin with local twin mind truth.
- Confirm the twin dock uses the known profile/twin image, not a generic body, whenever profile/twin image truth is already known.
- Confirm partial world/profile bootstrap payloads cannot erase the known twin visual identity.
- Send a message offline.
- Confirm local twin register/model/fallback answers.
- Confirm `world_profile_load_failed` does not remain visible after the twin has answered from local truth or the online stream.
- Reconnect and confirm online stream still works without silently replacing settled local truth.

### Profile, live player, viewer, and receipt routes

- Navigate profile -> live player.
- Navigate artifact viewer -> live player.
- Confirm known media/player truth is reused and no route feels like it reloads from scratch.
- Confirm live player entry never shows a long white screen.
- Confirm a valid live player URL returns the player route shell, not the global 404 page, even while server enrichment is still resolving.
- Confirm live player first paint does not wait on full queue hydration, world profile preview, or profile enrichment when selected media truth is already known.
- Confirm the route may use only the real player backdrop while hydrating a genuine snapshot; it must not show fake progress copy.
- Confirm an artist-flow or creator-flow live player never collapses back to a single-track player after background hydration.
- At 621px-920px wide viewports, confirm live-player controls are reachable and not clipped/trapped inside the artwork/video panel.
- Confirm audio continues to the next queued song in the background with crossfade continuity while navigating, matching the `v64.6.0` behavior.
- Open verified viewer and receipt links from Market.
- Confirm deterministic route snapshots are used where available and server routes resolve cold opens.

### Mobile UI polish

- On small screens, confirm Market username badge text fits inside the badge.
- Confirm inputs never zoom the user on mobile.
- Confirm video popover back-card pills match front-card scale.
- Confirm bottom bar signed-in state is correct on Wallet, Market, World, Calendar, Contacts, Profile, Account, and Explore.

## Release Rule

Do not release `v68.0.0` if any test or manual smoke shows:

- local proof waits on server before projection
- server propagation does not start when online
- a stale cookie or stale PWA crashes the app
- generic bootstrap behaves like permanent proof without local identity backing
- server session/cookie paths from `v64.6.0` are skipped or broken
- wallet first paint drops known local graph truth
- market or tracked positions briefly drop known buy/sell truth
- Explore replaces useful visible content with weaker payload
- Explore media cards overlap or hide the creator `@` identity line
- profile actions require unrelated entitlement to hide/unhide owner content
- local post truth is blocked by upload publication
- online twin behavior is replaced instead of augmented
- deterministic media/viewer/player routes rediscover known truth
- live player valid routes fall into the global 404 page before local/player snapshots can restore
- live player entry shows a white frame or waits on queue/world augmentation before painting known media
- live player artist/video flow collapses into a single-track shell after it was already visible
- live player tablet widths trap play/queue controls inside a clipped panel
- route or primitive work stops global audio from advancing/crossfading to the next queued song
- known twin/profile image truth is replaced by a generic fallback rig
- version surfaces drift from `v68.0.0`

## Final Human Check

The app should feel like `v64.6.0` at its best, with less waiting and stronger continuity.

If it feels like a different product, the release is wrong.

If it feels like the same product finally obeying its own proof, the release is correct.
