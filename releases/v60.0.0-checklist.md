# Receiz v60.0.0 Release Lock Checklist

Release date target: April 18, 2026

## Purpose

This checklist locks `v60.0.0` to one exact major-stable release state without allowing regression in live front-door truth, profile-original authority, twin portrait separation, market auth and funding continuation, production RLS and index hardening, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v60.0.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
pnpm exec tsc --noEmit --pretty false
npx tsx --test app/lib/market/__tests__/marketTradeContinuationContract.test.ts app/lib/market/__tests__/marketConformanceRouteContract.test.ts app/lib/world/__tests__/worldAvatarFreshnessContract.test.ts app/lib/world/__tests__/worldAvatarGenerateRouteContract.test.ts app/lib/world/__tests__/worldAvatarTransparencyProcessing.test.ts app/lib/__tests__/supabaseRlsInitPlanMigration.test.ts app/lib/__tests__/supabaseHighValueFkIndexesMigration.test.ts
pnpm build
pnpm release:lock
pnpm test:release-freeze
```

If the candidate deploy still requires signed release attestation, run it only after the release commit is locked and tagged with the expected version.

## Production Environment + Governance Preflight

For the `v60.0.0` release-prep state, confirm the target environment already includes the `v50.0.0` through `v58.0.0` baseline migrations and also includes:

- `supabase/migrations/20260418001500_market_conformance_history_rls.sql`
- `supabase/migrations/20260418003000_auth_rls_initplan_policy_fix.sql`
- `supabase/migrations/20260418004500_high_value_fk_indexes.sql`

Before release, confirm:

- the homepage hero loads live proof, live twin preview, and direct market action instead of a static-only shell
- market buy continues cleanly through passkey and Stripe funding fallback instead of dying on wallet-bootstrap gaps
- profile uploads still seal through the canonical path, and legacy streamed originals can still download as verifiable Receiz artifacts
- a forged twin portrait exposes its own verify file and does not become the public profile picture unless the user explicitly applies it
- certificate mobile surfaces and market wallet presentation stay compact and legible
- public version badges, plans and upgrade labels, offline verifier artifacts, service worker defaults, package metadata, and release docs all show `v60.0.0`

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- load `/` and confirm the hero shows real proof playback, the live twin dock, and direct market action
- sign in through the market buy flow, trigger a funding shortfall, and confirm the flow resumes through the Stripe funding path instead of dying after passkey completion
- upload a profile original, then download it and confirm the result remains a verifiable Receiz artifact
- trigger a world twin portrait generation, confirm the portrait exposes a verify route, and confirm the public profile picture changes only after the explicit apply action
- open a share certificate on mobile-width layout and confirm the framing, QR area, and certificate labeling stay clean
- public version badges, plans and upgrade labels, service worker defaults, offline verifier surfaces, and release docs all show `v60.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release commit for `v60.0.0`.
2. Create the release tag from that locked commit: `v60.0.0`.
3. Run the `Release Governance Attestation` workflow if this cut still requires a signed attestation.
4. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
5. Confirm the attestation references the published key ID and the expected release version.
6. Store the attestation path and approval reference with the release record.

## Release Rule

If any live-front-door regression, profile-original verification regression, twin-portrait publishing regression, market continuation regression, migration or policy regression, version-surface mismatch, release-lock failure, or release-freeze failure occurs, `v60.0.0` is not released from that commit.
