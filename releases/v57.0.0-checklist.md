# Receiz v57.0.0 Release Lock Checklist

Release date target: April 17, 2026

## Purpose

This checklist locks `v57.0.0` to one exact release state without allowing regression in canonical managed-owner navigation, world/market fast-entry continuity, share certificate custody/history correctness, market-balance coherence after certificate flows, market inventory freshness, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v57.0.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
pnpm exec tsc --noEmit --pretty false
npx tsx --test app/lib/profile/__tests__/managedRoute.test.ts app/lib/profile/__tests__/profileRouteContinuityContract.test.ts
npx tsx --test app/lib/market/__tests__/marketConformanceRouteContract.test.ts app/lib/market/__tests__/shareCertificateMarketFlowContract.test.ts app/lib/market/__tests__/shareCertificateViewportContract.test.ts app/lib/market/__tests__/shareTradeHydrationContract.test.ts
pnpm build
pnpm release:lock
```

If the candidate deploy also requires broader freeze or governance checks, those must still pass before tag creation.

## Production Environment + Governance Preflight

For the `v57.0.0` release-prep state, confirm the target environment includes:

- `supabase/migrations/20260417123000_market_share_note_events.sql`

Before release, confirm:

- runtime owner entry resolves through the canonical managed route rather than a generic `/profile` fallback
- world and market runtime entry paths still use warmed fast-entry route truth on the target environment
- certificate mint/import/sell flows update the visible market balance immediately without requiring reload
- certificate public links, OG/Twitter cards, and verify/share targets resolve to canonical production URLs
- profile archive/unarchive and new public receiz creation update market inventory deterministically
- public/offline verifier surfaces, service worker version, `package.json`, and public version badges all show `v57.0.0`
- `public/governance/governance-keyring.v1.json` still contains the active release-attestation key ID if release attestation is produced for this cut

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- profile entry from live player, market, world, and auth surfaces lands on the same canonical managed-owner route behavior
- world and market bottom-bar entry settles immediately from warmed route truth instead of showing a weak loading shell
- certificate page shows live certificate value/history, chart overlay dock, copy/share controls, and in-place passkey auth behavior correctly on mobile
- minting a certificate removes shares from the live desk immediately, and importing/restoring adds them back immediately
- public market inventory still shows the full current public tradable set after publish/archive mutations
- market conformance passes including `M-010` and `M-017`
- public version badges, service worker version, offline verifier surfaces, and release docs all show `v57.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v57.0.0`.
2. Run the `Release Governance Attestation` workflow if this cut still requires a signed attestation.
3. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
4. Confirm the attestation references the published key ID and the expected release version.
5. Store the attestation path and approval reference with the release record.

## Release Rule

If any version-surface mismatch, managed-route regression, world/market fast-entry regression, certificate custody/history regression, market inventory regression, conformance trace regression, or release-lock failure occurs, `v57.0.0` is not released from that commit.
