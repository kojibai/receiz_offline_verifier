# Receiz v47.8.0 Release Lock Checklist

Release date target: April 10, 2026

## Purpose

This checklist locks `v47.8.0` to one exact release state without allowing regression in durable contacts persistence, guest/public-twin continuity, person-first follow-up actions, mobile-visible identity context, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v47.8.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
npx tsx --test app/contacts/__tests__/contactsBootstrapContract.test.ts
npx tsx --test app/lib/world/__tests__/worldProfileRouteContract.test.ts
pnpm exec tsc --noEmit --pretty false
pnpm exec next build --webpack
pnpm release:lock
pnpm test:release-freeze
```

Environment and deployment preflight for production lanes:

- the production database must already include `supabase/migrations/20260409163000_contacts_crm.sql`
- server-side Supabase admin/service-role credentials must remain available for authenticated contact sync and world action-ledger projection
- world runtime payloads must continue to provide `visitorKey`, `threadKey`, and visitor identity fields for guest continuity

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- `/contacts` loads as the one-viewpoint contacts workspace without clipping or losing primary controls on mobile
- opening the Contacts popover shows the full contact list cleanly
- guest public-twin visitors materialize into contacts after sync without creating obvious duplicate records
- a guest contact shows continuity identity context and searchable visitor/thread continuity keys
- tapping `Message` on a guest-only twin contact reopens the saved twin thread
- tapping `Call` on a guest-only twin contact reopens the same continuity lane for live follow-up when a direct phone or chat lane is unavailable
- direct contacts with linked accounts, email, or phone still open the correct Receiz thread, email composer, SMS composer, or dialer
- public version badges and offline verifier surfaces all show `v47.8.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v47.8.0`.
2. Run the `Release Governance Attestation` workflow with the required approval identities and signing secrets.
3. Verify the uploaded release attestation artifact.
4. Store the attestation path and approval reference with the release record.

## Release Rule

If any contacts-durability regression, guest-continuity regression, follow-up-action regression, version-surface mismatch, release-lock check, or attestation step fails, `v47.8.0` is not released from that commit.
