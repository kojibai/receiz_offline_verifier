# Receiz v87.0.0

## Goal

Ship `v87.0.0` as the release that makes offline sealed proof objects first-class on `/v`.

The release goal is precise:

```txt
offline sealed proof object
compact self-contained /v transport
Signature V4 verification on /v
Proof badge for valid offline proof
QR scans to proof, not historical coordinates
offline verifier/studio parity
service-worker cache coherence
```

The release also carries the post-v86 Sports QR/value cockpit, World Estate slice, market/profile asset security hardening, and RLS cleanup.

## What Worked

- Treating the sealed proof bundle as the authority made the `/v` fix straightforward.
- The route could verify offline-generated proof by reusing the existing Signature V4 proof primitive instead of adding a new authority layer.
- Compact binary transport kept the link self-contained while making it practical for QR, text, and social sharing.
- Removing duplicated `a/ms` query state shortened the URL without removing proof truth because that truth is already inside the signed proof bundle.
- Keeping legacy `rpb1:` decode support preserved compatibility while allowing the shorter token to become the preferred transport.
- Updating offline verifier/studio HTML kept downloaded/offline flows aligned with app `/v` semantics.
- The live QR fix worked when QR generation used the final proof-bearing `anchoredUrl` after sealing.
- Sports QR work remained safe because it improved the single-card public proof surface without replacing vault card authority.
- World Estate stayed scoped by gating only `/world/estate`, leaving `/world` untouched.
- RLS/security work stayed compatible when public asset loading continued to protect deterministic public proof truth.

## What Did Not Work

- Coordinate-only `/v` links were not enough for offline proof objects that already carried Signature V4 proof truth.
- Treating all proof query payloads as wallet transaction payloads made real offline proof bundles fall through to `Historical`.
- Reconstructing links from coordinates after sealing erased the proof-bearing URL that the object already had.
- QR could lag behind proof truth when the visible card still pointed at an older coordinate URL.
- A formal service-worker cache/version move is needed when offline HTML changes, otherwise installed PWAs can keep serving stale verifier/studio files.
- Trying to make a self-contained offline proof link as short as a profile-composer link would require external lookup or removing proof material. That would violate the proof boundary.

## Practical Rule For Future Work

For offline proof links:

```txt
verify the sealed proof object first
project a proof-bearing /v link from that object
compact the transport only after preserving proof material
do not require profile publication
do not require server lookup
do not downgrade to historical when Signature V4 verifies
```

For QR:

```txt
QR is transport
the sealed proof object is truth
the QR URL must point at the strongest available proof-bearing /v link
coordinate-only QR is historical compatibility only
```

For `/v`:

```txt
route params bind identity
c= carries compact proof transport
decoder reconstructs receiz.proof_bundle
Signature V4 verifies authority
valid signed offline proof displays Proof
historical fallback stays below signed proof
```

For releases:

```txt
package version
service worker version
public labels
offline verifier labels
release note
product truth
checklist
process
release lock
focused proof contracts
visual proof evidence
production cache confirmation
```

The `v87.0.0` rule is:

```txt
If a proof object proves itself offline, /v must honor that proof directly.
Everything weaker may publish, index, hydrate, or append beneath it.
```
