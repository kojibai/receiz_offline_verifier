# Receiz v81.0.0

## Goal

Ship `v81.0.0` as the Signal Circuit provenance release that locks the 35-commit post-`v80.0.0` candidate ending at `23eec6d23`: Card DNA/history, rewarded-card payment-before-reveal, wallet/calendar card activity, transfer deduplication, explicit card discovery navigation, mobile game chrome stability, and release-surface coherence all under the same Receiz law.

## What Worked

- Treating Signal Cards as Receiz assets kept the release honest. The new history surface had to preserve sealed origin, market events, auction state, bid state, sale history, and arena appends without rewriting the original claim.
- Adding a dedicated Card DNA route made provenance inspectable instead of burying it inside card UI. That gives users a place to understand card value, movement, and lineage.
- Making rewarded-card sealing pay before reveal aligned the game loop with wallet truth. The route verifies the claim, checks wallet lanes, returns precise funding deltas, and charges idempotently.
- Wallet and calendar activity integration made card movement part of account history instead of game-only state.
- Same-session discovery memory worked because it follows explicit user expansion. It preserves the cards a user intentionally loaded without allowing background discovery to take over first paint.
- The mobile polish mattered because Signal Circuit is now a primary product surface. Popovers, bottom chrome, proof manifest height, route color isolation, and stage fit are part of product trust.
- Release coherence improved by moving package version, public labels, service-worker defaults, offline verifier labels, README pointer, release docs, and worker release markers in one governed bump.

## What Did Not Work

- Treating card history as a generic feed would have been wrong. The route has to be card provenance: origin first, verified additions around it, no claim mutation.
- Letting rewarded cards reveal before payment authority would have weakened the value loop. The paid seal path belongs before reveal, with idempotency and wallet shortfall handling.
- Hiding card purchases, seals, and transfers inside `/game` would have made the platform feel fragmented. Card events belong in Wallet and Calendar because they are account history.
- Duplicate receive activity was a product bug, not just a display issue. Duplicate movement makes ownership feel ambiguous.
- More discovery data is not allowed to erase user intent. Same-session memory is acceptable only because manual `More cards` came first.
- Premium visual polish cannot cover proof or actions. Popovers, loaders, and stage treatments have to stay inside the settled chrome contract.
- Release labels are product surfaces. Version drift between package metadata, service workers, badges, offline verifiers, docs, README, and resident workers is a release blocker.

## Practical Rule For Future Work

Every future Signal Circuit release should ask one question first: does this make the card more valuable as a verified Receiz asset without making the original claim less stable?

The right pattern is additive provenance: sealed origin first, wallet-funded movement second, verified market/arena/transfer additions third, and richer inspection surfaces around that truth. If a feature makes users watch the app rediscover card truth, duplicates ownership movement, reveals value before payment authority, or lets chrome cover the proof, it is not a Receiz-quality release.
