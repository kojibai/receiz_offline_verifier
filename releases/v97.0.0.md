## Title

Receiz `v97.0.0` - Public App-State Projection Rail And SDK Ecosystem Release

## Summary

`v97.0.0` is the release that makes Receiz a practical substrate for developers building advanced public apps with `@receiz/sdk`.

The release locks the durable public app-state projection rail, the app-state SDK surface, the OIDC consent repair, the public proof projection correctness fixes, admin performance measurement cleanup, and release identity alignment into one documented release.

This release does not create a new authority layer. The public app-state rail is a projection/read surface beneath Receiz primitives. Proof objects, identity primitives, ownership appends, settlement rows, durable proof memory, Kai/Klok state, and verified appends remain stronger truth.

## What Changed Since v96.1.0

- `@receiz/sdk@97.0.0` exposes `receiz.appState.publish()`, `publishRecord()`, `byUrl()`, `byHost()`, `byCreator()`, `byNamespace()`, and `byId()`.
- The backend exposes delegated public app-state publishing through `POST /api/public-proof/registry/feed`.
- Public app-state reads resolve through URL, host/domain, creator, namespace, and object id read paths.
- App-state projections are URL-normalized so `https://example.app` and `https://example.app/` resolve to the same public projection.
- The SDK ships typed constants and helpers for canonical app-state envelopes so developers and AI agents can publish public app state without hand-assembling feed boilerplate.
- `receiz.publicProof.registryFeed(feed)` remains supported as an alias for existing integrations.
- OIDC Connect consent uses correct browser redirect semantics and button readability.
- Public Receiz record projection responses use the canonical absolute verification URL and preserve owner-scoped behavior.
- Admin performance analytics now measures the user-visible app experience instead of misleading internal/admin surfaces.
- Release identity is locked across app version, SDK version, runtime/service-worker version, public release labels, and release documents.

## Developer Outcome

Developers can now build robust apps that recover public state from Receiz rails after a cold start:

```ts
import {
  RECEIZ_PUBLIC_STORE_STATE_PROJECTION_SCHEMA,
  createReceizClient,
} from "@receiz/sdk";

const receiz = createReceizClient({ accessToken });

await receiz.appState.publishRecord({
  sourceUrl: "bjklock.receiz.app",
  externalCreatorId: "bjklock.receiz.id",
  title: "BJ Klock storefront",
  platform: "Receiz.app Commerce Cloud",
  schema: RECEIZ_PUBLIC_STORE_STATE_PROJECTION_SCHEMA,
  data: { storeStateRecord },
});

const restored = await createReceizClient().appState.byHost("bjklock.receiz.app");
```

That path lets apps publish and recover storefronts, marketplaces, games, rewards, profiles, directories, blogs, access systems, and custom-domain applications without adding Supabase, Redis, Vercel KV, Shopify, or a custom database for public app-state recovery.

## Product Law Preserved

- Proof objects remain the source of truth.
- Kai Klok remains the proof object state machine.
- Durable proof memory remains first admission only, then append forever.
- Connect remains delegated API access, not the identity proof root.
- App-state projections remain public projection/index rows, not proof authority.
- Public reads recover app surfaces; delegated writes publish app projections.
- Server, database, SDK, CLI, and OpenAPI docs do not become truth authority.
