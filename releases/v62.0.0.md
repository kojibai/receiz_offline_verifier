# Receiz v62.0.0

Release date: April 19, 2026

## Overview

`v62.0.0` is the stabilization release that turns the calmer `v60.2.0` continuity state into the new canonical current baseline after comparing the current `HEAD` line against the settled `v60.0.0` and `v60.2.0` behavior. The release keeps the continuity fixes that clearly improved what users feel, adds one surgical home-hero market warm so the back-of-card chart is more likely live by the time the user flips, removes the extra home/profile/runtime wrappers that added indirection without a user-visible win, and cuts every governed version surface together to `v62.0.0`.

This release is intentionally subtractive. The current `HEAD` line carried more aggressive twin/runtime wrappers, service-worker experiments, and profile/media continuity layers. Some of that work addressed real edge cases, but it also changed behavior or added hidden state. `v62.0.0` keeps only the parts that clearly improved settled-screen continuity without making the product feel more engineered.

The product-truth freeze for this release is defined in `docs/releases/v62.0.0-product-truth.md`. The process retrospective for how the repository moved from the current `HEAD` experimentation line to this stabilized cut is recorded in `docs/releases/v62.0.0-process.md`.

## What Defines v62.0.0

- Settled home and profile twin shells keep the explicit activation rule proven in the `v60.2.0` continuity fixes, but the extra wrapper and handoff machinery is gone.
- The visible home twin path is simple again: direct `DeferredWorldProfileModule`, explicit `activated` gate, no hidden mounted loaded module, and no remembered session-open visible promotion.
- The public profile route uses the stronger direct `ProfileClient` and trusted-stats merge path instead of a second lazy wrapper around already-known truth.
- The homepage hero now warms the shared market asset in the background so the back-of-card chart is more likely live by the time the user flips, without adding a visible loading stage.
- Deterministic market sigil scope IDs stay in place across hero, profile, player, and artifact surfaces.
- The release keeps the low-risk service-worker hardening that returning installed users actually need: app-shell navigations prefer fresh network truth and fall back to cached shell/offline documents only when the network document is unavailable. The more behavior-changing dev reset script is still intentionally out of scope.
- Release-governed version surfaces now move together to `v62.0.0` across package metadata, service-worker defaults, public verifier artifacts, home, make, footer, legal, trust, plans, upgrade, and the active release docs.

## Since The Current HEAD Line

### 1. The Real Continuity Fixes Stayed

The keep-set from the `v60.2.0` continuity line is narrow and deliberate:

- the deferred world twin only renders the loaded module after explicit activation
- the settled dock shell keeps the shared rig instead of letting hidden preload take over
- profile route truth stays on the direct `ProfileClient` plus trusted-stats merge path
- deterministic market sigil scope IDs continue to prevent hydration drift across surfaces

This is the behavior that actually made home and profile feel calmer again.

### 2. The Extra Wrapper Churn Was Removed

The code removed from the release-prep tree is the work that looked clever but did not clearly pay for itself in user experience:

- dynamic home twin loader caching in `HomeHeroTwinDock`
- hidden mounted `prewarmOnly` world module paths in `DeferredWorldProfileModule`
- auto-open and handoff choreography in `HomeHeroVideoDeferred` and `HeroLogoVideo`
- lazy `ProfileClient` and idle follow-up merge loading in `UserProfileRouteClient`
- dev-time service-worker reset and forced reload behavior in `app/layout.tsx`
- the extra hero trigger stylesheet that only existed to support the removed handoff path

The resulting runtime is simpler: fewer moving parts, fewer hidden promotion paths, and less opportunity for the page to feel like it loaded twice.

### 3. Hero Chart Parity Was Fixed Surgically

`v60.2.0` still let the home hero back chart lag behind the already-live market pills. `v62.0.0` fixes that without reintroducing wrapper churn:

- `HeroLogoVideo` now warms the shared market asset in the background on homepage mount
- the chart still renders from the shared authoritative market asset
- no visible shell swap or staged loading theater was added

The user-facing effect is straightforward: when the card flips, the chart is much more likely to already be on live data.

### 4. Current-Head Learnings Were Captured, Not Shipped Blindly

The comparison against current `HEAD` produced a clear split:

- what worked: explicit activation, hidden-only warming, direct truth reuse, deterministic sigil IDs, the hero chart warm, and production app-shell navigation hardening that preserves offline fallback
- what did not clearly help: wrapper-driven speed choreography, hidden visible-takeover paths, and behavior-changing dev service-worker cleanup logic
- what remains meaningful but risky: the larger profile/media continuity stack, which solved narrower edge cases at the cost of extra behavior or complexity

The service-worker learnings were split more cleanly for this release:

- kept: production navigation hardening that prevents online stale-shell boot while preserving offline fallback
- kept: localhost static offline pages no longer register the shared worker automatically
- not kept: dev-time forced reload and reset choreography

This release records those learnings in the repository instead of letting them stay implicit.
