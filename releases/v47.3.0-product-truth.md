# Receiz v47.3.0 Product Truth

Release date: April 9, 2026

This document freezes the primitives and invariants that `v47.3.0` is allowed to extend but not weaken.

## Frozen Primitives

| Primitive | Locked truth | Primary enforcement |
| --- | --- | --- |
| Canonical Receiz auth authority | `receiz.com` remains the identity authority for passkeys, session proofing, and signing even when a user starts on a verified custom domain. | `app/lib/auth/bridge.ts`, `app/auth/bridge/page.tsx`, `app/auth/verify/route.ts` |
| Custom-domain local session continuity | A verified custom domain may mint its own local session only after canonical Receiz authority hands off a short-lived bridge token. | `app/api/auth/bridge/issue/route.ts`, `app/api/auth/bridge/redeem/route.ts`, `scripts/test_custom_domain_auth_bridge.ts` |
| Embedded-browser passkey honesty | In-app browsers that are likely to break WebAuthn must route users toward email fallback with honest guidance instead of presenting passkey failure as a product error. | `app/signin/passkeyAuth.ts`, `app/signin/SignInClient.tsx`, `app/components/auth/SignInPopover.tsx`, `app/upgrade/page.tsx`, `app/signin/__tests__/passkeyBrowserSupport.test.ts` |
| Canonical PBI signing realm | Business-domain signing may feel native on a custom domain, but the WebAuthn / PBI assertion still resolves through the canonical Receiz signing realm. | `app/utils/pbi/webauthn.ts`, `app/auth/bridge/pbi/page.tsx` |
| Custom-domain profile media continuity | Branded-domain profile hero media must resolve with the same first-load continuity as canonical Receiz-hosted profile surfaces. | `app/api/media/image/route.ts`, `app/api/media/image/__tests__/route.test.ts` |
| Release-governed version surfaces | Package version, release docs, public version badges, and service worker defaults must move together on each release. | `scripts/check_release_lock.mjs` |

## Frozen Invariants

| Invariant | What must not regress | Locked by |
| --- | --- | --- |
| Existing Receiz auth cannot be rewritten away | The bridge layer may wrap the auth flow, but it must not replace the canonical Receiz passkey and verify routes for primary authority. | `app/api/auth/request/route.ts`, `app/auth/verify/route.ts`, `scripts/test_custom_domain_auth_bridge.ts` |
| Verified custom domains cannot require a second account | The same Receiz account identity must remain valid on both `receiz.com` and the bound custom domain. | `scripts/test_custom_domain_auth_bridge.ts` |
| Embedded-browser sign-in cannot dead-end on passkey | Sign-in surfaces must surface email immediately when likely limited app browsers are detected. | `app/signin/__tests__/passkeyBrowserSupport.test.ts` |
| Custom-domain first-load media cannot intermittently break | Branded profile media requests must trust the current forwarded host instead of only the canonical host. | `app/api/media/image/__tests__/route.test.ts` |
| Profile browser-contract tests stay type-correct | The profile live-player contract harness must remain compatible with the current DOM typing model. | `app/lib/profile/__tests__/livePlayerShareContract.test.ts` |
| Release surface inventory stays canonical | The committed release surfaces and version badges must align with the package and service worker release. | `scripts/check_release_lock.mjs` |

## Release Gates

`v47.3.0` is considered locked only if these pass together:

- `npx tsx scripts/test_custom_domain_auth_bridge.ts`
- `npx tsx --test app/signin/__tests__/passkeyBrowserSupport.test.ts app/api/media/image/__tests__/route.test.ts app/lib/profile/__tests__/livePlayerShareContract.test.ts`
- `pnpm exec tsc --noEmit --pretty false`
- `pnpm exec next build --webpack`
- `pnpm release:lock`
- `pnpm release:attest`
- `pnpm release:verify-attestation <path>`

The manual release procedure is defined in `docs/releases/v47.3.0-checklist.md`.

## Residual Risks

The following areas are materially stronger after this release but are not yet absolute:

- The custom-domain bridge keeps Receiz as the source of identity, but production continuity still depends on valid domain binding, canonical origin reachability, and popup / redirect allowances in the real browser.
- Embedded-browser detection is intentionally conservative; some app browsers may evolve, so the safe bias remains email-first when the user agent looks like a limited webview.
