## Title

Receiz `v97.2.0` - Enterprise App Runtime SDK Release

## Summary

`v97.2.0` expands `@receiz/sdk` from App State and Commerce Cloud rails into a full enterprise app runtime surface. Developers can build public apps, storefronts, marketplaces, customer portals, rewards systems, access products, media workflows, audit trails, notification flows, search surfaces, and offline-first proof queues with typed Receiz primitives instead of hand-rolled glue.

This release keeps the same authority boundary as the product: SDK helpers are convenience rails beneath Receiz proof objects, identity primitives, ownership appends, settlement rows, durable proof memory, Kai/Klok state, public projections, and verified appends. The SDK does not create a new truth authority.

## What Changed Since v97.1.0

- `@receiz/sdk@97.2.0` adds tenant-scoped customer session helpers, customer portal helpers, merchant onboarding helpers, and expanded commerce primitives for refunds, subscriptions, shipping, tax, discounts, gift cards, access passes, inventory, fulfillment, and payouts.
- The SDK adds enterprise rails for RBAC, audit logs, risk checks, compliance exports, store portability, notifications, search, release pinning, media transforms, jobs, event replay, and offline proof queues.
- Every new write helper accepts idempotency material so third-party apps can safely replay writes after serverless cold starts, browser interruptions, or offline queue recovery.
- `@receiz/sdk/react` ships React hooks and hosted component factories for identity, checkout, app-state, media, domain connection, wallet balance display, and Twin writer surfaces.
- App Runtime docs now show how a developer builds a Commerce Cloud grade app with app-state, checkout, media, events, jobs, RBAC, audit, search, compliance, offline proof replay, and release pinning.
- SDK tests cover the new enterprise rails, React subpath, idempotency headers, and offline queue replay.

## Developer Outcome

Developers can now build a Receiz app runtime with one coherent SDK:

```ts
import { createReceizClient } from "@receiz/sdk";

const receiz = createReceizClient({ accessToken });

await receiz.merchants.onboard({
  tenantHost: "bjklock.receiz.app",
  merchantId: "merchant:bjklock",
  settlementWalletId: "wallet:bjklock",
  defaultStore: { title: "BJ Klock storefront" },
  idempotencyKey: "merchant:onboard:bjklock:v1",
});

await receiz.inventory.reserve({
  tenantHost: "bjklock.receiz.app",
  sku: "shirt-black-xl",
  quantity: 1,
  idempotencyKey: "inventory:reserve:order_1001",
});

await receiz.audit.append({
  tenantHost: "bjklock.receiz.app",
  action: "order.fulfilled",
  actorId: "staff:1",
  subjectId: "order_1001",
});
```

React apps can use the optional React subpath:

```tsx
import { ReceizCheckoutButton, useReceizAppState } from "@receiz/sdk/react";
```

## Product Law Preserved

- Proof objects remain the source of truth.
- Kai Klok remains the proof object state machine.
- Durable proof memory remains first admission only, then append forever.
- Public app-state projections remain recovery projections beneath proof objects.
- Connect remains delegated API access, not identity proof authority.
- SDK helpers remain convenience composition beneath Receiz primitives.
