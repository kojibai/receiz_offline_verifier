# Receiz v88.0.0

## Goal

Ship `v88.0.0` as the release that locks post-v87 live truth and regression lessons.

The release goal is precise:

```txt
preserve v87 offline proof
lock Sports live field truth
preserve locked single-game first paint
preserve individual player scores through hydration
restore exact 87.0.3 field layout
harden runtime/SW version coherence
harden durable session clear/migration
document DB timeout hardening
make regression lessons release-lock material
```

## What Worked

- Returning to the `87.0.3` Sports field baseline gave a concrete visual and code authority instead of guessing at a new layout.
- Separating score plate, center pitch-by-pitch, and bottom last play removed the ambiguity that caused field regressions.
- Model tests caught the stronger-truth rule for selected games: incoming refreshes may not erase detailed field state already held for selected games.
- First-paint bootstrap tests locked the single-game entry path so durable local/proof truth paints before slower private hydration.
- Route contracts locked the exact live pitch-by-pitch source: `currentPlayPitchCommentary` with batter, pitcher, count, result, inning, score, and base state.
- The `/baseball` route was added as a direct entry alias only after the contract proved it reused the canonical `/game/sports/baseball` page and chrome.
- Runtime coherence was improved by deriving `NEXT_PUBLIC_SW_VERSION` from `package.json` when the environment omits it.
- Durable session clear was fixed by removing runtime-scoped legacy keys, not only current and unscoped keys.
- DB timeout hardening stayed scoped to migrations/tests and did not mutate an unknown production project.
- Release lock now requires a regression lessons document for the current version.

## What Did Not Work

- Treating "the live module has data" as enough let later hydration overwrite a better selected-game field.
- Fixing locked entries without preserving the exact field layout caused pitch-by-pitch, base state, and last play to drift into the wrong places.
- Using fallback cards or fallback field text while real live field truth was available violated the selected-game proof surface.
- Letting reward/final labels share the same visible path as active live state caused "Reward ready" to appear during started games.
- Assuming Main Event behavior proved single-game behavior missed the fact that single games needed their own locked-entry projection path.
- Relying on manually supplied `NEXT_PUBLIC_SW_VERSION` was not enough for deterministic runtime/profile/session projection coherence.
- Clearing only current and unscoped durable session keys left runtime-scoped keys able to re-migrate later.
- The first DB timeout hardening pass did not include older chat RLS helper/policy paths.
- Broad Supabase Advisor/linter scans are not safe production proof by themselves; they can become the source of red timeout logs.

## Practical Rule For Future Work

For Sports selected-game field:

```txt
/baseball is an alias, not a fork
use the selected game's detailed field first
preserve detailed field across scoreboard refreshes
score plate shows base state
center recap shows real pitch-by-pitch
bottom ticker shows last play
player cards show real batter and pitcher when known
fallback only when detailed truth is genuinely absent
```

For single-game entries:

```txt
stored joined game entry is durable local/proof truth
paint locked immediately when known
hydrate private user-live beneath it
never let a partial refresh erase locked state or player scores
```

For runtime/session coherence:

```txt
package version is release truth
SW version and runtime keys derive from package release when env omits them
clear/logout removes every durable session key shape that can re-migrate
```

For DB timeout hardening:

```txt
patch specific policies, functions, and indexes
prefer init-plan-safe auth reads inside RLS
do not run broad production advisor scans as proof without explicit scope
do not claim production DB fixed until the migration is applied to the exact production project
```

For releases:

```txt
release note
product truth
process
checklist
regression lessons
version surfaces
release lock
focused contracts
full promotion gates
production migration/deploy evidence
```

The `v88.0.0` rule is:

```txt
Hydration may enrich live truth. Hydration may not reinterpret live truth.
```
