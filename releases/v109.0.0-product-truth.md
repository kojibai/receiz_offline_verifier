# Receiz v109.0.0 Product Truth

## Frozen Primitives

The proof object—not the SDK, MCP response, server, session, or database—is the authority.

`verifyArtifact()` and `verifyAndOpen()` recompute the proof object's carried truth locally. A successful result requires carrier integrity, payload digest binding, Signature V4 certificate/root verification, Groth16 verification, canonical identity/path fields, and applicable portable ownership continuity. Payload bytes are unavailable until the complete enclosing artifact passes.

`ARTIFACT-011` states: complete SDK artifact verification and extraction must remain available without network, session, database, or server state. The denial code is `LOCAL_ARTIFACT_VERIFICATION_MUST_NOT_REQUIRE_WEAKER_STATE`.

## Frozen Invariants

Native Record -> Seal issuance is unchanged. Receiz.com behavior is unchanged while its visible release labels, service worker, and runtime-coherence version identify v109. The database is unchanged. v109 adds developer verification capability without creating a new authority layer or weakening sealed artifact truth.

Historical sealed proof remains readable under its verified carried authority. Obsolete SDK and MCP entry points are not an authority requirement and are not shipped.

## Release Gates

Qualification requires the SDK and MCP builds and suites, zero-fetch production-signed verification, one-byte mutation rejection, package tarball inspection, coordinated external installation, AI-skill validation, the v109 registry lock, the inherited release freeze, and governance checks.

## Residual Risks

npm publication and release signing are separately authorized external actions. The inherited profile-database fixture requires an isolated local database endpoint; production data is forbidden for that test. These boundaries do not weaken local artifact verification or make server state authoritative.

The public `@receiz/sdk` preserves durable proof memory as first admission only, then append forever.
