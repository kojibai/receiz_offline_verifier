# Receiz v92.0.0

## Goal

Ship `v92.0.0` as the release that documents and locks the post-`v91.0.0` Receiz logic shift:

```txt
primitive first
strongest truth first
weaker layers beneath
visual evidence as product truth
contracts that catch legacy reasoning
version surfaces moved together
```

The release goal is not only to list Pitch Command, Pack Derby, value panel, vault, profile, and live-player changes. It is to freeze the reasoning that made those fixes work:

```txt
Receiz pages become faster and more reliable when implementation mechanics submit to implemented Receiz primitives.
```

## What Worked

- The `v91.0.0` scale law made rich deterministic surfaces legitimate, so v92 could improve pages without cutting primitives.
- Public proof first-paint fixes worked when the resolver order started from deterministic proof/player truth instead of session/private route work.
- Profile and live-player bugs became clear once `/<username>` and public proof routes were treated as canonical truth surfaces.
- Sports vault work became cleaner when ownership rows were treated as custody projection beneath vault objects, not as the artifact itself.
- The value panel fix became small when the settlement chart was protected and event memory was made scrollable beneath it.
- The event proof z-index fix stayed small because it kept the existing proof desk and changed only the card-viewer stacking relationship.
- Pitch Command date switching became fast when admitted historical proof painted before fresh slate rediscovery.
- The `source_date` error was solved without a DB migration because append proof payload truth already carried `sourceProof.sourceDate`.
- Pack Derby visual bugs were treated as product-truth contradictions, so field geometry was repaired instead of explained away.
- `docs/receiz-reasoning-kernel.md` compressed the doctrine into an operational loop that lowers future cognitive load.

## What Did Not Work

- Legacy reasoning kept trying to promote DB/session/API/hydration state into authority.
- Early fixes sometimes landed in the wrong selector or surface, such as styling `sportsSingleCardValuePanel` before the actual card viewer path applied that class.
- The event proof desk rendered behind the fixed value panel because the render stack and fixed z-index relationship were not contract-locked.
- Browser visual verification for sports routes was blocked locally by the sealed-route map returning 404 for `/sports`, `/game/sports`, and `/arena`.
- Generic "latest data wins" instincts would have produced a DB migration for `source_date`, which would have moved truth to the wrong layer.
- Generic performance instincts would have made historical proof, value charts, and public proof pages wait for rediscovery or hydration.

## Practical Rule For Future Work

For any Receiz surface:

```txt
do not ask weaker layers to decide product truth
```

Use the v92 kernel:

```txt
Primitive:
Strongest truth:
Weaker layers:
Forbidden downgrade:
Append/enrichment path:
Contract/visual evidence:
Exact uncertainty:
```

For profile and public proof routes:

```txt
paint public proof truth first
session and private owner state append after
```

For vault:

```txt
vault object first
ownership rows filter custody beneath it
```

For value:

```txt
settlement chart first
event memory scrolls or overlays beneath the same proof/value truth
```

For game:

```txt
known proof and settled visual truth first
live motion appends without making known truth feel missing
```

For DB fixes:

```txt
inspect whether proof payload truth already carries the field
index truth only after the primitive is named
```

For releases:

```txt
document the logic wins
document the exact regressions
document the exact contracts
move version surfaces together
name unverified visual and production boundaries exactly
```

## Release-Lock Requirement

`pnpm release:lock 92.0.0` must require `docs/releases/v92.0.0-regression-lessons.md` and `docs/releases/v92.0.0-commit-history.md`. A `v92.0.0` release that omits either document is incomplete because the regression lessons and exact commit-history boundary would not be explicit release truth.
