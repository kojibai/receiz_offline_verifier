# Receiz v47.7.0 Release Lock Checklist

Release date target: April 9, 2026

## Purpose

This checklist locks `v47.7.0` to one exact release state without allowing regression in durable contacts persistence, historical contact backfill, import/export portability, person-first timeline logging, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v47.7.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
npx tsx --test app/contacts/__tests__/contactsBootstrapContract.test.ts
pnpm exec tsc --noEmit --pretty false
pnpm exec next build --webpack
pnpm release:lock
pnpm test:release-freeze
```

Environment and deployment preflight for production lanes:

- the production database must include `supabase/migrations/20260409163000_contacts_crm.sql`
- server-side Supabase admin/service-role credentials must remain available for authenticated contacts operations
- browser-native contact import depends on client support for the Contacts Picker API; unsupported clients must still use file import/export

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- `/account` still links cleanly into `/contacts`
- `/contacts` loads as the one-viewpoint contacts workspace without losing core controls on mobile
- historical message participants and historical original collectors appear in contacts for signed-in users after bootstrap
- CSV, JSON, vCard, and native phone import all merge into the same person records without creating obvious duplicate contacts
- editing notes, stage, next step, and invite state survives refresh
- logging a note, email, call, meeting, or invite appends the person-first timeline and rolls up last interaction correctly
- public version badges and offline verifier surfaces all show `v47.7.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v47.7.0`.
2. Run the `Release Governance Attestation` workflow with the required approval identities and signing secrets.
3. Verify the uploaded release attestation artifact.
4. Store the attestation path and approval reference with the release record.

## Release Rule

If any contacts-durability regression, history-backfill regression, import/export regression, version-surface mismatch, release-lock check, or attestation step fails, `v47.7.0` is not released from that commit.
