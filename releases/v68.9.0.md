## Title

Receiz `v68.9.0` - Governed Profile Law Release

## Summary

`v68.9.0` is the governed public release of the restored `v68.8.1` profile-law state.

This release freezes the behavior that was already clearly best in use:

> `/<username>` is the canonical profile truth. `?manage=1` appends owner power. Signed-in users never wait longer than public users for the same truthful shell.

The release does not invent a new profile model. It packages the already-correct one under governed release surfaces, keeps the stronger auth/runtime recovery work already present on the line, and moves package metadata, service-worker defaults, public version badges, and release docs together to `v68.9.0`.

## What This Release Locks

### Profile law is literal again

- `/<username>` remains the only canonical profile route.
- `/<username>?manage=1` remains the same canonical profile with owner permissions layered on top.
- Public and owner routes first-paint the same profile identity, showcase, and settled shell.
- Hidden inventory and owner-only controls remain additive; they do not redefine the base profile.

### Signed-in continuity is not allowed to be slower

- Logged-in profile entry continues to open from already-known truth instead of waiting for server rediscovery.
- The fast route remains the username-managed route, not `/profile`.
- Background hydration is still allowed to append owner-only truth and server additions after the settled shell is already there.

### Auth/runtime hardening stays

- Receiz ID remains local-proof-first with server propagation.
- PBI recovery and Receiz ID binding remain append semantics, not session issuance.
- Stale runtime recovery, service-worker coherence, and proof-backed bootstrap remain part of the release baseline.

### Release surfaces are governed again

- `package.json`
- service-worker version defaults
- public version badges
- offline verifier surfaces
- release docs

All move together to `v68.9.0`.

## Release Principle

This release is the practical merge of two truths:

- the best profile behavior was the restored `v68.8.1` law
- the best runtime/auth direction is the proof-first recovery work already on the current release line

`v68.9.0` is the governed release that keeps both without letting interpretation outrank encoded product law.
