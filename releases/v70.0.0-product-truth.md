## Frozen Primitives

| Concept | v70.0.0 truth | Must not regress |
| --- | --- | --- |
| Product law | Literal doctrine and user-stated release law outrank code convenience. | Do not weaken an exact law into a looser approximation. |
| Receiz ID | One-click Receiz ID is the primary login and continuation rail. | Do not make server permission the moment local identity becomes real. |
| Signin route | `/signin` is a pass-through into Receiz ID continuity. | Do not render a manual signin destination page or park users there. |
| PBI/keyfile/email | PBI, keyfile, email, and magic links are recovery and interoperability rails. | Do not remove them and do not make them supersede local Receiz ID for ordinary login. |
| Canonical profile | `/<username>` is canonical profile truth. | Do not let owner/session/workspace state replace the public shell. |
| Manage overlay | `/<username>?manage=1` is canonical profile truth plus owner powers. | Do not let manage mode redefine identity, showcase, media, or first paint. |
| Local truth root | Verified local identity, proof bundles, wallet graphs, route snapshots, twin state, and market actions are real held truth. | Do not ask weaker network payloads for permission before using stronger known truth. |
| Verified append | Later server/global data may append verified additions only. | Do not replace verified prefixes with weaker or merely fresher transport. |
| Settlement/Reserve | Settlement is proof-native value; Reserve is funded action capital. | Do not credit unfunded value as Reserve or make notes carry Settlement. |
| Market and certificates | Shares and certificates bridge live proof-native value into buyer-funded Reserve. | Do not pay sellers as Reserve before funding exists. |
| Runtime recovery | Stale runtime/session/service-worker states recover through local proof, server resync, and route preservation. | Do not strand users on app errors or reset them to home when the current route is known. |
| Twin identity | Twin image truth is locked per user and twin. | Do not fall back to profile pictures as twin identity on any page. |
| Proof bundles | Downloaded proof packages must verify file-by-file and as a bundle. | Do not ship ZIPs that contain unverifiable files. |
| Release surfaces | Package version, service worker defaults, public badges, verifier surfaces, and release docs move together. | Do not ship version drift. |

## Frozen Invariants

- A settled visible shell is immutable unless the user explicitly interacts.
- Truthful first paint beats richer rediscovery.
- Stronger known deterministic truth is reused before any network rediscovery.
- Weaker partial payloads never downgrade stronger known truth.
- Authenticated/private enrichment may append only; it may not slow or replace the base public shell.
- `/signin` must immediately run Receiz ID continuation when no server session is already present.
- Existing account login must route directly to the managed profile.
- Cold Receiz ID account creation must route directly to the new managed profile.
- `/<username>` remains the canonical profile route.
- `?manage=1` remains a permission overlay.
- PBI recovery proves the legacy account; Receiz ID proves the local device; binding is an append, not the login root.
- `/api/auth/receiz-id/bind` must not depend on replay-guard storage.
- Server propagation is required when online, but it is propagation, discovery, cookie minting, mirroring, recovery, or verified addition.
- Settlement and Reserve metadata must survive every send, buy, sell, certificate, note, and wire path.
- Notes issue from Reserve only.
- Market sells split funded Reserve from remaining Settlement.
- Certificate sale credits the seller only after buyer or liquidity funding exists.
- Runtime recovery preserves the current safe route.
- Service-worker and app-version defaults must match the release version.
- Twin image fallback to profile avatar is forbidden.
- Mind trail downloadable ZIPs must include Receiz proof manifests for every file.
- PWA-only visual fixes must stay scoped to PWA-only selectors when that is the release law.

## Residual Risks

- The release is not fully locked until `pnpm test:release-freeze` passes after the `v70.0.0` surface bump.
- `pnpm release:attest` depends on release attestation signing environment outside the codebase.
- Browser-only Receiz ID pass-through still depends on Web Crypto support; unsupported browsers cannot create local device keys.
- Production database, Redis, Stripe, and external provider health remain deployment concerns, but they must not become permission to believe already-held local proof.
- Any future change that makes a visible shell load twice, lets `/signin` become a page again, or makes owner truth redefine public profile truth is release-blocking.
