# Receiz v110.0.0 Regression Lessons

The `@receiz/sdk` durable proof memory law remains **first admission only, then append forever**.

## What Regressed

The v109 MCP artifact-seal planner embedded the v108 registry digest even though its schema promised the current signed registry.

## Why It Regressed

The planner imported a release-specific constant and the existing test asserted the same stale constant, so implementation and test agreed with each other while disagreeing with current constitutional truth.

## Locked Rules

Current confirmation bases use the active registry; admission verifies exact bytes first; bearer authority cannot escalate; recovery mutation requires SDK plan custody, verified capability, idempotency, expected head, and atomic commit; explanations remain non-authoritative.

## Contract Gates

The v110 MCP adapter test, v110 release lock, AI validation, coordinated package tests, and full release freeze enforce these rules.

- A tool description promising the current registry is insufficient; its confirmation basis must import the active digest. The artifact-seal planner’s stale v108 binding is now locked by a direct test.
- Verification, classification, explanation, planning, and mutation must remain distinct authority layers.
- A bearer proof can restore carried continuity without granting canonical account, signing, publishing, or settlement authority.
- Recovery plans need runtime custody just as artifacts and capabilities do; copied JSON cannot become commit authority.
- Idempotency must bind the same operation identity across SDK, MCP, and AI tooling.
- Proof history convergence must be deterministic across applications and transport order while preserving unknown namespaces.
- Read-only convenience APIs must not accept stores, capabilities, idempotency keys, or mutation callbacks.
- A version surface may advance without changing Receiz.com behavior or database logic; release locks must verify that boundary directly.

## Release-Lock Requirement

`pnpm release:lock 110.0.0` must require this regression record for `v110.0.0` and later release cuts. Omitting it would allow stale registry confirmation, authority-layer collapse, bearer escalation, copied-plan mutation, non-deterministic history, or hidden writes to return without an explicit release contradiction.
