# Receiz v101.0.0 Performance Findings

`v101.0.0` preserves Receiz scale law across portable asset creation and admission: repeated verification may add bounded work, but no convenience path may skip the complete sealed proof object or delay known file truth behind server discovery.

## Findings

- Proof-object creation seals a canonical continuity document, reopens the returned file, and runs the existing artifact verifier before authenticated admission.
- Server admission repeats the same artifact verification and owner binding before the database append; duplicated mechanics confirm but do not redefine file truth.
- Canonical payload bytes/hash, ownership/custody, provenance/appends, and settlement travel together so cross-platform portability does not require broad rediscovery.
- Manifest-only and v1 RPC paths are closed, removing a faster but weaker path that could bypass complete continuity.
- Historical artifacts and the existing single verification verdict remain unchanged, avoiding new work or semantic churn for already-valid files.

## Performance Boundary

- Known verified truth must paint before hydration, analytics, SEO, session checks, lazy imports, global sync, chart hydration, ledger pagination, or server reconciliation.
- Offline file verification remains primary and immediate; SDK, API, server, and database work may append admission only after the file passes.
- Canonicalization, transport, authenticated runtime calls, RPC execution, and service-worker continuity are weaker mechanics beneath proof primitives.
- Visual evidence can block release even when static contracts pass.

## Release-Lock Requirement

`pnpm release:lock 101.0.0` must require this document for `v101.0.0` and later release cuts. A release that omits performance findings is incomplete because local round-trip verification, server re-verification, complete continuity transport, revoked manifest shortcuts, and verifier/service-worker release identity would lack explicit proof-authority boundaries.
