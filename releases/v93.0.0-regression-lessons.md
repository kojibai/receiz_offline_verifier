# Receiz v93.0.0 Regression Lessons

Release date: June 25, 2026

## What Regressed

- Sports value surfaces drifted between deterministic proof value, seller ask, internal economics, and display value.
- Some card backs and logged-in views dropped full player bio rows while public/single-card paths remained correct.
- Event and history panels could show less than the full proof chain because a route showed the latest proof rather than walking the prior proof lineage.
- Sports Economy and raw proof pages produced timeout or slow-query pressure from broad array filters and JSON payload aggregation.
- Active Sports views could wait behind background append guards even while the user expected current proof appends.
- Pack Derby opening and reveal paths could be destabilized by adjacent scene/load work.
- Deploys with newly added RPCs could temporarily show partial events/history when route code reached Supabase before the function/schema cache was ready.
- Visual fixes sometimes passed contracts while screenshots still showed a lower-quality surface.

## Why It Regressed

- Some code paths treated proof projection as local UI state instead of shared primitive projection.
- Some DB queries bounded by business intent but not by planner-friendly identity pages, causing payload-heavy scans.
- Some performance fixes focused on shorter timeouts or less data instead of preserving the full proof payload with a better read shape.
- Some visible surfaces inherited background policies that were right for pressure but wrong for active first-person use.
- Some UI changes optimized a panel without protecting the settled surface around it.
- Some release steps assumed deploy ordering was atomic even though Supabase RPC visibility and serverless code rollout are separate runtime events.

## Locked Rules

- If two surfaces carry the same proof object, they must share proof completeness unless a law explicitly scopes one surface.
- Do not replace proof history with the latest proof when the prior proof hash chain is available.
- Do not show internal economics as buyer-facing proof value.
- Do not use a broad PostgREST JSON aggregate shape when a bounded RPC can page identity first.
- Do not add proof payload JSON into wide covering indexes as a substitute for joining payloads after bounded row selection.
- Do not let background stale guards delay foreground proof appends.
- Do not call a new RPC from deployed code before the migration is live and visible to PostgREST.
- Do not claim a visual pass when screenshot/browser evidence contradicts it.

## Release-Lock Requirement

`pnpm release:lock 93.0.0` must require this document for `v93.0.0` and later release cuts. A release that omits regression lessons is incomplete because the value, lineage, visible append, bounded read, foreground cadence, Pack Derby, deploy-order, and visual-evidence regressions have not been made explicit.
