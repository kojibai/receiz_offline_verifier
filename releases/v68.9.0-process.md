## Goal

Package the restored `v68.8.1` profile-law behavior as a governed public release without sacrificing the stronger auth/runtime recovery work already present on the line.

In concrete terms:

- keep `/<username>` canonical
- keep `?manage=1` additive
- keep signed-in first paint as fast as public
- keep Receiz ID, recovery, and stale-runtime hardening intact
- move all governed version surfaces together to `v68.9.0`

## What Worked

- Treating the encoded product law literally made the profile-route target unambiguous.
- Restoring the pre-`v68.8.2` profile snapshot and owner-overlay behavior brought the code back to the settled route law.
- Aligning the contracts back to the restored code made the release candidate explicit instead of interpretive.
- Keeping the stronger runtime/auth recovery work while restoring the better profile discipline preserved the best advancements from the line.
- Governing the version surfaces again turns an internal best-known state into a real release candidate instead of an untracked repo state.

## What Did Not Work

- Letting interpretation outrun encoded law created profile-route behavior that was mechanically clever but product-wrong.
- Treating owner truth as something that could reshape the base route made the profile less truthful and less stable.
- Any release line where version surfaces still said `68.0.0` while behavior had moved far beyond that was mechanically unguided.
- Green contracts were not enough when the contracts themselves had drifted away from the restored release target.

## Practical Rule For Future Work

For Receiz release work, use this order:

1. identify the best known product-law state in code
2. restore or preserve that state exactly
3. align contracts to that literal state
4. move governed version surfaces together
5. run the release gates
6. only then call it a release

If a future release requires interpretation to explain why the code is right, the release target is not frozen tightly enough yet.
