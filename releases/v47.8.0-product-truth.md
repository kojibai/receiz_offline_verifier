# Receiz v47.8.0 Product Truth

Release date: April 10, 2026

This document freezes the primitives and invariants that `v47.8.0` is allowed to extend but not weaken.

## Frozen Primitives

| Primitive | Locked truth | Primary enforcement |
| --- | --- | --- |
| Atlas-first world districts | The district workspace must remain a map-first world-reading surface with one coherent atlas rather than regressing back into a flat stacked-card shell. | `app/world/WorldExperience.tsx`, `app/world/WorldExperience.module.css` |
| Concierge twin-quarterback runtime | The existing concierge shell may start live twin work, but that work must stay operator-triggered, visibly tracked, and grounded back into the same transcript context. | `app/api/world/concierge/stream/route.ts`, `app/api/world/concierge/tasks/route.ts`, `app/lib/world/conciergeTasks.ts`, `app/lib/world/intelligence.ts`, `app/world/WorldExperience.tsx` |
| Durable person-first contacts CRM | Contacts must remain a first-class, person-first operating view with durable persistence, import/export portability, and per-person timeline memory rather than falling back to a local-only shell. | `app/contacts/ContactsPageExperience.tsx`, `app/api/contacts/route.ts`, `app/api/contacts/import/route.ts`, `app/api/contacts/[contactId]/route.ts`, `app/api/contacts/[contactId]/events/route.ts`, `app/lib/contacts/server.ts`, `supabase/migrations/20260409163000_contacts_crm.sql` |
| Public-twin continuity inside contacts | Public twin and guest activity must materialize into the same contact records that hold direct messages, notes, and meetings rather than living in a detached world-only history. | `app/lib/contacts/server.ts`, `app/lib/world/runtime.ts`, `app/contacts/ContactsPageExperience.tsx` |
| Release-governed version surfaces | Package version, service worker version, public badges, offline verifier surfaces, and release docs must move together on each release. | `scripts/check_release_lock.mjs` |

## Frozen Invariants

| Invariant | What must not regress | Locked by |
| --- | --- | --- |
| Contacts cannot fall back to local-only truth | Signed-in contact state must persist through the server-backed contacts API and remain intact after refresh. | `app/contacts/ContactsPageExperience.tsx`, `app/api/contacts/route.ts`, `app/lib/contacts/server.ts` |
| Guest continuity must remain deterministic | A guest public-twin visitor must remain addressable by saved continuity keys so future events and authenticated upgrades land on one contact record. | `app/lib/contacts/server.ts`, `app/lib/world/runtime.ts` |
| Contact follow-up must remain person-first | Message, call, invite, note, meeting, and twin follow-up must continue to originate from the person record and update the same relationship history. | `app/contacts/ContactsPageExperience.tsx`, `app/lib/contacts/server.ts` |
| Historical backfill must stay additive | Existing user relationship history from messages, originals, calendar, and world twin activity must continue to enrich contacts without erasing operator edits. | `app/lib/contacts/server.ts`, `app/contacts/__tests__/contactsBootstrapContract.test.ts` |
| Release surface inventory stays canonical | The committed release surfaces and version badges must align with package and service worker release truth. | `scripts/check_release_lock.mjs` |

## Release Gates

`v47.8.0` is considered locked only if these pass together:

- `npx tsx --test app/contacts/__tests__/contactsBootstrapContract.test.ts`
- `npx tsx --test app/lib/world/__tests__/worldProfileRouteContract.test.ts`
- `pnpm exec tsc --noEmit --pretty false`
- `pnpm exec next build --webpack`
- `pnpm release:lock`
- `pnpm test:release-freeze`
- `pnpm release:attest`
- `pnpm release:verify-attestation <path>`

The manual release procedure is defined in `docs/releases/v47.8.0-checklist.md`.

## Residual Risks

The following areas are materially stronger after this release but are not yet absolute:

- Guest contacts can now reopen the saved twin thread, but reverse outreach still depends on the public twin lane unless the visitor later provides direct contact details.
- Contact continuity for request and reply pairs uses the action-ledger event timestamp, not an independently captured per-bubble timestamp stream.
- Production release attestation still depends on the required approval identities and signing env being present at release time.
