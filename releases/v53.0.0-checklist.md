# Receiz v53.0.0 Release Lock Checklist

Release date target: April 14, 2026

## Purpose

This checklist locks `v53.0.0` to one exact release state without allowing regression from the `v52.0.0` market baseline, the `v52.5.0` hardening pass, or the current release-candidate additions in public market data, historical conformance, prediction precision, passkey account attachment, Enterprise entitlement handling, beginner-first public entry, and public-key release attestations.

Product-truth reference:

- `docs/releases/v53.0.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
pnpm exec tsc --noEmit --pretty false
npx tsx --test app/lib/premium/__tests__/plan.test.ts
pnpm exec tsx --tsconfig tsconfig.scripts.json scripts/test_send_receiz_auth_e2e.ts
pnpm test:market-conformance
pnpm test:release-freeze
pnpm release:lock
pnpm run build
```

If the candidate deploy also requires world, settlement, verification, contacts, or governance release gates beyond the list above, those must still pass before tag creation.

## Manual Smoke

Confirm these flows on the candidate without changing code:

- `/market` renders as a public, indexable market desk and presents the new Open Graph card correctly
- `/market/assets/[receizId]` renders a focused public market asset page with valid metadata and JSON API output
- `/api/market/public-snapshot` returns current live and prediction market data for public items
- market pill interactions feel correct on touch hardware and do not double-fire actions
- player surfaces can open the market pill and route into the correct viewer or desk state
- market conformance history grows after hitting `/api/market/conformance`, and rollups render daily counts instead of a single synthetic row
- prediction contracts reject new wagers after expiry and resolve close-call outcomes correctly from precise price comparison
- an existing email user cannot start passkey registration from the public sign-in route, but can connect Face ID / passkey from Account or profile-owner surfaces
- `/login` stays a noindex developer-login embed surface and points regular users toward `/signin`
- `/receiz-world` redirects to `/public-twin#world`
- home, start-here, seal, make, and “what is a receiz” surfaces all lead with Verify / Seal / Record / Public Twin rather than advanced jargon
- Business is presented as the common next upgrade and World is framed as an advanced expansion layer
- public version badges, service worker version, and offline verifier surfaces all show `v53.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v53.0.0`.
2. Run the `Release Governance Attestation` workflow with the required approval identities, risk-owner values, and `Ed25519` signing material.
3. Verify the generated artifact with `pnpm release:verify-attestation <path>`.
4. Confirm the attestation references the published key ID and the expected conformance results.
5. Store the attestation path and approval reference with the release record.

## Release Rule

If any market determinism regression, public market feed regression, conformance-history persistence regression, prediction-finalization ambiguity regression, passkey-account-attachment regression, Enterprise entitlement regression, public-entry simplification regression, version-surface mismatch, release-lock failure, or release-attestation verification failure occurs, `v53.0.0` is not released from that commit.
