# Receiz v60.0.0 Product Truth

Release date: April 18, 2026

This document freezes the primitives and invariants that `v60.0.0` is allowed to extend but not weaken after the `v58.0.0` release-cut baseline and the cumulative `v50.0.0` through `v58.0.0` product line.

## Frozen Primitives

| Primitive | Locked truth | Primary enforcement |
| --- | --- | --- |
| Release surfaces stay canonical | `package.json`, service worker defaults, public version badges, plans and upgrade labels, offline verifier artifacts, and active release docs must all expose `v60.0.0` together when the cut is made. | `package.json`, `public/sw.js`, `app/components/pwa/ServiceWorkerRegistration.tsx`, public page/version surfaces, `scripts/check_release_lock.mjs` |
| The front door is live product, not a dead shell | The homepage hero must boot from real proof, world, and market data when that data is knowable, rather than regressing to a detached marketing-only shell. | `app/page.tsx`, `app/components/HomeHeroVideoDeferred.tsx`, `app/components/HeroLogoVideo.tsx`, `app/components/HomeHeroTwinDock.tsx`, `app/components/market/HeroMarketBuyStrip.tsx` |
| Profile originals remain canonically sealable and downloadable | Profile uploads and profile-original downloads must preserve or restore canonical Receiz verification instead of leaving legacy streamed originals outside the proof model. | `app/u/[username]/ProfileClient.tsx`, `app/api/profile/originals/download/route.ts`, `app/lib/documentSeal/receizeDocumentBytes.ts`, `app/lib/receizArtifactVerifier.ts` |
| Twin portrait output stays distinct from the live twin | Forged twin portraits are separate profile-ready artifacts with their own verify path and must not silently overwrite the public profile picture without explicit user action. | `app/api/world/avatar/generate/route.ts`, `app/world/WorldExperience.tsx`, `app/api/profile/me/route.ts` |
| Market continuation remains coherent across auth and funding | Hero and market buy surfaces must survive passkey and funding fallbacks instead of dying on missing wallet/bootstrap state after sign-in. | `app/components/market/HeroMarketBuyStrip.tsx`, `app/components/market/MarketValuePill.tsx`, `app/lib/market/tradeContinuation.ts`, `app/lib/market/walletSummaryClient.ts` |
| Certificate and wallet presentation stay compact and truthful | Certificate/mobile surfaces and market wallet presentation must remain owner-aware, compact, and aligned to live custody truth rather than drifting into decorative or misleading states. | `app/market/c/[noteToken]/MarketShareCertificateExperience.tsx`, `app/market/MarketDeskExperience.tsx`, `app/components/market/MarketValuePill.tsx` |
| Production governance hardening is part of release truth | Market conformance history RLS, auth init-plan policy safety, and high-value foreign-key indexes are part of the release boundary, not post-release cleanup. | `supabase/migrations/20260418001500_market_conformance_history_rls.sql`, `supabase/migrations/20260418003000_auth_rls_initplan_policy_fix.sql`, `supabase/migrations/20260418004500_high_value_fk_indexes.sql`, migration contract tests |

## Frozen Invariants

| Invariant | What must not regress | Locked by |
| --- | --- | --- |
| Version surfaces do not drift | Public pages cannot advertise one release while `package.json`, service worker defaults, plans and upgrade labels, or offline verifier artifacts advertise another. | `scripts/check_release_lock.mjs` |
| The homepage cannot rediscover already-known hero truth | If live proof, live twin preview, or live market entry data is already known at render time, the front door must not fall back to a weaker static shell first. | homepage hero loaders plus current hero runtime components |
| Profile uploads and downloads cannot escape the proof model | Uploaded originals and downloaded originals must remain canonical, verifiable artifacts even when legacy streamed-original repair paths are required. | profile upload/download routes and artifact-verification helpers |
| Twin portraits remain opt-in as profile identity | Generating a twin portrait must not automatically replace the user-facing profile picture. Explicit apply action remains required. | world forge UI plus profile patch path |
| Passkey and funding continuation cannot dead-end | A successful authentication step or funding step must not strand the user away from the pending market action because of missing wallet bootstrap state. | market continuation and wallet-summary contracts |
| Production policy and index hardening remain deployed requirements | The release is not fully live if the RLS, auth-policy, or high-value-index migrations are missing from the target environment. | migration contract tests plus deployment procedure |

## Release Gates

`v60.0.0` is considered locked only if these pass together:

- `pnpm exec tsc --noEmit --pretty false`
- `npx tsx --test app/lib/market/__tests__/marketTradeContinuationContract.test.ts app/lib/market/__tests__/marketConformanceRouteContract.test.ts app/lib/world/__tests__/worldAvatarFreshnessContract.test.ts app/lib/world/__tests__/worldAvatarGenerateRouteContract.test.ts app/lib/world/__tests__/worldAvatarTransparencyProcessing.test.ts app/lib/__tests__/supabaseRlsInitPlanMigration.test.ts app/lib/__tests__/supabaseHighValueFkIndexesMigration.test.ts`
- `pnpm build`
- `pnpm release:lock`
- `pnpm test:release-freeze`

The manual release procedure is defined in `docs/releases/v60.0.0-checklist.md`.

## Residual Risks

- `v60.0.0` rolls up the full post-`v58.0.0` delta, including the absent `v59.x` public release interval. The release-freeze suite still matters because some drift classes only show up at the whole-repository boundary.
- Legacy streamed-original recovery is much stronger now, but it still depends on production storage/write access remaining healthy for the reseal and backfill path.
- Twin portrait separation is explicit in the current routes, but any future avatar path that writes directly to `avatarUrl` without the explicit apply step can reintroduce identity drift.
- The schema hardening in this release only helps once the target environment actually applies the three new migrations.
