## Frozen Invariants

- A settled visible shell is immutable unless the user explicitly interacts.
- Stronger known deterministic truth is reused before network rediscovery.
- Weaker partial payloads never downgrade stronger known truth.
- Authenticated enrichment may append only; it may not slow or replace the base shell.
- `/<username>` remains canonical profile truth.
- `/<username>?manage=1` remains canonical profile truth plus owner permissions.
- Verified stored truth is permanent known truth, not stale mutable cache.
- Identity Record, Identity Seal, Receiz Key, Receiz ID, PBI/passkey, email, and server-session paths converge on the same active signed-in shell after proof acceptance.
- `/api/auth/receiz-id/bind` remains an idempotent append and must not depend on Redis replay guards.
- Settlement and Reserve value paths preserve funded causality.
- Signal Cards remain durable Receiz assets with additive DNA.
- Sports Cards remain durable Receiz assets with additive issuance, career, market, and provenance.
- Sports base cards remain base cards unless an owned issued rarity variant exists.
- Player strength and card rarity stay separate.
- Published pack odds and observed pack simulation must agree within deterministic sampling tolerances.
- Pick 5 per-game and Main Event contexts stay separate.
- Live field state belongs to the selected game, not to entry lock state.
- Pregame field state shows selected matchup truth and team star cards without inventing live play.
- Active entries may drive "entered" state; they may not replace live play state.
- Finalized entries may drive proof/reward fallback when live game context is no longer enough.
- Main Event reward, payout, result, and play-again flows must resolve without trapping users in old state.
- Single-game finals remain selectable for reward claim when claim is pending.
- Claimed proof actions must not create new locks for old games.
- Card memory appends persist across days.
- Sports conformance history is signed/rolled up as official release evidence.
- Public hot snapshots may speed delivery; they may not become authority over stronger account truth.
- Scale hardening must reduce duplicate work without creating stale visible shells.
- Release labels and service-worker defaults must match the package version.
