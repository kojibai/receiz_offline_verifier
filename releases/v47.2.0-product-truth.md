# Receiz v47.2.0 Product Truth

Release date: April 9, 2026

This document freezes the primitives and invariants that `v47.2.0` is allowed to extend but not weaken.

## Frozen Primitives

| Primitive | Locked truth | Primary enforcement |
| --- | --- | --- |
| Bound custom-domain root routing | A verified Business custom hostname must resolve its root document request to the bound public profile surface instead of falling through to the generic site root or 404 path. | `scripts/test_proxy_security_headers.ts`, `app/_business-domain/[hostname]/page.tsx`, `proxy.ts` |
| Forwarded-host authority for edge routing | Production custom-domain routing must use the effective forwarded host when deciding whether to rewrite or canonicalize. | `scripts/test_proxy_security_headers.ts`, `proxy.ts` |
| Public-profile continuity on custom domains | Existing bound profile routes remain the truth behind the custom domain; the root entry path is a routing concern, not a second profile system. | `app/_business-domain/[hostname]/page.tsx`, `app/[username]/page.tsx` |
| DNS operator clarity | DNS guidance may stay provider-backed, but the account panel must expose the concrete record name and value cleanly enough to execute without transcription friction. | `app/account/page.tsx` |
| Release-governed version surfaces | Package version, release docs, public version badges, and service worker defaults must move together on each release. | `scripts/check_release_lock.mjs` |

## Frozen Invariants

| Invariant | What must not regress | Locked by |
| --- | --- | --- |
| Custom-domain root cannot fall through to Receiz 404 | A live custom domain at `/` must not render the generic Receiz not-found surface when a binding exists. | `scripts/test_proxy_security_headers.ts` |
| Forwarded-host custom-domain traffic cannot be canonicalized away | If the forwarded host is a bound custom domain, routing must stay on that domain instead of treating the request as canonical `receiz.com` traffic. | `scripts/test_proxy_security_headers.ts`, `proxy.ts` |
| Nested custom-domain routes still pass through | The root rewrite fix must not break non-root custom-domain routes that already work. | `scripts/test_proxy_security_headers.ts` |
| DNS setup remains provider-transparent but user-executable | Users can still see the required provider-backed records, but can copy them directly from the account panel. | `app/account/page.tsx` |
| Release surface inventory stays canonical | The committed release surfaces and version badges must align with the package and service worker release. | `scripts/check_release_lock.mjs` |

## Release Gates

`v47.2.0` is considered locked only if these pass together:

- `npx tsx scripts/test_proxy_security_headers.ts`
- `npx eslint proxy.ts app/account/page.tsx scripts/test_proxy_security_headers.ts`
- `pnpm exec next build --webpack`
- `pnpm release:lock`
- `pnpm release:attest`
- `pnpm release:verify-attestation <path>`

The manual release procedure is defined in `docs/releases/v47.2.0-checklist.md`.

## Residual Risks

The following areas are materially stronger after this release but are not yet absolute:

- The routing fix covers the forwarded-host mismatch proven by the proxy regression test, but production correctness still depends on a valid binding record and a genuinely live provider-side domain attachment.
- The DNS setup panel is easier to execute, but it still reflects Vercel-backed records; this release does not introduce a non-Vercel white-label DNS abstraction.
