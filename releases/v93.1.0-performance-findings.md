# Receiz v93.1.0 Performance Findings

Release date: June 26, 2026

## Executive Finding

`v93.1.0` makes the v93 speed lesson explicit across Receiz: admitted proof truth is durable local truth, not stale cache.

The fastest correct path is first admission, then append forever. Once a device has verified proof memory, routes should project that memory immediately and ask hidden work only for verified additions after the known head.

## Findings

### 1. Durable Proof Memory Is Faster Than Re-Proving Old Truth

Sports card history proved the rule. Reopening a known proof object should not wait for broad DB reads to rediscover the history already admitted locally.

Known Sports history, Calendar events, Account activity nodes, wallet settlement rows, and offline verifier release surfaces all preserve stronger truth by projecting the known object first.

### 2. Append Windows Beat Full-History Hydration

Full-history hydration remains valid for first admission and recovery. After admission, it is the wrong normal path.

The performant path is to keep the known proof head, then ask what verified additions exist after that head. This protects history while reducing redundant server, DB, and global-sync work.

### 3. Shared Proof Projection Removes Route Duplication

Calendar and Account became faster and more truthful when they stopped acting like separate route memories. They are projections over the same admitted event proof objects.

Projecting once from the durable proof register avoids duplicate hydration, route-specific drift, and weaker temporary state replacing stronger admitted truth.

### 4. Kai-Ledger Heads Are Settlement Performance State

Wallet history performs correctly when Kai pulse/upulse order is treated as ledger state. Chronos remains display and fallback context.

The wallet asks for additions after the known Kai ledger head. It does not sort old settlement truth into provisional Chronos windows before known Kai truth can paint.

### 5. Bounded Sports Reads Must Discover Additions, Not Rebuild Truth

Sports vault, event proof, visible append, and showcase hardening reduced DB pressure by bounding reads and using RPC projection. The release lesson is not to remove proof history.

The release lesson is to stop re-querying old proof and make hot paths discover additions beneath already-admitted proof memory.

### 6. SDK Convenience Must Not Become A New Cache Authority

The SDK, OpenAPI console, fixtures, and conformance tests improve developer speed only when they remain mechanics beneath sealed artifacts, proof bundles, verified registers, and route-bound proof payloads.

SDK responses may help project proof. They may not outrank the proof object.

### 7. Release-Version Coherence Is A Runtime Boundary

Service-worker version truth, package metadata, offline verifier labels, release docs, and SDK version constants must move together. Drift creates stale installed shells and ambiguous evidence.

The local offline verifier repository protects this boundary through release-surface checks and explicit v93.1.0 docs.

## Locked Performance Invariants

- No performance fix may treat admitted proof truth as stale cache.
- No normal reopen path may require full-history hydration when durable proof memory exists.
- No hidden network, DB, RPC, server, or global-sync work may erase stronger admitted truth.
- No shared proof object may fork into competing Calendar and Account authorities.
- No wallet settlement projection may make Chronos outrank Kai order when Kai state exists.
- No Sports hot path may restore broad proof reads to rediscover already-known appends.
- No SDK convenience payload may outrank sealed artifact truth, proof bundles, verified registers, or route-bound proof payloads.
- No release surface may advance package metadata without service-worker, offline verifier, release-doc, and public-label coherence.

## Release-Lock Requirement

`pnpm release:lock 93.1.0` must require this document for `v93.1.0` and later release cuts. A release that omits performance findings is incomplete because durable proof memory, append-window discovery, shared proof projection, Kai-ledger ordering, bounded Sports projection, SDK convenience boundaries, and service-worker release truth would not be explicit release truth.
