# Receiz v48.0.0 Release Lock Checklist

Release date target: April 10, 2026

## Purpose

This checklist locks `v48.0.0` to one exact release state without allowing regression in operator-first mobile contacts operation, mobile atlas readability, linked-account import merges, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v48.0.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
pnpm exec eslint app/contacts/ContactsPageExperience.tsx
pnpm exec eslint app/world/WorldExperience.tsx
pnpm test:contacts-release-contracts
pnpm test:world-release-contracts
pnpm release:lock
pnpm test:release-freeze
```

Environment and deployment preflight for production lanes:

- the production database must already include `supabase/migrations/20260409163000_contacts_crm.sql`
- server-side Supabase admin/service-role credentials must remain available for authenticated contact sync and world action-ledger projection
- service worker and public offline verifier artifacts must all expose `v48.0.0`

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- `/contacts` loads on mobile with the command shell and primary controls visible without an initial scroll wall
- the mobile contacts summary and tools overlays open cleanly and preserve access to edit, log, invite, schedule, and history
- the mobile contacts roster reads as an operator queue rather than a plain recent-contact stack
- `/world` loads on mobile with `World atlas` naming, a readable map frame, and no overlapping label clutter inside the atlas viewport
- the mobile world lower rail separates `Atlas focus` from `Return signal` instead of repeating the same summary twice
- importing data that upgrades an existing person to a linked account persists that identity upgrade
- public version badges and offline verifier surfaces all show `v48.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v48.0.0`.
2. Run the `Release Governance Attestation` workflow with the required approval identities and signing secrets.
3. Verify the uploaded release attestation artifact.
4. Store the attestation path and approval reference with the release record.

## Release Rule

If any contacts operator regression, world atlas readability regression, import-merge identity regression, version-surface mismatch, release-lock check, or attestation step fails, `v48.0.0` is not released from that commit.
