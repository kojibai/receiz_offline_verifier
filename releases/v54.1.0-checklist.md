# Receiz v54.1.0 Release Lock Checklist

Release date target: April 15, 2026

## Purpose

This checklist locks `v54.1.0` to one exact release state without allowing regression in authenticated world first paint, market-to-profile route continuity, snapshot-driven world loading, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v54.1.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
pnpm exec eslint app/world/loading.tsx app/lib/world/__tests__/worldRouteHydrationContract.test.ts
npx tsx --test app/lib/world/__tests__/worldRouteHydrationContract.test.ts
npx tsx --test app/lib/profile/__tests__/profileRouteContinuityContract.test.ts
pnpm test:page-first-paint
pnpm exec tsc --noEmit --pretty false
pnpm release:lock
pnpm test:release-freeze
```

Before release, confirm:

- no new migrations are required for this patch cut
- public/offline verifier surfaces, service worker version, and public version badges all show `v54.1.0`

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- a signed-in hard load of `/world` keeps authenticated shell chrome throughout first paint instead of flashing public-visitor state
- an anonymous hard load of `/world` still shows the public preview state
- the signed-in world route still reconciles into the richer bootstrap payload once `/api/world/me` resolves
- bottom-bar profile navigation from `/market` lands on `/profile` through client-side app navigation instead of a browser page load
- the world route does not fall back to a bespoke skeleton that differs from the real page shell
- public version badges and offline verifier surfaces all show `v54.1.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v54.1.0`.
2. Run the `Release Governance Attestation` workflow if this cut still requires a signed attestation.
3. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
4. Confirm the attestation references the expected release version and published key ID.
5. Store the attestation path and approval reference with the release record.

## Release Rule

If any signed-in world-shell regression, market-to-profile continuity regression, anonymous-preview regression, snapshot-loading regression, version-surface mismatch, or release-lock failure occurs, `v54.1.0` is not released from that commit.
