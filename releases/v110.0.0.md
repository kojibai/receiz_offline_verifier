# Receiz v110.0.0

Release date: 2026-07-18

Receiz v110 is the coordinated unified artifact admission and recovery release for `@receiz/sdk`, `@receiz/mcp-server`, and `@receiz/ai-skills`.

Durable proof memory remains **first admission only, then append forever**.

## Headline API

```ts
const { admission, recoveryPlan } = await receiz.artifacts.admitAndRecover(artifact);
```

The SDK verifies the exact enclosing artifact bytes locally, classifies the artifact as `canonical-identity`, `bearer-recovery`, `verified-legacy-read`, `foreign-owner`, or `invalid`, restores the latest verified state and proof history, and returns explicit `canRestore`, `canSign`, `canClaimOwnership`, `canPublish`, and `canSettle` actions. The coordinator performs zero writes and zero network calls.

## Developer capabilities

- `artifacts.admit` provides one typed admission boundary for Identity Records, native proof objects, supported legacy artifacts, portable vault/proof objects, and invalid card-only payloads.
- `artifacts.planRecovery` creates a deterministic, SDK-branded, read-only recovery plan with prior/next head, recovered state, parent-linked proof history, and bounded non-authoritative explanation.
- `artifacts.admitAndRecover` composes admission and planning once without hidden mutation.
- `artifacts.commitRecovery` accepts only an SDK-issued plan, runtime-verified Ed25519 capability, stable idempotency key, and caller-provided local atomic store.
- Recovery commit atomically records verification evidence, projected state, proof history, continuity head, receipt, idempotency record, outbox intent, and audit reference—or changes nothing.
- SDK and MCP expose the same operation identity for deterministic AI/tool replay protection.
- Multi-application conformance covers create, evolve, transfer, offline restore, reconnect, reverse transport order, unknown namespaces, and deterministic convergence.

## Authority guarantees

Bearer possession cannot become canonical identity, signing, publishing, or settlement authority. Legacy and foreign-owner admissions remain read-only. Raw capability-shaped JSON is rejected. Explanations and plans never become proof authority; the verified enclosing proof object and verified history remain stronger truth.

## MCP and AI

MCP adds `receiz_artifact_admit`, `receiz_artifact_recovery_plan`, `receiz_artifact_admit_and_recover`, and `receiz_artifact_recovery_commit`. Commit JSON carries no raw capability or store. Artifact, identity, ownership, continuity, migration, testing, app-builder, MCP-agent, and release skills teach the same verdicts, actions, history, error, idempotency, and authority boundaries.

## Receiz.com and database boundary

No Receiz.com application behavior, API route, database schema, migration, query path, or production data changed. Visible product surfaces only identify the coordinated v110 release. Existing native Record → Seal issuance logic is unchanged. V109 local verification mechanics remain the foundation and are composed by v110 admission.

Repository qualification does not publish npm packages, deploy Receiz.com, issue a signed release attestation, or mutate production data. Those are separate authorized external actions.
