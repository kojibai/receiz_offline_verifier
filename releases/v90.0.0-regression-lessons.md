# Receiz v90.0.0 Regression Lessons

Release date: June 14, 2026

This document is part of the release lock. It records the regression classes found or carried into the `v90.0.0` release arc and the rules that must prevent the same mistakes.

## What Regressed

### Release Documentation Lagged Behind Product Truth

The repository had already moved beyond the last release story. `package.json`, service worker, runtime coherence, public labels, offline verifier labels, and post-v89 Sports proof work were not all presented as one official stable release.

That is a release regression because Receiz release docs are product-truth artifacts, not marketing notes. A major release must make the implemented primitive arc understandable and governed.

### Version Surfaces Were Split

The package and service-worker/runtime defaults were already at `89.0.1`, while public release labels and offline verifier surfaces still displayed `v89.0.0`.

The risk was runtime-coherence drift:

```txt
package/runtime truth says one release
public/offline surfaces say another release
```

For Receiz, release labels are product surfaces. They must move with package and service-worker identity.

### Sports Single-Game Entry Trusted Stale Client Lineup Transport

The post-v89 Sports path exposed the stale lineup payload boundary.

A selected game could be valid, the user could own visible matchup Sports Vault proof cards, and the server could still reject because the client submitted stale `selectedCardIds` from a previous matchup.

That treated client payload transport as stronger than the settlement admission primitive.

Correct boundary:

```txt
verified game identity
owned Sports Vault proof card identity
server-side card claim verification
matchup team normalization
lineup composition from verified owned cards
client selectedCardIds as useful transport only
```

### Sports Ownership Transfer Needed The Existing Append Boundary

Claimed/uploaded card images could not become signed-in local vault truth before the existing offline-transfer ownership append route admitted the transfer.

The implemented primitive already existed. Adding or relying on a new direct local custody shortcut would create a new authority layer above the ownership append.

### Sports Vault Projection Needed Owner Rows And Transferred-Out State

Sports Vault projection needed owner-row projection, `SportsVaultSurface`, transferred-out vault IDs, and normalized team-code behavior so proof/custody state remained stable across vault, reveal, transfer, and public projection.

Without this, a user could see a vault state that was legible as UI but not precise enough as ownership truth.

### Event Proof Metadata Could Drift From Public Proof Projection

Sports event proof metadata needed absolute URLs so public/social metadata projects the proof route correctly.

Metadata is still a lower layer:

```txt
event proof object -> public proof route -> metadata projection
```

Metadata may project public proof. It does not define proof.

## Why It Regressed

### Release Mechanics Were Treated As Separate From Product Truth

Version labels, service worker defaults, offline verifier labels, and release docs were easy to treat as mechanical. In Receiz, they are release surfaces because they define what runtime and verifier the user is seeing.

### Client Experience State Was Treated As Admission Truth

The stale Sports lineup case happened because selected client card IDs were allowed to carry too much authority.

For a settlement primitive, transport and experience state can suggest intent. They cannot outrank verified game identity and verified owned card truth.

### Generic Local Projection Was Too Tempting

For Sports ownership transfer, the tempting shortcut was to update signed-in local vault state directly. The correct path is stricter: use the implemented offline-transfer ownership append route and then project local vault truth.

### Projection Completeness Was Not Always Treated As Ownership Truth

Vault projection, owner rows, transferred-out IDs, team-code normalization, pack reveal state, and public vault surfaces all look like UI details. They are ownership and artifact-system projection details. If they are incomplete, ownership truth becomes harder to read.

### Metadata Was Close To The Public Route Surface

Open Graph and public metadata are route-adjacent, so they can be mistaken for public proof authority. They are not authority. They must project the public proof route that is already true.

## Locked Rules

### Release Rules

- Major releases must include release note, product truth, process, checklist, regression lessons, and commit history when closing a long arc.
- `v90.0.0` must document every commit from `v80.0.0` through the release candidate.
- Package version, service worker version, runtime coherence defaults, public labels, offline verifier labels, and release docs must move together.
- Release lock must fail when required release documents or version surfaces are missing.

### Sports Entry Rules

- Single-game Sports entry is a settlement primitive.
- Client lineup payload is transport/experience state.
- Server entry admission must verify selected game identity.
- Server entry admission must reject final games.
- Server entry admission must verify owned Sports Vault proof cards.
- If submitted IDs are stale, the server must keep valid owned matchup cards and fill from the owner’s verified visible matchup proof cards before rejecting.

### Sports Ownership Rules

- Sports card custody is an ownership surface.
- Claimed/uploaded card images must go through the existing offline-transfer ownership append route before signed-in local vault commit.
- Do not create a new authority layer when an implemented Receiz ownership primitive already carries the boundary.
- Transferred-out IDs must prevent transferred custody from resurfacing as owned vault truth.

### Sports Vault Rules

- Sports Vault is an artifact system.
- Sports Vault proof truth starts from sealed manifest/image and verified vault card identity.
- Owner-row projection, `SportsVaultSurface`, normalized team codes, transfer state, and public owner vault routes are projection obligations beneath vault proof truth.
- Browser payload, DB row, session state, and UI cache may not replace sealed vault proof truth.

### Event Proof And Metadata Rules

- Sports event proof is a proof object.
- Event proof hero card, tool dock, link actions, rarity odds, tier, impact, headline labels, media, and metadata project proof truth.
- Exact moment evidence outranks broad highlight enrichment.
- Metadata URLs must be absolute and public.
- Metadata must never become proof authority.
