# Receiz v47.6.0 Release Lock Checklist

Release date target: April 9, 2026

## Purpose

This checklist locks `v47.6.0` to one exact release state without allowing regression in atlas-first world operations, concierge twin-quarterbacking, contacts CRM behavior, outreach integrity, checkout-completion truth, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v47.6.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
npx tsx --test app/lib/world/__tests__/conciergeQuarterback.test.ts app/lib/world/__tests__/outreachDraft.test.ts app/lib/world/__tests__/outreachRouteContract.test.ts app/payments/embed/checkout/__tests__/completionVerificationContract.test.ts
pnpm exec tsc --noEmit --pretty false
pnpm exec next build --webpack
pnpm release:lock
pnpm test:release-freeze
```

Environment preflight for production lanes:

- `RESEND_API_KEY` and `RECEIZ_FROM_EMAIL` must be present anywhere Receiz email outreach is intended to be live.
- Twilio credentials and one sender lane must be present anywhere World SMS outreach is intended to be live.
- Favorite-app webhook destinations must remain public, operator-controlled destinations.
- Browser-native contact import depends on client support for the Contacts Picker API; unsupported clients should still be able to use file import/export.

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- `/world` districts load into the atlas-first world map and district focus changes without layout regression
- The concierge still renders in the same shell, and a detailed prompt can start a visible live twin run with stage and log updates
- `/contacts` imports and exports contacts cleanly, keeps notes and last-meeting state person-first, and launches from `/account`
- Private/internal webhook targets are rejected before send, and failed email sends do not leave orphaned outreach chat artifacts behind
- Completion-only embedded checkout returns only report success when the verified session is actually paid
- Public version badges and offline verifier surfaces all show `v47.6.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v47.6.0`.
2. Run the `Release Governance Attestation` workflow with the required approval identities and signing secrets.
3. Verify the uploaded release attestation artifact.
4. Store the attestation path and approval reference with the release record.

## Release Rule

If any atlas regression, concierge orchestration regression, contacts-surface regression, outreach-integrity regression, checkout-truth regression, release-lock check, or attestation step fails, `v47.6.0` is not released from that commit.
