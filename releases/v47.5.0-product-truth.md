# Receiz v47.5.0 Product Truth

Release date: April 9, 2026

This document freezes the primitives and invariants that `v47.5.0` is allowed to extend but not weaken.

## Frozen Primitives

| Primitive | Locked truth | Primary enforcement |
| --- | --- | --- |
| World revenue-skill policy | Revenue behavior in the public twin must remain explicitly owner-configured policy, normalized server-side, and grounded into runtime behavior rather than being inferred from UI state alone. | `app/lib/world/revenueSkills.ts`, `app/lib/world/policy.ts`, `app/lib/world/server.ts`, `app/lib/world/runtime.ts` |
| Multi-channel outbound twin delivery | Receiz email, SMS, and app-webhook outreach must remain explicit per-channel owner policy and must not send from unsaved local state. | `app/lib/world/outreach.ts`, `app/api/world/outreach/route.ts`, `app/world/WorldExperience.tsx` |
| Owner-side revenue playbooks | Post-call closeout, warm lead follow-up, and renewal nudges must load concrete drafts and route users toward one clear next step using existing pay, schedule, proof, or twin rails. | `app/lib/world/revenuePlaybooks.ts`, `app/world/WorldExperience.tsx` |
| Public twin action-lane expansion | Quote, deposit, proof, and renew rails must remain additive expansions on top of the existing intake, routing, handoff, schedule, and pay model rather than replacing or weakening it. | `app/lib/world/runtime.ts`, `app/u/[username]/WorldProfileModule.tsx`, `app/lib/world/types.ts` |
| Productized business-entry surfaces | Proof kits, action-card embeds, booking confirmation packs, call closeout packs, and the developer site kit remain productized wrappers around existing primitives rather than new trust models. | `app/lib/recordKits.ts`, `app/action-card/`, `app/calendar/confirmation/`, `app/components/chat/ReceizCallPanel.tsx`, `app/developers/site-kit/` |
| Release-governed version surfaces | Package version, service worker version, public badges, offline verifier surfaces, and release docs must move together on each release. | `scripts/check_release_lock.mjs` |

## Frozen Invariants

| Invariant | What must not regress | Locked by |
| --- | --- | --- |
| Revenue skills cannot silently activate | The public twin may not behave as if revenue skills are live unless the saved world policy says they are. | `app/lib/world/policy.ts`, `app/world/WorldExperience.tsx`, `app/api/world/outreach/route.ts` |
| Outbound channels stay governed | Email, SMS, and app-webhook delivery must each fail closed when their corresponding saved policy lane is not live. | `app/api/world/outreach/route.ts`, `app/world/WorldExperience.tsx` |
| Operator playbooks must resolve into real next steps | Playbooks must not collapse into generic copy; they must preload a concrete goal, channel, and route-aware CTA. | `app/lib/world/revenuePlaybooks.ts`, `app/world/WorldExperience.tsx` |
| Public twin still stays conversational first | Revenue skills may sharpen buying flows, but the twin must still avoid forcing scheduling, payment, or sales when the visitor is just having a normal conversation. | `app/lib/world/runtime.ts`, `app/lib/world/__tests__/showcaseSelectionContract.test.ts`, `app/lib/world/__tests__/worldConversationRelayContract.test.ts` |
| Expanded action rails remain additive | New quote / deposit / proof / renew actions must not remove or break the earlier intake / routing / handoff / schedule / pay lanes. | `app/lib/world/runtime.ts`, `app/u/[username]/WorldProfileModule.tsx` |
| Release surface inventory stays canonical | The committed release surfaces and version badges must align with package and service worker release truth. | `scripts/check_release_lock.mjs` |

## Release Gates

`v47.5.0` is considered locked only if these pass together:

- `npx tsx --test app/lib/world/__tests__/policyContract.test.ts app/lib/world/__tests__/revenueSkills.test.ts app/lib/world/__tests__/revenuePlaybooks.test.ts app/lib/world/__tests__/outreachDraft.test.ts app/lib/world/__tests__/worldConversationRelayContract.test.ts app/lib/world/__tests__/showcaseSelectionContract.test.ts`
- `pnpm exec tsc --noEmit --pretty false`
- `pnpm release:lock`
- `pnpm test:release-freeze`
- `pnpm release:attest`
- `pnpm release:verify-attestation <path>`

The manual release procedure is defined in `docs/releases/v47.5.0-checklist.md`.

## Residual Risks

The following areas are materially stronger after this release but are not yet absolute:

- Operator playbooks are productized presets, not a background scheduler; they reduce repetitive work but still depend on an owner-triggered send.
- SMS and app-webhook delivery still depend on deployment configuration and external provider availability even when the product surface is correct.
- The revenue-skill runtime is materially sharper, but future message patterns may still need more deterministic intent coverage as more verticals use the twin for closing work.
- `pnpm test:release-freeze` remains the full release freeze gate and can still fail outside this local scope if unrelated repository-wide surfaces drift before tag time.
