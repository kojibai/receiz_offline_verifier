# Receiz v93.0.0 Performance Findings

Release date: June 25, 2026

## Executive Finding

Receiz became faster when performance work followed product truth instead of fighting it.

The strongest pattern was not generic caching. The strongest pattern was: use already-known proof immediately, keep the visible shell settled, then bound the hidden transport so the database hydrates only the proof payload rows that matter.

## Findings

### 1. Known Proof Is Faster Than Rediscovery

When the app already has sealed artifact truth, deterministic first-paint proof bundles, local route snapshots, archived pitch proofs, visible card appends, or entry summaries, the fastest correct path is to paint that truth immediately.

Rediscovering DB/session/global truth before rendering made pages feel slower and sometimes created visible second-load behavior. Reusing known proof first made the experience faster and more trustworthy.

### 2. Bounded RPCs Beat Broad JSON Aggregation

The slow-query work showed repeated pressure around queries that were logically scoped but physically heavy:

- broad `collectible_id = any(...)` reads ordered by `kai_upulse`
- event proof reads ordered by `created_at`
- owner/sport/card/game append reads ordered by `observed_at`
- raw proof reads over event and append payload rows
- queue reads with broad OR patterns

The winning pattern was ID-first paging:

1. Use a targeted index or materialized CTE to select row IDs.
2. Apply keyset cursors instead of offset where the path can page.
3. Join the proof table back by ID.
4. Return the exact same authoritative payload rows.

This reduced pressure without hiding history or creating weaker cache truth.

### 3. Payload Authority Should Not Become Index Cargo

Adding `append_json`, `proof_json`, or `source_json` into broad covering indexes would make reads look faster at the cost of write weight and index bloat. The better invariant is to index identity/order columns and hydrate payloads after the page is bounded.

This keeps proof payload authority in the ledger row and keeps indexes as mechanics.

### 4. Foreground And Background Need Different Cadence

Background stale guards reduce duplicate sync work. They are good pressure controls. They are wrong when they delay visible proof updates for an active user.

The release separates the two:

- background work uses leases and cooldowns
- foreground proof viewing may warm scoped appends immediately
- visible surfaces keep settled truth while new verified appends attach behind it

### 5. Full History Must Be Resolved Before It Is Compressed

The lineage and event-history fixes showed that latest-proof display is not enough when the object carries deeper history. Summaries are acceptable after resolution. Summaries are not acceptable as a substitute for resolving the proof chain.

### 6. UI Speed Comes From Stable Surfaces

Several experience gains came from not moving the surface:

- popovers inspect rather than remount
- overlays expand without pushing the command surface away
- value basis starts compact and opens intentionally
- Pack Derby keeps the first scene stable
- Sports Economy panels show proof detail without replacing the stage

This is performance as continuity, not just milliseconds.

### 7. Deploy Timing Is A Performance Boundary

When new SQL functions back live routes, the deploy must respect DB-first ordering. If route code reaches a fresh serverless instance before the RPC is visible to PostgREST, users may see partial enrichment even though proof rows still exist.

The release rule is: migration first, deploy second, observe errors third, reset slow-query stats fourth.

## Locked Performance Invariants

- No slow-path fix may remove proof history to look faster.
- No first-paint surface may wait for weaker truth when stronger local/artifact truth exists.
- No broad proof payload read may remain on a hot path when an equivalent bounded RPC exists.
- No index may become the product authority for proof payloads.
- No foreground proof surface may wait behind a background stale guard.
- No deploy should reset pg_stat_statements before the DB migration and route deployment are both live.

## Release-Lock Requirement

`pnpm release:lock 93.0.0` must require this document for `v93.0.0` and later release cuts. A release that omits performance findings is incomplete because bounded proof-read projection, payload authority, foreground append cadence, and deploy-order timing would not be explicit release truth.
