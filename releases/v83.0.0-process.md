# Receiz v83.0.0

## Goal

Ship `v83.0.0` as the stable live-proof release after the `v82.0.0` Sports launch.

The release candidate is `223b2ddbbd` plus the final release-lock worktree. The release result is a Sports, profile, Explore, Market, OTC, and conformance system that preserves instant first paint, truthful known state, and proof continuity while hiding hydration and enrichment behind settled surfaces.

## What Worked

- Treating first paint as product law made the Sports fixes sharper. The question was not "can this eventually hydrate?" but "what truthful shell can the user see immediately?"
- Sharing the single-game tab path with the scoreboard path fixed the first selected-game score without inventing a second model.
- Separating entry lock state from scoreboard and field state removed a class of misleading Sports UI.
- Preserving legitimate zero scores avoided fake loading and made live scoring easier to trust.
- Fixing card reveal input state at the stage level protected purchase cards, reward cards, drag/drop, click-to-open, and deselection with one behavior contract.
- Moving issue number into proof-event continuity made saved cards behave like assets instead of screenshots.
- Keeping live-player playback on canonical public profile truth restored iMessage behavior without making owner/manage state part of the hot path.
- Cleaning SEO hot-path work without moving canonical truth kept profile and market surfaces fast.
- Treating Market conformance as stale rather than slowing `/market` kept test law aligned with product law.
- Centralizing callback policy made interoperability conformance inspect the real source of truth.
- OTC proof objects worked because they treated offline verification as a first-class outcome, not a temporary cache before chain reconciliation.

## What Did Not Work

- Rechecking known Sports lock state before first paint created the wrong user feeling. Known state should project, and unknown enrichment should hydrate later.
- Allowing single-game tab selection to behave differently from scoreboard selection made the first game look broken even when the data existed.
- Auto-selecting after a user deselected a card contradicted the interaction truth. Deselection is an explicit user state.
- Viewer close logic that left drag/drop degraded showed that popovers must not leak input state into the stage.
- "Pending" issue copy on saved cards was a proof transfer failure, not a copy problem.
- JSON-LD and structured proof data were too heavy for hot paths when the user needed the shell first.
- Conformance that expects removed hot-path artifacts is worse than no conformance because it pressures the product toward slower behavior.
- Live-player fixes that restore iMessage but crash the page are not fixes.

## Practical Rule For Future Work

The `v83.0.0` rule is:

```txt
paint known truth first
hydrate hidden data next
append verified additions only
never replace a settled shell to prove something already known
```

For Sports, the order is:

```txt
selected context
known lock state
known score state
visible shell
hidden hydration
lineup availability
live deltas
final proof
card memory
market value
```

For profile, Market, Explore, and proof objects, the same rule applies with different nouns. The release should feel stable because the product stops asking the user to wait while it rediscovers truth it already owns.
