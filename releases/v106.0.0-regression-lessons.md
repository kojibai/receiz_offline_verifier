# Receiz v106.0.0 Regression Lessons

## What Regressed

For a logged-in owner, the newest profile proof object could appear in Showcase while its PBI signature fell off and its Live Player route reported unavailable. Logged-out global projection still worked.

## Why It Regressed

The local publish path returned as soon as it found an existing canonical profile row. That row proved global visibility, but it did not prove that the current device had admitted the new proof object, merged its accepted PBI authorship, or written its Live Player snapshot. A weaker older snapshot was allowed to defeat a stronger append.

## Locked Rules

- Global visibility and local admission are distinct append obligations.
- A canonical row may enrich local truth; it may not short-circuit it.
- Accepted PBI authorship cannot disappear during reconciliation.
- The newest admitted proof object must remain available to Showcase and Live Player.
- Proof memory is **first admission only, then append forever**.
- `@receiz/sdk` and product implementation are corrected before contracts are strengthened.

## Contract Gates

Profile release contracts now bind local admission, PBI signature persistence, Live Player projection, and stale-snapshot exclusion. The v106 release lock requires those gates inside the full release freeze alongside constitutional, migration, package, governance, and performance evidence.

## Release-Lock Requirement

`pnpm release:lock 106.0.0` must require this document for `v106.0.0` and later release cuts. A release that omits these lessons is incomplete because canonical-row visibility could again be mistaken for owner-local admission, allowing weaker server or snapshot state to erase an accepted PBI signature or Live Player append.
