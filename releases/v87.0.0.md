## Title

Receiz `v87.0.0` - Offline Proof Links, QR Proof Transport, And Post-v86 Public Proof Lock

## Summary

`v87.0.0` documents and locks the work from the `v86.0.0` tag through the current release candidate.

The post-`v86.0.0` arc spans 17 commits and, before the final v87 release docs/version edits, 77 changed files, 6,903 insertions, and 455 deletions. The main product result is specific: offline sealed proof objects and offline moments now produce compact self-contained `/v` proof links that the `/v` public proof surface verifies as `Proof` from the embedded Signature V4 proof bundle, without requiring profile-composer publication or server-side rediscovery.

The release also locks the post-v86 Sports single-card QR cockpit, Sports card value surface repairs, World Estate slice, market/profile asset and first-paint hardening, and RLS/security cleanup.

## v86.0.0 Baseline

`v86.0.0` froze Receiz as a proof-native artifact system:

- deterministic proof and artifact truth outrank server/session/cache state
- first paint is protected across proof, identity, ownership, settlement, market, vault, and public witness surfaces

`v87.0.0` does not replace that baseline. It extends it by making offline-generated proof objects honor their own sealed proof truth on `/v`.

## Offline Sealing And Offline Moments

This is the headline v87 proof upgrade.

Before this work, an offline sealed file or offline moment could carry a real Signature V4 proof bundle and still land on a `/v` route that displayed `Historical`. The root cause was route recognition: `/v` treated the existing proof query payload path as wallet transaction payload only, then fell back to coordinate-only offline evidence. Coordinate-only evidence is historical compatibility evidence; it is not the same as a signed proof bundle.

The v87 behavior is now direct:

- offline sealing produces a proof-bearing `/v` URL
- offline moments preserve that proof-bearing `/v` URL
- `/record` and `/make` keep the proof-bearing URL instead of reconstructing coordinate-only links
- `/v` decodes the compact proof token, reconstructs the signed `receiz.proof_bundle`, verifies Signature V4, checks route binding, and classifies the page as `Proof`
- the proof object does not need to be published from profile composer before `/v` can recognize it
- `a` and `ms` query fields are removed from the transport URL once their truth is carried inside the signed proof bundle
- legacy `rpb1:` compressed JSON proof tokens remain decodable
- old coordinate-only evidence remains historical compatibility evidence, not upgraded authority

The compact production proof link measured for the signed fixture is 575 characters including `https://receiz.com`. That is short enough for QR code, text message, and social sharing while still carrying the self-contained proof material needed for offline verification.

## Compact Proof Token

`v87.0.0` adds binary compact proof token transport for public `/v` links.

The token is transport only. It is not a new authority layer. Authority remains the sealed proof object:

- `receiz.proof_bundle`
- Signature V4 bundle signature
- Signature V4 device certificate
- payload hash
- artifact basis hash
- claim identity
- route-bound slug, code, and pulse
- proof digests needed for public proof inspection

The binary transport derives route truth where possible, stores only the required bytes, and reconstructs the proof bundle before the existing verifier performs proof validation. The default pinned Signature V4 issuer is reconstructed by the decoder instead of repeated in the token. Non-default issuer IDs remain stored.

This preserves the proof boundary:

```txt
sealed proof bundle -> compact URL transport -> /v decoder -> Signature V4 verification -> public proof surface
```

It does not introduce:

- server lookup as authority
- profile publication as permission
- a new issuer/signature layer
- a weaker metadata wrapper
- coordinate-only proof promotion

## QR Code Behavior

The live offline moment card QR now uses the proof-bearing compact `/v` URL after sealing. This fixes the user-visible failure where the card QR could still scan to a historical coordinate URL after the sealed proof object itself verified offline.

The important distinction remains:

- live card QR: points to the compact self-contained proof URL
- sealed artifact truth: remains bound to the artifact basis and proof bundle
- `/v` route: verifies the proof bundle and shows `Proof`

The QR update is aligned with artifact truth because it projects the proof object link already carried by the seal result. It does not weaken file verification or make QR transport stronger than the sealed artifact.

## `/v` Public Proof Surface

`/v/[slug]/[code]/[pulse]` now accepts compact proof tokens from offline-generated proof objects and uses them before historical fallback.

The public proof surface now distinguishes these cases:

- signed proof bundle with valid Signature V4 and matching route: `Proof`
- legacy coordinate evidence without the signed proof bundle: `Historical`
- malformed or mismatched proof material: rejected or not elevated

The route therefore aligns with Receiz law:

```txt
offline proof is not fallback mode
sealed artifact truth outranks server/session/profile state
weaker historical coordinates cannot downgrade a stronger signed proof bundle
```

## Product Statement

`v87.0.0` exists because offline proof is real proof.

If a proof object is sealed offline and carries the proof material needed to verify itself, `/v` must honor that proof object as proof. The server, profile composer, and database may publish, index, or hydrate beneath it, but they are not permission for the proof object to be true.
