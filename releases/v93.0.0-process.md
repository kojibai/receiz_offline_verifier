# Receiz v93.0.0

Release date: June 25, 2026

## Goal

Ship `v93.0.0` as the Sports Proof Economy release after `v92.0.0`.

The goal is not a generic major bump. The release locks a concrete product truth:

- Sports Economy is a public proof surface.
- Sports cards carry deterministic value basis, score memory, lineage, and visible append history.
- Play Command, Pitch Command, Pack Derby, and physical activation are proof-native surfaces.
- DB performance hardening is beneath proof authority and preserves the same visible truth.
- The speed improvement comes from less rediscovery, less broad payload scanning, and more direct reuse of known proof.

## What Worked

- The strongest speed gains came from honoring the truth hierarchy. When known proof painted first and live work appended behind it, the experience stopped feeling like it loaded twice.
- Bounded RPCs worked because they matched the actual hot path: select the indexed identity page first, then join back to authoritative payload JSON.
- Performance improved without cache downgrades. The DB became lighter while `append_json`, `proof_json`, source payloads, and sealed artifacts remained authoritative.
- Shared projection fixed drift. Card bio, value basis, event proof, and lineage bugs repeatedly came from route-local projections that forgot proof fields already available elsewhere.
- Visual evidence found real bugs. Screenshots and live behavior showed when a claimed pass missed value chart continuity, player bio details, lineages, or surface hierarchy.
- Foreground append cadence fixed active-user delay. Background stale guards were valid pressure controls, but they were not valid reasons to defer proof updates while the user watched an active Sports surface.
- Deployment order became explicit. New RPC-backed routes require the function to exist live before code paths call it.

## What Did Not Work

- Treating value as a generic market number caused drift between proof value, seller ask, customer price, cost, and margin.
- Route-specific card backs lost bio rows even though shared card renderers already knew how to display them.
- Showing only the latest transfer proof was truthful but incomplete when the card-wrapped lineage carried prior proof hashes.
- Broad PostgREST reads over JSON payloads created slow-query pressure even when the underlying proof design was correct.
- Background pressure guards delayed foreground user-visible proof updates.
- UI overlays that pushed or remounted surfaces degraded experience even when the underlying data was correct.
- Fresh deployments can temporarily expose partial live enrichment if code and DB schema visibility are not aligned.

## Practical Rule For Future Work

When Receiz already has proof truth, speed comes from using it sooner and moving weaker work behind it.

Do not solve slow surfaces by removing proof, hiding history, flattening value, replacing append ledgers with cache truth, or letting a summary outrank the proof object. Instead, identify the existing primitive, reuse the shared projection, bound the transport, hydrate payloads only after identity is bounded, and keep visible continuity stable.
