# Receiz v80.0.0

## Goal

Ship `v80.0.0` as the major stable release that documents and locks the complete product trail from `v70.0.0` public alpha through the `v78.9.1` stabilization candidate at `997418abb`: identity continuity, canonical profile truth, local proof, file restore, wallet and market value, World agent loops, Public Proof, Creator OS, Signal Circuit, Vault export, card ownership, public conformance, and runtime coherence all under one release law.

## What Worked

- Keeping `v70.0.0` as the named baseline made the release story honest: v80 is the stable system that grew from public alpha doctrine, not an unrelated feature bundle.
- The v70 invariant register gave the release a real spine: profile truth, verified local truth, append-only history, PBI/Receiz ID binding, value-loop causality, and version coherence are still the laws.
- The v74-v75 identity work made restore feel complete at proof admission. That is the product standard users actually feel: no manual refresh, no half-login, no stale cookie reasserting an older account.
- The v76 World, Public Proof, Market Twin, wallet, and Status work gave the platform visible value loops rather than abstract proof plumbing.
- The v77 Signal Circuit and Creator OS work proved Receiz can host new product categories while preserving the same truth law: cards, creator output, vault packages, document seals, and runtime recovery all append or project verified truth.
- The v78 card-world/export/arena/exchange work made Signal Cards an owned-asset network instead of a local inventory feature.
- The post-v78 stabilization was valuable because it tightened the real edges users touch: share popovers, dashboard load behavior, Vault Verify feedback, flip gestures, card discovery, Arena append proof, and official game-front copy.
- Release coherence improved by moving package version, public labels, service-worker defaults, offline verifier labels, README pointer, release docs, and worker release markers in one governed release bump.

## What Did Not Work

- A release note that only lists recent commits would undersell the product. The correct story is the v70-to-v80 platform arc: verified truth first, network append second, settled surface preserved.
- Treating dashboard speed work as inherently better was too broad. The stable release keeps the route shape that preserves the settled first screen.
- Card proof popovers cannot become full-screen replacement surfaces just because they contain richer proof data. Richer proof is still an overlay around sealed card truth.
- Vault verification feedback cannot start before user intent. The opening state belongs after click, not as fake readiness.
- Signal Card interaction cannot be generic card flipping if it risks layout churn. Gestures need a dedicated contract so interaction is explicit and stable.
- Battle and Arena appends cannot be expressed by mutating the original card. The proof manifest has to show appended state while leaving original sealed truth intact.
- Release labels are product surfaces. Version drift between package metadata, service workers, badges, offline verifiers, docs, README, and resident workers is a release blocker.

## Practical Rule For Future Work

Every future major release should explain what new product category became governed truth, then prove that it obeys the same Receiz law: settled visible shells stay stable, verified truth projects immediately, later work appends verified additions, weaker payloads cannot downgrade stronger truth, and release surfaces move together. The more powerful the product becomes, the less acceptable it is to make users watch the system rediscover what it already knows.
