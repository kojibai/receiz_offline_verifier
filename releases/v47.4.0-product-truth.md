# Receiz v47.4.0 Product Truth

Release date: April 9, 2026

This document freezes the primitives and invariants that `v47.4.0` is allowed to extend but not weaken.

## Frozen Primitives

| Primitive | Locked truth | Primary enforcement |
| --- | --- | --- |
| Effective entitlement truth | Business must survive native session-derived flows as a first-class plan and must not be flattened into Premium compatibility output when a surface needs the effective entitlement. | `app/lib/premium/plan.ts`, `app/api/session/route.ts`, `app/api/profile/bootstrap/route.ts`, `app/api/world/me/route.ts`, `app/lib/premium/__tests__/plan.test.ts` |
| Affiliate entitlement continuity | Affiliate surfaces may remain Premium-compatible for old callers, but native affiliate dashboard state must read the effective entitlement so Business and Pro are not lost. | `app/affiliate/AffiliatePageExperience.tsx`, `app/lib/affiliate/types.ts`, `app/api/affiliate/summary/route.ts` |
| Receiz-owned scheduling surface | Upgrade and WireProof scheduling flows must open the Receiz calendar embed instead of an external Calendly surface. | `app/upgrade/page.tsx`, `app/wireproof/page.tsx`, `app/wireproof/SchedulerPopover.tsx` |
| Public payments embed continuity | Public payment widgets must launch Stripe embedded checkout inside a Receiz-hosted surface instead of navigating the buyer directly to Stripe-hosted checkout. | `public/receiz-pay-embed.js`, `app/api/payments/embed/checkout/route.ts`, `app/payments/embed/checkout/page.tsx`, `app/payments/embed/checkout/EmbedCheckoutPageClient.tsx` |
| Release-governed version surfaces | Package version, service worker version, public badges, offline verifier surfaces, and release docs must move together on each release. | `scripts/check_release_lock.mjs` |

## Frozen Invariants

| Invariant | What must not regress | Locked by |
| --- | --- | --- |
| Business cannot disappear inside compatibility layers | A Business user may still carry a legacy-compatible base session value, but effective plan output for native surfaces must remain Business. | `app/api/session/route.ts`, `app/lib/premium/__tests__/plan.test.ts` |
| World access gating cannot misread Business as Premium-only legacy state | Gated-district access must recognize Business as an entitled plan with Business-specific behavior and not force a downgrade to free or Premium compatibility. | `app/lib/world/access.ts`, `app/lib/world/__tests__/worldDistrictRouteContract.test.ts`, `app/lib/world/__tests__/worldMeRouteContract.test.ts` |
| Scheduling cannot depend on Calendly in these upgrade rails | Upgrade and WireProof CTAs must not reintroduce Calendly URLs or Calendly-branded popovers. | `app/upgrade/page.tsx`, `app/wireproof/page.tsx`, `app/wireproof/SchedulerPopover.tsx` |
| Payment widget checkout cannot hard-redirect buyers out of the embed flow | The public payments widget must keep the secure checkout inside the Receiz-hosted popup surface whenever Stripe embedded checkout is available. | `public/receiz-pay-embed.js`, `app/api/payments/embed/checkout/route.ts` |
| Release surface inventory stays canonical | The committed release surfaces and version badges must align with package and service worker release truth. | `scripts/check_release_lock.mjs` |

## Release Gates

`v47.4.0` is considered locked only if these pass together:

- `npx tsx --test app/lib/premium/__tests__/plan.test.ts`
- `npx tsx --test app/lib/world/__tests__/worldDistrictRouteContract.test.ts app/lib/world/__tests__/worldMeRouteContract.test.ts`
- `pnpm exec tsc --noEmit --pretty false`
- `pnpm exec next build --webpack`
- `pnpm release:lock`
- `pnpm release:attest`
- `pnpm release:verify-attestation <path>`

The manual release procedure is defined in `docs/releases/v47.4.0-checklist.md`.

## Residual Risks

The following areas are materially stronger after this release but are not yet absolute:

- Some older internal or external consumers may still read the legacy-compatible base session shape; native Receiz surfaces now preserve effective entitlement truth, but compatibility consumers should continue migrating toward explicit entitlement reads.
- Embedded checkout continuity depends on Stripe embedded-checkout support and browser popup / iframe allowances; redirect-required payment methods are routed back into the Receiz-hosted checkout surface, but browser policy can still affect the feel of the handoff.
- Replacing Calendly on the targeted upgrade rails removes the remaining known external scheduler dependency there, but future scheduling entry points still need the same discipline to avoid drift.
- Discovery-feed cache revalidation can still log transient upstream fetch failures during production builds; current behavior does not block the build, but it remains an operational signal worth watching outside the release lock itself.
