# Receiz v47.8.1 Product Truth

Release date: April 10, 2026

This document freezes the primitives and invariants that `v47.8.1` is allowed to extend but not weaken.

## Frozen Primitives

| Primitive | Locked truth | Primary enforcement |
| --- | --- | --- |
| Durable person-first contacts CRM | Contacts must remain a first-class, person-first operating view with durable persistence, import/export portability, and per-person timeline memory rather than falling back to a local-only shell. | `app/contacts/ContactsPageExperience.tsx`, `app/api/contacts/route.ts`, `app/api/contacts/import/route.ts`, `app/api/contacts/[contactId]/route.ts`, `app/api/contacts/[contactId]/events/route.ts`, `app/lib/contacts/server.ts`, `supabase/migrations/20260409163000_contacts_crm.sql` |
| One-view mobile contacts operator shell | Mobile contacts must keep the primary operating shell compact enough to expose the main command surface without forcing the full stats and detail stack into the first viewport. Secondary detail must move behind overlays rather than being removed. | `app/contacts/ContactsPageExperience.tsx`, `app/contacts/contacts.module.css` |
| Contact identity upgrade merge | Imported contacts must preserve authenticated identity upgrades when an existing contact gains a `linked_user_id`, even if no other major field changes in the merge payload. | `app/lib/contacts/server.ts`, `app/lib/contacts/__tests__/serverImportMergeContract.test.ts` |
| Public-twin continuity inside contacts | Public twin and guest activity must continue to materialize into the same contact records that hold direct messages, notes, and meetings rather than living in a detached world-only history. | `app/lib/contacts/server.ts`, `app/lib/world/runtime.ts`, `app/contacts/ContactsPageExperience.tsx` |
| Release-governed version surfaces | Package version, service worker version, public badges, offline verifier surfaces, and release docs must move together on each release. | `scripts/check_release_lock.mjs` |

## Frozen Invariants

| Invariant | What must not regress | Locked by |
| --- | --- | --- |
| Mobile primary controls stay visible first | The mobile contacts entry view must keep the command shell, dock, and selected-contact primary actions visible before deep detail overlays are opened. | `app/contacts/ContactsPageExperience.tsx`, `app/contacts/contacts.module.css` |
| Mobile compaction cannot remove features | Edit, invite, log, schedule, recent activity, and full history must remain reachable on mobile even if they are hidden behind overlay controls. | `app/contacts/ContactsPageExperience.tsx` |
| Identity upgrades persist through import merge | A linked-account upgrade on an imported contact must trigger a durable server update instead of being dropped as a no-op merge. | `app/lib/contacts/server.ts`, `app/lib/contacts/__tests__/serverImportMergeContract.test.ts` |
| Contacts cannot fall back to local-only truth | Signed-in contact state must persist through the server-backed contacts API and remain intact after refresh. | `app/contacts/ContactsPageExperience.tsx`, `app/api/contacts/route.ts`, `app/lib/contacts/server.ts` |
| Release surface inventory stays canonical | The committed release surfaces and version badges must align with package and service worker release truth. | `scripts/check_release_lock.mjs` |

## Release Gates

`v47.8.1` is considered locked only if these pass together:

- `pnpm exec eslint app/contacts/ContactsPageExperience.tsx`
- `pnpm test:contacts-release-contracts`
- `pnpm release:lock`
- `pnpm test:release-freeze`
- `pnpm release:attest`
- `pnpm release:verify-attestation <path>`

The manual release procedure is defined in `docs/releases/v47.8.1-checklist.md`.

## Residual Risks

The following areas are materially stronger after this release but are not yet absolute:

- Extremely dense contact records still depend on truncation and overlay access patterns on narrow mobile screens, so a populated-contact smoke test remains required before tagging.
- Import merge persistence now catches `linked_user_id` upgrades, but it still depends on the incoming import row carrying a reliable authenticated identity signal.
- Production release attestation still depends on the required approval identities and signing env being present at release time.
