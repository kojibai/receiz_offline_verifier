# Receiz v109.0.0 Process

## Goal

The release followed design approval, a committed implementation plan, TDD for zero-network verification, package typechecks, package suites, AI-skill validation, registry sealing, release-lock validation, and tarball qualification.

## What Worked

Receiz.com and database paths were held outside the implementation scope. Issuance remained authenticated Record -> Seal; verification/opening moved to packaged SDK-local recomputation.

The verifier reused the implemented artifact carrier, Signature V4, Groth16 seal, portable continuity, and capability boundaries. Clean-before-build packaging prevented deleted version-specific outputs from surviving in `dist`.

## What Did Not Work

The first package contracts still required obsolete v107 subpaths, and the first release-document draft lacked the generic release-lock headings. Those stale contracts were corrected to enforce the v109 law. The isolated profile-database gate cannot lawfully use the live database because its fixtures write and delete rows.

## Practical Rule For Future Work

Verify and classify sealed bytes locally before projection. Keep issuance and mutation explicit, capability-gated operations. Preserve historical proof evidence without carrying obsolete developer APIs into current packages, and never use production state to satisfy a mutating fixture gate.

The public `@receiz/sdk` preserves durable proof memory as first admission only, then append forever.

## Locked Release Records

The release process is incomplete without the [regression lessons](v109.0.0-regression-lessons.md), [performance findings](v109.0.0-performance-findings.md), [invariant register](v109.0.0-invariant-register.md), and [commit-history boundary](v109.0.0-commit-history.md). The standalone verifier runs `pnpm release:lock 109.0.0` against its local release-identification surfaces; production signing, npm publication, deployment, database work, production verification, and push remain separate actions.
