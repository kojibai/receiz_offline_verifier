# Receiz v64.0.0 Product Truth

Release date: April 21, 2026

Canonical invariant reference:

- `docs/value-loop-invariants.md`

`v64.0.0` is the Complete Value Loop release.

This release freezes the product/economic logic that makes Receiz one coherent system:

- Proof-native value is Settlement. Funded fiat is Reserve.
- Reserve funds notes, sends, positions, and wire transfers.
- Settlement is proof-native value.
- Price moves in Settlement until funded.
- Notes stay Reserve-only.
- Settlement shares/certificates are the bridge that lets a buyer fund proof-native value into Reserve.
- Reserve handles funded fiat conversion.
- Settlement is proof-native completion. Reserve makes fiat conversion ready.

## Frozen Primitives

| Concept | v64 truth | Must not regress |
| --- | --- | --- |
| Reserve | Funded fiat value for notes, sends, positions, and wire transfers. It supplies action capital. | Credit Reserve only from funded deposits, funded sales, funded transfers, or funded certificate purchases. |
| Settlement | Proof-native value resolved by deterministic ledger math. | Present it as completed proof-native value, never as a lower-value balance. |
| Live portfolio value | Current market value of held positions/certificates/contracts. | Show market movement as Reserve only when funded. |
| Notes | Portable Reserve instruments. | Issue notes from Reserve only. |
| Settlement shares/certificates | Buyer-funded ownership/liquidity objects. | Seller receives Reserve only after buyer/funded purchase. |
| Fiat conversion | Reserve-backed conversion from proof-native Settlement into fiat movement. | Keep Settlement primary. |

## Frozen Invariants

### Send

Sends consume Reserve first and fill the remainder with Settlement.

The receipt must preserve the split.

### Buy

Market buys deploy Reserve first, then Settlement covers remaining market balance.

If external/card funding is needed, the funding creates Reserve and resumes the pending action.

### Sell

Market sells split by funded liquidity.

- Funded slice posts to Reserve.
- Unfunded slice returns as Settlement.
- UI patches both balances immediately from known trade truth.

### Note

Notes are Reserve-only.

### Certificate

Certificates lock or package shares/value.

- Issue locks shares/value.
- Sale pays the seller after funding clears.
- Buyer receives the claim/shares.
- Duplicate capacity issuing is forbidden.

## User-Facing Language

Use this action language consistently:

> Hold Settlement. Deploy Reserve. Issue shares into certificates. Use Reserve for notes, positions, and wire transfers.

The loop goes both ways: Reserve builds Settlement shares. Reserve proceeds issue notes.

Use this outcome language:

- Settlement is real proof-native completion.
- Certificates make ownership fundable.
- Notes carry funded Reserve.
- Fiat movement follows funded Reserve.

## Known Non-Goals For This Release

- Do not add another value state.
- Do not let Settlement issue notes directly.
- Do not add automatic external withdrawal from live growth.
- Do not make public certificate purchases create Receiz-funded liquidity unless Receiz funded the buy side.
- Do not add creator asset withdrawal beyond the existing creator reserve / certificate capacity rules without new capacity accounting.

## Residual Risks

The remaining launch risk is not the economic model. The remaining risk is enforcement coverage.

Any future wallet-writing endpoint, admin path, migration, payment path, certificate path, or market path that moves value must answer:

1. How much is Reserve?
2. How much is Settlement?
3. What funded the Reserve portion?
4. What receipt metadata preserves the split?
5. Can the user understand the next valid action from the surface?

If those questions are not answered, the change is incomplete.
