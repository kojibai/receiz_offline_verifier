# Receiz v97.6.0 Invariant Register

`v97.6.0` carries forward the proof-native artifact system, public app-state projection, enterprise SDK, and MCP agent rail invariants from `v97.5.0` and adds the invariants needed for signed merchant public-store publishing, public proof graph inspection, service-worker release continuity, and Sports saved-card language.

## Locked Invariants

1. Receiz remains a proof-native artifact system. Product truth is carried by files, identities, vault objects, ownership states, settlement states, public witness surfaces, and verification routes.
2. Proof objects remain stronger truth than SDK helpers, MCP tools, server rows, service-worker state, public-store projections, analytics, SEO, session state, or generic display state.
3. Kai Klok remains the proof object state machine. `kaiPulseEternal` / `kai_upulse` remains the pulse unit for complete proof objects.
4. Durable proof memory remains first admission only, then append forever. Hidden work may only append verified additions after the known head.
5. Merchant-owned public-store state may be published through Identity Seal / Receiz Key proof authority without `RECEIZ_ACCESS_TOKEN`.
6. Delegated agent tokens remain scoped Connect/OIDC mechanics for agent/server work, not merchant proof authority.
7. Public-store projection rows remain recovery projections beneath sealed artifacts, identity proofs, ownership appends, settlement rows, verified local truth, and verified appends.
8. Public proof graph data must remain a public proof surface carrying real release/API/product proof data and live Kai pulse state, not decorative simulation.
9. MCP and SDK rails expose and compose Receiz primitives; they do not rename, replace, or outrank proof objects, identity primitives, ownership surfaces, settlement primitives, public proof surfaces, or verification primitives.
10. Service-worker release identity must match app release identity while remaining runtime continuity beneath proof authority.
11. Sports saved-card base issue probability must be labeled as base issue, not scarce rarity odds.
12. Static verifier, offline verifier, record-seal studio, settlement entrypoint, Sports card verifier, package metadata, release docs, and service-worker runtime version must move together.
