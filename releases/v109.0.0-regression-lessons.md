# Receiz v109.0.0 Regression Lessons

## What Regressed

- A public `verifyArtifact` name is insufficient if its verdict still depends on an API response.
- Deleted version-specific source remained packable when stale compiler output was not cleaned.

## Why It Regressed

- Verification had been routed through weaker server state even though the sealed proof object carried the required verification material.
- TypeScript compilation did not remove outputs for source files deleted between releases.

## Locked Rules

- Offline proof requires packaging every verification dependency, including pinned roots, Groth16 verification material, and runtime code.
- Issuance-time server coordination and artifact-time local verification are different boundaries and must remain visibly separate.
- Historical proof evidence does not require an installable historical developer SDK.
- Version bumps must not rename unchanged product payload schemas; coordinated developer and visible release-identification surfaces move to v109 without changing product behavior.

## Contract Gates

- Production-signed artifacts verify with zero fetch calls; a one-byte mutation fails with zero fetch calls.
- SDK and MCP builds clean `dist` before compilation.
- Tarball contracts reject v107 exports, source outputs, operation modules, and developer documentation.
- AI contracts require the v109 registry digest, `ARTIFACT-011`, zero-network evidence, and the local verifier result.

The public `@receiz/sdk` preserves durable proof memory as first admission only, then append forever.

## Release-Lock Requirement

`pnpm release:lock 109.0.0` must require this regression record for `v109.0.0` and later release cuts. Omitting it would allow a network-routed verifier, stale packable output, issuance/verification boundary collapse, or historical developer surface to return without an explicit release contradiction.
