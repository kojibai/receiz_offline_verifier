## Title

Receiz `v94.0.0` - Official SDK And Durable Proof Operating Model Release

## Summary

`v94.0.0` is the clean major release after the `v93.x` proof-memory and developer-platform arc. It locks the official `@receiz/sdk` launch together with the production lessons learned from Sports proof history, wallet Kai ordering, calendar/account proof projection, vault ownership projection, Sports Economy read hardening, and database pressure closeout.

The release does not redefine Receiz as an API-first app. Receiz remains a proof-native artifact system. The SDK is the official developer convenience layer for using Receiz proof objects correctly: verify, project, admit, persist, append, and render from known verified truth first.

The documented implementation baseline is `v93.0.0..HEAD` before the `v94.0.0` release-lock edits. That range spans `37` commits, `207` files changed, `82,794` insertions, and `1,174` deletions. The final six commits after `v93.2.0` close Kai child-offset ordering, tournament/vault RPC projection, mobile dock spacing, default lineup selection, and live registry valuation.

## What We Learned From v92 Through v94

### Receiz Gets Faster When It Honors Its Ontology

The largest performance gains did not come from hiding truth, trimming history, or treating proof data as disposable cache. They came from letting each primitive do its actual job:

- sealed artifacts and proof objects carry truth
- local admitted proof memory carries durable known truth
- server and database paths discover verified additions
- RPCs and indexes bound the search for additions
- Chronos displays user-facing time
- Kai pulse/upulse orders Receiz state

This is the operating rule that emerged from production evidence:

```txt
Use known verified truth immediately. Ask only for verified additions after the known head.
```

### Proof Memory Is Not Cache

Sports card history exposed the decisive invariant. Once full event history was admitted on a card, reopening the card should not rediscover that history. It should render immediately and append new verified events. The same rule applies to wallet ledger history, calendar points, account brain activity nodes, public proof objects, and developer SDK registers.

Known verified truth is never stale. Later truth may append, extend, or enrich it. A weaker partial payload must never erase a stronger admitted proof object.

### Kai Is The State Coordinate

`v93.x` made the Kai/Chronos distinction operational. Chronos remains useful display context, but Kai pulse/upulse is the ordering primitive for Receiz proof objects and settlement ledgers. The Kai child-offset repair closed the case where multiple child proof objects born under the same Chronos millisecond could be forced onto the same stored coordinate by a trigger. The fix was exact: preserve the app-supplied Kai child coordinate and enforce uniqueness at the ledger boundary.

### Database Work Is Projection, Not Authority

The database hardening did not move truth into indexes or RPCs. It moved expensive discovery beneath the proof model:

1. Bound the read path by owner, sport, collectible, tournament, or Kai head.
2. Page by indexed proof coordinates.
3. Join or project only the rows needed for the visible surface.
4. Preserve JSON proof payloads as source material, not broad scan cargo.
5. Treat failures as projection pressure, not proof invalidation.

This pattern closed slow-query pressure without degrading user experience because the user still receives the same proof truth sooner.

### Developer Experience Must Carry Product Law

The SDK launch is not only typed clients. It packages the hard-won Receiz integration law for third-party developers:

- validate Receiz manifests and webhook events
- project deterministic rows from proof objects
- admit proof truth once
- merge duplicate transport without dropping known fields
- ask for additions after a known head
- keep SDK convenience beneath sealed artifacts, verified appends, ownership appends, and settlement ledger rows

The goal is that developers choose Receiz because it is stronger and easier.
