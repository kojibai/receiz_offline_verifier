# Receiz v63.0.0 Release Lock Checklist

Release date target: April 19, 2026

## Purpose

This checklist locks `v63.0.0` to one exact release state without allowing regression in settled twin-shell continuity, direct profile-route truth, composer publish continuity, route-local feedback, same-surface market continuation, hero open responsiveness, hero chart live parity, value-state truth, funded market/certificate realization, installed-browser stale-shell protection, or release-governed version surfaces.

- `docs/releases/v63.0.0-product-truth.md`

Process retrospective:

- `docs/releases/v63.0.0-process.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
pnpm exec tsc --noEmit --pretty false
pnpm test:page-first-paint
pnpm test:continuity-release-contracts
pnpm test:deterministic-surfaces
npx tsx --test app/lib/wallet/__tests__/valueStateContract.test.ts app/lib/wallet/__tests__/valueStateFramingContract.test.ts app/lib/market/__tests__/shareCertificateMarketFlowContract.test.ts app/lib/market/__tests__/marketConformanceRouteContract.test.ts
pnpm build
pnpm release:lock
pnpm test:release-freeze
```

If the candidate deploy also requires broader governance or environment checks beyond the list above, those must still pass before tag creation.

## Production Environment Preflight

The production environment must already include the active baseline plus the `v63.0.0` market settlement migration:

- `supabase/migrations/20260419173000_market_share_certificate_wallet_purchase.sql`

Before release, confirm:

- home and profile still settle once and stay visually immutable until explicit interaction
- hidden twin warming does not auto-promote the visible shell
- the public profile route still resolves truthful content directly without a visible second client assembly stage
- publishing from the profile composer keeps the surrounding profile current instead of feeling like a same-surface reload
- button-local route feedback still feels immediate and does not spill into fake page loading
- passkey, funding, and magic-link fallback all preserve the exact pending market action on the same chart or desk surface
- the homepage hero opens through the explicit-open deferred path and repeat open stays instant-feeling after the first successful warm
- the homepage back chart is already live or nearly live by the time the card flips and matches the same live market truth the pill uses
- wallet and market show settled balance separately from live portfolio value and unrealized P/L
- direct market sells settle only when backed same-asset liquidity exists
- public certificate links let signed-in buyers fund available shares with settled wallet balance and credit sellers only after the funded purchase clears
- bank backup/wire copy stays downstream of settled value and does not become the governing product frame
- installed returning browsers use fresh online document truth and only fall back to cached shell/offline documents when the network document is unavailable
- offline `/` reopens from the canonical cached root shell instead of a mismatched visited-route document or interruption boundary
- public and offline verifier surfaces, service-worker version, `package.json`, and release docs all show `v63.0.0`

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- load `/` and a public profile, let both pages fully settle, and confirm the visible shell does not perform a second visual reload or runtime takeover
- explicitly activate the home twin dock and confirm the richer twin opens only after that user action
- tap the homepage logo and confirm the hero opens immediately; close it and tap again to confirm reopen stays immediate
- flip the homepage hero card and confirm the market chart is already live or catches up without a staged loading frame and without visibly disagreeing with the pill
- open `/wallet`, `/market`, and a share certificate and confirm the user sees settled balance, live portfolio value, and unrealized P/L as separate states
- attempt a direct market sell larger than backed liquidity and confirm it rejects instead of crediting wallet value
- open a public certificate as another signed-in account and confirm buying shares debits the buyer settled balance, credits the seller settled balance, and reduces the certificate remaining shares
- open profile, player, and artifact surfaces and confirm market pills still render without SVG ID drift
- open the global composer from profile, publish, close, and confirm the surrounding profile remains current
- account and other route-entry buttons show route-pending feedback on the button surface itself rather than creating a larger fake page-loading effect
- signed-out market buy runs through passkey or magic-link fallback, funding return resumes the exact pending action, and the flow stays on the same surface
- with an installed browser already carrying older state, online navigation opens the fresh site document rather than a stale cached shell
- after opening `/` online once, go offline and confirm `/` reopens from the same root shell without the interruption boundary
- public version badges, service-worker defaults, offline verifier surfaces, package metadata, and release docs all show `v63.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v63.0.0`.
2. Run the `Release Governance Attestation` workflow if the release process for this cut still requires a signed attestation.
3. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
4. Confirm the attestation references the published key ID and the expected release version.
5. Store the attestation path and approval reference with the release record.

## Release Rule

If any twin-shell continuity regression, profile-route continuity regression, composer-publish continuity regression, route-feedback regression, market continuation regression, hero-open responsiveness regression, hero-chart parity regression, value-state framing regression, funded-realization regression, stale-shell navigation regression, offline-root-shell regression, version-surface mismatch, release-lock failure, or release-freeze failure occurs, `v63.0.0` is not released from that commit.
