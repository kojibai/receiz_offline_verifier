## Release Law

The SDK accepts only a complete sealed proof object. Its file carries payload, ownership/custody, provenance with append history, and settlement state, and passes the existing Receiz verifier before admission. `@receiz/sdk`, MCP, server admission, and database state remain weaker mechanics beneath that file-carried truth. Durable proof memory remains **first admission only, then append forever**.

## Frozen Primitives

| Primitive | v101.0.0 implementation truth | Forbidden downgrade |
| --- | --- | --- |
| Sealed proof object | The existing offline verifier validates the sealed file and embedded proof. | Server, SDK, DB, or session state replacing the file verdict. |
| Complete proof-object continuity | Canonical payload bytes/hash, ownership/custody, provenance/appends, and settlement travel in the artifact. | Treating a proof-only or self-hashable manifest as an admitted object. |
| Admission | Server re-verification and owner binding append the already verified asset to the runtime register. | Admission inventing proof, ownership, or continuity. |
| Historical artifact compatibility | Existing valid artifacts continue to verify as artifacts. | Retroactively failing or relabeling historical files. |

## Frozen Invariants

1. An SDK-created proof object cannot be admitted without a verified enclosing artifact.
2. A valid manifest alone cannot enter the proof/custody register as an asset.
3. SDK creation returns only after seal, local artifact round-trip, verifier continuity verdict, and authenticated admission all succeed.
4. The server repeats the existing artifact verification and binds continuity ownership to the authenticated Receiz identity.
5. The database accepts only continuity-complete v2 admission and the v1 execution path is closed.
6. Historical artifact verification and the Receiz.com verification surface remain unchanged.
7. Offline verification remains primary; online verification mirrors it for automation.

## Release Gates

- Portable document canonicalization and media-hash tamper tests.
- SDK seal/verify/admit order and failure tests.
- Offline verifier, API response schema, server admission, owner binding, and migration contracts.
- Primitive-binding, verification-conformance, SDK release, release-lock, full freeze, governance, and attestation gates.
- Visual evidence confirms `/verify` retains its existing single verified verdict.

## Residual Risks

- Live Supabase migration state is unknown until the migration succeeds against the linked project.
- npm package availability is unknown until both registry publications and read-back checks succeed.
- deployed frontend/service-worker identity is unknown until deployment and production smoke checks succeed.
- Historical third-party SDK clients must migrate away from manifest admission before adopting v101.
