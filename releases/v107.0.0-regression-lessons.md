# Receiz v107.0.0 Regression Lessons

## What Regressed

1. A typed `@receiz/sdk` surface is incomplete when Receiz.com, MCP, AI skills, frontend, service worker, emulator, generator, migration, and package output do not use the same schemas and errors.
2. Username availability cannot be implemented as a reservation. Only the atomic commit establishes global uniqueness, and the old name is released after that commit.
3. Raw avatar upload and proof-reference reuse cannot be separate profile mutation paths. They share one staged, atomic profile/media/head operation.
4. A self-hashable embedded payload is not the bearer ownership authority boundary. The enclosing artifact must verify first.
5. A signed offline command is still only a proposal. UI copy, SDK results, migration, MCP, and emulator must preserve `globallyCommitted: false` until verified admission.
6. Local restore cannot wait for network rediscovery. Known verified carried truth paints immediately.
7. MCP plan/execute is incomplete without exact confirmation, idempotency, canonical execution, and verified returned receipts.
8. Emulator-specific outcome labels must never imply production verification; deterministic simulation remains explicitly beneath proof authority.
9. A generator is not complete when it emits sketches. It must produce compiling routes, adapters, offline storage, capability declarations, and conformance tests while preserving developer extensions.
10. Version alignment includes runtime labels, service worker, capability ranges, registry digest, AI manifests, packed dependencies, and clean external installation.
11. Migration cannot rewrite witnessed history or convert pending work into a commitment claim. It preserves artifacts/receipts/heads and signs queued proposals.
12. Visual evidence is product truth. A changed identity, ownership, media, hierarchy, or first paint fails v107.0.0 even if tests pass.

## Why It Regressed

Prior release layers evolved independently, leaving source-only features, stale package assertions, incomplete package documentation, and UI evidence paths that did not yet cover the unified operation language. Generic embedded-payload and queued-work assumptions also risked outranking the enclosing proof and canonical receipt boundaries.

## Locked Rules

The v107 registry locks same-identity username append, commit-time uniqueness, atomic profile/media effects, carried portable continuity, enclosing-artifact bearer authority, signed-but-uncommitted offline proposals, stable errors, exact idempotency, canonical receipt verification, and stronger-truth first paint.

## Contract Gates

`test:v107-release-lock`, `test:v107-visual-evidence`, `test:v107-mcp-parity`, `test:v107-ai-skills`, `test:v107-generated-integration`, `test:v107-migration`, SDK/MCP package release contracts, the inherited v106 lock, and `test:release-freeze` prevent recurrence.

The durable rule remains first admission only, then append forever.

## Release-Lock Requirement

`pnpm release:lock 107.0.0` must require this document for `v107.0.0` and later release cuts. A release that omits these lessons is incomplete because username availability, split profile/media mutations, embedded bearer payloads, queued proposals, network-delayed restore, unverified MCP execution, emulator labels, incomplete generation, version skew, migration rewrites, or contradictory visual evidence could again outrank implemented Receiz primitives.
