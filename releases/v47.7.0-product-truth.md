# Receiz v47.7.0 Product Truth

Release date: April 9, 2026

This document freezes the primitives and invariants that `v47.7.0` is allowed to extend but not weaken.

## Frozen Primitives

| Primitive | Locked truth | Primary enforcement |
| --- | --- | --- |
| Atlas-first world districts | The district workspace must remain a map-first world-reading surface with one coherent atlas rather than regressing back into a flat stacked-card shell. | `app/world/WorldExperience.tsx`, `app/world/WorldExperience.module.css` |
| Concierge twin-quarterback runtime | The existing concierge shell may start live twin work, but that work must stay operator-triggered, visibly tracked, and grounded back into the same transcript context. | `app/api/world/concierge/stream/route.ts`, `app/api/world/concierge/tasks/route.ts`, `app/lib/world/conciergeTasks.ts`, `app/lib/world/intelligence.ts`, `app/world/WorldExperience.tsx` |
| Durable standalone contacts CRM | Contacts must remain a first-class, person-first operating view with durable persistence, import/export portability, and per-person timeline memory rather than falling back to a local-only shell. | `app/contacts/ContactsPageExperience.tsx`, `app/api/contacts/route.ts`, `app/api/contacts/import/route.ts`, `app/api/contacts/[contactId]/route.ts`, `app/api/contacts/[contactId]/events/route.ts`, `app/lib/contacts/server.ts`, `supabase/migrations/20260409163000_contacts_crm.sql` |
| Historical relationship backfill | Signed-in users must continue to see contacts materialized from message history and original-collection history rather than starting from an empty CRM. | `app/lib/contacts/server.ts` |
| Release-governed version surfaces | Package version, service worker version, public badges, offline verifier surfaces, and release docs must move together on each release. | `scripts/check_release_lock.mjs` |

## Frozen Invariants

| Invariant | What must not regress | Locked by |
| --- | --- | --- |
| Contacts cannot fall back to local-only truth | Signed-in contact state must persist through the server-backed contacts API and remain intact after refresh. | `app/contacts/ContactsPageExperience.tsx`, `app/api/contacts/route.ts`, `app/lib/contacts/server.ts` |
| Contacts must stay person-first and portable | Import/export and per-person notes/history must remain core behavior even when native phone-contact APIs are unavailable. | `app/contacts/ContactsPageExperience.tsx`, `app/account/page.tsx` |
| Historical backfill must stay additive | Existing user relationship history from messages and originals must continue to enrich contacts without erasing operator edits. | `app/lib/contacts/server.ts`, `app/contacts/__tests__/contactsBootstrapContract.test.ts` |
| Timeline actions must roll up consistently | Logging invites, messages, meetings, notes, or twin actions must continue to update last-interaction, meeting, and interaction-count rollups coherently. | `app/lib/contacts/server.ts`, `supabase/migrations/20260409163000_contacts_crm.sql` |
| Release surface inventory stays canonical | The committed release surfaces and version badges must align with package and service worker release truth. | `scripts/check_release_lock.mjs` |

## Release Gates

`v47.7.0` is considered locked only if these pass together:

- `npx tsx --test app/contacts/__tests__/contactsBootstrapContract.test.ts`
- `pnpm exec tsc --noEmit --pretty false`
- `pnpm exec next build --webpack`
- `pnpm release:lock`
- `pnpm test:release-freeze`
- `pnpm release:attest`
- `pnpm release:verify-attestation <path>`

The manual release procedure is defined in `docs/releases/v47.7.0-checklist.md`.

## Residual Risks

The following areas are materially stronger after this release but are not yet absolute:

- Contacts durability is now server-backed, but the history-sync path still runs as an online bootstrap flow rather than a dedicated background backfill job with operator-visible progress.
- Native mobile address-book access still depends on browser and device support for the Contacts Picker API.
- The contacts data model is enforced through authenticated server routes and service-role writes; production correctness still depends on those ownership checks staying intact.
- Discovery-feed cache revalidation can still log transient upstream fetch failures during production builds; current behavior does not block the build, but it remains an operational signal worth watching outside the release lock itself.
