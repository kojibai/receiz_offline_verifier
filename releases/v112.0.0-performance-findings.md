# Receiz v112.0.0 Performance Findings

`v112.0.0` makes the artifact authority hierarchy executable while preserving known exact artifact truth as deterministic first paint. Durable proof memory remains first admission only, then append forever.

## Findings

- Exact artifact bytes are verified before profile admission, planning, staging, commit, receipt projection, session work, or UI state.
- The asynchronous admission engine preserves verification/admission separation instead of turning store availability into artifact authority.
- The IndexedDB admission ledger survives reload while storing no proof objects or sealed artifact bytes; durable coordination does not duplicate or weaken proof custody.
- Portable transition identity excludes idempotency and external effects, so the same proposed successor meaning remains stable across retries.
- Registry-derived effects prevent callers from expanding work and make plan review proportional to the pinned law set.
- Neutral staging durably preserves exact candidate bytes without advancing a named head or projecting acceptance.
- Commit independently resolves and verifies staged bytes, preventing a locator, plan, cache, receipt, or caller assertion from bypassing artifact verification.
- Named-domain head, event, receipt, effects, idempotency, and outbox updates are atomic, so expected failures leave no partial coordination state.
- Unknown namespaces remain byte-preserved while registered reducers handle only known namespace transitions.
- Ed25519 and P-256 identity-key capability signing binds the exact plan without elevating runtime key wrappers into identity truth.
- v111 artifact compatibility reuses exact bytes and current verification/admission; it does not carry historical runtime authority into v112.
- The visible standalone verifier change is release identification only. First paint, page hierarchy, media, routing, and verifier semantics remain unchanged.

## Performance Boundary

- Known artifact truth paints before IndexedDB hydration, network work, session checks, global reconciliation, analytics, or server projection.
- Remote global admission and offline-to-global reconciliation are absent rather than partially implemented.
- The local ledger coordinates admission state but never becomes a proof-object store or accepted artifact authority.
- This repository validates static release surfaces and constitutional records; it does not reproduce upstream package benchmarks or global-admission behavior.
