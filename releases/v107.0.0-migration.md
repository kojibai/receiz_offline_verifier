# Receiz v106.0.0 to v107.0.0 migration

Receiz v107.0.0 is a forward-only coordinated migration from v106.0.0 for Receiz.com, `@receiz/sdk`, `@receiz/mcp-server`, and `@receiz/ai-skills`. It preserves every sealed artifact, canonical receipt, proof head, verified local projection, and witnessed history byte-for-byte. Migration never treats package metadata, a queue record, or its own verification result as proof authority.

## Exact workflow

Preview without writing:

```bash
receiz migrate v106-v107 --dry-run --root ./my-app --snapshot ./.receiz/v106-state.json
```

The preview includes the exact source digests, destination digests, all 73 classified v106 SDK write surfaces, history-preservation digests, pending-outbox count, and one `planDigest`. Review it before applying.

Apply only the reviewed plan:

```bash
receiz migrate v106-v107 --apply --confirm <plan-digest> \
  --root ./my-app \
  --snapshot ./.receiz/v106-state.json \
  --identity ./my-receiz-key.json
```

If the key is encrypted, add `--passphrase`. The migrator refuses a changed source or wrong plan digest. Compatible v106 pending operations are signed with the carried Receiz identity and become `receiz.offline.signed-command.v107` queue records. Their authority boundary is exact: `signedCommandIsGlobalAdmission: false`, status `queued`, and zero globally committed claims. Canonical admission can happen only during a later connected resume with a verified receipt.

Verify after apply:

```bash
receiz migrate v106-v107 --verify --root ./my-app --snapshot ./.receiz/v106-state.json
```

Verification confirms exact coordinated package versions, preservation digests, signed queued proposals, forward-only recovery, and absence of migration-created commitment claims.

## Compatibility and history

- v106 public SDK methods remain explicit compatibility adapters.
- New development uses the v107 typed identity, profile, media, continuity, bearer-ownership, offline-command, receipt, emulator, and generator operations.
- A username rename stays bound to the original Receiz identity and releases the old username only after the canonical append commits.
- Raw profile image files and prepublished media proof references both use the same atomic profile append.
- Generic bearer claims verify the enclosing Receiz artifact before interpreting any embedded payload.
- Migration is not verification or admission. The sealed proof object remains stronger truth.

Recovery is `roll-forward-from-verified-checkpoint`. Existing witnessed history is never rewritten or deleted.
