# Receiz v47.5.0 Release Lock Checklist

Release date target: April 9, 2026

## Purpose

This checklist locks `v47.5.0` to one exact release state without allowing regression in world revenue-skill policy control, owner-side revenue playbook UX, multi-channel outbound delivery, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v47.5.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
npx tsx --test app/lib/world/__tests__/policyContract.test.ts app/lib/world/__tests__/revenueSkills.test.ts app/lib/world/__tests__/revenuePlaybooks.test.ts app/lib/world/__tests__/outreachDraft.test.ts app/lib/world/__tests__/worldConversationRelayContract.test.ts app/lib/world/__tests__/showcaseSelectionContract.test.ts
pnpm exec tsc --noEmit --pretty false
pnpm release:lock
pnpm test:release-freeze
```

Environment preflight for production lanes:

- `RESEND_API_KEY` and `RECEIZ_FROM_EMAIL` must be present anywhere Receiz email outreach is intended to be live.
- Twilio credentials and one sender lane must be present anywhere World SMS outreach is intended to be live.
- Favorite-app webhook destinations should be verified as operator-controlled before exposing that lane in production.

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- Saving world agent policy persists the selected revenue skills and the same saved state reloads in `/world`
- Loading each operator playbook in `/world` pushes the correct goal, channel, subject, message, and next-step CTA into the outreach composer
- Receiz email, SMS, and favorite app webhook outreach remain disabled until the matching saved policy lane is live
- Public twin flows can still surface the older intake / routing / handoff rails, and now also surface quote / deposit / proof / renew rails when the message and saved configuration justify them
- Proof-kit landing pages still open prefilled record flows cleanly
- The action-card, booking confirmation, and call closeout surfaces still open without route regressions
- Public version badges and offline verifier surfaces all show `v47.5.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v47.5.0`.
2. Run the `Release Governance Attestation` workflow with the required approval identities and signing secrets.
3. Verify the uploaded release attestation artifact.
4. Store the attestation path and approval reference with the release record.

## Release Rule

If any world revenue-routing regression, outreach-channel regression, operator-playbook regression, release-lock check, or attestation step fails, `v47.5.0` is not released from that commit.
