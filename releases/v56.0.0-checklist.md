# Receiz v56.0.0 Release Lock Checklist

Release date target: April 17, 2026

## Purpose

This checklist locks `v56.0.0` to one exact release state without allowing regression in canonical version surfaces, world route/cache stability, concierge entitlement gating, fresh-account analytics resilience, or the `v55.1.0` continuity baseline.

Product-truth reference:

- `docs/releases/v56.0.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
pnpm exec tsc --noEmit --pretty false
npx tsx --test app/lib/world/__tests__/worldConciergeTaskControlsContract.test.ts
npx tsx --test app/lib/profile/__tests__/profileAnalyticsTrackContract.test.ts
pnpm build
pnpm release:lock
```

If the candidate deploy also requires broader freeze or governance checks, those must still pass before tag creation.

## Production Environment + Governance Preflight

For the `v56.0.0` release-prep state, confirm the target environment includes:

- `supabase/migrations/20260417094500_profile_analytics_activation_event_support.sql`

Before release, confirm:

- fresh-account login does not emit `500` responses from `/api/profile/analytics/track`
- owner-side activation analytics persist once the new event-name constraint migration is applied
- signed-in non-Automation users do not repeatedly request `/api/world/concierge/tasks`
- public/offline verifier surfaces, service worker version, `package.json`, and public version badges all show `v56.0.0`
- `public/governance/governance-keyring.v1.json` still contains the active release-attestation key ID if release attestation is produced for this cut

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- fresh-account login completes without `/api/profile/analytics/track` 500s
- first owner profile load on a new account stays usable even if analytics tracking is best-effort
- logged-in non-Automation world entry does not show repeating concierge task 403s in the console
- Automation-entitled world users still get live concierge task polling and controls
- `/profile`, public profile navigation, market continuation, sell gating, and live-player fit still match the `v55.1.0` continuity baseline
- public version badges, service worker version, offline verifier surfaces, and release docs all show `v56.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v56.0.0`.
2. Run the `Release Governance Attestation` workflow if this cut still requires a signed attestation.
3. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
4. Confirm the attestation references the published key ID and the expected release version.
5. Store the attestation path and approval reference with the release record.

## Release Rule

If any version-surface mismatch, world-hydration regression, concierge-entitlement regression, fresh-account analytics regression, or release-lock failure occurs, `v56.0.0` is not released from that commit.
