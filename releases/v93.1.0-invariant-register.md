# Receiz v93.1.0 Invariant Register

Release date: June 26, 2026

This register makes explicit the v93.1 invariants proven by the work after `v93.0.0`.

## 1. Durable Proof Memory Is Local Truth

Once proof truth is admitted on a device, it is durable local truth and must project before weaker rediscovery work.

Violation: a route treats admitted proof memory as stale cache and hides known history until server, DB, global sync, or session work completes.

## 2. First Admission Then Append Forever

Full-history hydration is for first admission and recovery. Normal reopen paths ask for verified additions after the known head.

Violation: a known proof object rehydrates old history on every open instead of appending additions beneath admitted truth.

## 3. Calendar And Account Share Event Proof Projection

Calendar points and Account brain activity nodes project the same admitted event proof objects.

Violation: Calendar and Account fork the same event into separate route-local authorities or allow one projection to erase the other's stronger known truth.

## 4. Account Brain Activity Nodes Are Proof Nodes

Account brain activity nodes are proof projections over admitted account/calendar actions.

Violation: account activity is reduced to generic route hydration, timeline UI, or inferred display state.

## 5. Wallet Kai-Ledger Head Controls The Append Window

Wallet settlement rows that carry Kai pulse/upulse state use Kai as the ordering primitive and append-window head.

Violation: wallet history makes Chronos the first backend ordering authority when Kai state exists.

## 6. Chronos Remains Display And Fallback Context

Chronos helps users read settlement time and can serve as fallback context. It does not replace Kai causality for proof objects that carry Kai state.

Violation: Chronos windows reorder, overwrite, or hide known Kai-ordered settlement truth.

## 7. Bounded Sports Projection Preserves Proof History

Sports vault, event proof, visible append, and showcase paths use bounded reads and RPC projection beneath proof authority.

Violation: a pressure fix drops history, hides appends, or restores broad proof reads to rediscover already-known rows.

## 8. SDK Convenience Stays Beneath Proof Authority

`@receiz/sdk`, developer docs, OpenAPI console, fixtures, and conformance tests are integration mechanics beneath Receiz proof primitives.

Violation: an SDK response or convenience projection outranks sealed artifact bytes, proof bundles, verified registers, ownership proof, settlement proof, or route-bound payloads.

## 9. Fixture Construction Must Not Weaken Production Proof Contracts

Tests must move toward complete proof object construction when production contracts become stricter.

Violation: production media, base card, pack, proof, or witness fields are made optional to satisfy stale Sports fixtures.

## 10. Known Truth Paints Before Enrichment

Deterministic file proof, local verified registers, route snapshots, public witness surfaces, and admitted proof memory paint before hydration, analytics, SEO, session checks, lazy imports, or reconciliation.

Violation: a release surface shows a loading state or weaker placeholder when stronger known proof truth exists.

## 11. Release Identity Moves As One Surface

Package metadata, service-worker version, offline verifier labels, offline record seal labels, release notes, docs indices, and release-lock documents move together.

Violation: package or service-worker version advances without matching public/offline labels and v93.1.0 release documents.

## 12. Offline Verification Remains File-Authoritative

The offline verifier remains a deterministic file-proof surface. Release docs and labels can change; verifier proof authority remains in the selected file bytes and embedded proof material.

Violation: the offline verifier presents release metadata, cache state, external service state, or generic display state as stronger than file proof.

## Release-Lock Requirement

`pnpm release:lock 93.1.0` must require this document for `v93.1.0` and later release cuts. A release that omits the invariant register is incomplete because durable proof memory, first-admission append windows, shared event proof projection, account activity proof nodes, Kai-ledger ordering, bounded Sports projection, SDK convenience boundaries, strict fixture construction, release identity coherence, and offline file authority would not be explicit release truth.
