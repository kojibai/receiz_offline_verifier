## Goal

Cut `v70.0.0` as the public alpha release without weakening the product laws that made the current system work.

The process target is exact:

- preserve the settled visible product
- fix only obvious release-blocking violations
- move all governed version surfaces to `v70.0.0`
- document the growth from `v60.0.0` to `v70.0.0`
- run release gates from the exact release candidate

## What Worked

- Treating product law as literal made profile, runtime, login, proof, and value-loop decisions unambiguous.
- Local proof first plus server propagation preserved both Receiz's offline authority model and production interoperability.
- Keeping `/<username>` as canonical profile truth restored the best visible profile behavior while still allowing owner controls.
- The `v70.0.0-invariant-register` turned repeated release lessons into explicit non-negotiable contracts.
- Release-freeze caught a chaos-failure test configuration mismatch before final lock.
- The signin issue was fixed by reusing the existing Receiz ID continuation primitive instead of creating a second login model.

## What Did Not Work

- Letting `/signin` render a manual destination created a place where users could be parked even though Receiz ID already supports one-click local continuation.
- Version surfaces were still locked to `68.9.0` after the v70 invariant work existed, which made the repository internally consistent but not v70 release-ready.
- Some conformance and chaos checks had become sensitive to environment/config defaults instead of representing the intended local contract clearly.
- Any approach that explained product behavior as "basically the same" was too weak for this release line.

## Practical Rule For Future Work

For future Receiz release work:

1. identify the literal law first
2. identify the exact code path implementing or violating it
3. add or preserve a contract that fails if the law breaks again
4. make the smallest behavior-preserving change that satisfies the law
5. move release surfaces only when the release candidate is real
6. run release gates after the final surface bump

If a proposed release fix requires broad visual redesign, new authority semantics, or a weaker test to pass, it is not a release-prep fix.
