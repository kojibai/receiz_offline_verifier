# Receiz v102.0.0 Regression Lessons

These lessons keep `@receiz/sdk` beneath artifact continuity and durable proof memory as **first admission only, then append forever**.

The coordinated `@receiz/ai-skills@102.0.0` package is an exact-version dependency of SDK and MCP so installed agent doctrine cannot silently remain on the removed verification shortcuts.

## What Regressed

- Integrity-valid artifacts could receive top-level verified success with no continuity carrier.
- Generic SDK sealing could produce proof-looking assets without authenticated ownership or a prior continuity head.
- Receiz.com and downloadable offline verifiers inherited the byte-first verdict.
- MCP described public-proof resolution and local manifest shape validation as verification.
- A World trail ZIP marked integrity-sealed members and the package verified without an enclosing owner-continuity carrier.

## Why It Regressed

- Integrity and continuity were checked in different stages, so applications could stop after the weaker result.
- The generic document sealer was exposed at a proof-object developer boundary.
- Owner strings and session/account identifiers were not uniformly distinguished from a real profile-backed Receiz ID.
- “Verify” was used for lookup and schema inspection even though neither operation checked the enclosing artifact chain.
- Release conformance covered the primary verifier but the full freeze was required to find a secondary internal package expectation.

## Locked Rules

1. `verification.ok` is true only when integrity and existing Receiz continuity both verify.
2. Continuity-capable sealing rejects before producing bytes when real owner, namespace, deterministic identity, or prior-head/genesis evidence is missing or mismatched.
3. The SDK exposes `assets.createProofObject(...)`, not generic `sealArtifact(...)`, as its proof-object path.
4. Generic document sealing is integrity-only and rejects portable proof-object payloads.
5. MCP lookup and inspection tools never use verification names or return a verified proof-object claim.
6. Restore and admission consume the shared successful continuity verdict; they cannot manufacture continuity afterward.
7. App, SDK, MCP, runtime, service worker, offline verifier labels, and release records move together.

## Contract Gates

- Shared-verifier integrity-only rejection and positive native/portable continuity tests.
- Pre-seal invocation-order, authenticated owner, namespace, prior-head, and migration checks.
- Downloadable offline verifier and standalone offline Record/Seal continuity contracts.
- MCP tool-name, result-semantic, package-version, and generated distribution tests.
- World package enclosing-continuity test.
- Release lock, verification conformance, primitive binding, visual evidence, repo-only reasoning, governance, and full release freeze.

## Release-Lock Requirement

`pnpm release:lock 102.0.0` must require these regression lessons in the standalone offline-verifier checkout. Omitting them would sever the v102 verifier identity from the correction history that prevents integrity-only evidence, arbitrary owner state, lookup, or shape inspection from becoming a verified proof object.
