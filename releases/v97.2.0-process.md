# Receiz v97.2.0

Release date: 2026-06-30

## Goal

Release the enterprise app runtime SDK as one coherent developer surface. A developer should be able to build public apps, storefronts, marketplaces, customer portals, rewards systems, access products, media workflows, audit trails, notification flows, search surfaces, and offline-first proof queues with typed Receiz primitives instead of hand-rolled glue.

## What Worked

- `v97.0.0` already locked the public app-state projection rail and SDK app-state convenience beneath proof authority.
- The existing SDK identity, Connect, public proof, app-state, Twin, World, wallet, and commerce direction gave the enterprise app runtime surface a primitive-backed foundation.
- The enterprise rails could be framed as typed composition over existing proof, identity, ownership, settlement, public projection, durable proof memory, and verified append primitives.
- Idempotency material gives third-party apps a practical replay contract after serverless cold starts, browser interruptions, and offline queue recovery.
- The optional React subpath can reduce integration time while staying subordinate to proof objects and durable proof memory.

## What Did Not Work

- App State and Commerce Cloud rails alone still forced builders to assemble tenant sessions, merchant onboarding, portal flows, media workflows, audit trails, search, events, jobs, compliance, release pinning, and offline proof queues by hand.
- Enterprise writes without mandatory idempotency material would make retry and offline replay behavior too ambiguous for production app builders.
- React integrations without a package-level subpath would make every app recreate the same hooks and hosted component boundary.
- A release note without checklist, regression lessons, performance findings, invariant register, and commit-history boundary would hide the authority limits that keep SDK helpers below proof truth.

## Practical Rule For Future Work

When a developer capability is added beneath an implemented Receiz primitive, make the primitive easy to use without making a new authority:

1. Name the primitive directly.
2. Expose typed SDK convenience.
3. Require idempotency material for retryable writes.
4. Preserve the canonical lower-level route for advanced use.
5. Document public reads, delegated writes, replay behavior, and authority boundaries.
6. Add contract coverage so convenience cannot drift from primitive law.
7. Version-lock app, SDK, runtime, public labels, and release docs together.

The SDK remains convenience. It must make proof-native development easier without creating a new source of truth. The durable proof memory law remains first admission only, then append forever.
