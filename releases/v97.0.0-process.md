# Receiz v97.0.0

Release date: 2026-06-30

## Goal

Release the public app-state projection rail and `@receiz/sdk` app-state surface as a coherent developer substrate. A developer or AI agent should be able to build a real app, publish public app state through delegated Receiz access, and cold-recover that public app from Receiz rails without adding a separate public-state database.

## What Worked

- `v96.0.0` and `v96.1.0` already locked `@receiz/sdk` as a proof-native developer surface with CLI conformance, proof memory, Twin, World, Connect, identity, wallet, sports, and public proof primitives.
- The existing public proof table could host app-state projection fields as additive projection columns instead of introducing a new authority table.
- The existing Connect/OIDC bearer model already supplied delegated write authority through `receiz:record`.
- Public reads could reuse established public proof lookup patterns: by URL, by creator, by namespace, and by id.
- SDK ergonomics could improve materially with `publishRecord()` and `byHost()` without changing server semantics.
- OIDC consent and public projection correctness fixes were small but important release-quality repairs for external apps.
- The live `receiz.app` direction proves the developer substrate is practical: public app-state, storefront surfaces, rewards, assets, and custom-site launch paths can sit on Receiz rails.

## What Did Not Work

- Shipping only `publish(feed)` forced developers and AI agents to hand-assemble repetitive feed envelopes for the common one-record app save path.
- Shipping only `byUrl()` forced serverless apps to synthesize full URLs when most custom-domain and subdomain routing code starts from a request host.
- Leaving SDK at `96.2.0` while the app-state rail became public would have made npm identity lag behind implemented developer capability.
- A public app-state backend without release docs would have hidden the most important developer story: apps can recover public first paint from Receiz projections without making DB state the proof authority.

## Practical Rule For Future Work

When a developer capability is added beneath an implemented Receiz primitive, make the primitive easy to use without making a new authority:

1. Name the primitive directly.
2. Expose a typed SDK convenience.
3. Preserve the canonical lower-level route for advanced use.
4. Document the public read path and delegated write path.
5. Add contract coverage so convenience cannot drift from primitive law.
6. Version-lock app, SDK, runtime, public labels, and release docs together.

The SDK remains convenience. It must make proof-native development easier without creating a new source of truth. The durable proof memory law remains first admission only, then append forever.
