# Receiz v61.0.0 Release Lock Checklist

Release date target: April 19, 2026

## Purpose

This checklist locks `v61.0.0` to one exact release state without allowing regression in governed historical continuity, deterministic market witness and exit navigation, settled twin-shell continuity, compact verified-player control clarity, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v61.0.0-product-truth.md`

Run from the exact release commit only:

```bash
pnpm exec tsc --noEmit --pretty false
```

If the candidate deploy also requires broader governance or attestation checks beyond the list above, those must still pass before tag creation.

## Production Environment + Governance Preflight

The production environment must already include the active `v60.0.0` baseline. `v61.0.0` adds no new database migrations on top of that cut.

Before release, confirm:

- public profile asset markets still show holder count and latest witness timing from live public holder activity
- deterministic market exits to Receiz routes stay on the shared stable client-navigation path instead of degrading to weaker navigation behavior
- home and profile twin shells do not visually remount or upgrade after settlement, and compact touch devices keep full twin hydration on-demand
- the home dock stays on the lightweight closed shell until explicit interaction, while hidden warming remains hidden
- compact verified-player audio controls remain legible and do not crowd pulse, claim, or transport details
- the public historical cohort remains exactly the governed frozen set, and the pinned historical governance artifact plus payload hashes still match the committed snapshot
- historical downloads preserve raw bytes, carry the sealed governance bundle, and do not silently mint a new current proof identity
- public and offline verifier surfaces, service worker version, `package.json`, and release docs all show `v61.0.0`

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- open a public profile market asset with live activity and confirm holder count, witness timing, and verify/open-source exits stay truthful and continuous
- load `/` and a public profile on compact touch layout, let both pages fully settle, and confirm the visible twin shell does not perform a second visual reload or runtime takeover
- explicitly activate the home twin dock and confirm the full module opens only after that user action
- open a verified player at compact width and confirm the audio control glass, identity dock, queue badge, pulse pill, and transport controls remain clean
- open a historical pre-v4 record and confirm product surfaces explain `Historical` as a governed status rather than current proof
- download a historical original and confirm the result is a `.historical.receiz-package.zip` package that preserves the raw file and includes the sealed governance bundle
- public version badges, service worker defaults, offline verifier surfaces, package metadata, and release docs all show `v61.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release tag from the locked commit: `v61.0.0`.
2. Run the `Release Governance Attestation` workflow if the release process for this cut still requires a signed attestation.
3. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
4. Confirm the attestation references the published key ID and the expected release version.
5. Store the attestation path and approval reference with the release record.

## Release Rule

If any historical-cohort regression, historical-download regression, market-witness regression, market-navigation regression, twin-shell continuity regression, compact-player layout regression, version-surface mismatch, release-lock failure, or release-freeze failure occurs, `v61.0.0` is not released from that commit.
