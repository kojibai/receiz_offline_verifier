# Receiz v93.2.0 Regression Lessons

Release date: 2026-06-26

## What Regressed

No product runtime regression is introduced by this SDK slice. The regression risk addressed here is developer-side: apps integrating Receiz could treat proof objects as mutable API feed rows if the SDK only exposed low-level clients.

## Why It Regressed

The SDK was useful but incomplete as a law carrier. It exposed clients, validators, and docs, but it did not yet package the pattern that made Receiz fast and coherent in production:

```txt
known verified truth first -> additions only after the known head
```

When that pattern is left implicit, external apps can rediscover known truth, drop admitted history after partial responses, or split a proof object into unrelated UI state.

## Locked Rules

- SDK projections must be deterministic and derived from validated manifests.
- SDK proof memory must preserve first admitted truth for a stable ID.
- Later duplicate transport must not erase fields from the admitted object.
- SDK schemas help tooling but do not outrank artifact verification.
- The SDK must name Receiz primitives directly: proof object, Sports card proof object, durable proof memory, verified append, ownership append, settlement ledger.

## Contract Gates

- SDK tests must cover schema exports and projection rows.
- SDK tests must cover duplicate proof-register admission preserving known truth.
- SDK build must emit clean JavaScript and declaration files.
- Release lock must pass for the app/SDK version after any publishable SDK change.

## Release-Lock Requirement

`pnpm release:lock 93.2.0` must require this document for `v93.2.0` and later release cuts. A release that omits regression lessons is incomplete because the SDK newest-response-wins, implicit proof-memory, partial-response, and version-drift failure modes would not be explicit release truth.
