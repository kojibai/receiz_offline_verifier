## Frozen Primitives

The implemented primitive is the native Receiz proof object created by Profile Composer, Record, and Seal. Its sealed proof bundle, Kai coordinates, Receiz claim identity, Record ownership projection, and verification path are stronger truth than SDK wrappers or DB convenience rails.

`@receiz/sdk` composes that existing primitive. It does not create a second proof-object schema. `@receiz/mcp-server` and `@receiz/ai-skills` remain mechanics beneath the same native proof.

Historical native artifacts remain valid. Verification cannot require metadata introduced after those artifacts were sealed.

## Frozen Invariants

- A native proof object verifies through its existing sealed proof bundle and canonical Record/Seal coordinates.
- SDK proof-object creation resolves the authenticated account's real Receiz ID and creates the Record before Seal.
- No developer-supplied owner, namespace, or parallel continuity object can become proof authority.
- Existing Profile Composer, Record, and Seal creation semantics remain unchanged.

## Release Gates

- Native verifier and document-seal regression contracts.
- Profile release contracts and verification conformance.
- SDK, MCP, AI-skill distribution, lint, build, and release-freeze gates.

## Residual Risks

The v103 implementation is not production-verified until it is deployed. Production v101 rollback behavior is the confirmed live baseline.

Durable proof memory remains **first admission only, then append forever**.
