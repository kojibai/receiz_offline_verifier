## Goal

Make Receiz `v66.0.0` the release where the implementation obeys the product truth more completely:

> Known local truth should move immediately. Everything else must justify its existence.

The goal was not to remove the server, database, or online twin runtime. The goal was to stop treating them as authority when the local node already holds stronger verified truth.

## What Became Clear

The right model is not stale cache versus fresh server data.

The right model is:

- verified local prefix
- deterministic projection
- verified additions
- propagation
- merge
- replay

Once that model is accepted, many product bugs become obvious:

- a wallet graph applied in an effect is a correction frame when it could have been first paint
- a local twin register reply followed by a silent online replacement violates the settled visible answer
- a lagging market snapshot dropping a just-created position is weaker transport overriding known action truth
- a receipt proof surface that treats older evidence as equal to verified-register proof hides the actual authority ladder
- a missing session response should not make a locally proven identity disappear

## What Worked

- Treating the AGENTS.md rules as product law made the patch scope clearer.
- Moving wallet durable truth into initial state removed a first-render gap without changing the wallet feature set.
- Letting the local twin projection end the visible stream path protected the settled reply while keeping the online stream available when no local register truth exists.
- Renaming Explore's network request as verified additions made the code language match the architecture.
- Adding offline-persist headers to deterministic/private snapshot routes gave the service worker explicit truth-carrying signals.
- Preserving market positions until weaker payloads prove themselves made trade continuity match the user's known action.
- Keeping the `v64.0.0` value-loop truth intact prevented this release from turning offline capability into economic drift.

## What Did Not Work

- Treating hydration as harmless because it was not a literal browser refresh missed the user-visible problem.
- Letting server/session payloads narrate identity after local proof existed created avoidable fragility.
- Allowing online twin streaming to continue after a local register projection risked replacing an already-settled answer.
- Applying wallet route truth after mount was too late for Receiz's continuity standard.
- Calling network payloads "authoritative" in Explore obscured that the visible snapshot is already local truth and the network is only discovering additions.

## Banned Future Reasoning

Do not say:

- "The network is fresher, so it can replace local proof."
- "The screen only corrected for a moment."
- "It is only a cache seed."
- "The online stream is ready, so it can take over."
- "Session says anonymous, so local identity should disappear."
- "The trade will come back on the next snapshot."

Those are regressions against `v66.0.0`.

## Correct Future Reasoning

Every code path should answer:

1. What stronger truth is already local?
2. What verified prefix does the node hold?
3. Is the network delivering a verified append or merely retelling known truth?
4. Can this background work alter a settled visible surface?
5. Can this payload downgrade a stronger local block?
6. Would this still project correctly with the server gone?

If the answer is that the server is involved only to reconfirm local truth, the path is wrong.

## Release Decision

`v66.0.0` is launch-ready when:

- version surfaces move together
- local proof first-paint checks pass
- twin mind portable verification checks pass
- market and wallet continuity checks pass
- deterministic surface checks pass
- value-loop checks from `v64.0.0` still pass
- manual PWA/offline tests confirm no settled screen is visibly redefined by background hydration

## Practical Rule For Future Work

Build every new Receiz surface from held proof first, deterministic projection second, verified additions third, and transport last.
