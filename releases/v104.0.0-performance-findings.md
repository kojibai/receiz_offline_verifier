# Receiz v104.0.0 Performance Findings

`v104.0.0` preserves Receiz scale law by compiling one typed application contract through a deterministic SDK rules engine shared by thin CLI and MCP interfaces. Inspection, planning, findings, generation, and upgrade guidance reduce integration work without entering the artifact-verification path.

## Findings

- The SDK owns one deterministic compiler core so CLI and MCP do not duplicate scope, environment, finding, conformance, or upgrade logic.
- Repository inspection is bounded to the explicit root and remains read-only until an exact preview digest is confirmed.
- Generated Next.js App Router wiring uses small modules, excludes browser secrets, and does not add database proof authority.
- Existing witnessed history is never rewritten; missing truth appends and incorrect projections rebuild from stronger truth.
- Packed tarballs and empty external-project installation expose dependency defects that source-workspace tests can mask.

## Performance Boundary

- Known verified truth must paint before hydration, analytics, SEO, session checks, lazy imports, global sync, chart hydration, ledger pagination, or server reconciliation.
- Offline artifact verification remains primary and separate from compiler inspection or conformance findings.
- Compiler execution, repository scans, preview hashing, file writes, framework adapters, CLI/MCP calls, package installation, and service-worker continuity remain weaker mechanics beneath proof primitives.
- Visual evidence can block release even when static contracts pass.

## Release-Lock Requirement

`pnpm release:lock 104.0.0` must require this document for `v104.0.0` and later release cuts. A release that omits performance findings is incomplete because shared compiler determinism, root-confined inspection, digest-confirmed changes, small generated modules, append-only history, external packed installation, and verifier/service-worker identity would lack explicit proof-authority boundaries.
