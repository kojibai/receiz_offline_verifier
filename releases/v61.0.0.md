# Receiz v61.0.0

Release date: April 19, 2026

## Overview

`v61.0.0` is the governed historical-continuity and runtime-discipline release on top of the `v60.0.0` stable baseline. Since the last tagged cut, Receiz has tightened the rule that settled surfaces must stay settled, kept public market witness and market exits on stronger deterministic paths, turned pre-v4 receiz into a first-class governed historical cohort instead of a loose legacy exception path, and published the standards and evidence that now define those behaviors.

`v60.1.0` has a release note in the repository, but the post-`v60.0.0` line has no matching repository tags and no dedicated `v60.2.0` through `v60.4.x` public release notes. This `v61.0.0` note therefore rolls up the full repository delta after `v60.0.0`, including the `v60.1.0` documented interval, the undocumented `v60.2.0` through `v60.4.x` interval, and the current release-prep working tree, into one canonical release record.

The product-truth freeze for this release is defined in `docs/releases/v61.0.0-product-truth.md`.

## What Defines v61.0.0

- Historical pre-v4 receiz are now governed as an explicit public historical cohort with frozen membership, signed governance artifact support, historical trust-state presentation, and verified transport packaging that preserves original bytes.
- Settled home and profile twin shells now enforce explicit-intent rules more strictly: compact touch devices keep full twin hydration on-demand, the home dock stays on its closed audit shell until explicit activation, and hidden warming stays hidden.
- Public market witness truth from `v60.1.0` remains the floor, and market desk exits now keep deterministic internal routes on the shared stable-navigation client path instead of dropping back to weaker navigation behavior.
- Receiz now publishes the operating standards behind that behavior: experience-first engineering, the expensive truth map, the fast page playbook, and the historical continuity standard are part of the repository truth for this release line.
- The current release-prep tree also tightens compact verified-player audio controls and home twin dock image sizing without changing the closed-shell continuity model.
- Release-governed version surfaces now move together to `v61.0.0` across package metadata, service worker defaults, public verifier artifacts, home, make, footer, legal, trust, plans, upgrade, and the active release docs.

## Since v60.0.0

### 1. Public Market Truth And Exit Continuity Tightened

The `v60.1.0` and `v60.2.0` work made market both more truthful and more continuous.

Public profile asset markets now prefer live holder-side evidence over creator-side fallback clocks:

- holder count comes from public positions with non-zero quantity
- latest witness timing prefers public buys, owner position-open time, and matching share-trade history
- witnessed duration no longer stays stranded at zero when the system already knows real activity

The public and signed-out desk was also retuned for real compact usage:

- chart dock spacing, timestamps, and proof metadata tiles fit better in narrow widths
- compact hero and public stat cards are denser and easier to read
- signed-out and public summary copy wastes less space

After that floor was set, `v60.2.0` tightened the exit path:

- `Verify asset` and `Open source` actions now force the shared stable client-navigation path for deterministic Receiz routes
- `/v/...`, `/r/...`, verified-player, and original-viewer targets no longer risk falling back to a weaker navigation mode
- market continuation and non-holder behavior remain coherent: buy continues through the same surface model, and sell stays hidden when there is no position

This matters because the market now prefers stronger known truth in both what it says and how it moves.

### 2. Settled Twin Shells Became Hard Runtime Boundaries

The twin/runtime line since `v60.0.0` has one product rule: once a shell feels loaded, background work does not get to visually redefine it.

The earlier continuity fixes from `v60.1.0` established the floor:

- the visible home and profile dock shell stays on the stable `WorldAgentDockRig`
- background preload may warm code and data, but it may not auto-promote the full module
- remembered twin session state no longer reopens the richer runtime on load

The later `v60.4.1` follow-up tightened that boundary further:

- compact touch devices now keep full twin background hydration on-demand instead of auto-starting it
- the hidden-preparation path prefetches world data without eagerly warming the full runtime graph

The current release-prep tree finishes the home dock handoff:

- `HomeHeroTwinDock` now keeps the lightweight audit shell mounted until explicit activation
- runtime warming can start on hover, focus, touch, or pointer intent without taking over the visible shell
- explicit activation loads the deferred module and opens it through `openOnMount`, instead of letting hidden preparation promote the surface by itself
- the audit shell now uses an exact-fit proxied dock render image, and compact verified-player audio controls are tightened so the mobile control glass stays legible

The user-facing effect stays simple: if the page already looks loaded, it does not load again until the user explicitly opens the richer twin surface.

### 3. Pre-v4 Receiz Became A Governed Historical Cohort

The largest post-`v60.0.0` addition is the historical continuity system landed in the legacy-repair merge.

Receiz now treats the first pre-v4 transition as governed historical continuity instead of a vague legacy exception:

- a frozen public cohort, `bjklock_pre_v4_2026_04`, is locked by exact immutable membership
- the public governance payload contains `61` record IDs and is fingerprinted and hash-locked
- historical trust presentation is explicit: `Historical` is distinct from `Verified` and `Warning`
- current verifier semantics stay strict; non-members do not get historical status by shape

That governance model now drives the product and tooling:

- stale localhost verify URLs in pre-v4 bundles can be canonicalized to the production origin without changing proof identity
- historical downloads can ship as verified `.historical.receiz-package.zip` bundles that preserve untouched raw bytes and add a separately sealed governance artifact plus package manifests
- receipt and proof surfaces can explain why a record is historical instead of presenting it as broken or silently current
- repair, packaging, and cohort integrity are all covered by dedicated scripts and tests:
  - `repair:pre-v4-receiz`
  - `build:receiz-historical-governance-artifact`
  - `test:receiz-historical-records`
  - `test:receiz-historical-package`
  - `test:receiz-legacy-repair`

This is the key trust shift in `v61.0.0`: authentic history is preserved without falsifying what current Receiz Signature v4 verification means.

### 4. Historical Continuity And Fast-Page Discipline Became Published Standards

The runtime changes above are no longer just code paths. They are now documented operating rules.

This release line adds or expands:

- `docs/experience-first-engineering.md`
- `docs/expensive-truth-map.md`
- `docs/fast-page-playbook.md`
- `docs/governance/historical-continuity-standard.v1.md`
- `docs/verified-history-first-principles.md`
- governance document-register and artifact-manifest updates

Together they freeze the current Receiz stance:

- deterministic surfaces never use fake loading theater
- stronger known truth beats weaker later payloads
- background warming stays hidden until explicit user intent
- historical cohorts are governed, additive, and never inferred by broad legacy shape

The repository also now carries the performance evidence behind that model, including saved Lighthouse artifacts and a rendered playbook PDF. The fast-page playbook records the concrete route-entry delta against the cited baseline commit: home route-entry raw bytes are `74.2%` smaller and profile route-entry raw bytes are `99.1%` smaller.
