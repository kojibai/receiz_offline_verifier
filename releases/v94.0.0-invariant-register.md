# Receiz v94.0.0 Invariant Register

Release date: 2026-06-27

This register makes explicit the v94.0 invariants proven by the official SDK and durable proof operating model release.

## 1. Proof Objects Remain Authority

Receiz proof objects carry deterministic identity, verifiable payload/proof material, provenance, inspection behavior, durable state, and stronger source-of-truth status than generic display state.

Violation: an API response, SDK projection, index row, cache, or route-local state replaces sealed or admitted proof truth.

## 2. Durable Proof Memory Is Append-Only Admission

First verified admission creates known truth for a stable proof identity. Later verified additions append after the known head.

Violation: reopening a proof surface rediscovers old truth before rendering, or a newer partial payload erases a fuller admitted proof object.

## 3. First Paint Uses Known Verified Truth

Deterministic file proof, verified local register truth, route-bound payloads, public witness snapshots, and admitted proof memory paint before weaker reconciliation work.

Violation: first paint is downgraded to loading, placeholder, analytics/session dependency, or server rediscovery while known proof truth exists.

## 4. Kai Is The State Coordinate

Kai pulse/upulse orders Receiz proof objects and settlement ledger additions when present. Chronos is display context and compatibility fallback.

Violation: backend state ordering overwrites app-supplied Kai child coordinates or treats Chronos as the primary state machine where Kai exists.

## 5. Projection Mechanics Stay Beneath Proof Authority

Database indexes, RPCs, route snapshots, and SDK projections may make verified additions faster to discover and render. They do not define Receiz truth.

Violation: a projection row, summary, timeout, or external payload is treated as stronger than the proof object, verified append, ownership append, or settlement ledger row.

## 6. Official SDK Is Convenience, Not Authority

`@receiz/sdk@94.0.0` is the official developer convenience package for verifying, projecting, admitting, persisting, appending, and rendering Receiz proof objects correctly.

Violation: SDK helpers become a truth engine above sealed artifacts, proof bundles, verified appends, ownership appends, or settlement ledger rows.

## 7. Sports Proof Economy Preserves Ledger Truth

Sports cards, Sports Economy, tournaments, event proofs, card appends, ownership appends, showcase projections, and live valuation are proof surfaces over ledger truth.

Violation: read hardening drops history, hides appends, changes ownership truth, or replaces proof payloads with summary-only state.

## 8. Wallet Settlement Uses Verified Addition Windows

Wallet history is settlement proof projection ordered by Kai heads and append windows.

Violation: wallet surfaces ask whether old truth is still true instead of asking for verified additions after the known settlement head.

## 9. Calendar And Account Share Event Proof Projection

Calendar points and account brain activity nodes are projections of admitted proof objects.

Violation: the same event proof object is duplicated into competing route-local truth systems.

## 10. Offline Verification Remains File-Authoritative

This offline verifier can document platform SDK behavior, but verification authority remains in selected file bytes and embedded proof material.

Violation: release metadata, SDK docs, service-worker cache state, external registry state, or route copy is presented as stronger than file proof.

## Release-Lock Requirement

`pnpm release:lock 94.0.0` must require this document for `v94.0.0` and later release cuts. A release that omits the invariant register is incomplete because proof object authority, append-only proof memory, first-paint truth, Kai ordering, projection boundaries, SDK convenience, Sports proof economy, settlement windows, shared event projection, and offline file authority would not be explicit release truth.
