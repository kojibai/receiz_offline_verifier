## Title

Receiz `v90.0.0` - Official Stable Proof-Native Artifact System Release

## Summary

`v90.0.0` is the official stable release that locks the full Receiz arc from the first proof product through the current proof-native artifact system.

The heavily documented `v80.0.0` to current arc spans `723` commits, `1,043` files changed, `339,345` insertions, and `6,263` deletions before the v90 release-lock edits. The mechanical commit ledger is preserved in `docs/releases/v90.0.0-commit-history.md`.

The post-`v89.0.0` arc spans `29` commits, `100` files changed, `9,949` insertions, and `1,259` deletions before the v90 release-lock edits. That slice locks the final Sports proof, ownership transfer, event proof, live value, conformance, market-depth, competition-room, and metadata repairs that promote `v90.0.0` from release note to stable release.

The v90 statement is direct:

```txt
Receiz is a proof-native artifact system.
Files, identities, ownership states, settlement states, public witness surfaces, vault objects, cards, passports, market objects, Sports event proofs, and verification routes carry product truth.
Server, database, session, service worker, outbox, global sync, metadata, analytics, and UI cache are mechanics beneath that truth.
Known stronger truth paints first.
Verified additions append after it.
No weaker layer may rename, delay, replace, or downgrade it.
```

## What Receiz Is

Receiz is not a generic app with proof features.

Receiz is a proof-native artifact system where:

- a file can carry verifiable truth
- a public route can project proof without becoming the proof
- a Receiz ID can carry local identity continuity
- PBI, Receiz Key, Identity Record, and Identity Seal can prove recovery and authorship
- ownership can move as proof, not only as account state
- settlement can move as governed value, not only as a balance display
- a public profile can be canonical truth, not a mutable social shell
- a card, passport, certificate, vault object, or original can remain legible as itself across private, public, market, and offline surfaces
- offline proof is deterministic projection from verified truth, not degraded fallback

The product matters because it makes digital ownership, identity, authorship, settlement, and public verification behave like durable reality instead of temporary application state.

## The Whole Arc

The repository history records a continuous climb from proof object to operating system.

### v1 to v9 - The Proof Object Foundation

The first era built the core mint, verify, share, receipt, QR, PNG, signature, and proof-bundle rails. The work established the first user-visible proof object: a minted Receiz artifact that could be rendered, downloaded, shared, and verified.

The important outcomes were:

- receipt and verification routes
- magic-link and premium account mechanics
- deterministic PNG output
- QR and public proof links
- sigil/Kai timing foundations
- Groth16/Poseidon proof metadata experiments
- embedded proof bundles
- PBI signing and authorship flow
- WireProof and AssetSeal as early proof/custody primitives
- offline verifier beginnings

This era proved the first claim: Receiz could put proof into an artifact instead of leaving truth inside the app.

### v10 to v20 - Proof Hardening, Profiles, Originals, Identity, And Auth

The commit history records `v10.0.0` and `v20.0.0` milestones even though the current repository does not retain dedicated v10/v20 release-note files.

- SVG, JSON, ZIP, and folder proof handling grew out of the file-first model

The v13 to v20 path added product identity around that proof:

This era made proof usable by people and accounts, not only by files.
