## Title

Receiz `v97.6.0` - Signed Merchant Public-Store, Live Proof Graph, MCP, SDK, and Service Worker Law Release

## Summary

`v97.6.0` locks the next developer-platform layer after `v97.5.0`: merchant-owned public-store publishing through Identity Seal / Receiz Key proof authority, a live public proof graph surface, MCP and SDK release alignment, service-worker runtime alignment, and Sports vault saved-card language that separates base issue probability from scarce rarity odds.

This release preserves the authority hierarchy. The SDK, MCP, server, service worker, database, and public projection rail are convenience, sync, indexing, and inspection layers beneath proof objects, verified local truth, Kai/Klok state, ownership appends, settlement primitives, and public proof surfaces.

## What Changed Since v97.5.0

- `@receiz/sdk` now exposes the no-env-token signed merchant public-store rail:
  - `publicStore.signPublish(...)`
  - `publicStore.publishSigned(...)`
  - `publicStore.publishWithIdentityProof(...)`
  - `publicStore.restoreLatest(...)`
  - `publicStore.assertSignedPublishContract(...)`
- Signed merchant public-store publish uses the merchant's Identity Seal / Receiz Key proof object as the authority. It does not require `RECEIZ_ACCESS_TOKEN`, and the SDK suppresses configured bearer tokens for that lane.
- `POST /api/public-proof/registry/feed` accepts signed public-store envelopes, verifies feed digest, challenge digest, tenant host, merchant Receiz ID, public key identity, and signature, then appends the public projection beneath Receiz proof rails.
- The signed publish result returns the append coordinate:
  - `kaiPulse`
  - `appendAnchorId`
  - `appendProof`
  - `knownHead.afterKaiUpulse`
- `/proof-graph` became a public one-viewport proof graph organism: live `☤KAI` pulse, proof/economy/flow/asset/agent views, mobile-first graph navigation, and in-page public proof network inspection.
- `/exchange` now exposes the proof graph from the file footer proof rail.
- `@receiz/mcp-server` remains the official agent wrapper over the SDK. Documentation now separates delegated agent/server writes from merchant proof-owned public-store publishing.
- Service worker runtime identity moved to `97.6.0` and offline verifier labels moved to `v97.6.0` so runtime continuity, offline proof verification, and app release identity remain aligned.
- Sports saved-card export now renders base cards as `BASE 1:1` / `Base issue 1:1`. Rare and premium cards keep `ODDS` and rarity odds language.

## Developer Outcome

Apps can publish durable storefront state with the merchant's proof object, then recover it publicly after a cold start:

```ts
import { createReceizClient } from "@receiz/sdk";

const receiz = createReceizClient({ baseUrl: "https://receiz.com" });

const result = await receiz.publicStore.publishWithIdentityProof({
  keyFile,
  tenantHost: "bjklock.receiz.app",
  merchantReceizId: "bjklock.receiz.id",
  storeStateRecord,
});

console.log(result.knownHead.afterKaiUpulse, result.appendAnchorId);

const restored = await receiz.publicStore.restoreLatest({
  host: "bjklock.receiz.app",
  requiredSchema: "receiz.app.store_state.v1",
});
```

Agent hosts still use delegated tokens for agent/server tasks:

```json
{
  "mcpServers": {
    "receiz": {
      "command": "npx",
      "args": ["-y", "@receiz/mcp-server"],
      "env": {
        "RECEIZ_BASE_URL": "https://receiz.com",
        "RECEIZ_ACCESS_TOKEN": "<delegated agent token>"
      }
    }
  }
}
```

The distinction is release law: merchant-owned public store state should be signed by merchant proof authority; delegated tokens are for scoped agent/server operations.

## Product Law Preserved

- Proof objects remain the source of truth.
- Kai Klok remains the proof object state machine.
- Durable proof memory remains first admission only, then append forever.
- Identity Seal / Receiz Key signed publish is proof authority for merchant-owned public-store sync.
- Public app-state projections remain recovery projections beneath proof objects.
- MCP remains agent tooling beneath SDK/API rails.
- Delegated agent tokens remain scoped Connect/OIDC access, not merchant proof authority.
- Service worker state remains runtime continuity and offline projection support, not proof authority.
- Base-card probability is not renamed as scarce rarity odds.
