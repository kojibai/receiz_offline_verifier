## Title

Receiz `v68.0.0` - Local Truth Production Baseline

## Summary

`v68.0.0` is the production release that joins the `v64` value-loop baseline with the local-proof/offline-primitive work completed through the official Receiz ID branch.

The release target is exact:

> Keep the excellent `v64.6.0` production behavior and server function, then move known local truth earlier everywhere it is already proven.

This release does not replace the server. It demotes the server from permission to propagation when the device already holds proof. The server still mints cookies, publishes appends, writes global mirrors, runs recovery, serves global discovery, powers online twin/runtime paths, and syncs new verified additions. The user simply does not wait on those jobs before seeing or acting on truth the local node already holds.

## Baseline

### `v64.0.0`

`v64.0.0` is the Complete Value Loop baseline:

- Settlement is proof-native value.
- Reserve is funded value for external rails, notes, sends, positions, and wire transfer.
- Notes remain Reserve-only.
- Certificates bridge proof-native Settlement to buyer-funded Reserve.
- Market sells split funded Reserve from remaining Settlement.
- Sends and buys consume Reserve first, then Settlement.
- Wallet and Market must preserve lane truth and quote math.

`v68.0.0` keeps those economic invariants intact.

### `v64.6.0`

`59cc4b7` / `v64.6.0` is the stable production UX/server behavior baseline:

- logged-in and logged-out routes were stable
- stale-cookie PWAs did not crash the app
- server session/cookie/API paths continued doing the normal production work
- the app functioned well as a web product

`v68.0.0` treats `v64.6.0` as the behavior we preserve, not as something to discard. The local-first work is additive: it removes waits for already-held proof without weakening server sync, global publication, or user experience.

## Full Change Record From v64 To v68

### 1. Product truth was formalized

What changed:

- Added and expanded `docs/offline-verified-register.md`.
- Connected it to `docs/experience-first-engineering.md`, `docs/deterministic-surfaces.md`, and `docs/verified-history-first-principles.md`.
- Froze the language that known verified truth is a verified prefix, not stale mutable data.
- Added the rule that sync asks for verified additions, not permission to believe known truth.

Why:

Receiz primitives already worked offline. The app needed to stop behaving like a conventional server-authority web app when it already held stronger proof locally.

What this preserves:

Server/global systems still move truth globally. They no longer get to delay or downgrade local proof.

### 2. Local first, server always became the governing shape

What changed:

- User actions commit/project local proof first.
- The same action path starts server/global propagation when network exists.
- Server work is classified as publication, recovery, cookie minting, mirroring, discovery, or verified addition.
- Server failure, 401, 404, timeout, or missing-cookie state cannot roll back verified local truth.

Why:

Treating offline primitives as a reason to skip server work was an over-correction. Treating server work as permission was the legacy mistake. The correct product shape is both:

```txt
local proof -> local commit/projection now -> server/global propagation starts -> verified append/recovery admitted later
```

What this preserves:

`v64.6.0` server behavior remains valuable and active. The user simply does not wait on it for truth already held locally.

### 3. Receiz ID became the primary one-click login rail

What changed:

- One-click Receiz ID login now creates or reuses a local Ed25519 device identity.
- If the device already holds a Receiz ID, the client signs a fresh local challenge and projects immediately.
- If the device has no key yet, the browser creates the Receiz ID locally before server publication.
- `/api/auth/receiz-id/continue` verifies the same signed proof, creates or continues a global account, mints the server cookie, and returns account bindings.
- Server publication runs immediately in the background after local projection.

Why:

Identity is local. Proof is local. A user with valid local device proof should not wait for the server before the app can project their account shell.

What this preserves:

Server session cookies still exist. They are minted as soon as the server validates the local proof. Cookie/session remains a convenience and global interoperability rail, not the first source of identity truth.

### 4. PBI and magic-link login became interop/recovery rails

What changed:

- PBI stays in the product.
- Magic link stays in the product.
- Both become Account recovery/interoperability paths rather than primary everyday login.
- Legacy PBI users can recover their old account and bind it to the current Receiz ID.
- Legacy email users can still open a Receiz email link; after the session exists, the local Receiz ID can bind naturally.

Why:

The product needed a primary Receiz-native identity path while keeping existing users whole.

What this preserves:

No legacy login logic is thrown away. Existing email/PBI users still recover and continue. Future login becomes one press after binding.

### 5. Multi-account binding was added under one Receiz ID

What changed:

- A local account-binding register stores multiple accounts under one Receiz key.
- Active account selection projects locally first.
- Server switch route verifies the local device signature.
- Server storage supports many account bindings per Receiz key.

Why:

Users can legitimately have multiple passkeys, emails, and accounts. Receiz ID should become the local continuity root that can switch between bound accounts without making the user rebuild identity each time.

What this preserves:

Server-side account rows and legacy identities still exist. The binding register makes them easier to manage from the local identity root.

### 6. Durable session bootstrap was corrected into projection, not proof

What changed:

- Session bootstrap can still seed fast first paint.
- Durable signed-in bootstrap is now written only when the projection is backed by local proof roots: Receiz ID, PBI/keyfile, offline identity register, or account binding.
- Generic server/session bootstrap remains a fast projection convenience, but does not become indefinite durable authority by itself.
- Signed-in bootstrap snapshots without a real UID are rejected.
- `GlobalAuthChrome` no longer lets unbacked bootstrap skip the normal server session path.

Why:

The first bad post-`v64.6.0` regression came from making signed-in bootstrap durable before runtime recovery and proof boundaries were strict enough. That made stale projection able to survive bad runtime/session edges.

What this preserves:

`v64.6.0` session behavior remains intact. Proof-backed local continuity is added without letting a stale projection masquerade as identity proof.

### 7. Stale runtime and stale cookie recovery was hardened

What changed:

- Added runtime recovery for React #185, max-depth loops, and null `addEventListener` stale-runtime crashes.
- Recovery order is fixed:
  1. read valid local proof/Receiz ID
  2. resync from server session if online
  3. clear stale projection and land stable on home
- Service-worker runtime chunks, styles, and workers are network-first while online.
- Cross-origin requests are left browser-owned instead of being manufactured by the service worker.
- Stale service-worker/runtime state must never crash the app.

Why:

A stale logged-in cookie or stale PWA should never bring down the application. If local proof exists, use it. If local runtime state is suspect but server session exists, recover from the server. If neither can be safely admitted, the user lands on stable home.

What this preserves:

Offline route snapshot behavior remains. App-shell runtime coherence is no longer treated as user truth.

### 8. Service worker storage became explicit truth carriage

What changed:

- Deterministic and private snapshots opt into offline persistence through `x-receiz-offline-persist`.
- The service worker carries deterministic/private truth only when routes explicitly mark it.
- Local actor namespace can be set from locally verified identity.
- Anonymous server session payloads cannot clear a local actor namespace that came from verified local identity.

Why:

The service worker should be a vault for verified local truth, not generic mutable cache.

What this preserves:

Server routes still return normal API data. Offline persistence becomes explicit where data has deterministic/private truth value.

### 9. Wallet first paint and local graph continuity improved

What changed:

- Wallet route hydration can project held wallet graph state before server reconciliation.
- Known signed-in zero state is `$0.00`, not loading theater.
- Account value must not drop to a label when balance is absent; if no balance exists, value is zero until proven otherwise.
- Active Wallet views use the lean wallet-summary sync path so newly received funds update the visible balance without a page refresh.
- Wallet UI refinements reduced vertical weight in the market wallet panel and moved add funds / open wallet / import certificate actions above the chart so the action result appears in the same viewport.
- "Send via Receiz" copy was simplified to "Send".

Why:

The wallet already knows the user's local graph and zero state. It should not make the user watch the app rediscover what it holds.

What this preserves:

Server wallet API still syncs balances, ledger rows, notes, transfers, and global wallet state. Local first paint simply stops waiting on that sync when truth is already held.

### 10. Market positions and tracked positions now preserve local action truth

What changed:

- Buy/sell actions update local positions immediately.
- Lagging market snapshots cannot drop a just-created position.
- Tracked positions follow the same preservation rule as wallet and ledger.
- Market first paint can carry active share/prediction ownership before full snapshot hydration.
- Active Market views subscribe to wallet-summary truth and force a lean summary before buy shortfall math, so a second buy cannot trust an old balance and dead-end on a server correction.
- Wallet/market value pills were refined for logged-in and logged-out states.
- The username badge was constrained so it fits without handing off the badge on small screens.

Why:

A local buy or sell is action truth. A delayed snapshot that does not include it yet is weaker transport, not stronger truth.

What this preserves:

The market snapshot API, live chart, playback, wallet summary, and server settlement routes still run. They can append or enrich; they cannot remove known local position truth without stronger proof.

### 11. Explore was moved toward verified additions

What changed:

- Explore snapshots became local route truth instead of a brittle remote-first surface.
- Hydration selects the strongest visible Explore view instead of replacing useful content with weaker network payloads.
- Missing exports and brittle route-hydration helpers were corrected.
- Single-view Explore cards reserve separate media and owner-identity lanes so image/audio cards cannot cover the creator `@` handle.

Why:

Explore should not visually regress just because the network is discovering additions. Known content is known content.

What this preserves:

`/api/explore/view` still discovers global additions. It just does not get to downgrade already useful visible truth.

### 12. World/profile route continuity improved

What changed:

- World route snapshots were made referentially stable for `useSyncExternalStore`.
- Background hydration is not allowed to pull the user back to the beginning of a settled World surface.
- Local profile route seeds can project the newly created Receiz ID profile immediately.
- Managed username routes remain the primary user-facing route shape.
- Profile/account creation can land the user directly on the managed profile route after local identity creation.

Why:

If the app already knows the user's profile truth, first navigation should feel instant. Background server work should not remount or reset the visible surface.

What this preserves:

Server profile and world routes still load global/public profile data and online enrichments.

### 13. Profile PBI signing was placed on the public profile surface

What changed:

- First-time profiles can show a compact PBI signing rail above the fold on the profile surface.
- If a user has a passkey, the ceremony attempts to select/recover with that passkey first.
- If no passkey exists, the flow can create/sign as appropriate.
- After successful signing, the prompt disappears and the profile shows PBI-backed verification/blue-check style proof.

Why:

The profile is the user's identity surface. Presence-bound proof belongs there as a compact action until it is done.

What this preserves:

Existing PBI ceremony logic remains. The release surfaces it where it teaches the product and completes local proof.

### 14. Hide/unhide and profile visibility moved back to deterministic action truth

What changed:

- Profile visibility actions are treated as deterministic owner actions.
- Non-entitled users can hide/unhide posts without being blocked by glyph-economy entitlement checks.
- Glyph economy entitlement remains available for economy features, but hiding a user's own post is not treated as a paid entitlement requirement.

Why:

The owner has local action truth over visibility of their own page. The server syncs the action globally; it does not decide whether the local action happened.

What this preserves:

Server publication and global profile visibility still sync. Local owner action no longer waits on unrelated entitlement state.

### 15. Profile posting moved toward local truth before upload propagation

What changed:

- The product rule was clarified: sealing and local post truth happen locally first.
- Upload/server sync is propagation.
- The server should still start immediately when online and should still have a valid session/cookie whenever possible.
- Upload failures must not make a logged-in local user appear logged out.

Why:

Posting is a proof-bearing local action. The server distributes and stores it globally; it is not the moment truth becomes real.

What this preserves:

The upload pipeline and server publication remain required for global reach. The local composer should not block the proof/action truth on that publication.

### 16. Local signed outbox direction was added

What changed:

- Local signed action outbox contracts were added for private server actions.
- Identity publication and other proof-bearing writes can be modeled as local signed appends first, server propagation second.
- Server routes still verify signatures and publish.

Why:

The app needed a broad primitive for "truth first, sync next" that can eventually cover every private server action.

What this preserves:

Private server actions still run. The outbox is the bridge that prevents them from being permission gates for local proof.

### 17. Public twin gained offline/local continuity

What changed:

- The twin can project from local twin register truth.
- The online stream remains intact.
- Local projection is used first when a verified local twin mind register exists.
- Known profile/twin image identity is preserved across partial world payloads.
- A partial world bootstrap cannot suppress a deterministic profile/twin image and force a generic rig.
- A stale `world_profile_load_failed` bootstrap result cannot remain visible after the twin has already produced a local or streamed answer.
- Same-rank local model output cannot overwrite profile, receipt, memory, wallet, market, or twin-mind truth without domain-specific merge/proof rules.

Why:

The twin should not die when the network drops if the node already has local authored truth. It also should not forget its known visual identity just because a later world/profile payload is incomplete.

What this preserves:

The current online AI twin brain stays remarkable and intact. Local/offline capability is additive continuity.

### 18. Twin mind portability moved from server-secret HMAC to public-key verification

What changed:

- Twin mind PNG import/export authenticity moved toward portable public-key verification.
- Production now requires:
  - `RECEIZ_WORLD_TWIN_MIND_SIGNING_PRIVATE_KEY_PEM`
  - `RECEIZ_WORLD_TWIN_MIND_SIGNING_PUBLIC_KEY_PEM`
  - `RECEIZ_WORLD_TWIN_MIND_KEY_ID`

Why:

If a PNG carries portable twin mind truth, offline nodes must be able to verify authenticity without a server-held secret.

What this preserves:

Server-authenticated import/export still exists as online convenience. Portable verification becomes the authority path for offline truth.

### 19. Live player, viewer, receipt, and deterministic media rules were tightened

What changed:

- Deterministic viewer/player routes must reuse known artifact/player truth.
- Snapshots are allowed as additive transition fast paths, not fake placeholders.
- Live player entry no longer waits on full queue hydration or world preview augmentation before the selected original can paint.
- Live player routes no longer convert an incomplete/momentarily unresolved route into the app-level 404 before the client can restore from held player snapshots.
- If no snapshot is present yet, live player may show only the real route backdrop, never a white frame, spinner, or fake loading copy.
- Full queue, world preview, artwork augmentation, and profile enrichment hydrate after the selected player truth is visible.
- A held multi-track artist/video flow cannot be downgraded by a later single-track server payload; hydration may extend or complete the queue, not collapse the visible playback mode.
- Media/poster/artwork identity should come from deterministic blocks and route snapshots where possible.
- The 621px-920px live-player breakpoint keeps audio controls reachable by letting the route scroll and by docking the transport outside the constrained square art frame.
- Global audio playback remains provider-owned, so the next song can continue/crossfade in the background the way it did in the stable `v64.6.0` baseline.
- Market receipt and playback links are treated as deterministic routes that should not probe server truth before rendering known local truth.

Why:

Profile -> live player and artifact viewer -> live player should both honor the same route truth. If the previous surface already knows the media block, the next surface should open from it. A route transition that shows a long white screen violates the surface even when the eventual player resolves correctly.

What this preserves:

Server playback and media APIs still resolve cold opens, global discovery, and missing blocks.

### 20. UI polish was folded into the truth work

What changed:

- Market badge text sizing was adjusted so usernames fit their badge.
- Market wallet panel tabs were made smaller and more refined.
- Video popover back-card pills were reduced to match the front.
- Mobile input zoom was hardened beyond the simple `16px` approach.
- Bottom bar signed-in awareness was fixed on Contacts/Calendar/Wallet/Market/Profile surfaces.

Why:

Primitive correctness is not enough if the user experience regresses. Law first, then elite user experience.

What this preserves:

The polished `v64.6.0` feel remains the target.

## Exact Corrections From Over-Corrections

### Durable bootstrap correction

Problem:

The first bad commit after `v64.6.0` made signed-in bootstrap durable before proof boundaries and stale-runtime recovery were strict enough.

Correction:

Durable signed-in bootstrap now requires proof-backed input. Generic server/session bootstrap remains fast but not indefinite authority.

### Server omission correction

Problem:

Some local-first paths drifted toward treating the server as optional.

Correction:

The release freezes "local first, server always." User never waits for server when proof is held, but server propagation still starts when network exists.

### Runtime recovery correction

Problem:

Stale SW/runtime/cookie state could produce React #185 or application failure for logged-in users.

Correction:

Runtime recovery now tries local proof, then server session resync, then stable home. App-shell chunks are network-first while online.

### Unbacked bootstrap correction

Problem:

An unbacked bootstrap username could skip normal server session hydration.

Correction:

`GlobalAuthChrome` only bypasses the normal server session path when the bootstrap is backed by local offline identity proof.

## What v68.0.0 Intentionally Does Not Replace

- It does not replace the `v64.0.0` value loop.
- It does not remove server routes.
- It does not remove session cookies.
- It does not remove database/global mirrors.
- It does not remove online AI twin behavior.
- It does not remove PBI.
- It does not remove email/magic-link recovery.
- It does not remove payment or settlement server rails.

## Release Principle

`v68.0.0` is the release where Receiz becomes coherent about its own power:

- local proof is real
- local truth moves first
- server propagation always starts when possible
- server sync moves truth globally
- settled visible surfaces do not get redefined by background work
- stale runtime/session state never crashes the app
- `v64.6.0` production stability remains the baseline

If a future change makes the user wait on the server to believe proof already held locally, it violates `v68.0.0`.

If a future change skips server propagation when network exists, it also violates `v68.0.0`.

The law is both.
