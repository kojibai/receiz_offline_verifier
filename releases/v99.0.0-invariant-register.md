# Receiz v99.0.0 Invariant Register

`v99.0.0` carries forward the proof-native artifact system, public app-state projection, enterprise SDK, MCP agent rail, and signed merchant public-store invariants from `v97.x` and locks the full post-`v97.0.0` developer-runtime, proof graph, ledger, transaction, sports, profile, market, webhook, skill, and Pitch Command archive working set.

## Locked Invariants

1. Receiz remains a proof-native artifact system. Product truth is carried by files, identities, vault objects, ownership states, settlement states, public witness surfaces, verification routes, public proof surfaces, and append-only history.
2. Proof objects remain stronger truth than SDK helpers, MCP tools, OIDC tokens, webhook endpoints, server rows, service-worker state, public-store projections, app-state rows, charts, analytics, SEO, session state, or generic display state.
3. Kai Klok remains the proof object state machine. `kaiPulseEternal` / `kai_upulse` remains the pulse unit for complete proof objects.
4. Durable proof memory remains first admission only, then append forever. Hidden work may only append verified additions after the known head.
5. Deterministic first paint must project known proof, identity, ownership, settlement, profile, market, ledger, transaction, Sports, public graph, and archive truth before weaker work.
6. `@receiz/sdk` remains typed developer convenience beneath proof objects, verified appends, identity primitives, ownership appends, settlement rows, public proof surfaces, and offline verification.
7. `@receiz/mcp-server` remains an agent rail over SDK/API mechanics and must name its authority boundary.
8. Delegated OIDC agent tokens remain scoped, revocable, one-time-visible mechanics for MCP, CI, and trusted local agents; they do not replace Receiz identity proof.
9. Developer webhooks sign delivery events beneath delegated scopes; endpoint state, secrets, test events, and delivery logs do not become proof authority.
10. Merchant-owned public-store state may be published through Identity Seal / Receiz Key proof authority without delegated bearer-token authority.
11. Public app-state and public-store rows remain recovery projections beneath sealed artifacts, identity proofs, ownership appends, settlement rows, verified local truth, and verified appends.
12. `/proof-graph` remains a public proof surface carrying real release/API/product proof data, proof/economy/flow/asset/agent state, and live Kai pulse state.
13. Ledger, wallet, transaction, market-cap, and settlement rollup surfaces must preserve Kai/proof coordinates, proof-bundle inspection, and settlement language.
14. Sports vault and arena performance work must preserve known proof, media, ownership, closeout, rollup, and settlement truth before background work.
15. Profile and market public projection freshness is a write-side obligation after deterministic read paths are protected from rediscovery.
16. Pitch Command day-proof archives are proof archive material with strict manifest/projection contracts.
17. Receiz AI skills must teach implemented proof primitives directly and must not soften Receiz into generic app categories.
18. Static verifier, offline verifier, record-seal studio, settlement entrypoint, Sports card verifier, package metadata, release docs, and service-worker runtime version must move together.
