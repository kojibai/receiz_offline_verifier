# Receiz v64.0.0 Release Process

Release date: April 21, 2026

Product-truth reference:

- `docs/value-loop-invariants.md`
- `docs/releases/v64.0.0-product-truth.md`

## Goal

Make Receiz `v64.0.0` the first release where the product is documented as a complete value system, not just a set of advanced primitives.

The central problem was not whether Receiz could prove records, move wallet value, price assets, issue notes, or trade Settlement shares. Those primitives already existed.

The central problem was making the economic loop explicit enough that users and future contributors cannot accidentally collapse every value state into one wire-transfer number.

## What Became Clear

The full product needed one governing distinction:

- Reserve is funded value for notes, sends, positions, and wire transfers.
- Settlement is proof-native value.
- Price can move without becoming unfunded Reserve.
- Certificates are the buyer-funded bridge.
- Notes are Reserve-only.
- Wire transfer is a downstream conversion path.

Once this distinction became explicit, the product stopped feeling like separate surfaces and started acting like one system.

## Why This Matters

Without the distinction, product success could pressure Receiz into becoming a fiat wire-transfer surface.

That would be the wrong center of gravity because:

- market growth is not automatically external funding
- proof-native value should not be framed as fake or lesser
- Wire transfer should not define whether Receiz value is real
- the platform should not create unfunded obligations
- users need a path to liquidity that does not break causal accounting

The value loop solves this:

1. Reserve handles funded actions and wire transfers.
2. Settlement is proof-native value that can be sent, used, or certified now.
3. Certificates let a buyer fund conversion.
4. Sellers receive Reserve only after funded purchase.

## What Worked

- Keeping notes Reserve-only made notes simple and trustworthy.
- Keeping Settlement proof-native avoided making wire transfer the center of the product.
- Using certificates as the bridge turned live/unfunded value into an action users can share, sell, and fund.
- Splitting market sells into Reserve and Settlement let users realize value where funded liquidity exists without trapping the rest or creating fake Reserve.
- Making sends and buys Reserve-first, Settlement-second gave users maximum flexibility while preserving accounting truth.
- Updating Market and Wallet charts to use real quote math prevented the product from visually lying about Φ.
- Making the Settlement wallet card tappable gave users an explicit value surface instead of a hidden lane.

## What Did Not Work

- A single balance number carried too much meaning.
- Provider-specific settlement language was too implementation-facing.
- "Usable inside Receiz" framed Settlement too low; the correct frame is proof-native value.
- Wire/external withdrawal language pulled attention toward wire transfer instead of the full Receiz value loop.
- Reconstructing the Settlement chart from partial recent receipts could produce misleading visible spikes; the chart needed to reflect current lane truth unless full historical lane truth is available.
- Showing Φ by swapping symbols instead of converting through the account quote was a product-truth bug.

## Banned Future Reasoning

Do not say:

- "If it has USD reference value, it should be Reserve."
- "If it cannot wire out instantly, it is not real."
- "If a certificate is issued, the creator should be credited."
- "If a market price moved, the gain should become external-ready Reserve."
- "If a buyer funds a peer certificate, platform direct-sell liquidity increased."
- "It is fine to show Φ by replacing `$` with `Φ`."

Those are regressions against the `v64.0.0` product truth.

## Correct Future Reasoning

Every value movement should answer:

1. What value state is debited?
2. What value state is credited?
3. What funded the Reserve portion?
4. What remains Settlement?
5. Does a certificate lock capacity or complete a funded sale?
6. Can the user understand the next valid action from the surface?

If those answers are explicit, the change likely belongs in the Receiz economy.

If those answers are ambiguous, the change is not ready.

## Release Decision

`v64.0.0` should be treated as launch-test ready when:

- the invariant docs are present
- value-state tests pass
- Market and Wallet show real Reserve/Settlement math
- certificate buyer/seller flows preserve funded causality
- stale browser/PWA checks pass
- real users can use the loop without needing the founder to explain the economics

This release should not add more economic primitives before launch testing.

The correct next phase is:

1. Use the product with real trusted users.
2. Observe where users hesitate.
3. Patch only the edge that failed.
4. Preserve the core loop.

## Practical Rule For Future Work

Ship only changes that make the next valid action feel obvious while preserving the lane truth already visible on the screen.
