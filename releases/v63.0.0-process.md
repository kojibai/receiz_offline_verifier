# Receiz v63.0.0 Official Release Process

Release date: April 19, 2026

## Goal

Turn the current good runtime into an official release without pretending the whole evolution was linear or obviously correct.

The `v63.0.0` cut is based on user experience facts:

- `v55.1.0` had the strongest earlier continuity discipline across more flows than just the homepage.
- `v58.0.0` was the cleanest historical release before the later homepage/profile reload-bug family appeared.
- `v60.2.0` was the practical calm baseline after that homepage/profile bug family was corrected.
- the current runtime at `62.0.2` added meaningful fixes `v60.2.0` did not yet have, especially the restored instant-feeling hero path and the production stale-PWA navigation hardening.

`v63.0.0` makes that current runtime official and governs it more broadly.

## Comparison Windows Used

This release cut was reviewed across four explicit comparison windows:

1. `v55.1.0`
   - Used as the continuity constitution reference.
   - It was unusually strict about composer publish continuity, route-local feedback, search/profile parity, and same-surface market continuation.
2. `v58.0.0`
   - Used as the clean historical stability reference.
   - This is the last release before the homepage/profile second-load bug family entered the repo.
3. `v58.0.0..v60.2.0`
   - Used to isolate what changed after the clean baseline and which fixes were needed to get back to a calm homepage/profile experience.
4. `v60.2.0..62.0.2`
   - Used to identify which later additions were meaningful to users and which were extra complexity.

## Facts Observed From User Experience

### 1. `v55.1.0` governed more continuity surfaces than later releases

The strongest thing about `v55.1.0` was not that its homepage was better than current `HEAD`.
It was that it explicitly locked more user journeys:

- profile truth continuity
- composer publish continuity
- route-local feedback
- same-surface market continuation
- sell-lane realism
- live-player fit

That release behaved more like a continuity constitution than a single-surface fix.

### 2. `v58.0.0` is the clean historical release before the later homepage/profile bug family

The visible homepage/profile second-load and remount issue was introduced after `v58.0.0`.
That means `v58.0.0` remains the least-caveated earlier release from a pure stability perspective.

This release record does not rewrite that. It uses it.

### 3. The post-`v58.0.0` speed-wrapper era created real churn

Between `v58.0.0` and `v60.2.0`, the repo accumulated:

- homepage/profile speed wrappers
- lazy handoff logic
- hidden preload-driven behavior
- repeated fixes for second reload, remount, and visible second-load symptoms

That is a fact pattern, not an interpretation. The commit history in that band repeatedly names the same symptom class.

### 4. `v60.2.0` was the first corrected calm baseline

By `v60.2.0`, home and profile were back in a state that users experienced as calm again.

That made `v60.2.0` the best practical baseline for rebuilding forward:

- explicit activation stayed in control
- the visible shell felt settled again
- the second-load bug family was no longer the active experience

### 5. Later work solved different problems, not the same one

After `v60.2.0`, the important later changes were not mostly about the original homepage remount bug.
They were different bug families and experience gaps:

- stale installed-browser / stale-PWA shell boot
- hero video open responsiveness
- hero back-chart parity with the live pill

Those later fixes mattered, but they were not a license to bring back wider wrapper churn.

### 6. The first `v63.0.0` candidate exposed a real offline-worker regression

During the official release cut, offline root was rechecked instead of assumed.

That check found a factual service-worker regression:

- newer navigation caching logic was writing every visited navigation document into the canonical cached `/` shell slot
- the same worker also let unrelated routes fall back to that canonical shell slot
- offline `/` could therefore reopen from the wrong previously visited document instead of the real root shell
- the visible symptom was the generic interruption boundary while offline

This was not guessed from theory. It came from comparing the current worker to the older working releases and reading the actual navigation-cache write path.

The release decision was therefore explicit:

- do not ship `v63.0.0` until offline root is corrected
- keep the online stale-PWA hardening
- restore a canonical offline-root rule: exact-route offline documents stay exact-route only, and `/` stays `/`

## What Was Kept For `v63.0.0`

### 1. The current runtime, not the earlier package label

The code on `HEAD` was already at commit `62.0.2`, but the governed version surfaces still said `62.0.0`.

`v63.0.0` fixes that mismatch by making the current runtime an official release instead of leaving it as an in-between state.

### 2. The current good homepage/profile/hero/PWA behaviors

`v63.0.0` keeps:

- explicit twin activation only
- direct profile-route truth
- explicit-open deferred hero path
- active hero media prewarm for fast reopen
- hero back-chart parity from the same live market truth as the pill
- production app-shell navigation hardening so online installed users do not boot stale cached shell first

### 3. The broader `v55.1.0` continuity constitution

This release also re-promotes the older but stronger continuity lessons into the active release boundary:

- composer publish continuity
- route-local feedback
- bottom-bar navigation continuity
- same-surface market continuation

The point is simple: the homepage can feel good and the product can still regress elsewhere. `v63.0.0` does not allow that blind spot.

## What Was Not Changed For This Release

### 1. No new runtime rewrite

This official release does not change the runtime direction again.
It does not re-open the old debate between `v58.0.0`, `v60.2.0`, and the current head by rewriting the app.

It governs the current winning runtime more clearly.

### 2. No forced service-worker swap behavior

The production stale-PWA fix stays network-first for online navigations, but this release still does not add forced `skipWaiting` or visible mid-session reload semantics.

That tradeoff remains deliberate:

- keep offline behavior
- prevent stale online shell boot
- do not add a forced visible swap

### 3. No claim that newer history was always better

This release does not claim:

- that `v58.0.0` was wrong
- that every post-`v58.0.0` change added value
- that the later wrapper-heavy era was justified because some of it was eventually corrected

The factual record is:

- `v58.0.0` was the cleaner earlier release
- later work introduced real churn
- some later fixes were still worth keeping
- `v63.0.0` is the governed current release built from those facts

## What Worked

- Treating `v55.1.0` as a continuity constitution instead of just an old release
- Treating `v58.0.0` as the clean historical stability baseline
- Treating `v60.2.0` as the practical calm corrected baseline
- Keeping the current explicit-open hero path and hot-reopen behavior
- Keeping hero chart parity with the same live market truth as the pill
- Keeping production app-shell navigation hardening with offline fallback preserved
- Correcting the canonical offline-shell regression before tagging the release
- Moving wallet and market language to value states instead of making wire or a single balance number carry the whole economy story
- Making direct market sells reject when backed liquidity is unavailable instead of crediting wallet value from an unfunded mark
- Letting public certificate links become a real funded realization path: buyer settled balance moves to seller settled balance, and the buyer receives the shares
- Turning continuity lessons into active release gates instead of leaving them as oral history

## What Did Not Work

- Broad speed-wrapper churn that made the product feel more engineered
- Treating homepage calmness as sufficient while other continuity surfaces drifted
- Leaving the repo on a runtime that was effectively `62.0.2` while governed version surfaces still said `62.0.0`
- Treating historical release quality as if it could be judged only by newness instead of by what users actually felt
- Assuming offline still worked because online stale-PWA hardening looked correct; the worker had to be re-checked against actual offline root behavior
- Letting one wallet number imply too much. The launch product now separates settled balance from live portfolio value and unrealized P/L so users feel the value is real without Receiz creating unfunded cash obligations.
- Trying to fold creator-owned asset withdrawal into this release would have been premature. That model needs issuance caps and creator reserve accounting so a creator cannot repeatedly withdraw the same live asset mark into new certificates.

## Release Boundary Chosen For `v63.0.0`

The official `v63.0.0` release boundary is:

- current runtime behavior from `62.0.2`
- official version surfaces at `v63.0.0`
- current homepage/profile/hero/PWA behavior kept
- canonical offline-root shell behavior corrected and frozen
- value-state framing made explicit across wallet, market, and certificate surfaces
- direct market sells capped to backed same-asset liquidity
- public certificate links able to realize value only through a funded buyer using settled wallet balance
- broader continuity gates pulled forward from `v55.1.0`
- factual historical record kept in release docs so this cut can be understood later without guesswork

## Practical Rule For Future Work

If a future change makes the homepage benchmark better but weakens composer publish continuity, route-local feedback, same-surface trade continuation, or installed-browser truthfulness, the change is not an improvement.

For Receiz, user-perceived continuity is not a homepage-only concept. `v63.0.0` is the release where the repository records that clearly.
