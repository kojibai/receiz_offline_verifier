# Receiz v95.0.0 Regression Lessons

Release date: 2026-06-27

## What Regressed

No product runtime behavior regressed in this release. The regression was language and public contract drift:

- SDK `ReceizProofBundle` typing was softer than the canonical internal proof bundle.
- Some documentation described Kai with conditional phrasing.
- The public report needed to state Kai/Klok as the state machine, not merely a coordinate.

## Why It Regressed

The weaker public SDK type was treated like a developer-friendly projection layer. That was wrong. The SDK is convenience beneath Receiz primitives; it is not a weaker ontology.

## Locked Rules

- Kai Klok is the proof object state machine.
- `kaiPulseEternal` / `kai_upulse` is the unit.
- Complete Receiz proof objects carry Kai.
- Offline proof requires a living coordinate that works without consensus.
- SDK types must mirror canonical internal proof contracts.
- Historical compatibility may be explicitly marked as legacy, but a canonical proof object cannot be loose.

## Contract Gates

- `pnpm release:lock 95.0.0`
- SDK tests must validate canonical proof bundle fixtures.
- Release lock must pass for `95.0.0`.
- The generated PDF must render after the Kai/Klok wording change.
- Future docs must not reintroduce "when present" wording for complete proof objects carrying Kai.

## Release-Lock Requirement

`pnpm release:lock 95.0.0` must require this document for `v95.0.0` and later release cuts. A release that omits regression lessons is incomplete because SDK proof bundle softness, Kai conditional phrasing, public report wording drift, and canonical proof object completeness would not be explicit release truth.
