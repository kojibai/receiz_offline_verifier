# Receiz v113.0.0 Release Process

## Goal

The implementation followed the frozen design in `docs/superpowers/specs/2026-07-19-v113-production-global-reconciliation-design.md` and its file plan. Production Global Reconciliation is scoped to the explicitly named `receiz.com/global/v1` coordination domain and does not claim universal consensus.

## What Worked

The work froze release identity and limits first, then implemented exact carried-history custody, key-state evidence, provider-neutral SDK orchestration, private atomic persistence, Receiz.com routes, MCP parity, AI distribution, executable packed documentation, ownership/two-device/first-paint conformance, and release gates.

## What Did Not Work

The first umbrella run exposed incomplete release-record linkage and one TypeScript library-target incompatibility. Both were corrected without weakening the authority model or changing visible product behavior.

## Practical Rule For Future Work

Tests were run at each boundary before the umbrella release freeze. The migration remains unapplied. Package publishing, tagging, pushing, and production deployment are separate explicit actions.

Release records: [product truth](v113.0.0-product-truth.md), [regression lessons](v113.0.0-regression-lessons.md), [performance findings](v113.0.0-performance-findings.md), [invariant register](v113.0.0-invariant-register.md), and [commit history](v113.0.0-commit-history.md).

## Release-Lock Requirement

`pnpm release:lock 113.0.0` must require the complete v113 record set. Omitting a record would leave named-domain meaning, exact-history custody, migration separation, retry/indeterminate outcomes, or the deployment boundary outside explicit release truth.
