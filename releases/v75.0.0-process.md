# Receiz v75.0.0

## Goal

Ship the full post-`v74.0.0` local-proof, file-restore, market-continuity, Explore, and upgrade-parity work as an official `v75.0.0` release while preserving the v70 public alpha law and the v74 Receiz Key / identity-artwork baseline.

## What Worked

- Measuring the release from the real `v74.0.0` tag made the release scope concrete: the pre-bump candidate contains 34 commits, 74 changed files, 5,135 insertions, and 437 deletions after that tag.
- Receiz ID login matched product law when local device proof projected first and server continuation moved into background publication instead of becoming permission for identity to exist.
- Restored-account parity became stable when Identity Record, Identity Seal, and Receiz Key restore admitted the recovered account locally before import, bind, wallet, or cookie refresh jobs could finish.
- The local account-binding register made stale server/session echoes tractable because account authority can be compared by source and actor instead of treating the latest transport response as truth.
- Old-browser support stayed aligned when PBI/passkey, email, and identity-file recovery moved to a dedicated backup route while the primary sign-in surface remained Receiz ID pass-through.
- Market continuity worked when wallet state, owned positions, prediction history, share-trade history, and sold-position evidence were classified as same-actor local truth rather than disposable route payload.
- Explore became less fragile when it reused held route state, bounded swipe rendering, kept preview fallbacks deterministic, and packaged read / play / save / share / proof controls inside the card.
- Plans and Upgrade parity was easier to lock when immediate payment behavior and fallback access were covered by a shared contract.

## What Did Not Work

- Waiting for server Receiz ID continuation or treating remote reconfirmation as proof permission caused visible identity churn and contradicted the local-proof law.
- File restore remained incomplete while import, bind, wallet sync, or stale cookies could keep the UI half-logged-in after proof acceptance.
- Reusing old sign-in fallbacks on the primary route would have weakened the one-press Receiz ID primitive; the fallback needed its own sanitized route.
- Market snapshots regressed whenever lagging or anonymous payloads could overwrite richer same-actor signed-in wallet truth.
- Sold-position continuity could not rely on live market payloads alone because older active-position data can arrive after a local sell.
- Explore could feel over-engineered when mobile scroll, rich discovery loading, or card utilities changed the settled first frame instead of staying contained.

## Practical Rule For Future Work

Classify account, market, and discovery data before it reaches the visible shell. Receiz ID proof, file restore proof, local account bindings, wallet state, sold-position evidence, deterministic previews, and canonical profile truth are authority. Server sessions, import jobs, global sync, payment context, discovery refreshes, and owner controls may append or enrich that authority, but they must not hold first paint hostage, downgrade known truth, or hide release version drift.
