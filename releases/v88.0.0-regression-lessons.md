# Receiz v88.0.0 Regression Lessons

Release date: June 3, 2026

This document is part of the release lock. It records what regressed during the post-v87 arc, why it regressed, and the rules that must prevent the same mistakes.

## What Regressed

### Sports Selected-Game Field Drift

The Sports field drifted away from the exact `87.0.3` field behavior while locked-entry and hydration fixes were being added.

The visible regression was specific:

- pitch-by-pitch was not always the real detailed pitch-by-pitch in the center field recap
- base state text was allowed to appear where pitch-by-pitch belongs
- last play appeared in the center or score area instead of staying in the bottom last-play ticker
- fallback cards appeared when the real live batter/pitcher cards should have been projected
- switching games could make a selected field drop from detailed truth to fallback truth

### Locked Single-Game First Paint

Single-game locked entries did not always show locked immediately after cache clear, app close, and reopen.

The data was not gone. The break was projection order: the UI could read a stale or incomplete selected-game state before stronger durable proof/local entry truth was restored.

### Individual Player Score Hydration

Individual player rows and live cockpit player scores could understate or hide proof-backed scores while append hydration was still catching up.

Main Event totals looked correct sooner because that path already had stronger score projection. Single-game and card-row projections needed their own protection.

### Reward State Over Active Live State

Started games could show reward/final language such as "Reward ready" in places where live game state should remain primary.

The regression came from allowing finalized/reward state to share a visible path with active game state instead of keeping active live state above unresolved reward recovery.

### Service-Worker And Runtime Version Drift

Deterministic session/profile projection keys depended too much on a manually supplied `NEXT_PUBLIC_SW_VERSION`.

When the environment omitted or lagged that variable, package release truth and runtime projection truth could diverge.

### Durable Session Clear Gap

The migration from older runtime-scoped durable session keys was added, but the clear path initially removed only the current key and the unscoped legacy key.

That left stale `receiz.session.bootstrap.durable.v1.*` entries able to survive logout/clear and re-migrate later.

### DB Timeout Hardening Gaps

The first DB hardening pass covered newer advisor-flagged owner/public-live/push paths, but older chat RLS helper functions and policies still carried direct auth reads.

The logs showed `canceling statement due to statement timeout`. The pasted logs did not include SQL text, so the repo fix had to be scoped to concrete known migration gaps, not guessed production queries.

## Why It Regressed

### Stronger Truth Was Not Always Chosen First

Some paths allowed newer or later-arriving transport state to replace stronger selected-game, proof-backed, or durable local truth.

Correct order:

```txt
selected detailed field truth
durable local/proof entry truth
verified appends
authenticated snapshot
server/DB/session refresh
temporary UI state
```

Wrong order:

```txt
latest partial feed refresh
generic selected scoreboard state
fallback UI text
then maybe detailed truth later
```

### Field Roles Were Treated As Flexible

The Sports field roles are not flexible:

```txt
score plate: inning/status, score, base state/countdown
center recap: real pitch-by-pitch
bottom ticker: last play
side cards: real batter and pitcher when known
```

When those roles were treated as interchangeable text slots, one fix could move data into the wrong visible location.

### Fixes Were Too Local To The Symptom

Some patches fixed the immediate symptom but did not protect the upstream contract.

Examples:

- showing locked entries without locking selected-game field preservation
- fixing Main Event total without proving single-game player rows
- changing visible labels without proving the actual `playByPlay` data source
- hardening some RLS policies without searching older helper/policy paths

### Fallbacks Were Used Where Real Truth Was Needed

Fallbacks are allowed only when stronger truth is genuinely absent. They are not a repair strategy when the app already has detailed live field, locked entry, player card, or proof append truth.

### Production State Was Confused With Repo Truth

The repo can prove migrations, contracts, and code paths. It cannot prove a live DB is fixed until the migration is applied to the exact production Supabase project and production evidence confirms it.

## Locked Rules

### Sports Field Rules

- The selected game owns the field.
- Incoming public refreshes may enrich the selected field but may not erase detailed selected-game field truth.
- Score plate subtext is base state or countdown.
- Pitch-by-pitch belongs only in the center field recap.
- Bottom ticker is last play.
- Real live batter and pitcher cards must remain when known.
- Fallback field copy and fallback cards are allowed only when detailed field/player truth is genuinely absent.

### Sports Entry And Score Rules

- Stored joined game entries are durable local/proof truth for first paint.
- Locked single-game entries must show locked immediately when known.
- Player card rows must use proof-backed card scores while appends hydrate.
- Partial append windows may not zero or understate stronger proof scores.
- Active live game state outranks unresolved reward recovery for visible live gameplay.
- Finalized unresolved rewards stay claimable until opened.
- Opened rewards close stale recovery.

### Runtime And Session Rules

- Package version is release truth.
- `NEXT_PUBLIC_SW_VERSION` must be injected from package version when the environment omits it.
- Service worker version, runtime coherence, release labels, and projection keys move together.
- Explicit clear/logout must remove every durable session key shape that can re-migrate.
- Runtime-scoped legacy session keys are not allowed to survive clear.

### DB Hardening Rules

- Search older helper functions and policies, not only latest migrations.
- RLS auth reads in hardening migrations must be init-plan-safe.
- New hardening migrations must not reintroduce direct `auth.uid()` calls.
- Add indexes for hot FK/member lookup paths when migrations define them.
- Do not run broad Supabase Advisor/linter scans against production as release proof unless explicitly scoped and approved.
- Do not claim live DB fixed until the exact production project has the migration applied.

### Release Process Rules

- Regression lessons are a release artifact.
- The release note must reference the regression lessons.
- The checklist must reference the regression lessons.
- `pnpm release:lock` must fail when the lessons document is missing.
- Convenience routes such as `/baseball` must reuse the canonical proof surface directly and keep the same route chrome; they must not fork logic or introduce a second source of Sports truth.

## Contract Gates

The following gates exist to keep these mistakes from returning:

```bash
pnpm release:lock
pnpm test:game-release-contracts
pnpm test:page-first-paint
npx tsx --test app/game/__tests__/sportsArenaConsoleModel.test.ts app/game/__tests__/sportsArenaFirstPaintBootstrap.behavior.test.ts app/game/__tests__/sportsArenaRouteContract.test.ts
npx tsx --test app/lib/__tests__/supabaseRlsInitPlanMigration.test.ts app/lib/__tests__/supabaseHighValueFkIndexesMigration.test.ts scripts/test_supabase_advisor_guard.ts
git diff --check
```

The Sports route contract must continue to prove:

- `/baseball` reuses `/game/sports/baseball` page logic and route chrome without redirecting or forking
- `playByPlay` comes from `currentPlayPitchCommentary`
- the source includes batter, pitcher, count, result line, inning, score line, and base state
- `playByPlay` is not replaced by event line, live pulse, or count

The Sports model/first-paint contracts must continue to prove:

- incoming live refresh cannot erase detailed selected-game fields
- locked entry scoreboard projection preserves detailed live field
- single-game proof projection restores locked entries when selected state is stale
- individual card rows use proof-backed scores while appends hydrate

The DB contracts must continue to prove:

- hardening migrations use `(select auth.uid())` where auth identity is read
- new hardening migrations do not reintroduce direct `auth.uid()`
- broad Supabase Advisor/linter automation remains blocked

If visual evidence contradicts any of those contracts, the product is failing and the contract must be strengthened without weakening the product.
