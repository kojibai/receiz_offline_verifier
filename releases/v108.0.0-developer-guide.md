# Receiz v108.0.0 Developer Guide

## Authority boundary

A sealed Receiz asset is a proof object, and proof-object continuity remains inside that enclosing artifact. The dependency direction is:

```txt
Receiz law and invariants
-> sealed proof object and verified carried continuity
-> admitted same-account binding or authenticated Receiz owner
-> Receiz.com native profile and Record -> Seal operations
-> @receiz/sdk
-> @receiz/mcp-server
-> AI skills
```

The SDK, MCP server, AI skills, server, database, and session are adapters or projections. They do not become proof authority.

## Update the authenticated profile

Receiz.com UI and developer integrations call the same neutral profile command. The server derives the actor from the admitted session or scoped OIDC bearer. The caller supplies no UID, identity key, or separate operation authority.

```ts
import { createReceizClient } from "@receiz/sdk";

const receiz = createReceizClient({
  accessToken: process.env.RECEIZ_ACCESS_TOKEN,
});

const result = await receiz.profile.update({
  username: "new_name",
  displayName: "New Name",
  bio: "Proof remains authority.",
});

console.log(result.accountUid, result.profilePath);
```

The operation validates the same profile patch, coordinates username uniqueness at commit, preserves the admitted UID, and returns the refreshed profile/public projection. `receiz.identity.updateProfile(...)` is a current alias with the same neutral input and result.

## Create, verify, open, and download a proof object

`assets.createProofObject()` returns only after native Record, Seal, integrity verification, Signature V4, owner/record/claim/path binding, and payload binding all pass. Its creation request and standard transport parameters do not create a second proof, ownership, or continuity authority.

```ts
const sealed = await receiz.assets.createProofObject(createInput, createOptions);
/*
createInput = {
  assetType: "document",
  payload: {
    bytes: new Uint8Array(await sourceFile.arrayBuffer()),
    mimeType: sourceFile.type,
  },
};
createOptions identifies the output filename and standard transport mechanics.
*/

await receiz.artifacts.download(sealed); // exact complete artifact bytes

const opened = await receiz.artifacts.verifyAndOpen(sealed.artifact);
const payload = opened.verifiedPayload; // exposed only after enclosing verification
```

Payload bytes are a verified projection of the proof object. They are never a substitute for the sealed asset.

## Claim bearer ownership lawfully

Bearer ownership accepts the complete SDK-verified proof object. The server verifies the enclosing artifact, verifies the portable document, requires carried bearer custody, derives the prior ownership reference from that verified document, binds the authenticated Receiz owner, appends ownership, and performs native Record -> Seal again.

The object is portable across lawful platforms; it is not captive to its creator. A platform may verify and append authenticated ownership or history only by preserving the same immutable object identity, payload, provenance root, prior history, and unknown application namespaces and returning another complete verified proof object.

```ts
const opened = await receiz.artifacts.verifyAndOpen(bearerFile);

const claimed = await receiz.ownership.claimBearerAsset({
  artifact: opened.sealedArtifact,
});

await receiz.artifacts.download(claimed);
```

The operation accepts only the complete verified artifact and the authenticated Receiz owner binding. The result is a new complete verified artifact; no database table, storage bucket, or row authorizes the ownership change.

## MCP

The default MCP registry exposes current profile and bearer proof-object plan/execute tools plus the complete artifact custody contract. Planning is local confirmation mechanics only. Execute calls the current SDK methods:

- profile plan accepts profile intent only;
- profile execute returns the same-UID profile result;
- bearer plan accepts complete artifact bytes and their carrier metadata only;
- bearer execute verifies and opens those exact bytes, then returns the new complete sealed artifact.

The artifact contract is exact:

- `receiz_artifact_record_seal_plan`
- `receiz_artifact_record_seal_execute`
- `receiz_artifact_verify`
- `receiz_artifact_extract_verified`
- `receiz_artifact_round_trip_check`
- `receiz_artifact_explain`

Verification and extraction use the same SDK artifact verifier. Extraction never runs before enclosing verification, round-trip checks compare the exact complete bytes, and explanation is evidence only.

MCP never asks a weaker registry which admitted identity or carried owner is real.

## Profile media

Publish media through its proof operation, then reference the verified URL in `profile.update()`. Media verification remains its own proof boundary and cannot select or replace the profile actor.
