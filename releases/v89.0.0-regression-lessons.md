# Receiz v89.0.0 Regression Lessons

Release date: June 13, 2026

This document is part of the release lock. It records what regressed during the post-v88 arc, why it regressed, and the rules that must prevent the same mistake.

## What Regressed

### Foreground Anchor Publication Was Admitted To Generic Outbox

`/api/receiz/anchor` was allowed into the generic local signed-action/outbox boundary.

That was the regression. The anchor server route was not the broken authority. Production route evidence showed the route could return readable JSON. The composer still surfaced:

```txt
The anchor route returned an unreadable response. Refresh and retry.
Could not anchor this memory payload.
```

The failure was the weaker generic outbox path getting access to the foreground anchor response boundary before the composer could read the route-owned anchor result.

```

Fixed by:

Exclude anchor route from local signed outbox
```

### Profile Original Global Completion Was Too Easy To Underclaim

The owner could see a post locally or the DB could contain a row, but that did not prove global completion.

The correct completion boundary is:

```txt
foreground anchor publication
canonical profile snapshot append/refresh
public profile market append source append/refresh
visible local route snapshot update
live-player object shape compatibility
```

If any part is missing, the post can disappear after refresh, PWA close/reopen, profile navigation, showcase reload, or market cold paint.

### Response Readability Was Treated As The Root

Adding better error reading did not fix the primitive violation. It only made the failure easier to describe.

The root was not "the server returned unreadable JSON" as a product cause. The root was that a weaker local outbox/replay layer could touch a foreground proof publication response that it did not own.

### Service Worker Was A Plausible Mechanic But Not The Root Evidence

The service worker can affect installed PWA fetch behavior and cache activation, so it remains a hydration/runtime risk. In this regression, the repo evidence identified the local signed-action outbox boundary as the actual root.

The locked conclusion is:

```txt
Do not blame SW/cache when the foreground proof route has been admitted to a weaker local outbox path.
Do not ignore SW/cache as a runtime risk after versioned release changes.
```

### Brand-New Sports Vault Accounts Had No Human Public Vault Route

Saved profile usernames produced `/vault/{sport}/{username}` links. Brand-new accounts without a saved profile username passed `null` through the Sports Vault URL helper, so the helper returned no public owner route and callers fell back to `/game/sports/{sport}/vault/{vaultCardId}`.

The missing boundary was default profile username projection. A public vault proof URL must not require a manual profile edit when the session already carries the identity material needed to project the repo default username.

### Sports Single-Game Entry Trusted Stale Client Lineup Payloads Too Much

COL / ATH on June 13, 2026 exposed a stale single-game entry boundary.

Production evidence showed:

```txt
POST /api/game/sports/baseball/entries -> 409
sourceDate: 2026-06-13
gamePk: 824995
matchup: COL / ATH
user: bjklock
```

The user owned verified visible COL and ATH Sports Vault proof cards, and MLB schedule truth showed the game was scheduled, not final. The failure was not the game, not the date, and not missing owned cards.

The root was that the client could submit stale `selectedCardIds` from a previous matchup while the selected game had already changed. The first repair reconciled the client at submit time, but an already-open runtime could still post stale card ids. The server route then rejected the stale payload with `sports_entry_matchup_cards_required`, even though the stronger admission facts were valid:

```txt
verified MLB game identity + owned visible matchup proof cards
```

The route had treated the submitted lineup payload as the whole admission boundary. That was weaker than the implemented primitive. For single-game entries, the server must compose from the current verified game and the owner's verified Sports Vault proof cards before rejecting a stale client payload.

The verified fix created entry:

```txt
entry: 9ab6ddd3-5da3-4d30-b173-7961249156d8
gamePk: 824995
arenaKey: mlb:2026-06-13:824995
lineupScope: game
lineupSize: 5
enteredAt: 2026-06-13 17:18:09Z
```

## Why It Regressed

### A Weaker Layer Was Allowed Above A Stronger Boundary

Correct order:

```txt
Receiz law
sealed/deterministic proof publication
foreground anchor response
profile original projection
verified local/route snapshots
server append/global sync
generic signed-action outbox
service worker/runtime cache
UI render state
```

Wrong order:

```txt
generic signed-action outbox
foreground anchor response
then maybe proof publication
```

The wrong order caused the composer to see an unreadable anchor response even when the server route could answer.

### Generic `/api/receiz` Reasoning Was Too Broad

The local signed-action classifier got too broad for a namespace that contains both foreground proof publication routes and background/local sync candidates.

The namespace is not the primitive. The route's authority boundary is the primitive.

### The June 6 Working Shape Was Not Treated As The Contract Early Enough

Existing showcase posts already established the object shape and navigation behavior. New posts needed to be admitted into that same shape and projection path, not patched with local-only behavior.

### Server Append Was Confused With Source Of Truth

Server append/global sync is required for public projection. It is not above foreground proof publication or sealed/local proof truth.

When the server is unavailable or delayed, local proof can still exist. When global publication is required, the write path must project that proof into canonical profile and market surfaces. Neither layer replaces the other.

### Client Payload Was Confused With Settlement Admission Truth

Single-game Sports entry admission is a settlement primitive. The client payload is experience state and transport. It is not stronger than:

```txt
verified schedule game identity
owned Sports Vault proof card identity
server-side card claim verification
matchup team normalization
```

A stale client-selected lineup may be useful input, but it cannot be the only authority that decides whether a valid game can be entered. If the server can verify the selected game and can verify enough owned visible matchup proof cards, it must compose the lockable lineup from those stronger facts instead of making the user reopen or refresh the app.

## Locked Rules

### Anchor Rules

- `/api/receiz/anchor` is foreground proof publication.
- `/api/receiz/anchor` is excluded from local signed-action outbox admission.
- The anchor route owns its readable response contract.
- Generic outbox/replay/sync code may not inspect, wrap, consume, or replay the foreground anchor response.
- No broad `/api/receiz` classifier may re-admit anchor publication.
- Better error text is not a fix if the weaker layer remains in the foreground route path.

### Profile Publication Rules

- A profile original is a proof object.
- A profile post is not globally complete because a DB row exists.
- A profile post is not globally complete because the owner can see local truth.
- The write path must append or refresh canonical profile snapshot truth.
- The write path must append or refresh public profile market append source truth.
- The write path must update local route snapshots for surfaces it makes visible.
- New profile originals must carry the same showcase/live-player object shape as existing profile originals.
- Refresh, app close/reopen, and public profile navigation must not drop a published object.

### Truth Hierarchy Rules

- Foreground proof publication outranks generic outbox mechanics.
- Sealed proof/artifact truth outranks server append.
- Verified local truth outranks UI cache.
- Server append/global sync is required projection, not source-of-truth replacement.
- Service worker/runtime cache may warm and activate, not redefine the proof object.

### Release Rules

- Regression lessons are a release artifact.
- Full commit history since the previous major release is a release artifact for `v89.0.0`.
- The release note must reference regression lessons and commit history.
- The checklist must reference regression lessons and commit history.
- `pnpm release:lock 89.0.0` must fail if required release documents or version surfaces are missing.

### Sports Vault Public URL Rules

- Saved profile username wins.
- If no saved username exists, the Sports Vault public route uses the default profile username projection from session email and user id.
- Public vault URL copy, live showcase publish, saved vault image, and public owner vault route must use the same effective public username.
- Canonical fight, baseball, and soccer Sports routes must all provide first-paint session/default-username bootstrap.

### Sports Single-Game Entry Rules

- Single-game Sports entry is a settlement primitive.
- The client lineup payload is transport/experience state, not the admission root.
- The server route owns final admission for `/api/game/sports/baseball/entries`.
- Admission must verify the MLB game, reject final games, verify owned Sports Vault card claims, and normalize matchup team codes.
- If a single-game payload contains stale or off-matchup card ids, the server must keep any valid owned matchup cards and fill from the owner’s verified visible matchup Sports Vault proof cards before rejecting the entry.
- Non-owned cards, hidden cards, unverifiable claims, and cards outside the verified matchup must still be rejected.
- Do not fix a stale single-game lock by inserting DB rows manually. The entry must be created through the settlement route so the lineup hash, entry hash, proof JSON, and card career memory remain coherent.
- Do not blame date handling, service worker cache, or game finality until production evidence checks the exact 409 guard and the owner’s visible matchup proof-card rows.

## Contract Gates

The following gates exist to keep this mistake from returning:

```bash
pnpm release:lock 89.0.0
npx tsx --test app/lib/offline/__tests__/localSignedActionOutboxContract.test.ts app/lib/profile/__tests__/composerPublishContinuityContract.test.ts app/lib/profile/__tests__/profilePublishGlobalSyncContract.test.ts app/lib/profile/__tests__/profileOriginalSignedUploadContract.test.ts app/components/pwa/__tests__/serviceWorkerNavigationContract.test.ts
pnpm test:page-first-paint
git diff --check
```

The outbox contract must continue to prove:

- `/api/receiz/anchor` is excluded before generic signed-action admission
- no broad `/api/receiz` proof classifier exists
- foreground anchor publication cannot be treated as a background replay/sync action

The profile contracts must continue to prove:

- composer publish continuity preserves the profile original object shape
- profile publish global sync refreshes canonical profile and market append sources
- profile original signed upload does not replace proof truth with weaker upload/session state
- live-player navigation keeps the same route/proof fields used by existing showcase posts

The PWA/navigation contract must continue to prove:

- service-worker navigation behavior does not intercept or redefine proof routes into stale shells
- runtime version coherence moves with release truth

The Sports single-game entry contract must continue to prove:

- submit-time client reconciliation refills stale single-game lineups from the current matchup
- the server composes stale submitted baseball card payloads from owned verified matchup proof cards
- canonical MLB team aliases such as `OAK -> ATH` and `AZ -> ARI` are normalized before admission
- single-game entries may use one to five owned matchup proof cards, while tournament/Main Event entry remains exactly five

If visual evidence contradicts any of those contracts, the product is failing and the contract must be strengthened without weakening the product.
