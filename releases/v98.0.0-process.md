# Receiz v98.0.0 Process

## Goal

Lock every committed change after `v97.0.0` into `v98.0.0` without downgrading proof objects, deterministic first paint, durable proof memory, merchant proof authority, delegated agent-token boundaries, public proof surfaces, ledger settlement truth, sports vault/arena truth, profile/market projection law, or Pitch Command proof archives.

`@receiz/sdk` remains a convenience rail beneath proof objects, and durable proof memory remains first admission only, then append forever.

## What Worked

- Existing Receiz primitives already carried the needed boundaries, so the release could add SDK, MCP, OIDC, webhook, graph, ledger, transaction, sports, and archive mechanics beneath those primitives.
- The public app-state and public-store rails gave developers recovery projections without making DB rows the source of truth.
- Identity Seal / Receiz Key proof authority could publish merchant-owned public-store state without delegated bearer-token authority.
- MCP could wrap SDK/API rails while explicitly reporting that it is not authority.
- Delegated OIDC agent tokens gave MCP and CI a scoped operation rail without replacing Receiz identity proof.
- Webhook signing, endpoint management, and test events could be added as delivery mechanics beneath Connect/OIDC scopes.
- The proof graph could become visually rich while staying a public proof surface with live Kai pulse and real proof/economy/network data.
- Ledger and transaction routes could project wallet, transfer, settlement, and proof-bundle truth without demoting settlement primitives into generic balances.
- Sports arena performance work reduced background pressure through bounded workers, deferred surfaces, dedupe, and image helpers while preserving proof/vault/settlement first paint.
- Pitch Command archives could be caught up as proof archive material with strict manifest/projection contracts.

## What Did Not Work

- Some intermediate release documents described correct facts but used minor-version scope; `v98.0.0` needed one full post-`v97.0.0` lock.
- Public proof graph visual polish touched many viewport-sensitive files, so static contracts are necessary but not enough for production promotion.
- Rich chart and ledger work created real computation pressure; the correct fix was downsampling and bounded projection beneath the primitive, not removing the visible proof surface.
- Developer rails introduced several lower-authority systems at once: SDK, MCP, OIDC tokens, webhooks, public-store projections, and service worker/runtime labels. The release lock has to state explicitly that none of those outrank proof truth.
- Pitch Command day-proof archive catch-up created large generated evidence changes; the review must classify them as proof archive material, not ordinary app payload churn.

## Practical Rule For Future Work

When a new developer, graph, ledger, sports, market, profile, archive, or webhook rail is added, identify the implemented Receiz primitive first and keep the new rail strictly beneath it. If the rail makes known proof, identity, ownership, settlement, profile, or public proof truth wait for a lower-authority system, the rail is wrong.

## Process Law

1. Baseline the release at `v97.0.0`.
2. Review every commit in `v97.0.0..HEAD`.
3. Group commits by primitive surface: SDK/MCP, OIDC/Connect, webhooks, public proof graph, ledger/transaction/settlement, market/profile first paint, sports vault/arena, Pitch Command archive, AI skills, and release identity.
4. Preserve direct primitive language for implemented proof objects, ownership surfaces, settlement primitives, identity primitives, public proof surfaces, durable local truth, and offline verification.
5. Update release identity to `98.0.0` only after the reviewed scope is clear.
6. Add release docs that satisfy `scripts/check_release_lock.mjs`.
7. Run targeted gates first, then the umbrella release freeze before promotion.
8. Treat visual evidence as product truth. If fresh browser evidence contradicts this lock, the release is blocked.
