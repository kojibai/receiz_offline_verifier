## Release Law

Verification is indivisible from continuity. A verified Receiz proof object proves its exact enclosing bytes and the existing continuity that binds deterministic identity, a real Receiz owner, namespace, and prior head or valid genesis. Integrity alone is not verification.

`@receiz/sdk` remains beneath that artifact truth, and durable proof memory remains **first admission only, then append forever**.

## Frozen Primitives

| Primitive | v102.0.0 implementation truth | Forbidden downgrade |
| --- | --- | --- |
| Verified proof object | Top-level verifier success requires integrity and a verified native or portable continuity carrier. | Calling valid bytes, a signature, an anchor, or a shape-valid manifest verified. |
| Identity/ownership continuity | Ownership resolves from the authenticated user to an existing profile-backed `<username>.receiz.id`; no UUID or arbitrary string fallback exists. | Developer input, session UID, placeholder, or MCP result becoming owner authority. |
| Pre-seal continuity | Owner, profile namespace, payload identity, and prior head/genesis validate before the proof-object sealer is invoked. | Sealing bytes first and creating continuity during admission or restore. |
| Native Record/Seal/Profile continuity | Existing profile composition, Record, Seal, ownership provenance, and card append carriers remain authoritative. | Replacing native continuity with a parallel SDK registry or generic security layer. |
| Offline verification | Downloadable verifiers independently recompute the same integrity-and-continuity verdict without server or database authority. | Treating offline verification as fallback or accepting integrity-only evidence. |
| MCP rail | MCP resolves and inspects; it does not claim verification for lookup or manifest shape validation. | `ok: true` lookup/inspection being described as a verified proof object. |
| Installed AI skills | SDK and MCP install the matching `@receiz/ai-skills@102.0.0` doctrine bundle. | An installed agent learning removed verification shortcuts or drifting from the runtime gate. |

## Frozen Invariants

1. No SDK, API, Connect, Receiz.com, offline, admission, restore, World package, or MCP path may report a verified proof object without continuity.
2. A real authenticated Receiz ID exists before continuity-capable sealing; there is no UID, arbitrary owner, placeholder, or synthetic owner fallback.
3. The continuity namespace is the real owner profile namespace and the prior-head/genesis relationship fails closed on mismatch.
4. The enclosing artifact remains stronger truth; a self-hashable subpayload, DB row, session, or MCP response cannot replace it.
5. Generic document sealing is an integrity operation and cannot accept portable proof-object payloads or return proof-object verification success.
6. Native Receiz Record, Seal, profile composition, ownership provenance, and card continuity stay in place and continue through their existing carriers.
7. SDK, MCP, and installed AI skills remain mechanics beneath identity, ownership, continuity, sealed artifact truth, and verified append history.
