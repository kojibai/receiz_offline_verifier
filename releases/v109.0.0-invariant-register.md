## Locked Invariants

1. Receiz remains a proof-native artifact system. Complete sealed artifact bytes, deterministic proof state, verified local truth, identity proof, ownership proof, settlement proof, and append-only history remain stronger than software or service projections.
2. `ARTIFACT-011` requires SDK verification and opening of complete artifacts without network, session, database, registry, or server state. Its denial code is `LOCAL_ARTIFACT_VERIFICATION_MUST_NOT_REQUIRE_WEAKER_STATE`.
3. `receiz.verification.verifyArtifact(file)` verifies complete `.receizbundle` bytes locally, including carrier integrity, payload digest binding, Signature V4 root/certificate continuity, Groth16 proof, canonical identity/path fields, and applicable portable continuity.
4. `receiz.artifacts.verifyAndOpen(file)` verifies before extraction and returns the complete enclosing artifact separately from the verified payload.
5. Artifact and payload are separate custody domains. Payload bytes are unavailable until the enclosing artifact passes every applicable verification layer.
6. Signature V4 roots, the document-seal Groth16 verification key, and the verification runtime are packaged dependencies, not network-time authority lookups.
7. One-byte mutation, malformed carrier, artifact-binding mismatch, Signature V4 failure, Groth16 failure, and invalid portable continuity fail closed with zero network calls.
8. MCP artifact operations compose the SDK verifier and remain orchestration beneath artifact truth. AI skills require explicit zero-network and local-verifier evidence before a production-ready claim.
