## Frozen Invariants

- Same-proof parity: a proof object available in one Receiz surface must project with equivalent proof completeness in public, economy, logged-in, export, and QR surfaces unless the route law explicitly scopes it.
- ID-first performance: hot proof reads must bound identity rows before hydrating payload JSON.
- Payload authority: `append_json`, `proof_json`, `source_json`, sealed file bytes, and embedded proof payloads remain source truth; indexes, RPCs, and summaries are projection mechanics beneath them.
- First-paint preservation: Sports Economy, Sports card, event proof, Pitch Command, Play Command, Pack Derby, physical activation, and vault surfaces must paint known truth before enrichment.
- Foreground append cadence: active proof viewing may warm hidden work immediately; background leases may suppress duplicate work but may not suppress visible proof updates.
- Value basis separation: deterministic value, seller ask, customer price, internal cost, margin, reserve, and settlement are different primitives and must not be merged in UI or docs.
- Full-history before compression: lineage/history may be summarized visually only after the full proof chain has been resolved or explicitly marked unavailable.
- Route-local projection cannot outrank shared proof projection: card backs, value panels, event popovers, economy rows, and public proof pages must reuse shared proof/card projection helpers where they exist.
- Deployment order matters: app code that calls a new Supabase RPC must deploy after the live function exists and PostgREST can see it.
- Visual evidence is binding: if screenshots or browser state show history, identity, score, value, proof media, or first paint regressing, passing contracts are insufficient.

## Residual Risks

- Production Supabase state must be observed after deployment for RPC visibility, schema-cache freshness, and slow-query disappearance.
- Vercel cold starts can temporarily show partial live enrichment while new serverless instances and PostgREST schema cache settle. This is deploy timing, not proof data loss, but it must be monitored.
- Live sports feeds and media/CDN availability remain external runtime inputs. They can confirm projection, but they do not define proof truth.
- Installed PWAs need the `93.0.0` service-worker update before stale `92.x` shells are guaranteed gone.
- The generated `.next/dev/types/validator.ts` file has previously produced local `TS1128` before source checking; release build must be verified from a clean/generated state.
