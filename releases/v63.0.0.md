# Receiz v63.0.0

Release date: April 19, 2026

## Overview

`v63.0.0` is the official release cut that turns the current `62.0.2` runtime into a governed release and expands its continuity boundary using the strongest lessons from `v55.1.0`, `v58.0.0`, and `v60.2.0`.

This release is not pretending every later change was a win. It records a simpler truth:

- `v55.1.0` had the strongest earlier continuity discipline
- `v58.0.0` was the clean historical release before the later homepage/profile reload-bug family
- `v60.2.0` was the practical calm corrected baseline after that bug family
- the current runtime solved real later problems that `v60.2.0` did not yet solve

`v63.0.0` makes that current runtime official and governs it with a broader continuity standard.

The product-truth freeze for this release is defined in `docs/releases/v63.0.0-product-truth.md`. The process record for how this official cut was chosen is in `docs/releases/v63.0.0-process.md`.

## What Defines v63.0.0

- Settled home and profile twin shells still obey explicit activation only.
- The public profile route still uses the direct strong `ProfileClient` path.
- The homepage hero still opens through the explicit-open deferred path and reopens hot after the first successful warm.
- The back-of-card hero chart now follows the same live market truth as the pill instead of visibly lagging on a weaker chart source.
- Wallet and market value states are now named and wired as product truth: settled balance is the money-like lane, live portfolio value is the held-market lane, and unrealized P/L is not treated as cash.
- Direct market sells now require backed same-asset liquidity from prior funded market buys; unfunded unrealized gains do not become settled wallet value just because the mark moved.
- Share certificate links now support funded buyer settlement: a signed-in buyer can buy available certificate shares with settled wallet balance, the seller receives settled value, and the buyer receives the share position.
- Installed online browsers now prefer fresh document truth for app-shell navigations instead of booting stale cached shell first.
- The canonical offline app shell now stays canonical: exact visited-route offline entries stay exact, but they do not overwrite cached `/` or become the fallback for unrelated routes.
- Composer publish continuity, route-local pending feedback, bottom-bar navigation continuity, and same-surface market continuation are now part of the official release boundary again, not just historical lessons.
- Release-governed version surfaces now move together to `v63.0.0` across package metadata, service-worker defaults, public verifier artifacts, home, make, footer, legal, trust, plans, upgrade, and the active release docs.

## Why This Release Exists

The repository had reached an awkward state:

- the runtime on `HEAD` was already better than the old `62.0.0` release label implied
- the official version surfaces still said `62.0.0`
- the homepage/profile/hero/PWA behavior had been improved
- but the official release boundary still was not broad enough to protect other continuity surfaces the product historically cared about

`v63.0.0` resolves that.

It does not just rename the current code. It turns the current good runtime into an official release and broadens the continuity constitution around it.

## What Stayed

### 1. The current winning runtime

This release keeps the parts of the current runtime that clearly improved what users feel:

- explicit twin activation
- direct profile-route truth
- explicit-open instant-feeling hero behavior
- hero media prewarm for hot reopen
- hero back-chart parity with the live pill truth
- explicit value-state framing across wallet, market, and certificate surfaces
- backed-liquidity market sells and funded certificate realization
- production stale-PWA navigation hardening with offline fallback preserved
- canonical offline-root shell preservation so offline `/` cannot be silently replaced by another visited route document

### 2. The subtractive calm baseline

This release also keeps the subtractive discipline that came out of the `v60.2.0` line:

- no wrapper-driven visible takeover of the settled shell
- no hidden promotion of richer twin shells
- no fake loading theater on deterministic surfaces
- no new runtime churn just to make the release look more modern

## What Expanded

The main expansion in `v63.0.0` is governance, not runtime spectacle.

The official release boundary now includes:

- composer publish continuity
- route-local pending feedback
- bottom-bar continuity
- same-surface market continuation
- hero responsiveness and chart parity
- stale installed-browser navigation truth
- value-state truth for settled balance, live portfolio value, and unrealized P/L
- liquidity-backed realization rules for direct market sells and certificate sales

That is the biggest difference between this release and the narrower `v62.0.0` cut.

## Facts From The Evolution

The factual read across the versions is:

- `v55.1.0` was the strongest earlier continuity constitution
- `v58.0.0` was the cleanest historical release before the homepage/profile reload-bug family
- the speed-wrapper era after `v58.0.0` introduced real visible churn and needed repeated fixes
- `v60.2.0` was the first corrected calm baseline after that bug family
- later work added a separate stale-PWA problem class and hero responsiveness gaps that were worth fixing
- the current runtime fixed those later issues without needing to bring back the wider wrapper churn
- the first `v63.0.0` release candidate also exposed a worker regression where newer writes could replace cached `/` with another visited document, which is why offline root had to be corrected before ship
- late launch review clarified that wire cannot be the governing frame for the economy; value states are the governing frame, and bank backup rails must stay downstream of settled value
- late launch review also clarified that a sell is not settled because the mark exists; it is settled only when backed market liquidity or a funded certificate buyer pays for it

That is why `v63.0.0` exists.

## Public Summary

Receiz `v63.0.0` is the official current release: it keeps the calm corrected home/profile baseline, preserves the instant-feeling explicit-open hero path, keeps the hero chart and market pill on the same live truth, hardens installed-browser navigation against stale cached shell boot, makes broader continuity flows part of the official release contract again, and separates settled value from live market value without pretending unrealized growth is cash. Direct market sells now require backed liquidity, and public certificate links give users a real funded sale path without creating unfunded platform obligations.

## Deployment Notes

This release adds one launch-critical market settlement migration on top of the existing baseline:

- `supabase/migrations/20260419173000_market_share_certificate_wallet_purchase.sql`

That migration must be deployed before relying on public certificate purchases or liquidity-capped market sells in production.

Before tagging `v63.0.0`, production should also confirm:

- home and profile still settle once and stay visually settled until explicit interaction
- hidden twin warming never auto-promotes the visible shell
- composer publish keeps the surrounding profile current
- route-entry feedback stays button-local and real
- the homepage hero opens immediately and repeat open stays hot
- the homepage back chart stays aligned with the live pill truth
- wallet and market surfaces show settled balance separately from live portfolio value and unrealized P/L
- direct market sells reject when backed same-asset market liquidity is unavailable
- public certificate links allow signed-in buyers to fund available shares with settled wallet balance and credit the seller only after the funded purchase clears
- bank wire remains framed as a backup exit lane for already-settled value, not the main economy story
- installed online browsers open the fresh site document rather than a stale cached shell
- offline `/` reopens from the canonical cached root shell instead of a mismatched visited-route document or interruption boundary
- public version badges, plans and upgrade labels, offline verifier artifacts, service-worker defaults, package metadata, and release docs all show `v63.0.0`
