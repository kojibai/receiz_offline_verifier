`v66.0.0` is the Local Proof Continuity release.

This release freezes the truth model that Receiz must obey after the `v64.0.0` Complete Value Loop:

- The local node holds verified truth.
- The verified local prefix never becomes untrue.
- Sync asks for verified additions, not permission to believe known truth.
- Background hydration may warm hidden state only.
- Stronger local proof outranks weaker remote narration.
- AI reasons over verified substrate; it does not become authority over it.

## Frozen Primitives

| Concept | v66 truth | Must not regress |
| --- | --- | --- |
| Local truth root | Verified local identity, proof bundles, route snapshots, wallet graphs, and register entries are real held truth. | Do not block first paint or action on remote confirmation when a stronger local block exists. |
| Verified prefix | A node may be missing later appends, but its admitted verified prefix remains true. | Do not describe verified local truth as stale mutable data. |
| Append-only register | New facts arrive as verified additions after the known prefix. | Do not replace an admitted prefix with a weaker whole-surface payload. |
| Deterministic projection | UI state projects from verified proof/state immediately. | Do not use fake loading theater when truth is already held. |
| Portable identity | Keyfile/PBI-backed identity can prove continuity locally, including brand-new local PBI enrollment before global publication. | Do not let anonymous or missing session payloads erase local identity proof, and do not make publication permission for local existence. |
| Portable artifacts | Receipt, ledger, and twin-mind artifacts carry verification material. | Do not treat server-secret verification as portable offline authority. |
| Local twin rail | The twin can project from local register truth and local model/fallback before network stream. | Do not let a later online stream silently overwrite a settled local register projection. |
| Market position truth | Local buy/sell position changes and active share/prediction ownership are known action truth. | Do not make ownership wait for background market hydration or let lagging snapshots drop visible positions without stronger proof. |
| Offline persistence | Service-worker persisted snapshots are deterministic or private truth carriers. | Do not turn offline persistence into generic mutable cache invalidation. |

## Frozen Invariants

- If stronger verified truth is already local, nothing weaker remote may delay, downgrade, replace, or reinterpret it.
- A settled visible shell is immutable unless the user explicitly interacts.
- Background preload, hydration, lazy imports, and remembered session state may warm hidden code, data, and state only.
- A later network response may append verified truth, but it must not replace a verified prefix with weaker or merely fresher transport data.
- Wallet, Market, World, Explore, Account, and Receipt surfaces must project known local truth before discovering additions.
- A missing, anonymous, or weaker session payload must not erase a locally proven identity/account state.
- Brand-new local PBI identity enrollment must write/project local truth first. Global registration is a verified append, not the moment the identity becomes real.
- Signed-in Market entry must carry active position ownership on first paint. Full market snapshot hydration is enrichment, not permission for positions to exist.
- Public twin local register projection is visible truth. Online stream output is allowed when no local projection exists or when a future stronger verified append contract exists.
- Same-rank model output must not overwrite verified profile, receipt, memory, wallet, market, or twin-mind truth without domain-specific merge and proof rules.
- Deterministic trust surfaces must never use spinners, skeletons, "Loading...", "Preparing...", or equivalent fake progress for already-held truth.
- Sync is propagation, discovery, and verified addition. It is not permission.
- Local-first and server-sync are both required. User truth commits and projects locally first; server propagation starts in the same action path when network exists and can only publish, recover, mirror, mint cookies, or return stronger verified additions.
- A stale cookie, stale service-worker runtime, stale app chunk, or stale server projection must recover through local proof first, server session resync second, and stable home third. It must never leave the user on an application error.
- A durable session bootstrap is projection for first paint, not the identity proof root. It cannot outrank Receiz ID/PBI/keyfile proof, verified account binding, or a recovered valid server session.
- `v64.6.0` remains the stable production UX/server baseline. Local-first changes are valid only when they preserve that server function and improve first paint/action continuity by honoring proof already held on the device.

## v64.6.0 Regression Trace

The rollback point matters:

- `59cc4b7` (`v64.6.0`) is last-known-good for the stale-cookie/service-worker resume crash.
- The first bad commit is `231127057` (`honoring the primitives`).
- The correct part of that commit was adding the offline identity/register direction: local proof, local route truth, durable register entries, and local projection.
- The over-correction was making the signed-in session bootstrap durable before the runtime had a safe recovery ladder and before the code made clear that bootstrap is projection, not proof.
- The durable bootstrap could then survive across deploy/runtime/session edges and collide with stale chunks or stale cookies, producing React #185/max-depth/null-listener application failure instead of recovery.
- The repair is to keep the primitive, not roll it back: admit real local proof first, start server propagation/resync immediately when online, keep runtime chunks network-first while online, and clear to stable home only when neither local proof nor server session can be safely admitted.
- The target is not a new app shape. It is `v64.6.0`'s stable production behavior plus earlier local truth projection and immediate server propagation.

## Residual Risks

- Local PBI account creation is canonical local truth in this release. Remaining work is recovery/onboarding polish, multi-device transfer ergonomics, and clearer global-publication status, not permission for local existence.
- Browser local model readiness depends on device WebGPU support, memory, and model availability. Deterministic local fallback remains required.
- Peer-to-peer presence still needs transport. Transport loss is not identity loss, but live communication still requires a path between devices.
- Public-key rotation for portable twin-mind artifacts must preserve historical verification.
- The node can know its verified prefix while still missing latest global additions until sync.
- Future same-rank local AI or enrichment writes need domain-specific merge/hash rules before they can touch verified truth domains.
