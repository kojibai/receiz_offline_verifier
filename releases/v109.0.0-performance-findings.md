# Receiz v109.0.0 Performance Findings

`v109.0.0` makes complete-artifact verification and opening locally computable from `.receizbundle` bytes. The performance boundary is constitutional: verification performs zero network calls and does not wait for access tokens, cookies, sessions, databases, registries, or a running Receiz.com process.

## Findings

- `receiz.verification.verifyArtifact(file)` recomputes carrier integrity, payload binding, Signature V4 roots, the document-seal Groth16 proof, canonical identity/path fields, and applicable portable continuity locally.
- `receiz.artifacts.verifyAndOpen(file)` verifies the enclosing artifact before extraction and returns artifact custody separately from the verified payload.
- Signature V4 roots, the pinned Groth16 verification key, and the `snarkjs` runtime ship with the SDK, eliminating verification-time server and registry latency.
- A one-byte mutation, malformed carrier, binding mismatch, Signature V4 failure, Groth16 failure, or invalid portable continuity fails closed without a fallback network verdict.
- MCP artifact operations call the same SDK verifier. AI skills require both `Network calls during verification: 0` and the recorded local-verifier result before a production-ready claim.
- Clean-before-build packaging prevents deleted version-specific outputs from surviving in distributable package graphs.
- Native issuance remains authenticated Record -> Seal and may coordinate with Receiz.com. Verification/opening is local; issuance authority is unchanged.
- Receiz.com application behavior, database schema, migrations, query paths, and production data remain the v108 implementation. Only developer verification capability and visible release identification advance.
- Historical sealed proof, receipts, registry records, and release evidence remain readable historical truth without retaining obsolete installable SDK/MCP implementations.

## Performance Boundary

- Known complete artifact bytes are verified directly before projection, extraction, session work, registry lookup, server reconciliation, or network sync.
- A local verifier result cannot be weakened by a missing or conflicting server, session, database, SDK, MCP, AI, or release-evidence prior.
- Artifact and payload remain separate custody domains; payload availability never races ahead of complete enclosing-artifact verification.
- Release qualification may validate local static surfaces, but it does not reproduce upstream production-artifact or package evidence and does not authorize production signing, npm publication, deployment, database mutation, production verification, or Git push.

## Release-Lock Requirement

`pnpm release:lock 109.0.0` must require this document for `v109.0.0` and later release cuts. A release without these findings would omit the zero-network latency boundary, packaged verification dependencies, fail-closed mutation behavior, custody separation, clean packaging rule, and unchanged issuance/application/database scope.
