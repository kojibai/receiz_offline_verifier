# Receiz v110.0.0 Invariant Register

`v110.0.0` locks unified artifact admission and recovery beneath exact sealed artifact bytes and verified history. An admission verdict, action list, recovery plan, explanation, SDK, MCP, AI skill, registry, server, session, database, or release record cannot replace stronger proof truth.

## Locked Invariants

1. Receiz remains a proof-native artifact system. Exact artifact bytes, deterministic proof state, verified history, identity proof, ownership proof, settlement proof, and append-only continuity remain stronger than software or service projections.
2. Durable proof memory remains first admission only, then append forever. Recovery restores verified carried state and appends new verified evidence; it never rewrites admitted history.
3. `artifacts.admit` verifies the complete enclosing artifact before classification or payload projection and returns exactly one verdict: `canonical-identity`, `bearer-recovery`, `verified-legacy-read`, `foreign-owner`, or `invalid`.
4. Every admission explicitly returns `canRestore`, `canSign`, `canClaimOwnership`, `canPublish`, and `canSettle`; absent authority is false rather than inferred.
5. `ARTIFACT-012` denies incomplete artifact admission with `COMPLETE_ARTIFACT_ADMISSION_REQUIRED`.
6. `ARTIFACT-013` denies bearer escalation with `BEARER_AUTHORITY_ESCALATION_FORBIDDEN`; possession alone never grants canonical identity, signing, publishing, settlement, or ownership authority.
7. `ARTIFACT-014` denies non-atomic or unbound recovery commit with `ATOMIC_CAPABILITY_RECOVERY_COMMIT_REQUIRED`.
8. `ARTIFACT-015` denies explanation authority with `PROOF_EXPLANATION_NOT_AUTHORITY`; SDK-branded plans and bounded explanations remain non-authoritative.

10. `artifacts.admitAndRecover` composes exact-byte admission and recovery planning once with zero network calls, zero writes, and no hidden mutation surface.
