# Receiz v93.1.0

Release date: June 26, 2026

## Goal

Ship `v93.1.0` as the durable proof memory release after `v93.0.0`.

The goal is not a cosmetic version bump. The release locks a concrete product truth:

- Proof objects are admitted once and appended forever.
- Calendar and Account are projections over the same admitted event proof objects.
- Wallet settlement ordering uses Kai head state beneath Chronos display.
- Sports card history, vault appends, event proofs, and showcase routes stay bounded without dropping proof.
- Developer tooling exposes Receiz primitives as a convenience layer, not a new authority.

A `v93.1.0` release that omits any of these documents is incomplete because the durable proof memory primitive, performance law, invariant register, regression lessons, and exact implementation boundary would not be explicit release truth.

## What Worked

- Sports card history revealed the full operating law: once known proof history is admitted, reopening the object should project that memory immediately.
- Calendar and Account became clearer when treated as two projections over the same event proof object register.
- Wallet settlement became more coherent when Kai pulse/upulse was treated as state ordering and Chronos as display/fallback context.
- DB pressure improved when old truth stopped being rediscovered and hot reads became bounded addition discovery.
- SDK and OpenAPI work became stronger when framed as developer convenience beneath proof authority.
- TypeScript cleanup improved release confidence by upgrading stale tests to strict proof object fixtures instead of loosening production types.

## What Did Not Work

- Treating route hydration as normal repeat behavior created avoidable rediscovery and made known history feel provisional.
- Letting Calendar and Account maintain separate temporary memories hid the fact that they project the same event proof objects.
- Chronos-first backend reasoning obscured Kai causality and forced extra cursor/tie-breaker thinking.
- Timeout/RPC fixes were incomplete when they still let fallback paths call broad proof reads during normal operation.
- Stale tests masked release hygiene by using incomplete proof objects after production contracts had become stricter.
- Version surfaces drifted when package/service-worker labels moved to `93.0.1` but release docs stayed at `93.0.0`.

## Practical Rule For Future Work

Every Receiz surface should start from the proof object it already has, not from the route that displays it.

If a user or device has admitted truth, project it immediately. Ask the network only what verified additions exist after the known head. Do not re-fetch old truth to prove it is still true. Do not create separate authorities for separate screens when they display the same proof object. Do not weaken production proof contracts to satisfy old tests.

## Release-Lock Requirement

Run `pnpm release:lock 93.1.0` from this offline verifier checkout after version surfaces and v93.1.0 docs are aligned. Broader platform promotion must also run the app TypeScript and focused contract gates named in `docs/releases/v93.1.0-checklist.md`.
