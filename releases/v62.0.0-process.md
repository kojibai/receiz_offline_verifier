# Receiz v62.0.0 Stabilization Process

Release date: April 19, 2026

## Goal

Turn the calmer `v60.2.0` continuity state into a governed release candidate, keep only the parts that clearly improved the settled user experience, and document the exact reasoning so the repository can return to this cut later without re-litigating what happened.

## Comparison Windows Used

This stabilization pass was done across two explicit windows:

1. `v60.0.0..dd508df0`
   - Used to isolate the real continuity fixes that made home and profile feel settled again.
   - `dd508df0` was treated as the practical `v60.2.0` runtime snapshot even though some version surfaces inside that snapshot still said `60.1.0`.
2. `HEAD..release-prep workspace`
   - Used to identify later additions that could be removed without losing those continuity fixes.
   - This is the comparison that showed which current-`HEAD` changes were meaningful, which were meaningful but risky, and which looked like extra churn.

One practical marker from that review: `6669539b` behaved like the last clearly settled tiny UI-only tweak before the home/profile speed-wrapper churn began.

## Decision Rules Used

The release cut followed the repository’s product rules directly:

- settled visible shell beats eager background cleverness
- background warming may prepare hidden code and data, but it may not visually take over
- stronger already-known truth should be reused directly
- if a page feels loaded and then feels like it loaded again, that is a regression

Those rules came from:

- `docs/experience-first-engineering.md`
- `docs/deterministic-surfaces.md`
- `docs/verified-history-first-principles.md`

## What Clearly Worked And Stayed

### 1. Explicit twin activation only

The real continuity fix was kept in `app/u/[username]/DeferredWorldProfileModule.tsx`:

- `LoadedWorldProfileModule` only renders when `activated` is true
- hidden warming may load the module, but it does not mount it visibly
- remembered session-open state does not silently promote the visible shell
- the runtime boundary `resetKey` includes `forceOpenSequence`, which keeps explicit-open transitions deterministic

This is the core rule that stopped the visible “loaded, then loaded again” feel.

### 2. Direct home twin path

`app/components/HomeHeroTwinDock.tsx` now uses the direct `DeferredWorldProfileModule` path again. The extra loader cache wrapper was removed.

That matters because the user only sees one stable entry path instead of a closed-shell component that later hands off to a second component tree.

### 3. Direct profile route truth

`app/[username]/UserProfileRouteClient.tsx` keeps the direct `ProfileClient` and `mergeProfileWithTrustedStats` path in the main route tree.

The lazy wrapper and idle follow-up merge loader were removed because they added indirection around already-known route truth without a clear user-facing benefit.

### 4. Deterministic sigil scope IDs

The deterministic market-pill `sigilScopeId` work stayed. It had real value and no meaningful downside for this release line.

### 5. Hero chart background warm

`app/components/HeroLogoVideo.tsx` now warms the shared market asset in the background after settlement so the back-of-card chart is more likely live by the time the user flips. This was kept because it improves parity with the already-live market pills without introducing visible choreography.

## What Clearly Did Not Pay For Itself And Was Removed

### 1. Home twin speed wrapper

Removed from `app/components/HomeHeroTwinDock.tsx`:

- dynamic import cache wrapper
- separate runtime-requested state
- second component boundary whose only purpose was to hand off into the deferred module

This made the code more clever, not the experience calmer.

### 2. Hidden mounted world module path

Removed from `app/u/[username]/DeferredWorldProfileModule.tsx`:

- hidden mounted `prewarmOnly` loaded module path
- auto-open-on-mount behavior
- extra state tied to hidden promotion of the richer module

This was the biggest continuity risk because it let hidden readiness start to resemble visible readiness.

### 3. Hero handoff choreography

Removed from `app/components/HomeHeroVideoDeferred.tsx`, `app/components/HeroLogoVideo.tsx`, and `app/components/HeroLogoTrigger.module.css`:

- pre-open lazy player handoff state machine
- inline trigger suppression and delayed reappearance
- extra trigger-only stylesheet that existed only for that choreography

This was wrapper-heavy benchmark logic with no clear benefit to the settled product feel.

### 4. Lazy profile-route wrapper

Removed from `app/[username]/UserProfileRouteClient.tsx`:

- lazy `ProfileClient`
- lazy trusted-stats merge import
- idle follow-up hydration scheduler whose main effect was to stage work around already-known truth

### 5. Localhost service-worker poisoning path

Kept out of the release in `app/layout.tsx`:

- dev-time service-worker reset script
- automatic reload behavior tied to stale service-worker cleanup

Those hooks solved a real local stale-worker problem, but they also changed visible runtime behavior. They were not folded into this stabilization release.

Instead, the release keeps the lower-risk local fix:

- the static offline verifier and record/seal HTML documents no longer register `/sw.js` on localhost-style origins

That closes the dev-origin poisoning path without changing production install behavior.

### 6. Larger profile/media continuity stack

The current `HEAD` line also carried a wider profile/original/media/stats stack:

- `app/lib/profile/originalAccess.ts`
- `app/lib/profile/originalComposer.ts`
- `app/lib/profile/playableMedia.ts`
- `app/lib/profile/statsContinuity.ts`

That work may still help narrower edge cases, but it was not a no-downside fit for this release. It added complexity and hidden merge policy without being necessary to preserve the calm home/profile baseline.

## What Was Meaningful But Risky

Two current-`HEAD` areas were explicitly treated as meaningful but risky rather than useless:

1. Service-worker stale-PWA fixes
   - Benefit: addressed old installed-browser shells booting stale state.
   - Tradeoff: some versions of the work also changed returning-user behavior at the service-worker layer and, in dev, added forced reload semantics.
2. Profile/media/original continuity stack
   - Benefit: stronger edge-case handling for downloads, playable sources, and stats continuity.
   - Tradeoff: significantly more policy and hidden merge complexity.

Those are not dismissed. `v62.0.0` keeps only the service-worker parts that fit the continuity rules:

- production app-shell navigations prefer the fresh network document and only fall back to cached shell/offline pages when the network document is unavailable
- localhost static offline pages do not register the shared service worker automatically

The more behavior-changing dev reset script remains out of scope because this release is optimizing for stable settled experience first.

## Net Result Of The Stabilization Cut

From current `HEAD` to the `v62.0.0` release-prep tree:

- `49` files changed
- `1,048` insertions
- `2,561` deletions

The important part is not just the numbers. The direction is subtractive:

- fewer wrappers
- fewer hidden promotion paths
- fewer behavior-changing experiments
- the real continuity fixes retained

## What Worked

- Explicit activation-only twin promotion
- Direct truth reuse instead of lazy rediscovery
- Deterministic sigil scope IDs
- Hidden market warming for the hero chart
- Release-governed version surfaces moving together
- Production app-shell navigation prefers fresh network truth with offline fallback preserved
- Localhost static offline pages no longer re-poison the dev origin with `/sw.js`

## What Did Not Work

- Speed-wrapper choreography that made settled surfaces feel staged
- Hidden preparation paths that could visually resemble a second load
- Lazy route wrappers around already-known truth
- Behavior-changing service-worker cleanup folded into a continuity release without enough separation

## Practical Rule For Future Work

If a future change improves a benchmark by adding a wrapper, handoff, hidden mount, or remembered auto-promotion path, the burden is on that change to prove it does not make the settled product feel more engineered.

For this release line, the repository answer is now clear: calm direct truth beat clever staging.
