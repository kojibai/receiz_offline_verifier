# Receiz v110.0.0 Release Process

## Goal

Coordinate the v110 SDK, MCP, and AI admission/recovery release without changing Receiz.com or database behavior.

The `@receiz/sdk` durable proof memory process remains **first admission only, then append forever**.

## What Worked

Contract-first implementation exposed the stale MCP registry binding, kept read-only recovery separate from commit, and made capability/idempotency/atomicity directly testable.

## What Did Not Work

Relying on a tool-schema description did not guarantee the artifact-seal confirmation basis used the active registry. Direct digest assertions were required.

## Practical Rule For Future Work

Bind every confirmation and operation identity to the active signed registry in executable tests, then verify no weaker layer became authority.

1. Verify the exact v109 baseline and preserve unrelated user files.
2. Add contracts before implementation for admission types, proof history, local classification, recovery planning, atomic commit, client conformance, and MCP orchestration.
3. Keep `admitAndRecover` read-only and make mutation an explicit capability-bound API.
4. Add a signed-registry chain entry with ARTIFACT-012 through ARTIFACT-015.
5. Bind SDK, MCP, AI skills, manifests, packages, and release labels to the one v110 digest.
6. Run focused tests, package suites, AI validation, release locks, package installation, full release freeze, governance, and diff checks.
7. Confirm no API/database/production-data path changed and no obsolete versioned SDK export reappeared.
8. Keep npm publication, deployment, attestation signing, and production verification outside repository qualification unless separately authorized.

## Locked Release Records

The release process is incomplete without the [regression lessons](v110.0.0-regression-lessons.md), [performance findings](v110.0.0-performance-findings.md), [invariant register](v110.0.0-invariant-register.md), and [commit-history boundary](v110.0.0-commit-history.md). The standalone verifier applies `pnpm release:lock 110.0.0` only to its local release-identification surfaces; production signing, npm publication, deployment, database work, production verification, and Git push remain separate actions.
