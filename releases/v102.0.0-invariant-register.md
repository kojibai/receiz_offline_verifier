5. Deterministic first paint must project known proof, identity, ownership, settlement, profile, market, ledger, transaction, Sports, public graph, and archive truth before weaker work.
6. `@receiz/sdk` remains typed developer convenience beneath proof objects, verified appends, identity primitives, ownership appends, settlement rows, public proof surfaces, and offline verification.
7. `@receiz/mcp-server` remains an agent rail over SDK/API mechanics and must name its authority boundary.
8. Top-level verifier success requires enclosing integrity and a verified native or portable continuity carrier.
9. A real authenticated profile-backed Receiz ID, profile namespace, deterministic identity, and prior head or valid genesis exist and match before continuity-capable sealing.
10. UUIDs, arbitrary owner strings, placeholders, developer input, session identifiers, lookup results, and manifests cannot become owner or verification authority.
11. Generic document sealing remains integrity-only, rejects portable proof-object payloads, and cannot return proof-object verification success.
12. Native Record, Seal, profile composition, ownership provenance, and card continuity remain authoritative through their existing carriers.
13. Downloadable offline verifiers recompute the same integrity-and-continuity verdict without server or database authority.
14. MCP resolves and inspects; lookup and shape validation are never named or returned as proof-object verification.
15. SDK, MCP, and installed matching AI skills remain mechanics beneath identity, ownership, continuity, sealed artifacts, and append history.
16. Static verifier, offline verifier, record-seal studio, settlement entrypoint, Sports card verifier, package metadata, release docs, and service-worker runtime version must move together.
