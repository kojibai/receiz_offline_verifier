# Receiz v108.0.0 Performance Findings

`v108.0.0` preserves Receiz scale and truthful-speed law by restoring the direct first-party profile operation and keeping proof-object custody, verified local truth, and admitted identity ahead of database projections and developer rails.

## Findings

- Inline username changes and full profile saves call `PATCH /api/profile/me` directly under the admitted account, preserving the same UID without a developer-operation preflight or `identity_keys` lookup.
- Commit-time username uniqueness and post-commit session/profile/public-projection refresh avoid reservation state and pre-commit projection churn.
- Missing, delayed, or failed identity-key projection cannot block first-party saves or trigger false identity mismatch; explicit service and projection errors remain distinguishable.
- Native proof-object creation performs Record before Seal and returns complete verified artifact bytes; download preserves exact bytes and opening verifies the enclosing carrier before extraction.
- Bearer ownership derives prior custody from the verified carried artifact, binds the authenticated owner, appends continuity, and returns another complete verified Record-to-Seal artifact.
- Known verified local truth paints immediately; queues, service workers, sessions, databases, registry rows, and background reconciliation may warm or append without replacing settled truth.
- The first-party service worker and root runtime do not mount the developer command queue merely to conform to SDK tooling.
- Historical v107 source, witnessed receipts, and never-applied SQL remain isolated evidence, avoiding current capability, migration, and runtime work for retired mechanics.
- The unversioned `/developers` route remains the maintained reference; redirects do not add a version-branded current product page.

## Performance Boundary

- Deterministic first paint must precede hydration, analytics, SEO, session checks, key projection reads, developer preflights, lazy imports, queue resume, or server reconciliation.
- Receiz.com reference behavior, SDK, MCP, AI skills, registry evaluation, server/database/session state, and release evidence remain weaker than sealed proof, deterministic proof state, verified local truth, and accepted identity continuity.
- Visual mutation evidence may block promotion when tests pass but the UID, network path, zero-key-row behavior, public projection, or mobile/desktop result contradicts the release law.
- Production signing, npm publication, deployment, production database state, and authenticated production evidence remain separate external boundaries.
