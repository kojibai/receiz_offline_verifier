# Receiz v94.0.0

Release date: 2026-06-27

## Goal

Ship `v94.0.0` as the official SDK launch and durable proof operating model release.

The goal is to make the production Receiz law explicit for both the app and external developers: first admitted proof truth paints immediately, verified additions append after the known head, and projection systems exist to make truth arrive sooner without replacing it.

## What Worked

- Sports card history exposed the correct proof-memory model because cards with full admitted event history became instant and stable.
- Moving Sports reads to bounded RPC/index projections reduced pressure without changing the user-facing proof object.
- Kai-ledger work clarified that Receiz ordering is a state coordinate, not a wall-clock guess.
- The SDK became a real public package, `@receiz/sdk`, with typed clients, schemas, fixtures, projections, and proof-memory helpers.
- Calendar, Account, wallet, Sports, and developer surfaces converged on one rule: render known verified truth first, then append verified additions.

## What Did Not Work

- Rediscovering old truth before rendering created avoidable lag and pressure.
- Treating route-local projections as separate truth objects made Calendar, Account, wallet, and Sports feel less coherent than the primitives already allowed.
- Letting DB triggers overwrite app-supplied Kai child coordinates created invalid ordering pressure even though the proof objects themselves were unique.
- Broad PostgREST reads made projection failures look like proof failures.
- SDK-only low-level clients were not enough; developers need the admission and append model packaged directly.

## Practical Rule For Future Work

Every feature must answer this before code changes:

```txt
What verified truth do we already have, and what verified additions exist after its known head?
```

If the feature asks whether old verified truth is still true, it is using the wrong model.

If the feature creates a new route-local copy of an existing proof object, it must be reduced to a projection.

If the feature adds a database/RPC/index path, it must stay below the proof object and make additions faster without replacing authority.
