## Title

Receiz `v101.0.0` — Portable Asset Continuity

## Summary

`v101.0.0` closes manifest-only asset admission. `@receiz/sdk` accepts only a complete sealed proof object whose file passes the existing Receiz verifier and carries canonical payload, ownership, provenance/appends, and settlement continuity.

The proof-object creation rail seals a canonical continuity document, re-opens and checks the returned file locally, requires the existing artifact verifier to pass, and only then admits the object through the authenticated runtime. The server repeats verification against the same Receiz artifact verifier before the database append. A self-hashable manifest is never sufficient authority.

Historical Receiz artifacts and Receiz.com verification behavior remain unchanged. The stricter boundary applies to new SDK proof-object creation and admission.

Across the app and `@receiz/sdk`, durable proof memory remains **first admission only, then append forever**.

## Breaking Boundary

- `ReceizProofRegister.admitAssetManifest` and Sports manifest admission reject with `complete_sealed_proof_object_required`.
- Asset creation must use `receiz.assets.createProofObject(...)`.
- `POST /api/document-verify` keeps its existing single verification contract.
- Database asset admission moves to `sdk_portable_asset_admit_v2`; execution of the v1 admission RPC is revoked.
