## Goal

Make Receiz `v68.0.0` the release where the app keeps the `v64.6.0` production feel while honoring local proof everywhere the local node already has authority.

The goal is not "offline mode."

The goal is:

```txt
local truth is real -> user sees/acts now -> server propagates globally -> verified additions merge later
```

## What Became Clear

The correct model is not client versus server.

The correct model is:

- local proof
- deterministic projection
- server propagation
- verified additions
- merge
- replay

That means the app must never wait for the server to confirm already-held proof. It also means the app must not skip server work when server work is the way local truth reaches the global network.

## What Worked

- Treating `v64.6.0` as the production UX/server baseline gave the release a stable target.
- Local Receiz ID login made the main sign-in path match the actual authority model.
- Keeping PBI/email as recovery and interop preserved existing users while simplifying future login.
- Multi-account binding made one Receiz ID capable of carrying multiple account continuities.
- Proof-backed durable bootstrap preserved local continuity without letting generic remembered state become fake proof.
- Stale runtime recovery closed the gap that could crash logged-in PWAs after a deploy or stale cookie edge.
- Service-worker network-first runtime chunks preserved app coherence while keeping explicit deterministic/private snapshot storage.
- Wallet and Market continuity improved once known wallet graphs, incoming wallet summaries, local positions, and buy shortfall corrections were treated as truth that moves on the active surface without route refresh.
- Explore, World, Profile, Viewer, Player, and Receipt routes became easier to reason about when network responses were classified as additions or cold-open resolution, not authority over already-held blocks.
- The public twin work stayed additive: local continuity without deleting the online brain.
- Live player entry became coherent once the selected original was treated as route truth and queue/world/profile data was treated as augmentation.
- Twin visual identity became deterministic once partial world/profile payloads were no longer allowed to hide known profile/twin images.

## What Did Not Work

- Making signed-in bootstrap durable for every session projection was too broad.
- Treating local-first as if the server could be forgotten was the wrong interpretation.
- Letting unbacked bootstrap skip the normal server session path drifted away from the `v64.6.0` baseline.
- Calling every API 404/401 a product truth issue missed the real distinction: local truth can project while server publication still needs to happen.
- Some route work initially focused on hiding network dependency instead of preserving server function and improving user experience together.
- Hydration fixes that did not preserve the settled visible screen were not sufficient, even if there was no literal browser refresh.
- Letting a live player transition expose a white frame was still a visible regression, even if the server eventually resolved the route.
- Letting a partial world profile response produce a generic twin rig was an authority inversion when the app already knew the user's image.

## Regression Trace

Last-known-good for the stale-cookie/PWA crash:

- `59cc4b7` / `v64.6.0`

First bad commit:

- `231127057` / `honoring the primitives`

What was right in that commit:

- local proof direction
- offline identity session
- local truth store
- deterministic route hydration
- service-worker explicit offline persistence

What was wrong:

- signed-in bootstrap became durable before it was clearly framed as projection, not proof
- runtime recovery did not yet guarantee non-crashing stale-cookie behavior

`v68.0.0` keeps the good part and corrects the bad part.

## Banned Future Reasoning

Do not say:

- "The server can be skipped because local proof exists."
- "The server is the source of truth for a user's own held proof."
- "A remembered bootstrap is proof."
- "It only crashed for stale cookies."
- "A lagging snapshot can briefly drop a known position."
- "The page only looked like it reloaded."
- "The online twin is ready, so it can replace the local answer."
- "A 401 means locally proven identity became untrue."
- "Upload failure means the post did not locally happen."
- "The queue/world profile is not ready, so live player can show a blank transition."
- "A partial world payload can replace known twin image identity."

Those are regressions against `v68.0.0`.

## Correct Future Reasoning

Every code path should answer:

1. What proof does the local node already hold?
2. Is the visible surface already settled?
3. What does the user experience right now?
4. Does local truth move immediately?
5. Does server propagation still start when network exists?
6. Is the server returning verified additions, recovery, publication status, or merely restating known truth?
7. Can a weaker response downgrade a stronger local block?
8. Can stale runtime/session state crash the app?
9. Does this preserve the `v64.6.0` production baseline?

If the answer fails either the primitive law or the user experience standard, the change is not finished.

## Release Decision

`v68.0.0` is launch-ready when:

- package/service-worker/public version surfaces are all `v68.0.0`
- release docs fully document the `v64` to `v68` arc
- local-first identity contracts pass
- stale runtime recovery contracts pass
- service-worker runtime coherence contracts pass
- wallet/market/explore/world/profile deterministic surface contracts pass
- value-loop gates from `v64.0.0` still pass
- manual smoke confirms the app feels like `v64.6.0` plus better local continuity

## Practical Rule For Future Work

Start from the user and the law:

```txt
law: local verified truth is real
experience: user never suffers for us honoring it
server: propagation starts immediately when possible
```

The release path to `v70.0.0` should expand this rule to every remaining private action through local signed outboxes and verified append contracts, without changing the stable product feel users already trust.
