# Receiz v58.0.0 Release Lock Checklist

Release date target: April 17, 2026

## Purpose

This checklist locks `v58.0.0` to one exact release state without allowing regression in authoritative market first paint, wallet/popover coherence, deterministic lean market transport, open-contract tracked-position semantics, or release-governed version surfaces.

Product-truth reference:

- `docs/releases/v58.0.0-product-truth.md`

## Pre-Tag Requirements

Run from the exact release commit only:

```bash
pnpm exec tsc --noEmit --pretty false
npx tsx --test app/lib/market/__tests__/marketRouteHydrationContract.test.ts app/lib/market/__tests__/marketSnapshotWalletContract.test.ts app/lib/market/__tests__/marketTradeContinuationContract.test.ts app/lib/market/__tests__/marketValuePillContract.test.ts app/lib/market/__tests__/walletHistoryContract.test.ts
pnpm test:market-conformance
pnpm build
pnpm release:lock
pnpm test:release-freeze
```

If the candidate deploy also requires release attestation, run that only after the release commit is built and tagged with the expected release version.

## Production Environment + Governance Preflight

For the `v58.0.0` release-prep state, confirm the target environment includes:

- `supabase/migrations/20260417194500_receiz_schema_drift_repair.sql`
- `supabase/migrations/20260417204000_world_district_baseline_contract.sql`
- `supabase/migrations/20260417225500_profile_analytics_last_event_at_repair.sql`
- `supabase/migrations/20260417233500_receiz_anchor_kind_repair.sql`

Before release, confirm:

- signed-in `/market` and signed-in market asset entry render wallet, positions, and history on first paint
- `/api/market/snapshot` continues to return the lean transport contract rather than a regressed expanded snapshot payload
- chart buy shows live wallet buying power without waiting on a full market refresh
- tracked positions count only share positions plus still-open contracts
- public/offline verifier surfaces, service worker version, `package.json`, home/make/footer/legal/trust badges, and plans/upgrade release labels all show `v58.0.0`
- `public/governance/governance-keyring.v1.json` still contains the active release-attestation key ID if release attestation is produced for this cut

## Manual Smoke

Confirm these flows on the release candidate without changing code:

- sign in and open `/market`; positions, history, and buying power are visible immediately instead of hydrating in late
- open a market asset and trigger the chart buy popover; the wallet state is already present and the buy flow does not stall on `Wallet: Loading...`
- inspect a user with only share positions and a closed/expired contract; tracked positions excludes the non-open contract
- refresh market after a signed-in transition and confirm route continuity keeps newer wallet/history truth intact
- public version badges, plans/upgrade release labels, service worker defaults, offline verifier surfaces, and release docs all show `v58.0.0`

## Tag + Attestation

After the pre-tag requirements pass:

1. Create the release commit for `v58.0.0`.
2. Create the release tag from that locked commit: `v58.0.0`.
3. Run the `Release Governance Attestation` workflow if this cut still requires a signed attestation.
4. Verify the generated artifact with `pnpm release:verify-attestation <path>` if an attestation is produced.
5. Confirm the attestation references the published key ID and the expected release version.
6. Store the attestation path and approval reference with the release record.

## Release Rule

If any market-authority regression, wallet/popover regression, transport bloat regression, tracked-position counting regression, version-surface mismatch, release-lock failure, or release-freeze failure occurs, `v58.0.0` is not released from that commit.
