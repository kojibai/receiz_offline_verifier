# Receiz v97.0.0 Regression Lessons

Release date: 2026-06-30

## What Regressed

No runtime product behavior is intended to regress in this release. The concrete risks closed by this release were:

- The SDK had the app-state rail but still required feed boilerplate for the most common one-record app-state publish path.
- Host/subdomain recovery required developers to build their own URL normalization before calling `byUrl()`.
- OIDC consent had a visible button-readability and redirect-behavior failure that could make delegated application login look broken.
- Delegated record projection could return a weaker verification URL than the persisted canonical URL.
- Owner-scoped Receiz projection lookup briefly risked treating another owner's slug/code row as a conflict.
- Admin performance analytics could report misleading scores by measuring surfaces that were not the real user app path.

## Why It Regressed

The app-state projection rail moved from internal capability into real ecosystem use. At that boundary, generic SDK shape was not enough. Developers and AI agents needed the canonical envelope, URL normalization, and recovery path encoded into the package so they do not have to recreate Receiz law by hand.

The OIDC and projection repairs came from the same lesson: external apps expose every ambiguity. A route can be technically present and still fail the developer surface if the button, redirect, or returned proof URL is weaker than the canonical path.

## Locked Rules

- `@receiz/sdk` must be version-locked to the app release.
- `RECEIZ_SDK_VERSION` must match `packages/receiz-sdk/package.json`.
- Public app release labels must match `package.json`.
- Service-worker and runtime coherence defaults must match the release.
- `receiz.appState.publishRecord()` must remain a convenience wrapper over the same delegated registry feed route.
- `receiz.appState.byHost()` must remain a convenience wrapper over public URL lookup, not a separate state model.
- App-state projections are public projection/index rows beneath proof objects and durable proof memory.
- Connect scopes authorize delegated API calls; they do not become identity proof.
- SDK docs must explain public reads, delegated writes, payload limits, and first admission only, then append forever.
- OpenAPI/status catalog entries must stay aligned with exposed developer routes.

- `scripts/test_release_freeze.ts`

These gates fail if the app and SDK version identity diverge, the SDK drifts from packaged evidence, the app-state helpers disappear, or public developer route documentation stops matching the implemented surface.

## Release-Lock Requirement

`pnpm release:lock 97.0.0` must require this document for `v97.0.0` and later release cuts. A release that omits regression lessons is incomplete because SDK app-state helper drift, host URL normalization, OIDC consent behavior, canonical verification URL projection, owner-scoped lookup, admin performance measurement, and app/SDK version lock lessons would not be explicit release truth.
