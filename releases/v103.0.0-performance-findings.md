# Receiz v103.0.0 Performance Findings

`v103.0.0` preserves Receiz scale law by removing the v102 post-creation portable-carrier rejection from native Profile Composer, Record, and Seal proof objects. Native sealed bundle and canonical-coordinate verification remains direct; SDK composition adds no parallel verification authority.

## Findings

- Existing native bundles verify through their sealed proof bundle, Receiz claim identity, Kai coordinates, Record ownership projection, and established verify path without a newly introduced carrier lookup.
- Profile showcase downloads reuse the same native verdict, preventing a redundant post-creation rejection and 409 for already-valid artifacts.
- SDK `assets.createProofObject` resolves the authenticated Receiz ID, persists the native Record before Seal, seals source bytes at matching coordinates, and verifies the returned native claim.
- Historical native artifacts are not forced through metadata or carrier work introduced after they were sealed.
- MCP and AI skills teach the single native SDK proof-object rail instead of adding parallel translation or continuity machinery.

## Performance Boundary

- Known verified truth must paint before hydration, analytics, SEO, session checks, lazy imports, global sync, chart hydration, ledger pagination, or server reconciliation.
- Offline native verification remains primary and recomputable from existing sealed proof bundles; server and database work may mirror but cannot add required post-seal authority.
- Authenticated account lookup, SDK orchestration, transport, package distribution, and service-worker continuity remain weaker mechanics beneath native Record/Seal proof truth.
- Visual evidence can block release even when static contracts pass.

## Release-Lock Requirement

`pnpm release:lock 103.0.0` must require this document for `v103.0.0` and later release cuts. A release that omits performance findings is incomplete because direct native verification, Profile showcase download continuity, Record-before-Seal SDK ordering, historical compatibility, offline recomputation, and verifier/service-worker identity would lack explicit proof-authority boundaries.
