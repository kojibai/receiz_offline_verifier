# Receiz v90.0.0

## Goal

Ship `v90.0.0` as the stable official Receiz release:

```txt
document every commit from v80.0.0 through the current release candidate
recap the entire product arc from first proof object through v90
make the implemented primitive model direct and understandable
preserve v89 foreground anchor and profile/global projection locks
lock post-v89 Sports ownership, event proof, live value, conformance, and market-depth work
move package, service worker, runtime, public labels, offline verifier labels, and release docs together
```

The release goal is not to invent a new story. The release goal is to name what the repository already implements:

```txt
Receiz is a proof-native artifact system.
```

## What Worked

- The doctrine files already state the exact product law: stronger truth must not be replaced by weaker truth, deterministic first paint is protected, and implemented primitives must be named directly.
- Existing release notes from `v40.0.0` through `v89.0.0` already document the governed product train, so v90 can reuse repo evidence instead of outside framing.
- The historical `v30.0.0` release note is recoverable from git and records the public standards-platform transition.
- The current git range from `v80.0.0` to `HEAD` gives an exact `723` commit ledger for the heavy v80-to-v90 arc.
- The post-`v89.0.0` git range identifies the final release slice: Sports Vault projection, ownership transfer, stale lineup repair, event proof UI, live value, market depth, conformance, competition rooms, and public metadata.
- The existing release-lock script already protects package version, release docs, public labels, service-worker defaults, runtime coherence defaults, and offline verifier labels.
- The v89 regression lessons already name foreground anchor publication, profile projection, Sports Vault public URLs, and Sports single-game entry admission as exact boundaries.

## What Did Not Work

- The repository retained release docs from v40 forward, but current docs do not retain v10, v20, or v30 release files. v30 exists historically in git; v10 and v20 are commit-history milestones.
- Release documentation lagged behind the amount of Sports, market, vault, profile, runtime, and conformance work landed after v80.
- Version surfaces were split: package/service-worker/runtime were already at `89.0.1`, while several public and offline verifier labels still displayed `v89.0.0`.
- The release-lock script did not previously require the v80-to-current commit ledger, even though v89 process language made full commit history a release artifact.
- Some commit subjects in the mechanical v80-to-current ledger are raw working commits. That is correct for evidence, but the official release note must group them into primitive outcomes so the arc is understandable.

## Practical Rule For Future Work

For release documentation:

```txt
use repo evidence first
name implemented primitives directly
document exact commit range and stats
include regression lessons
include commit history when a major release closes a long arc
name production, visual, and DB uncertainty exactly
```

For Sports proof and ownership work:

```txt
verify Sports card/vault/event proof truth first
reuse existing ownership transfer append boundaries
compose single-game entry admission from verified game identity and owned proof cards
project metadata and value beneath proof truth
lock conformance as release evidence
```

For version locks:

```txt
package version
service worker version
runtime coherence default
public labels
offline verifier labels
release docs
release lock
```

The `v90.0.0` rule is:

```txt
The whole product is governed by the proof-native artifact system truth hierarchy.
```
